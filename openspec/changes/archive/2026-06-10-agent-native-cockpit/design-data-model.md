# DataWeave 数据模型设计（企业级全量 schema）

> 本文是 DataWeave 数据库 schema 的**设计真相源**。DDL 实现见 `backend/dataweave-api/src/main/resources/schema.sql`，种子见 `data.sql`。
> 设计目标：标准企业级数据中台底座 —— 多租户 + 项目隔离 + 完整 RBAC + 数据源/任务流(DAG)/指标分层/资源诊断/告警，且任务流与节点两级状态机清晰可聚合。

---

## 1. 公共约定（每张业务表）

### 1.1 审计列（所有业务表统一带）

| 列 | 类型 | 说明 |
|----|------|------|
| `id` | BIGINT IDENTITY | 主键 |
| `tenant_id` | BIGINT | 租户隔离（全局表如 tenants/permissions/datasource_types/worker_nodes 除外） |
| `project_id` | BIGINT | 项目隔离（项目级资源才有；租户级表无此列） |
| `created_by` | BIGINT | 创建人（users.id） |
| `updated_by` | BIGINT | 最后修改人 |
| `created_at` | TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | 修改时间 |
| `deleted` | SMALLINT | 逻辑删除：0 正常 / 1 已删（查询默认带 `deleted = 0`） |
| `version` | INTEGER | 乐观锁版本号（更新 +1） |

### 1.2 隔离层级

```
tenant（租户）
   └── project（项目，资源隔离单元）
          └── datasource / task_def / workflow_def / metric / ...（项目级资源）
user ──< user_role >── role ──< role_permission >── permission     （租户级 RBAC）
user ──< project_member(role) >── project                          （项目内授权）
```

- **全局表**（无 tenant_id/project_id）：`tenants`、`permissions`、`datasource_types`、`worker_nodes`。
- **租户级表**（有 tenant_id，无 project_id）：`users`、`roles`、`user_role`、`role_permission`、`projects`、`project_member`、`notification_channels`。
- **项目级表**（tenant_id + project_id）：`datasources`、`task_def`、`workflow_def`、`workflow_node`、`workflow_edge`、`workflow_instance`、`task_instance`、`dimensions`、`atomic_metrics`、`derived_metrics`、`metric_dimension`、`metric_lineage`、`alert_rules`、`task_diagnosis`、`audit_log`。

### 1.3 DDL 可移植性

兼容 PostgreSQL（生产）与 H2（开发）：自增用 `GENERATED BY DEFAULT AS IDENTITY`；JSON 字段统一存 `VARCHAR`（不用 PG 专属 `JSONB`，保 H2 兼容）；状态/枚举用 `VARCHAR(32)`。

### 域 C · 任务与任务流 DAG（9）
| 表 | 职责 | 关键列 |
|----|------|--------|
| `task_def` | **任务定义**（单个可执行节点，**草稿态**可编辑） | name, type(SQL/SHELL/SPARK/PYTHON/SYNC), content, datasource_id, target_datasource_id, params_json, timeout_sec, retry_max, status(DRAFT/ONLINE/OFFLINE), **current_version_no, has_draft_change** |
| `task_def_version` | 任务定义**已发布版本快照**（不可变，可回滚） | task_id, version_no, content 等快照, published_by, published_at |
| `workflow_def` | **任务流定义**（DAG 容器，草稿态可编辑） | name, description, **schedule_type(MANUAL/CRON/DEPENDENCY), cron, schedule_start/schedule_end(生效区间)**, status, current_version_no, has_draft_change |
| `workflow_def_version` | 任务流**已发布版本快照** | workflow_id, version_no, **dag_snapshot_json**(冻结节点+边+各节点 task 版本), published_at |
| `workflow_node` | DAG 节点（把 task_def 挂到流上，**M:N 复用**） | workflow_id, task_id, node_key, name, pos_x, pos_y |
| `workflow_edge` | DAG **内**依赖边 | workflow_id, from_node_id, to_node_id |
| `workflow_dependency` | **跨任务流**依赖（DAG 间） | workflow_id(下游), node_id, depend_workflow_id(上游), depend_node_id, date_offset, dep_type |
| `workflow_instance` | 任务流运行实例 | workflow_id, **workflow_version_no**, trigger_type, state(见 §4), biz_date |
| `task_instance` | 节点运行实例（一次执行） | workflow_instance_id, workflow_node_id, task_id, **task_version_no, run_mode(NORMAL/TEST)**, state(见 §4), attempt, worker_node_code, log |

> 命名去歧义：**DAG 节点** = `workflow_node`（`workflow_node_id`）；**物理机器节点** = `worker_nodes`（`node_code`）。`task_instance.worker_node_code` 指向机器。
> `task_instance.workflow_instance_id` 可空：允许任务脱离工作流被临时单跑（ad-hoc）。
> 版本/发布/复用/跨流依赖语义见 **§3**。

## 3. 版本、发布、复用与跨流依赖

### 3.1 草稿-发布分离 + 版本快照（可回滚）

`task_def` / `workflow_def` 的 `content`/DAG 是**草稿（开发态）**，编辑不立即生效；点「发布」才冻结成一份不可变快照并对调度生效。

```
编辑草稿(has_draft_change=1) ──「发布」──▶ 写 *_version 快照(version_no=N)
                                          设 current_version_no=N, has_draft_change=0, status=ONLINE
                                                    │
调度 / 运行实例只认 current_version_no 指向的快照     │
                                          回滚：把 current_version_no 指回 N-1
```

- `task_def_version`：任务定义每次发布一份快照（content 等全量字段冻结）。
- `workflow_def_version`：任务流每次发布把整张 DAG 序列化进 `dag_snapshot_json`（节点 + 边 + **各节点引用的 task 版本号**）。
- 运行实例记录所跑版本：`workflow_instance.workflow_version_no`、`task_instance.task_version_no` → 可追溯「这次跑的是哪版」。
- `status`：`DRAFT`（从未发布）/ `ONLINE`（已发布且调度开启）/ `OFFLINE`（已发布但暂停调度）。

### 3.2 任务复用（M:N）

`task_def` 是**项目级可复用资源**；`workflow_node(workflow_id, task_id)` 是工作流↔任务的多对多连接 —— 同一个任务可被多个工作流的多个节点引用。

```
工作流A ─┐
         ├─(workflow_node)→ task_def「订单宽表加工」(可复用)
工作流B ─┘
```

**版本冻结避免连锁修改**：共享任务被改并发布新版本后，已发布的工作流仍跑其 `dag_snapshot_json` 里冻结的旧 task 版本；只有**工作流重新发布**才会拾取任务新版本。改一个任务不会意外影响所有引用它的工作流。

### 3.3 跨任务流依赖（DAG 间）

- **DAG 内**节点依赖 → `workflow_edge`。
- **跨 DAG**（任务流之间）→ `workflow_dependency`：下游工作流/节点 依赖 上游工作流(+节点)，带**业务日期偏移**（`CURRENT_DAY`/`LAST_DAY`…）与条件（`dep_type`，如 `ALL_SUCCESS`）。

```
下游日报工作流(今日) ──workflow_dependency(CURRENT_DAY, ALL_SUCCESS)──▶ 每日 GMV 工作流(今日) 须先成功
```

调度时下游实例进入 `WAITING`，直到匹配业务日期的上游实例满足条件，才推进到 `RUNNING`。

### 3.4 触发与调度

`workflow_def.schedule_type` 三类触发：
- `MANUAL` 手动：人工点「运行」才建实例。
- `CRON` 周期：调度器服务读 `cron`，在 `schedule_start ~ schedule_end` 生效区间内到点**自动建 `workflow_instance` 并推进**（「配好就自己跑起来」）。
- `DEPENDENCY` 依赖触发：不靠自身 cron，由 §3.3 跨流依赖在上游满足后驱动。

> 调度器是**运行态服务**（master 模块，搭积木阶段实现）：扫描 ONLINE 且在生效区间内的 CRON 工作流 → 到点创建实例 → 按 §4 状态机推进。schema 已备齐所需字段。

### 3.5 单任务试跑（run_mode）

`task_instance.run_mode`：
- `NORMAL` 正式：正式调度/上线运行，`task_version_no` 指向已发布版本，计入工作流聚合与生产概况。
- `TEST` 试跑：开发者调试单任务。可**直接跑草稿版**（`task_version_no=NULL`）、`workflow_instance_id=NULL`（脱离工作流）；**不计入**工作流状态聚合、不计入驾驶舱生产概况、不影响血缘，仅返回结果/日志给开发者。

## 4. 状态机设计（两级）

### 3.1 节点实例状态（`task_instance.state`）—— 单个可执行节点的一次运行

| 状态 | 含义 | 进入条件 |
|------|------|----------|
| `NOT_RUN` 未运行 | 实例已随工作流实例生成，但调度器**尚未评估**它（工作流未启动 / 还没轮到这一层） | 工作流实例创建时所有节点初始态 |
| `WAITING` 等待 | 调度器已评估，节点**就绪但 DAG 上游未全部成功**，在等上游 | 工作流运行中，本节点存在未完成（非 SUCCESS）的上游 |
| `RUNNING` 运行中 | 已派发到某 worker 正在执行 | 上游全部 SUCCESS 且分配到可用 worker |
| `SUCCESS` 成功 | 执行成功 | RUNNING 正常结束 |
| `FAILED` 失败 | 执行失败且重试耗尽（attempt ≥ retry_max） | RUNNING 异常结束、重试用尽 |
| `STOPPED` 停止 | 人工停止/杀死，或因**上游失败被取消** | 人工操作；或上游 FAILED 导致下游不再调度 |

**「未运行 NOT_RUN」 vs 「等待 WAITING」的关键区别**：
- `NOT_RUN`：工作流实例**还没开始评估**该节点（工作流未触发，或调度器尚未推进到它）。**它不在「待跑队列」里。**
- `WAITING`：工作流**已在运行**，调度器**已评估**该节点并确认它「该跑」，只是**卡在上游依赖**上。**它在「待跑队列」里等上游。**

节点状态转移：
```
NOT_RUN ──(工作流启动, 评估依赖)──▶ WAITING ──(上游全 SUCCESS + 有 worker)──▶ RUNNING
                                      │                                     │
                                      │                                     ├─ 正常结束 ─▶ SUCCESS
                                      │                                     └─ 异常&重试耗尽 ─▶ FAILED
   NOT_RUN/WAITING/RUNNING ──(人工停止 | 上游 FAILED 取消)──▶ STOPPED
```

### 3.2 工作流实例状态（`workflow_instance.state`）—— 由子节点状态聚合

聚合规则（**自上而下取第一个命中**，优先级从高到低）：

| 优先级 | 工作流态 | 聚合条件（看该实例下所有 task_instance） |
|--------|----------|-------------------------------------------|
| 1 | `STOPPED` 停止 | 实例被人工停止 |
| 2 | `RUNNING` 运行中 | 存在任一节点 `RUNNING`；**或** 已有节点 `SUCCESS` 且仍有 `WAITING/NOT_RUN`（推进中） |
| 3 | `FAILED` 失败 | 存在 `FAILED` 节点 且 **无** `RUNNING` 节点（已卡死、无法继续推进） |
| 4 | `SUCCESS` 成功 | 所有节点均 `SUCCESS` |
| 5 | `WAITING` 等待 | 已触发但**尚未开始任何节点**（等定时到点 / 等资源 / 等上游工作流），节点多为 WAITING |
| 6 | `NOT_RUN` 未运行 | 所有节点均 `NOT_RUN`（实例已建未启动） |

聚合矩阵（行=节点状态组合特征，列=结论）：

```
                       有RUNNING  有FAILED  有WAITING/NOT_RUN  全SUCCESS  人工停止
 全部 NOT_RUN              -          -            ✓(未启动)         -         -      → NOT_RUN
 已触发未跑首节点          -          -            ✓                 -         -      → WAITING
 部分成功+剩余在等         ✓或-       -            ✓                 -         -      → RUNNING
 有节点在跑                ✓          -/✓          -/✓               -         -      → RUNNING
 有失败且无在跑            ✗          ✓            -/✓               -         -      → FAILED
 全部成功                  -          -            -                 ✓         -      → SUCCESS
 被人工停止                -          -            -                 -         ✓      → STOPPED
```

> 聚合只统计 `run_mode=NORMAL` 的节点实例；`TEST` 试跑不参与工作流状态聚合。
>
> MVP 落地：调度执行仍为 mock，但 `task_instance.state` 与上述聚合规则在 `OpsService`/调度服务里按真实语义计算 `workflow_instance.state`，前端驾驶舱与 `/ops` 直接展示两级状态。后期接真实 Master↔Worker 调度时，状态来源从 mock 推进切换为真实回调，规则不变。

## 5. 关键外键与引用（逻辑约束，DDL 不强制 FK 以兼容 H2 重建）

```
projects.tenant_id          → tenants.id

project_member(user_id,role_id,project_id) → users/roles/projects
datasources.type_code       → datasource_types.code

workflow_node(workflow_id,task_id) → workflow_def.id / task_def.id
workflow_edge(from_node_id,to_node_id) → workflow_node.id
workflow_instance.workflow_id → workflow_def.id
task_instance(workflow_instance_id, workflow_node_id, task_id, worker_node_code)
                            → workflow_instance / workflow_node / task_def / worker_nodes.node_code
derived_metrics.atomic_refs_json → atomic_metrics.code[]
metric_lineage(metric_type,metric_id) → atomic_metrics/derived_metrics
task_diagnosis.task_instance_id → task_instance.id
alert_rules.channel_id      → notification_channels.id
```

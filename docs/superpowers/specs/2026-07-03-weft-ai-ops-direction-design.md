# Weft 方向文档：AI 时代的数据运维（北极星）

- 日期：2026-07-03
- 状态：方向已确认（产品层面），不含实施计划
- 性质：本文回答"Weft 是什么、不是什么、往哪走"；具体功能落地仍走 Spec Kit（`specs/NNN-*`）逐个立项

## 1. 命题

**Weft 不做"数据的管理平台"，做"agent 写的数据项目的运行时 + 信任层"。**

数据中台的产品形态（资产目录、指标市场、质量规则中心、数据服务超市）诞生于"写代码很贵"的年代，用表单、拖拽、目录替代代码。AI 让写代码变得便宜且可由 agent 代劳，这个前提已失效。中台模式的三个必然失败点，Weft 全部反着做：

| 中台老路 | Weft 的立场 |
|---|---|
| 元数据靠人肉维护（目录描述、质量规则、指标口径），必然腐烂 | **代码是唯一真相**；一切元数据要么写在代码里，要么从代码与运行时**推导**（如 Calcite 血缘），绝不允许第二份人肉真相 |
| 平台是目的地，考核资产数量/覆盖率 | **价值发生在消费点**；用消费度量替代资产盘点，没人用的数据应被降频、下线——持续收缩而非堆积 |
| 按角色切碎人（数开/运维/治理/分析各一套界面） | **单一用户**：懂代码懂业务的开发者，一个人 + 一个 AI 编队端到端拥有一条数据产品线 |

## 2. 用户

唯一的第一用户：**懂点代码和业务的开发人员**。不为独立的运维、数开、治理、分析角色建任何界面。

这个人只做两件事：
1. **表达业务意图**——在 repo 里与本地 AI vibecoding 出任务/任务流（Skill + dw CLI 黄金路径）；
2. **批准 AI 的重要提案**——在监督席上审查运维编队的诊断与处置。

## 3. 三层产品结构

```
懂代码懂业务的开发者
   │ vibecoding（本地 AI + Skill + dw CLI）
   ▼
 repo（唯一真相）──push──▶ Weft 运行时（调度/执行/freshness/血缘）
                              │ 事件流
                              ▼
                      多 agent 运维编队（外挂进程，MCP + 闸门）
                              │ 带证据的提案 / 自动处置 / 经验沉淀
                              ▼
                      人的监督席：异常队列 + 审批 + 成绩单
```

- **Author 层（已成型，平台主动退出）**：接数、处理、产出全部是 repo 里的代码；平台零 UI 介入 authoring。凡是让人在网页上"录入/维护"元数据的界面都是老路。
- **Run 层（已成型，持续做深)**：调度内核、worker、补数、重跑、freshness/SLA、血缘推导。物理地基，与 AI 无关，必须稳。
- **Trust 层（新大陆，本方向的建设主体）**：常驻服务端的多 agent 运维编队 + 人的监督席。详见 §4、§5。

## 4. 多 agent 运维编队

**定位：一支故障响应团队（incident response team），不是通用多 agent 框架。**

### 4.1 闭环

感知 → 诊断 → 决策 → 行动 → 沉淀：

- **感知**：订阅 EventBus 事件流（失败事件、freshness 延迟、SLA/ETA 预警、节点遥测）。事件中心从"给人看的列表"升级为"agent 的收件箱"。
- **诊断**：证据链 = 实例日志（MinIO）+ 任务代码 + 血缘（neo4j）+ 历史运行 + 节点心跳。第一步做**故障分型**：瞬态（OOM/掉线/抖动）vs 确定性（代码 bug/上游数据异常/schema 漂移）。
- **决策 = 分诊，目标是把损失降到最低，比"修好任务"大**：沿血缘算**爆炸半径**，对照下游 SLA 算**时间预算**。时间充裕 → 修根因；紧张 → 先重跑赌瞬态并行诊断；赶不上 → 降级供数（下游用昨日数据 + freshness 标记）+ 提前通知，把"事故"变成"已知延迟"。
- **行动**：梯度阶梯，全部走既有 PolicyEngine 闸门（见 §4.3）。
- **沉淀**：incident 收口后写回经验库（见 §6）。

### 4.2 编队结构

```
事件流 ──▶ 统筹 agent（Incident Commander）
              │  开一个 Incident 工单，全程握有唯一决策权/行动权
              ├─▶ 诊断 agent      日志指纹 + 代码 + 历史 → 故障分型
              ├─▶ 影响评估 agent   血缘爆炸半径 × SLA 时间预算 → 分诊优先级
              ├─▶ 修复 agent      生成 patch → TEST run 拿证据 → 修复提案
              ├─▶ 处置 agent      重跑/补数/hold 下游/降级供数（走闸门）
              └─▶ 记忆检索/写回    相似事故命中经验；收口后沉淀
```

三条设计判断：
1. **Incident 是一等领域对象**：证据、各 agent 结论、统筹决策、提案、审批、结果、复盘全挂其上。它同时是监督席上的一张卡片——agent 的工作空间与人的审查单位是同一个东西。
2. **只有统筹 agent 有行动权**：干活 agent 只产出结论与提案；所有写动作由统筹经 GatedActionService 提交。权责单点、审计干净、防止 agent 互相打架。
3. **第一版编队要小**：统筹 + 诊断 + 影响评估三个，固定流水线（不做动态编排）；修复与处置先作为统筹的直接能力，跑顺再拆。

### 4.3 行动梯度与渐进自主

| 梯度 | 动作 | 自主性 |
|---|---|---|
| L1 | 重跑、换节点重跑、调资源、hold 下游 | 全自动，事后可查 |
| L2 | 补数、改调度参数、降级供数 | 视信任度自动或审批 |
| L3 | 改业务代码：patch → TEST run 验证 → 附证据包提交审批 | 默认人审 |

- **agent 永远交"带证据的提案"，不交"结果"**——这是它敢改业务代码的前提。
- **信任是挣来的，不是配的**：某类故障的提案被人连续批准 N 次后，该签名的处置才升格自动执行。经验库的成绩单是升格依据。
- **成本分层**：确定性 playbook 引擎（签名匹配 → 既定动作，毫秒级零成本）打底；LLM agent 只处理新签名与复杂分诊。经验沉淀的过程就是把 LLM 判断结晶成 playbook 的过程。

### 4.4 形态：外挂 agent 进程

编队以独立进程运行（Claude Agent SDK 或同类），通过事件订阅 + MCP 工具与平台交互，不内建于 master。理由：agent 能力（多步推理、工具使用、子 agent 编排）演进快，外挂可独立升级；代价是多一个部署单元与更宽的 MCP 面（见 §7）。

## 5. 监督席（产品主页）

**打开 Weft 首先看到的是 incident 队列，不是 dashboard 集合。**

每张卡片 = 一次事故 + agent 诊断 + 处置提案 + 一键批准/驳回。freshness、ops、血缘、日志全部降级为从卡片下钻进去的**证据页**。alerts + event-center + freshness 告警面收敛进这条队列。

UI 的职责收窄为三件事：情境感知（数据准不准时、谁坏了）、审批闸门、事后核查 agent 干了什么。**UI 不再是工作台，是监督席。**

## 6. 经验库（incident 记忆）

每次事故收口写回一条结构化记录：**故障签名（任务 + 错误类型 + 日志指纹）→ 诊断结论 → 采取的动作 → 效果 → 人的批注**。

- 喂 agent：同签名故障直接命中经验，免重新推理；人的批准/驳回/修改本身是训练信号。
- 喂人：成绩单（编队最近处理了什么、准确率如何），支撑渐进自主权的升格决策。
- **经验库是本产品的核心 IP**：运行时谁都能做，一个越用越懂你业务的运维记忆无法被抄。是数据资产，不是日志，按可检索、可统计的结构设计。

## 7. 被迫重划的边界

1. **MCP 定位改写**：原约定"MCP 只做查询、绝不扩 authoring"。新边界：**authoring 永远走 Skill + CLI（底线不变）；运维动作（诊断查询、重跑、补数、修复提案）走 MCP + 闸门**。agent 修 bug 的 patch 不是 authoring，是带证据的变更提案。
2. **修复代码必须回流 repo**：修复提案被批准后，变更必须以 repo 侧 `dw pull` 可取回的形式存在（理想形态：agent 直接开 PR）。服务端永远不能出现 repo 追不回来的定义漂移，否则"代码是唯一真相"即破。
3. **质量与目录的转型**：quality 从"规则录入中心"变为"契约履约看板"——下游对上游的期待（到达时间/schema/断言）写成代码里的 data contract，运行时强制校验，违约进 incident 队列；catalog 从人肉资产登记变为由 task_table_io + 血缘 + schema 自动推导的"数据地图"。

## 8. 页面清单裁决

| 现有视图 | 裁决 |
|---|---|
| ops / workflow-canvas / fleet / instance-log / workflow-instance-detail | 保留（Run 层地基） |
| freshness | 保留；其告警面并入 incident 队列 |
| lineage | 保留（Trust 层推理原料 + 证据页） |
| alerts / event-center | **收敛**为 incident 队列（监督席主页） |
| quality | **转型**：契约履约看板，去掉规则录入 |
| catalog | **转型**：自动推导的数据地图，去掉人肉维护 |
| marketplace（指标市场） | **砍**：指标口径入代码 |
| reports | **砍**：BI 交给专业 BI |
| service | **砍**：数据服务超市是中台货架 |
| integration | **砍**：接数就是写 task |
| datasources / settings | 保留（连接凭据需中心化管理） |

## 9. 反目标（明确不做）

- 不做 Web IDE / 拖拽式 ETL / 表单化任务配置——authoring 只在 repo。
- 不做任何需要人肉维护的元数据录入界面。
- 不做通用多 agent 编排框架——只做故障响应这一个场景的编队。
- 不做面向多角色的权限化工作台矩阵。
- 不以资产数量为任何页面的核心指标。

## 10. 后续落地路径（方向级，非承诺排期）

1. **收缩**：按 §8 裁决砍/合并页面，产品面先瘦下来（独立 change）。
2. **Incident 域模型**：一等对象 + 事件流接入 + 监督席队列 UI（走 Spec Kit 立项）。
3. **编队 v1**：外挂进程，统筹 + 诊断 + 影响评估，L1 动作自动化，L3 提案走审批。
4. **经验库 + 渐进自主**：签名命中、成绩单、升格机制。
5. **契约与数据地图转型**：quality / catalog 按 §7.3 改造。

每步独立立项（`specs/NNN-*`），本文档只锚定方向。

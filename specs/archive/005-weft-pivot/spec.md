# Feature Specification: Weft —— 「任务即代码」范式转型(总纲)

**Feature Branch**: `005-weft-pivot`

**Created**: 2026-06-26

**Status**: Draft

**Input**: User description: "DataWeave 从『AI 数据中台』转型为『任务即代码(Tasks-as-Code)』平台,更名为 Weft。任务/任务流作为本地 git 仓库里的纯文本文件存在,对本地 AI 编程 agent(Claude Code/Codex)友好;开发者 pull 到本地开发调试,push 回服务器治理与运行。服务端 AI 整套删除。本总纲统领后续 A–E 五个子特性。"

## 概述 *(总纲说明)*

这是一份**总纲/愿景 spec**,定义产品的范式转变、不可动摇的核心原则、子特性边界与交付顺序。它**不展开**各子特性(A–E)的具体字段 schema、REST/CLI 契约、UI 细节 —— 那些由各自的子 spec 承接。本总纲的"完成"定义是:① 范式与原则被各子 spec 共同遵守;② A–E 全部落地后,"本地开发 → 提交 → 运行"主链路端到端跑通;③ 服务端 AI 痕迹清零。

**一句话定位**:Weft 让数据任务与任务流像本地代码一样开发 —— 服务器是织机(loom)负责治理与运行,开发者在本地用 AI 编程 agent 编写"纬线"(weft,即任务),`pull`/`push` 往返,本地真跑或提交 TEST 调试。

**A–E 子特性对照表**(各自由独立子 spec 承接,详见 FR-017):

| 代号 | 名称 | 一句话范围 |
|------|------|-----------|
| A | 服务端 AI 拆除 | 移除 chat/AG-UI/workhorse/IntentRouter/findings,净化环境,不伤运行态观测 |
| B | 文件化定义契约 | 任务/任务流/类目的纯文本文件格式 schema 与序列化往返 |
| C | pull/push API | 服务器侧同步端点 + 版本快照治理 + 租户/项目隔离 |
| D | CLI+本地 runtime | 本地真跑 + TEST 提交 + 日志流回传 |
| E | MCP 工具重塑 | 向本地 AI agent 暴露操作工具,删旧 agent 工具 |

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 把项目同步到本地,像写代码一样开发任务 (Priority: P1)

开发者从服务器把一个项目(受租户/项目隔离)`pull` 到本地,得到一个纯文本文件的工作副本:目录树即类目树,每个任务是一份元数据文件 + 一份脚本执行体,每个任务流是一份带 DAG 拓扑的定义文件。开发者用本地的 AI 编程 agent(Claude Code/Codex)或任意 IDE 直接读写这些文件,用 git 做版本与 review,然后 `push` 回服务器;服务器在保持隔离与版本治理的前提下接收并物化为可调度的定义。

**Why this priority**: 这是整个范式转变的主链路。没有"pull→本地编辑→push"的闭环,"任务即代码"无从谈起。它单独可用即构成 MVP:即便本地真跑(Story 2)尚未具备,开发者已能用 AI agent 在本地开发并提交。

**Independent Test**: 在一个已存在任务/任务流的项目上执行 `pull`,确认本地文件能完整、可读地表示该项目;用文本编辑器修改一个任务的脚本与元数据,`push` 回去,再次 `pull` 到一个干净目录,确认改动被忠实保留(round-trip 一致),且服务器侧生成了新的版本快照、隔离边界未被跨越。

**Acceptance Scenarios**:

1. **Given** 服务器上有一个含若干任务与任务流的项目,**When** 开发者在本地 `pull` 该项目,**Then** 本地得到一棵目录树,其结构与命名直接对应类目树,且每个任务/任务流都有确定的纯文本表示。
2. **Given** 本地工作副本,**When** 开发者修改一个任务的脚本与调度元数据并 `push`,**Then** 服务器接收改动、生成新版本快照,且不影响其它项目/租户。
3. **Given** 一份本地定义,**When** 它被 `push` 后又被 `pull` 到全新目录,**Then** 两份文件在语义上等价(序列化/反序列化 round-trip 无丢失)。
4. **Given** 开发者无权访问的另一个项目,**When** 其尝试 `pull`/`push` 该项目,**Then** 操作被拒绝,隔离边界生效。

---

### User Story 2 - 本地真跑调试,无需先提交服务器 (Priority: P1)

开发者在本地工作副本里,用 CLI 直接运行一个任务,任务在本机由内嵌的轻量 runtime 真实执行(复用平台的 SQL/Shell 执行器),连接本地或开发数据源,标准输出直接回到终端。这让开发者像运行本地代码一样快速迭代,无需每次都提交服务器。

**Why this priority**: "像写本地代码一样"的核心体感就是能本地跑。它与 Story 1 共同构成开发内循环。优先级 P1,因为缺了本地真跑,开发体验退化为"只能编辑+提交",与现状差异不大。

**Independent Test**: 在本地工作副本里对一个 SQL 任务和一个 Shell 任务分别执行本地运行命令,确认任务在本机进程内真实执行(而非提交服务器),日志/结果直出终端,退出码正确反映成功/失败。

**Acceptance Scenarios**:

1. **Given** 本地工作副本里有一个 Shell 任务,**When** 开发者执行本地运行命令,**Then** 任务在本机进程内执行,stdout/stderr 实时回显,进程退出码被忠实返回。
2. **Given** 一个绑定开发数据源的 SQL 任务,**When** 本地运行,**Then** 执行器连接所配数据源执行,结果/错误直出终端,不触达服务器调度。
3. **Given** 一个会超时的任务,**When** 本地运行,**Then** runtime 在超时阈值强制结束并明确报告超时。

---

### User Story 3 - 提交服务器跑 TEST 实例,在真实环境验证 (Priority: P2)

本地快跑之后,开发者需要在贴近生产的环境验证(真实数据源、真实调度上下文)。开发者用 CLI 把任务以 TEST 模式提交服务器,服务器跑一个 TEST 实例,日志流式回到本地终端。

**Why this priority**: 本地真跑解决迭代速度,但环境保真不足;TEST 实例补齐"真实环境验证"。P2,因为它依赖服务器侧已有的 TEST 运行态与日志流能力,且在 Story 1+2 之后才形成完整调试闭环。

**Independent Test**: 对一个已 `push` 的任务执行 `run --test`,确认服务器创建 TEST 实例、执行,日志实时流回本地终端,最终状态(成功/失败/超时)正确回报。

**Acceptance Scenarios**:

1. **Given** 一个已提交服务器的任务,**When** 开发者执行 `run --test`,**Then** 服务器创建并执行 TEST 实例,本地终端实时看到日志,结束时回报终态。
2. **Given** 一次进行中的 TEST 运行,**When** 开发者在本地查看日志,**Then** 能拉取/跟随该实例的运行日志。

---

### User Story 4 - 本地 AI 编程 agent 通过 MCP 操作平台 (Priority: P2)

开发者在 Claude Code/Codex 里让 AI agent 直接代为操作平台:列出项目/任务、提交改动、触发运行、读取日志。平台通过 MCP 暴露这组工具,所有写操作仍经过既有的写闸门(PolicyEngine)。AI 不再是服务端内置的角色,而是开发者本地编辑器里的助手。

**Why this priority**: 这是"对 AI 编程 agent 友好"的直接兑现,也是删除服务端 AI 后保留 AI 价值的方式。P2,因为它依赖 Story 1/3 的能力先以可被 agent 调用的形式存在。

**Independent Test**: 用一个 MCP 客户端连接平台,确认能列出/读取任务、提交一处改动、触发一次运行、读取其日志;确认每个写操作都留下审计且未绕过写闸门。

**Acceptance Scenarios**:

1. **Given** 一个连上平台 MCP 的本地 AI agent,**When** agent 请求列出某项目的任务,**Then** 返回该项目(受隔离约束)的任务清单。
2. **Given** 同一 agent,**When** 它提交一处任务改动或触发一次运行,**Then** 操作经过写闸门、留下审计,效果与开发者手动经 CLI 操作一致。
3. **Given** 一个被策略判定为高风险的写操作,**When** agent 发起它,**Then** 该操作按既有闸门策略被拦截/升级审批,不因来自 agent 而被放行。

---

### User Story 5 - 服务端 AI 痕迹被干净拆除,环境净化 (Priority: P1)

平台维护者运行一次性的拆除,把服务端 AI 整套(聊天驾驶舱 cockpit、AG-UI 协议、workhorse 真大脑桥接、IntentRouter、主动播报/findings)从前后端移除,使代码库瘦身、依赖清理、环境无残留;同时**不得**误伤运行态观测能力(ops 概览、metrics、运行日志、DAG 实例视图)。

**Why this priority**: 用户明确要求"直接删除、用脚本删除处理掉,避免环境污染"。它是范式转变的前置解耦动作,且越早做越能避免新旧两套 AI 概念并存造成的认知与维护负担。P1。

**Independent Test**: 拆除后,后端编译通过、可启动;前端类型检查通过、可构建并在浏览器渲染;代码库中不再存在 chat/AG-UI/workhorse/IntentRouter/findings 的活跃代码与依赖;运行态观测(ops/metrics/日志/DAG 实例)仍可用。

**Acceptance Scenarios**:

1. **Given** 当前含服务端 AI 的代码库,**When** 执行拆除,**Then** chat cockpit、AG-UI 端点与事件、workhorse 桥接、IntentRouter、主动播报/findings 均被移除,后端编译与启动正常。
2. **Given** 拆除后的前端,**When** 类型检查与构建,**Then** 均通过,且页面在浏览器正常渲染、无聊天残留入口与控制台报错。
3. **Given** 拆除后,**When** 验证运行态观测,**Then** ops 概览、metrics、运行日志、DAG 实例视图等非 AI 能力完好可用。

---

### Edge Cases

- **本地与服务器版本不一致**:开发者基于旧版本本地修改后 `push`,服务器已有更新版本 —— 因服务器为治理真相源且 `push` 为幂等覆盖+生成新快照,系统以"覆盖并新增版本"处理,**不做**双向合并;但必须让开发者在覆盖前可感知差异(避免无意覆盖他人改动)。
- **本地文件被改成无效定义**:手写文件破坏了必填元数据或脚本缺失,`push` 必须校验并以可定位的错误拒绝,而非静默接受半成品。
- **目录树与类目树语义冲突**:本地重命名/移动目录如何映射为类目变更,需有确定规则(本总纲只要求"目录即类目"语义成立,细则留子 spec B/C)。
- **本地 runtime 缺少数据源凭据**:本地真跑时若无可用数据源,需明确报告而非伪装成功。
- **拆除误伤**:拆除服务端 AI 时若与运行态观测共享代码,必须区分保留,任何"顺手删"导致 ops/metrics/日志/DAG 视图失效都视为缺陷。**已知耦合(经代码勘查)**:`OpsController` 把 workhorse supervisor 探活端点(`/ops/supervisor`)与观测端点(metrics/日志/DAG 实例流)混置一处,子 spec A MUST **逐端点裁剪**而非整体删该 Controller。(另:运维告警 `OpsAlertService` 现状走 REST 返回、并不依赖 AG-UI 事件管道,删 AG-UI 不影响告警,仅需清理其过时注释 —— 此为 A 的清理项,非阻塞依赖。)
- **隔离越权**:任何 `pull`/`push`/run/MCP 操作触达无权项目/租户,必须被拒绝。

## Requirements *(mandatory)*

### Functional Requirements

#### 范式与文件化(贯穿 A–E 的总纲约束)

- **FR-001**: 系统 MUST 让 project、catalog、task、workflow 都具有**确定的纯文本磁盘表示**;本地目录树 MUST 直接映射类目(catalog)树。
- **FR-002**: 文件表示 MUST 对 AI 编程 agent 与人类同等友好 —— 可读、可 diff、可 code review,任务的元数据与脚本执行体分离存放。
- **FR-003**: 系统 MUST 保证本地文件定义与服务器侧定义之间的序列化/反序列化 **round-trip 语义无丢失**(push 后再 pull 得到语义等价的定义)。

#### 治理与同步(服务器为真相源)

- **FR-004**: 服务器 MUST 作为治理真相源,保留**租户/项目隔离**与**版本快照**治理;本地为工作副本。
- **FR-005**: 系统 MUST 提供 git 式的 `pull`(取下项目为本地文件)与 `push`(提交本地文件回服务器)。`push` MUST 是**幂等覆盖**并生成新版本快照;系统 MUST NOT 实现双向同步或冲突合并。
- **FR-006**: `push` 前系统 MUST 让开发者可感知本地与服务器的差异,避免无意覆盖。
- **FR-007**: 所有 `pull`/`push`/run 与 MCP 操作 MUST 受隔离约束;越权访问 MUST 被拒绝。
- **FR-008**: `push` MUST 对本地定义做有效性校验,对无效/不完整定义以**可定位的错误**拒绝,不静默接受。

#### 本地与远端调试(两条腿)

- **FR-009**: CLI MUST 内嵌轻量 runtime,在开发者本机**真实执行**任务(复用平台 SQL/Shell 执行器语义),连接本地/开发数据源,输出直出终端,退出码忠实反映结果。
- **FR-010**: 本地 runtime MUST 能脱离服务器 worker 进程独立运行,且 MUST 与服务器执行语义保持一致(不另起一套不一致的语义)。鉴于 CLI 与服务器执行器跨语言(CLI 为 Go、执行器为 Java/worker 模块),此处"一致"为**契约级对齐**(退出码/stdout/stderr/超时/数据源驱动加载等行为对齐),而非代码级复用;具体实现选型留子 spec D。
- **FR-011**: CLI MUST 支持把任务以 TEST 模式提交服务器执行,并将运行日志流式回传本地终端。

#### AI 归位(服务端拆除 + 本地 MCP)

- **FR-012**: 系统 MUST 移除服务端 AI 整套:chat cockpit、AG-UI 协议端点与事件、workhorse 真大脑桥接、IntentRouter、主动播报/findings;移除 MUST 干净(代码与依赖无活跃残留)。
- **FR-013**: 拆除 MUST NOT 损伤运行态观测能力(ops 概览、metrics、运行日志、DAG 实例视图)与调度内核。
- **FR-014**: 平台 MUST 通过 MCP 向本地 AI agent 暴露操作工具(至少:列出/读取、提交、触发运行、读取日志);旧的服务端 agent 专用工具 MUST 被移除或重塑。
- **FR-015**: 所有经 MCP/CLI 发起的**写操作** MUST 仍经过既有写闸门(PolicyEngine)与审计,不因来源是 AI agent 而被放行。

#### 内核复用与边界

- **FR-016**: 转型 MUST 复用现有内核 —— 调度器(peer master + SKIP LOCKED + cron guard)、版本快照、SQL/Shell 执行器、写闸门、MCP server 框架 —— 而非重写。
- **FR-017**: 本总纲 MUST 把交付拆为五个有序子特性,各自由独立子 spec 承接:A 服务端 AI 拆除(前置)、B 文件化定义契约(地基)、C 服务器 pull/push API、D CLI+本地 runtime、E MCP 工具重塑;依赖关系 MUST 为:B 依赖 A 解耦后的净化环境;C 依赖 B;**D 的本地真跑子集(FR-009/010)仅依赖 B**(能解析文件格式即可),**D 的 TEST 提交子集(FR-011)依赖 C** —— 故本地 runtime 可与 C 并行开工;E 依赖 C。
- **FR-018**: 产品 MUST 更名为 **Weft**;代码 MUST 在当前仓库**原地重构**(不新建空仓库),定位词从"AI 数据中台"改为"任务即代码平台"。
- **FR-019**(硬规范,No Legacy Migration): 存量 DB 中的任务/任务流/类目数据 MUST 被直接删除,系统 MUST NOT 提供任何存量导出/迁移/新旧并存路径;文件格式与 pull/push MUST 仅服务于转型后新建或重建的定义。

### Key Entities *(include if feature involves data)*

- **Tenant(租户)**:隔离的最外层边界;一个租户下含若干项目;所有 `pull`/`push`/run/MCP 操作都受租户边界约束,跨租户访问 MUST 被拒绝。
- **Project(项目)**:隔离与治理的顶层单元,归属某租户;`pull`/`push` 以项目为粒度;承载其下的类目、任务、任务流。
- **Catalog(类目)**:任务/任务流的目录组织;在本地表现为**目录树**,目录层级即类目层级。
- **Task(任务)**:最小可执行单元;由**元数据**(类型、调度参数、数据源绑定、超时/重试等)与**脚本执行体**(SQL/Shell/Python 文本)组成;在本地表现为元数据文件 + 脚本文件。
- **Workflow(任务流)**:由任务节点与依赖边构成的 DAG;在本地表现为带拓扑(节点+边+各节点引用的任务)的定义文件。
- **Version Snapshot(版本快照)**:服务器侧每次 `push` 生成的不可变发布快照,作为运行期真相源;本总纲沿用现有快照机制。
- **Local Working Copy(本地工作副本)**:开发者本机的项目文件集合,受 git 管理,是本地开发/调试/AI agent 操作的对象。
- **MCP Tool(MCP 工具)**:平台向本地 AI agent 暴露的操作能力单元(列出/读取/提交/运行/日志),写操作经写闸门。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 开发者能在不打开任何 Web 编辑界面的情况下,完成"取下项目 → 本地编辑任务 → 提交 → 运行"的完整链路;该链路 100% 可用纯文件 + CLI/MCP 完成。
- **SC-002**: 对任意任务/任务流定义,`push` 后再 `pull` 到干净目录,得到的文件与原文件**语义等价**(round-trip 一致率 100%,无字段静默丢失)。
- **SC-003**: 一个标准的 SQL 或 Shell 任务,从本地发起真跑到看到首行输出,在开发者本机于数秒内完成,无需任何服务器往返。(具体通过/失败阈值,如 <3s/<5s,留子 spec D 明确。)
- **SC-004**: 服务端 AI 拆除后,代码库中不再存在 chat/AG-UI/workhorse/IntentRouter/findings 的活跃代码与依赖;后端编译/启动、前端类型检查/构建/浏览器渲染全部通过;运行态观测能力 100% 保留可用。
- **SC-005**: 本地 AI 编程 agent 经 MCP 能完成"列出 → 提交 → 运行 → 读日志"全流程;其中每一次写操作都留下审计且未绕过写闸门(抽样核对 100% 命中审计)。
- **SC-006**: 越权访问(跨租户/跨无权项目的 pull/push/run/MCP)100% 被拒绝。
- **SC-007**: 五个子特性 A–E 各自有独立子 spec,且彼此边界无重叠、依赖顺序自洽(交叉评审时无"单独编译却在兄弟特性落地后失效"的不闭环项)。

## Assumptions

- **文件格式倾向**:任务元数据与任务流定义采用人类可读的结构化文本(倾向 YAML),脚本执行体为独立的 `.sql`/`.sh`/`.py` 文件;具体 schema 留子 spec B 定义。本总纲只约束"纯文本、元数据与脚本分离、目录即类目"。
- **存量不予考虑(硬规范)**:现有由 Web 编辑器创建、存于 DB 的存量任务/任务流/类目数据**直接删除**,不做导出/迁移/并存。转型从 clean slate 开始,文件格式与 pull/push 只服务于转型后新建或重建的定义。子 spec B/C MUST NOT 把"存量迁移"作为需求。(见 constitution 硬约束 No Legacy Migration。)
- **Web 编辑器存废**:倾向**保留只读查看**(便于运行态/治理侧观察),**编辑能力全部转本地**;最终存废与细节留子 spec(不在本总纲拍死)。
- **数据源凭据**:本地真跑所需的数据源连接信息由开发者本机配置提供(沿用现有 `DW_DS_*` 类环境注入思路);凭据不随定义文件入库到 git。细则留子 spec D。
- **认证与权限**:沿用平台既有的租户/项目隔离与认证机制;`pull`/`push`/MCP 复用现有鉴权,不新设独立账户体系。
- **现有内核可用**:调度器、版本快照、SQL/Shell 执行器、PolicyEngine 写闸门、MCP server 框架均已存在且稳定,可被复用与适配(已由代码勘查确认)。
- **隔离推进**:鉴于本转型同时删除大量代码并改造主链路,整体在**独立 git worktree** 推进,避免与近期已合并的 distributed-cron、ops/instance-dag-viewer 等特性相互污染(遵循仓库的并行特性隔离规约)。
- **更名范围**:本总纲确立更名为 Weft 与原地重构;具体的二进制名/包名/品牌文案替换细则随各子特性推进,不在本总纲穷举。

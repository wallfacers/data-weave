# Feature Specification: 运营中心实例列表切换与 DAG 查看

**Feature Branch**: `003-instance-dag-viewer`

**Created**: 2026-06-26

**Status**: Draft

**Input**: User description: "运营中心 任务流实例，当前的查看的是任务实例列表，直接需要支持切换，查看任务实例列表，和任务流实例列表，基于表设计，一个任务流实例下有多个任务实例，任务流实例支持查看任务实例的DAG，并且都需要支持查看实例级别的DAG图，支持节点状态显示，支持查看实例的实际代码和配置，我们知道任务运行的时候，比如SHELL脚本，实际脚本，十二ehco '${yyyyMMdd}' 这一的参数会被替换，业务日期"

## Clarifications

### Session 2026-06-26

- Q: DAG 视图从列表点击行后以什么形式呈现？ → A: 全屏弹窗（Dialog），与现有周期任务流列表"查看 DAG"按钮打开 DagViewerDialog 的模式一致
- Q: DAG 弹窗中点击节点后详情（代码/配置）以什么形式呈现？ → A: 弹窗内侧边面板——DAG 图左移，右侧展开详情面板，点击其他节点切换内容
- Q: 参数替换各来源的优先级规则是什么？ → A: 不单独定义优先级，复用执行引擎实际使用的参数解析逻辑，保证展示结果与真正执行时一致
- Q: 任务流实例列表的状态字段来源？ → A: 直接使用 WorkflowInstance.state 存储值，与 task_instance.state 一样，无需实时聚合
- Q: 实例 DAG 拓扑应使用哪个版本的 DAG？ → A: 使用 WorkflowInstance.workflowVersionNo 对应的历史版本 DAG，即实例真实执行时的拓扑

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 实例列表视图切换 (Priority: P1)

运营人员在运营中心查看实例时，当前只能看到平铺的任务实例列表。但实际运维场景中，运营人员需要先了解哪些任务流实例在运行、哪些失败了，再下钻到具体的任务实例排查问题。系统需要提供"任务实例"和"任务流实例"两个列表视图，并支持一键切换。

**Why this priority**: 这是整个功能的入口。没有任务流实例列表视图，用户无法从任务流粒度感知整体运行状况。所有后续的 DAG 查看和代码查看都依赖先找到目标实例。

**Independent Test**: 打开运营中心实例页签，默认显示任务实例列表（保持现有行为）。点击"任务流实例"切换按钮，列表切换为任务流实例数据。切换回"任务实例"，恢复原列表。两个列表各自保持独立的筛选条件和分页状态。

**Acceptance Scenarios**:

1. **Given** 运营中心已打开，当前在"实例"页签，**When** 用户查看列表表头区域，**Then** 看到"任务实例"和"任务流实例"两个切换选项，当前选中"任务实例"
2. **Given** 当前选中"任务实例"并已设置筛选条件（如状态=FAILED），**When** 用户切换到"任务流实例"，**Then** 列表展示任务流实例数据，筛选条件重置为适用于任务流实例的默认条件
3. **Given** 当前在"任务流实例"视图，**When** 用户点击某一行，**Then** 以全屏弹窗打开该任务流实例下的任务实例 DAG 图（进入 User Story 2 的流程），与现有周期任务流列表"查看 DAG"的弹窗交互一致
4. **Given** 当前在"任务实例"视图，**When** 用户点击某一行，**Then** 以全屏弹窗打开该任务实例所属的任务流实例 DAG 图，并高亮当前选中的任务实例节点（进入 User Story 2 的流程）
5. **Given** 任务流实例列表已加载，**When** 用户浏览列表，**Then** 每行显示：任务流名称、触发类型（CRON/MANUAL/BACKFILL）、业务日期、状态、总任务数/完成数/失败数、开始时间、耗时

---

### User Story 2 - 实例级别 DAG 图与节点状态 (Priority: P2)

运营人员需要直观地看到任务流实例中各个任务节点的运行状态和依赖关系。DAG 图应以有向无环图的形式展示，每个节点显示其实时运行状态（等待中、运行中、成功、失败、跳过等），支持缩放和拖拽。

**Why this priority**: DAG 可视化是理解任务流实例运行情况的核心手段。有了 P1 的列表入口后，DAG 图是用户下钻后的第一需求，直接影响问题定位效率。

**Independent Test**: 从任务流实例列表点击任意一行，以全屏弹窗打开 DAG 视图（与现有 DagViewerDialog 交互一致）。DAG 图中每个节点显示对应任务实例的状态。状态为 FAILED 的节点以红色标记，RUNNING 的节点有动画指示。悬停节点显示详情浮层。

**Acceptance Scenarios**:

1. **Given** 用户从任务流实例列表点击某一行，**When** DAG 视图打开，**Then** 展示该任务流实例的 DAG 拓扑图，每个节点为一个任务实例，节点上显示任务名称和当前运行状态
2. **Given** DAG 视图已打开且该任务流实例正在运行，**When** 某个任务实例状态发生变化（如从 RUNNING 变为 SUCCESS），**Then** DAG 图中对应节点的状态标识实时更新
3. **Given** DAG 视图中有多个节点，**When** 用户悬停在某个节点上，**Then** 显示浮层信息：任务名称、状态、开始时间、结束时间、耗时、重试次数、执行节点
4. **Given** DAG 视图中有节点处于 FAILED 状态，**When** 用户查看该节点，**Then** 节点以醒目的红色标识，浮层中显示失败原因摘要
5. **Given** DAG 视图已打开，**When** 用户缩放画布，**Then** DAG 图按比例缩放，节点标签在缩小时适当隐藏以保持可读性
6. **Given** 用户从任务实例列表点击某一行（该任务实例属于某个任务流实例），**When** DAG 视图打开，**Then** 展示其所属任务流实例的完整 DAG，并以高亮边框标记被点击的任务实例节点，画布自动滚动使该节点居中可见

---

### User Story 3 - 实例实际代码与配置查看 (Priority: P3)

运营人员在排查任务失败或验证执行结果时，需要查看任务实例在运行时的实际代码和配置——不是任务定义的模板，而是参数替换后真正被执行的内容。例如 SHELL 任务定义中写的是 `echo '${yyyyMMdd}'`，运营人员需要看到业务日期替换后的实际脚本 `echo '20260626'`。

**Why this priority**: 这是问题排查的深度需求。大多数情况下，DAG 状态和日志已经足够判断问题；但当怀疑参数传递错误或配置覆盖时，查看实际执行的代码是唯一的确诊手段。

**Independent Test**: 在 DAG 视图中点击任意任务节点，打开详情面板，其中包含"实际代码"和"实际配置"两个子页签。对于 NORMAL 模式实例，展示参数替换后的完整脚本；对于 TEST 模式实例，展示测试时临时修改的内容并标注参数替换结果。

**Acceptance Scenarios**:

1. **Given** 用户在 DAG 视图中点击某个 SHELL 类型任务节点，**When** 打开详情面板的"实际代码"页签，**Then** 显示该任务实例被执行时的完整脚本内容，其中所有 `${...}` 参数已被替换为实际值
2. **Given** 用户查看某个任务的"实际代码"，**When** 脚本中包含未解析的参数（参数值在运行时不可用），**Then** 参数位置保留原始 `${...}` 占位符并附带视觉提示"未解析"
3. **Given** 该任务实例为 TEST 运行模式且携带编辑器临时内容，**When** 用户查看"实际代码"，**Then** 显示该临时内容（TEST 模式以临时内容为准），并标注"测试运行"标识
4. **Given** 用户在 DAG 视图中点击某个任务节点，**When** 打开详情面板的"实际配置"页签，**Then** 显示该任务实例运行时的完整配置（包括任务类型、超时时间、重试策略、资源限制），以及参数 JSON（所有变量已替换为实际值）
5. **Given** 用户查看某个任务的"实际配置"，**When** 任务有测试运行时的临时参数覆盖，**Then** 配置中标注"参数已被测试运行覆盖"，并同时展示原始参数和覆盖后的参数
6. **Given** 用户查看某个任务的"实际代码"，**When** 任务类型为 SQL 或 SPARK，**Then** 显示参数替换后的实际 SQL/Spark 代码

---

### Edge Cases

- 任务流实例的 DAG 节点全部被冻结/skip 时，DAG 视图显示"该实例无活跃任务节点"的空状态提示
- 任务实例不属于任何任务流实例（独立 MANUAL 或 TEST 运行）时，从任务实例列表点击后不显示 DAG 图，直接展示该任务实例的详情（代码和配置）
- 任务流实例中有大量任务节点（如超过 100 个）时，DAG 图默认以适合视口的缩放级别显示，并提供搜索/过滤节点功能保证可用性
- 任务流实例已被软删除但其 task_instance 记录仍存在时，DAG 视图仅展示任务实例节点，任务流名称显示为"已删除的任务流"，节点间连线仍正常显示
- 参数替换时遇到嵌套参数（如 `${${paramName}}`），系统最多解析一层嵌套，其余保留原始占位符并标注"嵌套参数未完全解析"
- 用户快速连续切换任务实例和任务流实例视图时，系统取消前一个未完成的请求，仅加载最新选择的视图数据
- 任务流实例的业务日期参数替换涉及特殊日期格式（如 `${yyyyMMdd-1}` 表示前一天）时，按运行时实际使用的值展示

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在运营中心"实例"页签提供"任务实例"和"任务流实例"两个列表视图，并支持一键切换
- **FR-002**: 任务流实例列表必须展示以下信息：任务流名称、触发类型、业务日期、状态、总任务数/已完成数/失败数、开始时间、耗时
- **FR-003**: 任务流实例列表必须支持按状态、触发类型、业务日期范围进行筛选，并支持分页
- **FR-004**: 从任务流实例列表点击任意行，系统必须展示该任务流实例的 DAG 拓扑图，每个节点代表一个任务实例并显示其实时运行状态
- **FR-005**: 从任务实例列表点击任意行，若该实例属于某个任务流实例，系统必须展示其所属任务流实例的 DAG 图并高亮该节点；若不属于任何任务流实例，直接展示该实例的详情（代码和配置）
- **FR-006**: DAG 图中的节点必须支持以下运行状态的视觉区分：未运行、等待中、已派发、运行中、成功、失败、已停止、已抢占、已暂停、已跳过
- **FR-007**: DAG 图中状态为"运行中"的节点必须支持实时状态更新
- **FR-008**: 用户点击 DAG 图中任意节点，系统必须在 DAG 弹窗内以侧边面板形式展示该任务实例的详情，包含"实际代码"和"实际配置"两个页签；点击其他节点时侧边面板内容切换为对应节点
- **FR-009**: "实际代码"页签必须展示参数替换后的完整执行脚本/SQL/代码，原始模板中的 `${...}` 占位符替换为运行时的实际值。参数替换必须复用执行引擎的解析逻辑，保证展示结果与任务执行时真正使用的脚本一致
- **FR-010**: "实际配置"页签必须展示任务实例运行时的完整配置，包括任务类型、超时时间、重试策略、资源限制以及参数替换后的参数 JSON
- **FR-011**: 对于 TEST 运行模式的任务实例，系统必须在代码和配置视图中标注"测试运行"，并明确标识测试时临时修改的内容与已发布版本原始定义的差异
- **FR-012**: DAG 视图必须支持缩放和拖拽画布操作
- **FR-013**: 系统必须支持在 DAG 视图中搜索/筛选节点（按任务名称）
- **FR-014**: 对于参数替换后仍未解析的占位符，系统必须以视觉提示标注"未解析"

### Key Entities

- **任务流实例 (WorkflowInstance)**: 一次任务流触发产生的运行记录。包含触发类型、业务日期、聚合状态、任务完成进度等属性。关联多个任务实例。
- **任务实例 (TaskInstance)**: 单个任务的运行记录。包含运行状态、业务日期、执行日志、退出码、失败原因、TEST 模式临时内容、TEST 模式临时参数等属性。隶属于一个任务流实例（或独立运行）。
- **任务定义版本 (TaskDefVersion)**: 任务在发布时的快照，包含任务类型、脚本/SQL 内容模板、参数定义、资源配置等。任务实例通过其关联到执行时使用的版本。
- **DAG 视图**: 任务流实例运行时的有向无环图快照，节点为任务实例（携带运行时状态），边为任务间的依赖关系（来源于工作流定义的 DAG 拓扑）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可在 2 秒内完成"任务实例"和"任务流实例"列表视图之间的切换（列表数据加载完成）
- **SC-002**: 用户从任务流实例列表点击进入 DAG 视图，20 个节点以内的 DAG 图在 2 秒内完成渲染并可交互
- **SC-003**: 100 个节点以内的大型 DAG 图在 5 秒内完成渲染并支持缩放/拖拽操作
- **SC-004**: DAG 图中节点的状态更新延迟不超过 3 秒（从任务实例状态实际变更到用户在 DAG 图上看到节点视觉更新）
- **SC-005**: 用户点击 DAG 节点后，"实际代码"和"实际配置"在 1 秒内完成加载和参数替换渲染
- **SC-006**: 运营人员通过 DAG 图定位失败任务的时间比当前纯列表方式减少 50%（当前需要逐行点击日志链接再返回）
- **SC-007**: 90% 的参数替换结果与任务执行时实际使用的值一致（剩余 10% 为运行时动态变量等无法事后还原的情况）

## Assumptions

- 现有的事件流机制可复用于 DAG 图的实时状态更新
- 现有的 DAG 渲染组件可扩展以支持实例级别的运行时状态叠加（在定义 DAG 拓扑上叠加 task_instance 的状态）
- 任务流实例的 DAG 拓扑从 WorkflowInstance.workflowVersionNo 对应的历史版本 dag_snapshot_json 中获取（实例真实执行时的拓扑），结合实际 task_instance 的运行状态叠加渲染；不依赖当前最新已发布版本
- 参数替换的解析逻辑复用执行引擎现有的参数解析实现，保证展示结果与真正执行时一致（不单独实现一套解析规则）
- 独立运行的任务实例（无 workflow_instance_id）不需要 DAG 视图，直接展示详情面板
- 运营中心的"实例"页签默认仍显示任务实例列表（保持向后兼容，不改变现有用户的默认体验）
- 本功能仅涉及查看操作，不改变任何实例的写入行为

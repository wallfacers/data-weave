# Feature Specification: Weft 子特性 C —— 服务器 pull/push 同步 API

**Feature Branch**: `008-weft-pull-push-api`

**Created**: 2026-06-27

**Status**: Draft

**Input**: User description: "Weft 子特性 C:服务器侧 pull/push 同步 API,把项目在『服务器治理真相源』与『本地纯文本工作副本』之间往返。承接总纲(005)FR-004/005/006/007/008 + SC-002/SC-006,复用已落地的 B(文件化定义契约)。"

## 概述 *(子特性说明)*

C 是 Weft「任务即代码」主链路的**服务器一端**:它让开发者把一个项目从服务器**取下(pull)**为一棵纯文本文件工作副本,在本地用 AI 编程 agent/IDE 编辑后,再**提交(push)**回服务器治理与运行。C 不发明文件格式 —— 文件格式由子特性 B(文件化定义契约)定义并已落地;C 负责把服务器侧的项目定义**装配/落库**到 B 的契约两端,并施加治理(版本快照)、隔离(租户/项目)与校验(可定位错误拒绝)。

**边界一句话**:B 是「定义 ↔ 文件」的纯转换;C 是「服务器真相源 ↔ B 的转换入口」之间的同步端点 + 治理。C 只提供服务器 HTTP 端点,**不含** CLI(留 D)、本地 runtime(留 D)、MCP 工具(留 E)。

**完成定义**:在一个已存在任务/任务流的项目上,通过 HTTP 完成 `pull → 本地改 → push → 再 pull` 闭环,round-trip 语义等价;push 生成新版本快照;隔离边界生效;无效定义被可定位错误拒绝且不部分落库。

## Clarifications

### Session 2026-06-27

- Q: push 的"生成版本快照"是否=自动发布上线(可调度)? → A: 否 —— push 落库 + 生成版本快照后,定义停在**草稿/新版本态,不自动晋级 ONLINE**;上线是与 push 解耦的另一个显式治理动作(经 UI/CLI publish,不在 C 范围)。push 不受"被引用任务必须 ONLINE"的发布前置约束拖累。
- Q: push 是否默认拒绝"陈旧覆盖"(并发防护)? → A: 是 —— 采用**乐观并发**:pull 返回该项目的基线版本令牌,push 携带它;若服务器侧当前版本已不等于基线(他人已改),**默认拒绝**并提示先 pull/看 diff;调用方可显式 `force` 覆盖。diff(US3)为配套知情手段。
- Q: push 对本地缺失的服务器定义如何处理(删除语义)? → A: 删除 + **引用完整性守卫** —— 本地缺失即删除,但若被删任务/任务流仍被**在线(ONLINE)任务流**引用,则整单 push 以可定位错误拒绝(指明被谁引用),要求先解除引用;历史版本快照按现有治理**保留**(软治理)。
- Q: pull/push 交换文件集的传输形态? → A: **单个 JSON 请求/响应**,body 含 `files{ 相对路径 → UTF-8 内容 }` 映射 + 元字段(projectId、基线令牌、force)。配**完整性校验**(文件数/大小核对,防截断丢失);push 为事务性全有或全无(强一致,非最终一致)。不用 tar/zip 打包或逐文件多请求。
- Q: 任务绑定的数据源"逻辑名"对应服务器哪个字段? → A: 逻辑名直接 = **`Datasource.name`**(项目内唯一,`existsByProjectIdAndNameAndDeleted` 保证);C 按 `(tenantId, projectId, name)` 解析为 `datasourceId`。不新增 `code` 字段、不在文件写数字 id —— 零 schema 变更。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 把项目取下为本地文件工作副本(pull) (Priority: P1)

开发者对一个有权访问的项目发起 `pull`。服务器把该项目下的**类目树**、**全部任务**(元数据 + 脚本执行体)、**全部任务流**(DAG 拓扑:节点 + 边 + 各节点引用的任务)、**标签与标签关联**装配成一份完整定义,序列化为一组纯文本文件返回。文件中任务对数据源的绑定以**逻辑名**出现(不含连接主机/凭据)。整个操作受租户/项目隔离约束。

**Why this priority**: pull 是主链路入口 —— 没有"把项目变成本地文件",本地开发无从开始。它单独可用即有价值(开发者可在本地只读地浏览/diff 整个项目定义)。

**Independent Test**: 对一个含若干任务与任务流的项目调用 pull,确认返回的文件集能完整、可读地表示该项目:目录树结构对应类目层级,每个任务有元数据文件 + 脚本文件,每个任务流有带拓扑的定义文件,标签与绑定的数据源逻辑名可见,且文件中不含任何数据源连接凭据。

**Acceptance Scenarios**:

1. **Given** 服务器上有一个含任务、任务流、类目、标签的项目,**When** 开发者 pull 该项目,**Then** 得到一组纯文本文件:目录树即类目树,每个任务/任务流有确定的纯文本表示,标签与数据源逻辑名忠实呈现。
2. **Given** 某任务绑定了源/目标数据源,**When** pull,**Then** 文件中以数据源**逻辑名**记录绑定,**不**包含主机、端口、用户名、密码等连接信息。
3. **Given** 开发者无权访问的另一个项目,**When** 其 pull 该项目,**Then** 操作被拒绝(隔离边界生效),不泄露该项目存在与否之外的任何定义内容。
4. **Given** 一个空项目(无任务/任务流),**When** pull,**Then** 返回的文件集仍能表达项目与类目骨架,不报错。

---

### User Story 2 - 把本地改动提交回服务器并生成版本快照(push) (Priority: P1)

开发者把本地文件工作副本 `push` 回服务器。服务器对文件集做**反序列化 + 有效性校验**:任何无效/不完整定义(缺必填元数据、脚本缺失、引用了项目内不存在的数据源逻辑名、任务流边指向不存在的节点等)以**可定位的错误**(指明文件 + 字段 + 原因)拒绝,**不**静默接受半成品,**不**部分落库。校验通过后,服务器把数据源逻辑名解析回该项目内的数据源,以**幂等覆盖**方式重建该项目的类目/任务/任务流/标签定义,并为受影响的任务与任务流**生成新版本快照**。整个操作受隔离约束。

**Why this priority**: push 是主链路的回写半边;与 pull 合起来才构成"任务即代码"的 pull→edit→push 闭环,是 MVP 的核心。push 必须保证治理(快照)与安全(隔离、校验),否则文件化失去服务器治理价值。

**Independent Test**: 对一个项目 pull 到本地,修改一个任务的脚本与调度元数据后 push,确认服务器接收改动、为该任务生成新版本快照、隔离未被跨越;再对一份故意写坏的定义 push,确认被可定位错误拒绝且服务器侧定义未发生任何变化。

**Acceptance Scenarios**:

1. **Given** 本地工作副本改了一个任务的脚本与超时,**When** push,**Then** 服务器更新该任务定义并为其生成新版本快照,其它项目/租户不受影响。
2. **Given** 本地文件缺了某任务的必填元数据或脚本体,**When** push,**Then** 操作被拒绝,错误信息可定位到具体文件与字段,服务器侧该项目定义**保持 push 前状态**(全有或全无)。
3. **Given** 本地某任务引用了项目内不存在的数据源逻辑名,**When** push,**Then** 被可定位错误拒绝(指明该任务与未知数据源名),不落库。
4. **Given** 一份本地定义,**When** 它被 push 后再 pull 到全新目录,**Then** 两份文件语义等价(round-trip 一致,无字段静默丢失)。
5. **Given** 开发者无权访问的项目,**When** 其 push,**Then** 被拒绝,隔离边界生效。
6. **Given** 本地新增/删除了任务或任务流,**When** push,**Then** 服务器侧据此新增/删除对应定义(幂等覆盖:以本地文件集为准重建),历史版本快照按现有治理保留;**但**若被删定义仍被在线任务流引用,push 以可定位错误整单拒绝(指明引用方),要求先解除引用。

---

### User Story 3 - push 前预览将被覆盖的差异(diff) (Priority: P2)

push 是**幂等覆盖**而非合并 —— 开发者基于旧状态本地编辑期间,服务器侧可能已被他人改动。为避免无意覆盖,开发者在 push 前可获取一份**差异预览**:本地文件集相对服务器当前定义,哪些任务/任务流/类目/标签将被**新增、修改、删除**。系统**不做**双向合并(总纲明确否决);diff 仅为告知,让开发者知情后再决定是否 push。

**Why this priority**: 总纲 FR-006 要求"push 前可感知差异,避免无意覆盖"。它是 push 安全性的补强,但不阻塞 pull/push 闭环本身可用,故 P2。

**Independent Test**: 在服务器侧定义与本地文件集存在差异(增、改、删各一)的情况下请求 diff 预览,确认返回的增/改/删清单与实际差异一致;确认 diff 是只读操作,不改变服务器侧任何定义。

**Acceptance Scenarios**:

1. **Given** 本地相对服务器有增、改、删三类变化,**When** 请求 diff 预览,**Then** 返回按实体粒度(任务/任务流/类目/标签)分类的新增、修改、删除清单。
2. **Given** 本地与服务器定义完全一致,**When** 请求 diff 预览,**Then** 返回空差异(无任何增改删)。
3. **Given** 任意 diff 预览请求,**When** 执行,**Then** 服务器侧定义不被改变(只读)。

---

### Edge Cases

- **无效/半成品定义 push**:必须校验并以可定位错误拒绝,且**事务性**保证——要么整份 push 全部生效,要么全部不生效,不留半成品(US2 场景 2)。
- **未知数据源逻辑名**:push 引用了项目内不存在的数据源名,以可定位错误拒绝(US2 场景 3);C 不创建数据源(数据源定义不在文件契约内,B 已定)。
- **越权同步**:任何 pull/push/diff 触达无权项目/租户,必须被拒绝(US1 场景 3、US2 场景 5)。
- **并发覆盖**:开发者基于旧状态 push 覆盖他人改动 —— 系统采用**乐观并发**:pull 发基线版本令牌,push 校验;基线陈旧(他人已改)时**默认拒绝**,提示先 pull/看 diff,调用方可显式 `force` 覆盖。**不做**双向合并;diff 预览(US3)是配套知情手段。
- **空项目 / 空类目**:pull/push 对空项目或仅含空类目的项目须正常工作(US1 场景 4),复用 B 对空类目的表达。
- **round-trip 字节稳定**:push→pull 的语义等价由 B 的序列化往返契约(R3 字节稳定)保证;C 不得在装配/落库环节引入字段丢失或顺序抖动。

## Requirements *(mandatory)*

### Functional Requirements

#### Pull(导出)

- **FR-001**: 系统 MUST 提供按 **project 粒度**的 pull 端点,把该项目下的类目树、全部任务(元数据 + 脚本体)、全部任务流(节点 + 边 + 节点→任务引用)、标签与标签关联,装配成一份完整定义并以纯文本文件集返回。
- **FR-002**: pull 返回的文件集 MUST 由子特性 B 的序列化契约产生(C 不另立文件格式);目录树 MUST 映射类目层级。
- **FR-003**: pull 文件中任务对数据源的绑定 MUST 以**逻辑名**表示,MUST NOT 包含任何数据源连接凭据(主机/端口/用户名/密码/连接串)。

#### Push(摄入 + 治理)

- **FR-004**: 系统 MUST 提供按 project 粒度的 push 端点,接收文件集,经子特性 B 反序列化为定义。
- **FR-005**: push MUST 对定义做有效性校验,对无效/不完整定义(缺必填元数据、脚本缺失、未知数据源逻辑名、任务流悬挂引用等)以**可定位错误**(文件 + 字段/实体 + 原因)拒绝,MUST NOT 静默接受。
- **FR-006**: push MUST 是**事务性**的全有或全无:校验或落库任一步失败,服务器侧该项目定义 MUST 保持 push 前状态,MUST NOT 部分落库。
- **FR-007**: push MUST 把任务绑定的数据源**逻辑名**(= 数据源的项目内唯一名 `Datasource.name`)按 `(租户, 项目, 名)` 解析为对应 `datasourceId`;逻辑名在项目内不存在时 MUST 以可定位错误拒绝,且 MUST NOT 创建新数据源、MUST NOT 在文件中使用数字 id。
- **FR-008**: push MUST 以**幂等覆盖**语义重建该项目的类目/任务/任务流/标签定义(以本地文件集为准:增、改、删),MUST NOT 实现双向同步或冲突合并。
- **FR-008a**(乐观并发): pull MUST 返回该项目的**基线版本令牌**;push MUST 携带该令牌并校验,服务器侧当前版本不等于基线(他人已改)时 MUST **默认拒绝**并提示先 pull/看 diff;调用方 MUST 能以显式 `force` 标志绕过该校验执行覆盖。
- **FR-008b**(删除引用守卫): push 对本地缺失的定义 MUST 执行删除,但若被删任务/任务流仍被**在线(ONLINE)任务流**引用,push MUST 以可定位错误拒绝(指明引用方)整单回滚,要求先解除引用;被删定义的历史版本快照 MUST 按现有治理保留(不物理清除)。
- **FR-009**: push 成功后 MUST 为受影响的任务与任务流**生成新版本快照**,复用现有版本快照治理机制(任务版本、任务流 DAG 快照)。push MUST NOT 自动把定义晋级为 ONLINE(可调度)状态:定义落库后停在草稿/新版本态,上线为与 push 解耦的显式治理动作(不在 C 范围)。故 push MUST NOT 受"被引用任务必须已发布(ONLINE)"的发布前置约束阻塞。

#### Diff(差异预览)

- **FR-010**: 系统 MUST 提供 push 前的差异预览:给定本地文件集,返回相对服务器当前定义的**新增 / 修改 / 删除**清单,按实体粒度(任务/任务流/类目/标签)分类。
- **FR-011**: diff 预览 MUST 是只读操作,MUST NOT 改变服务器侧任何定义。

#### 隔离与治理(贯穿)

- **FR-012**: 所有 pull/push/diff 操作 MUST 受**租户/项目隔离**约束;越权访问 MUST 被拒绝(复用平台既有租户上下文与鉴权)。
- **FR-013**: 服务器 MUST 保持为治理真相源:版本快照、租户/项目隔离均由服务器侧强制,本地为工作副本。
- **FR-014**(No Legacy Migration,承总纲 FR-019): C MUST NOT 提供任何存量迁移/导出旧数据/新旧并存路径;pull/push 仅服务于转型后新建或重建的定义。
- **FR-015**: round-trip 语义等价 MUST 由 B 的序列化往返契约保证;C 在装配(pull)与落库(push)环节 MUST NOT 引入字段静默丢失或顺序抖动,使得 push→pull 后的文件与 push 前语义等价。

#### 传输形态

- **FR-016**: pull/push MUST 以**单个 JSON 请求/响应**承载文件集,body 含 `files`(相对路径 → UTF-8 内容映射)与元字段(项目标识、基线版本令牌、`force`);MUST NOT 采用打包归档或逐文件多请求。
- **FR-017**: 传输 MUST 带**完整性校验**(如文件数/总字节核对),对截断或不完整的文件集 push MUST 以错误拒绝,不落库,防止内容静默丢失。

### Key Entities *(include if feature involves data)*

- **Project(项目)**:pull/push/diff 的粒度单元,归属某租户;承载其下类目、任务、任务流、标签。
- **Catalog / Task / Workflow / Tag(沿用总纲与 B)**:被同步的定义实体;在文件侧由 B 的契约表达,在服务器侧为治理真相源记录。
- **Datasource(数据源)**:任务绑定的逻辑引用;**定义不入文件契约**,C 只做项目内**逻辑名 ↔ 数据源**的解析。逻辑名 = `Datasource.name`(项目内唯一),解析键为 `(租户, 项目, 名)`。
- **Version Snapshot(版本快照)**:push 时为受影响任务/任务流生成的不可变发布记录,沿用现有任务版本与任务流 DAG 快照机制;作为运行期真相源。
- **Pull Bundle(导出文件集)**:一次 pull 产出的纯文本文件集合,是本地工作副本的初始内容;随附该项目的**基线版本令牌**,供 push 做乐观并发校验。
- **Push Result(提交结果)**:一次 push 的结果,包含校验结论、生成的版本快照引用、以及被新增/修改/删除的定义统计。
- **Diff Preview(差异预览)**:本地文件集相对服务器当前定义的增/改/删清单,只读。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对任意含任务/任务流的项目,`pull → 本地改 → push → 再 pull 到干净目录`,得到的文件与原文件**语义等价**(round-trip 一致率 100%,无字段静默丢失)。
- **SC-002**: 越权(跨租户 / 跨无权项目)的 pull/push/diff **100% 被拒绝**。
- **SC-003**: push 无效/不完整定义时,**100%** 被可定位错误(文件 + 字段/实体 + 原因)拒绝,且服务器侧定义**0 次**部分落库(全有或全无)。
- **SC-004**: 每次成功 push,为受影响的任务/任务流**各生成 1 个新版本快照**;抽样核对快照内容与提交定义一致率 100%。
- **SC-005**: pull 返回的文件中**0 处**出现数据源连接凭据(主机/端口/用户名/密码/连接串)。
- **SC-006**: diff 预览返回的增/改/删清单与实际差异一致率 100%,且 diff 执行后服务器侧定义**0 变化**(只读)。
- **SC-007**: 基于陈旧基线(pull 后他人已改)的非 `force` push **100% 被拒绝**,不发生覆盖;携带 `force` 或基线最新的 push 正常执行。

## Assumptions

- **复用 B**:文件序列化/反序列化与 round-trip 字节稳定由已落地的子特性 B(文件化定义契约)提供;C 直接复用其转换入口与"定义 ↔ 文件"两端,不重写文件格式。
- **复用版本快照内核**:任务版本与任务流 DAG 快照机制已存在且稳定(现有"发布"路径即生成快照),push 复用之,不新造快照机制。
- **复用隔离与鉴权**:租户/项目隔离与认证沿用平台既有机制(现有租户上下文 + JWT),C 不新设独立账户体系或新的隔离模型。
- **数据源以逻辑名标识**:数据源在项目内由唯一逻辑名标识;C 做逻辑名 ↔ 数据源解析,数据源的创建/连接凭据管理不在 C 范围(沿用现有数据源管理)。
- **传输形态**:pull/push 通过服务器 HTTP 端点以**单 JSON** 交换文件集(`files{ 相对路径 → 内容 }` + 元字段),带完整性校验(见 FR-016/017)。
- **No Legacy Migration(硬规范)**:存量 DB 数据按总纲直接删除,C 不提供任何迁移/并存路径(承 constitution)。
- **CLI / 本地 runtime / MCP 不在 C**:开发者侧的 clone/pull/push 命令与本地真跑属 D,MCP 工具属 E;C 只交付服务器端点,使 D/E 可在其上构建。
- **Web 编辑器**:编辑能力转本地后,服务器侧定义的写入主要经 push;Web 只读查看的存废沿用总纲(不在 C 拍死)。

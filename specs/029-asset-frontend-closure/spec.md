# Feature Specification: 资产目录 / 指标市场前端收口

**Feature Branch**: `029-asset-frontend-closure`

**Created**: 2026-06-30

**Status**: Draft

**Input**: User description: "资产目录/指标市场前端收口：把 023 已落地的只读资产目录与指标市场补成写侧闭环——资产创建/编辑/下线/对账、指标上架/下架、订阅退订、分面真过滤/分页/复用防环提示。纯前端，复用既有后端 17 端点，零后端与 schema 改动。"

## 背景与问题

023（资产目录 + 指标市场）已交付**只读**消费面：用户可以搜索资产/指标、看详情、看血缘与质量徽章、认证指标、复用指标、订阅资产。但写侧能力虽在服务端完整存在，前端却基本没有入口——用户**进不去也下不来**：

- 资产无法在界面里编目、修改元数据、下线、或对账（schema 漂移恢复）。
- 指标无法在界面里上架、下架。
- 订阅只能新增、不能退订；用户看不到自己的订阅清单。
- 搜索分面只显示计数不可点（owner/tag/status 无法过滤），结果写死单页 20 条、超出不可见；复用成环时只显示通用「操作失败」，用户不知道为什么。

本特性把这些缺口补齐，使资产目录与指标市场成为**真正闭环**的治理界面。范围限定为前端；所有服务端能力（含写操作的审批闸门与审计）已存在并复用。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 资产全生命周期管理 (Priority: P1)

数据管理员在资产目录里完成资产从编目到退役的全过程：新建资产并填写元数据、修改既有资产的元数据、把不再使用的资产下线、对疑似 schema 漂移的资产发起对账以恢复或标记其状态。

**Why this priority**: 这是把「只读目录」变成「可治理目录」的核心。没有它，资产目录无法承担其治理用途，023 的写侧服务端能力等于闲置。

**Independent Test**: 仅交付本故事即可独立验证——在界面新建一个资产、编辑它、下线它、对一个无血缘锚点的资产对账，全程不依赖指标市场或订阅功能即构成可用 MVP。

**Acceptance Scenarios**:

1. **Given** 用户在资产目录，**When** 点击「编目资产」、选择数据源、填写限定名（必填）与元数据后提交，**Then** 系统受理并在列表中出现新资产（若操作命中审批闸门则提示「待审批」而非立即出现）。
2. **Given** 用户提交的限定名在同数据源下已存在，**When** 提交，**Then** 系统拒绝并给出「资产已存在」的明确提示，不产生重复资产。
3. **Given** 用户打开某资产详情，**When** 点击「编辑」、仅修改描述与负责人后提交，**Then** 仅被修改的字段被更新，未触及字段保持原值。
4. **Given** 用户打开某 ACTIVE 资产详情，**When** 点击「下线」并确认，**Then** 资产状态变为 RETIRED 并在列表中以退役标识呈现。
5. **Given** 某资产缺少底层血缘锚点，**When** 用户对其发起「对账」，**Then** 系统将其标记为 STALE 并在界面反馈；锚点恢复后再次对账则回到 ACTIVE。

---

### User Story 2 - 指标上架 / 下架 (Priority: P1)

指标负责人把已定义的指标上架到市场供他人发现与复用，并在指标退役时将其下架。

**Why this priority**: 指标市场的供给侧入口。缺失则市场只能消费既有种子数据，无法由用户自助供给，023 US3 不闭环。

**Independent Test**: 仅交付本故事即可独立验证——从既有指标定义中选一个上架，确认它出现在市场列表；再将其下架，确认其退出在售状态。

**Acceptance Scenarios**:

1. **Given** 用户在指标市场，**When** 点击「上架指标」、从既有指标定义中选择一个并填写元数据后提交，**Then** 该指标进入市场列表（命中审批则提示「待审批」）。
2. **Given** 用户对同一指标重复上架，**When** 提交，**Then** 系统幂等处理（更新并复位为在售），不产生重复条目。
3. **Given** 用户打开某在售指标详情，**When** 点击「下架」并确认，**Then** 指标状态变为 DELISTED 并退出在售列表。
4. **Given** 用户发起复用且复用关系会在指标依赖图中成环，**When** 提交，**Then** 系统拒绝并给出「会形成循环依赖」的明确提示，而非通用失败信息。

---

### User Story 3 - 订阅生命周期 (Priority: P2)

用户查看自己对资产的订阅清单，并退订不再需要的变更通知。

**Why this priority**: 订阅当前能订不能退、且不可见，属于体验缺口。重要但不阻断资产/指标主流程，故 P2。

**Independent Test**: 订阅一个资产，在订阅清单中看到它，退订后它从清单消失，独立可测。

**Acceptance Scenarios**:

1. **Given** 用户已订阅若干资产，**When** 打开订阅清单，**Then** 列出当前用户的全部有效订阅及其目标与变更过滤项。
2. **Given** 用户在订阅清单或资产详情中，**When** 对某订阅点击「退订」并确认，**Then** 该订阅被移除，用户不再收到其变更通知。
3. **Given** 退订的目标不属于当前用户，**When** 尝试退订，**Then** 系统拒绝（沿用服务端属主校验），界面给出失败提示。

---

### User Story 4 - 检索体验补全 (Priority: P2)

用户用全部可用分面（敏感度、状态、负责人、标签、认证）精确过滤资产/指标，按质量分数收窄结果，并翻页浏览超过单页容量的结果集。

**Why this priority**: 服务端已支持的检索能力前端只暴露了一部分。提升可用性但不阻断写侧闭环，故 P2。

**Independent Test**: 对一个超过 20 条结果的资产集，点击 owner 分面过滤、翻到第 2 页、设置质量分数下限，结果集相应收窄，独立可测。

**Acceptance Scenarios**:

1. **Given** 资产搜索结果含多个负责人/标签/状态分面，**When** 用户点击某个分面值，**Then** 结果按该分面收窄且分面高亮为已选；再次点击取消过滤。
2. **Given** 结果总数超过单页容量，**When** 用户翻页，**Then** 展示对应页结果，页码与总数一致。
3. **Given** 指标市场有已认证与未认证条目，**When** 用户选择 CERTIFIED 分面，**Then** 仅展示已认证指标。
4. **Given** 质量分数过滤可用，**When** 用户设置分数下限，**Then** 结果按分数收窄；当当前环境无质量评分数据时，界面明示「暂无质量数据」而非静默返回空集。

---

### Edge Cases

- **闸门待审批**：任何写操作（创建/编辑/下线/对账/上架/下架/订阅/退订/复用）都可能因风险等级被闸门挂起为「待审批」而非立即生效；界面须如实区分「已执行 / 待审批 / 被拒/失败」三态，不可把待审批伪装成成功。
- **PII 资产可见性**：非 owner/steward 用户对 PII 资产无详情权限；界面须沿用服务端可见性结果，不旁路展示。
- **乐观失败 / 并发**：编辑一个已被他人改动或已下线的资产时，须以服务端返回为准给出可理解的失败提示，不静默丢弃用户输入。
- **空数据源 / 空指标定义池**：上架指标时若无可选指标定义、或编目时无可选数据源，须给出空态指引而非崩溃。
- **复用成环**：见 US2 场景 4，须可识别并专门提示。
- **结果截断**：当结果触达服务端硬上限被截断时，界面须明示「结果已截断」（沿用既有 truncated 标识）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 用户 MUST 能在资产目录界面新建资产，至少提供数据源、限定名（必填）、名称、描述、负责人、管家、敏感度、标签等元数据；限定名在同数据源下重复时 MUST 被拒并给出明确提示。
- **FR-002**: 用户 MUST 能编辑既有资产的元数据，且 MUST 采用部分更新语义——仅提交被修改的字段，未触及字段保持原值。
- **FR-003**: 用户 MUST 能将资产下线（置 RETIRED），操作前 MUST 二次确认。
- **FR-004**: 用户 MUST 能对资产发起 schema 对账，并在界面看到对账后的状态（ACTIVE/STALE）变化与原因提示。
- **FR-005**: 用户 MUST 能从既有指标定义中选择并上架指标到市场，重复上架 MUST 幂等（不产生重复条目）。
- **FR-006**: 用户 MUST 能将在售指标下架（置 DELISTED），操作前 MUST 二次确认。
- **FR-007**: 指标复用因成环被服务端拒绝时，界面 MUST 给出「会形成循环依赖」的专门提示，区别于通用失败。复用 MUST 支持选择消费方类型（METRIC/TASK/ASSET）。
- **FR-008**: 用户 MUST 能查看自己的资产订阅清单，并 MUST 能退订；退订前 MUST 二次确认。
- **FR-009**: 资产搜索 MUST 支持对敏感度、状态、负责人、标签全部分面的真过滤（可点选、可取消、已选高亮）；指标市场搜索 MUST 支持按认证状态分面过滤。
- **FR-010**: 资产与指标列表 MUST 支持分页浏览，替代写死的单页；MUST 显示当前页与结果总数，并在结果被服务端截断时明示。
- **FR-011**: 资产搜索 MUST 提供质量分数下限过滤入口；当当前环境无质量评分数据时 MUST 明示「暂无质量数据」，不静默返回空集。
- **FR-012**: 所有写操作的结果 MUST 如实区分「已执行 / 待审批 / 被拒或失败」三态并据此给出反馈与列表刷新行为；MUST NOT 把「待审批」呈现为「已执行」。
- **FR-013**: 本特性 MUST NOT 改动任何服务端行为、数据库 schema 或既有读路径契约；仅在前端补充写侧入口与检索交互，复用既有服务端能力与审批闸门/审计。
- **FR-014**: 新增的界面文案 MUST 同时提供 zh-CN 与 en-US 两套且 key 集一致；数据术语（如 sensitivity / certification / STALE / DELISTED）保持英文。

### Key Entities *(include if feature involves data)*

> 均为既有实体，本特性不新增数据结构，仅在前端读写。

- **资产 (Data Asset)**：被编目的数据表/数据集。关键属性：数据源、限定名（去重键之一）、名称、描述、负责人、管家、敏感度（PUBLIC/INTERNAL/CONFIDENTIAL/PII）、状态（ACTIVE/STALE/RETIRED）、标签、血缘锚点。
- **指标上架 (Metric Listing)**：指标定义在市场中的上架记录。关键属性：指标类型、关联指标定义、负责人、认证状态（NONE/CERTIFIED）、在售状态（LISTED/DELISTED）、新鲜度、复用计数。
- **订阅 (Subscription)**：用户对某资产的变更订阅。关键属性：目标类型/目标、变更过滤项、属主。
- **复用引用 (Metric Reuse Ref)**：指标被某消费方（METRIC/TASK/ASSET）复用的依赖边；指标间复用参与成环校验。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户可在界面内完成「编目一个新资产 → 编辑 → 下线 → 对账」的完整生命周期，无需任何界面外操作（CLI/直连接口）。
- **SC-002**: 用户可在界面内完成「上架一个指标 → 下架」的完整流程，无需界面外操作。
- **SC-003**: 用户可在界面内查看并退订自己的订阅，订阅清单与实际状态一致。
- **SC-004**: 资产/指标搜索的全部服务端可用分面在界面均可点选过滤，且超过单页容量的结果可经分页完整浏览。
- **SC-005**: 每一类写操作命中审批闸门时，界面均明确呈现「待审批」状态，且不出现「提示成功但实际未生效」的误导（三态如实）。
- **SC-006**: 复用成环、限定名重复等已知业务拒绝场景，界面均给出可理解的具体原因，而非通用「操作失败」。
- **SC-007**: 交付不引入任何服务端、数据库 schema 或既有读契约的改动（纯前端增量）。

## Assumptions

- 服务端 023 的全部资产/指标/订阅读写端点已存在且行为稳定，本特性仅消费之；写操作的风险闸门与审计由服务端既有机制承担。
- 既有「数据源列表」与「指标定义列表」能力可作为编目/上架时的选择来源。
- 质量评分（022 评分卡）在当前环境可能尚无数据；质量过滤入口按「通路就绪、待数据」交付，不依赖 022 落地。
- 沿用既有鉴权与租户/项目隔离（写经统一 Bearer 凭据，租户由服务端身份解析）。
- 移动端适配不在本特性范围；目标为桌面工作台内的资产目录 / 指标市场两个视图。
- 026/027（告警/事件中心）与本特性互不相干，可独立开发与合并。

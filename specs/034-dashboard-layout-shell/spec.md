# Feature Specification: Dashboard-01 外壳布局迁移（保留多 Tabs）

**Feature Branch**: `034-dashboard-layout-shell`

**Created**: 2026-07-01

**Status**: Draft

**Input**: User description: "一个项目，当前项目的里面内容，千奇百怪，不就方式，你看下：npx shadcn@latest add dashboard-01命令安装的页面，具体可以看：https://ui.shadcn.com/blocks A dashboard with sidebar, charts and data table模块，把这里的布局方式放到当前项目，并且，我的多Tabs还是要的，你来处理"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 顶部面包屑呈现当前位置 (Priority: P1)

用户在工作区内打开某个视图（如任务列表、指标概览、目录树等）时，内容区顶部会出现一条面包屑/标题行，清晰标出"项目 > 功能分组 > 当前视图"的层级路径，让用户即使在侧边导航收起（icon rail）时也能确认自己当前所在的位置。

**Why this priority**: 这是 dashboard-01 布局区别于当前项目"纯侧边导航、无顶部上下文"现状的核心可见变化，也是后续卡片/表格栅格对齐的容器基础，必须先落地。

**Independent Test**: 打开任意一个视图 Tab，无需查看侧边导航即可通过顶部面包屑判断当前处于哪个功能分组、哪个视图；折叠侧边导航后面包屑依旧可读。

**Acceptance Scenarios**:

1. **Given** 用户打开"任务列表"视图 Tab，**When** 内容区渲染完成，**Then** 顶部出现面包屑「任务调度 > 任务列表」，与左侧导航当前高亮分组一致。
2. **Given** 用户将左侧导航折叠为 icon rail，**When** 查看当前激活 Tab，**Then** 顶部面包屑内容不受侧边栏折叠状态影响，仍完整显示层级路径。
3. **Given** 视图不属于任何导航分组（如设置页），**When** 该视图被打开，**Then** 面包屑按兜底规则仅显示视图自身名称，不报错、不留空。

---

### User Story 2 - 多 Tabs 之间面包屑与布局互不串台 (Priority: P1)

用户同时打开多个不同类型的 Tab（例如"任务列表""指标概览""目录树"），在 Tab 之间来回切换时，每个 Tab 的顶部面包屑、卡片区块、数据表格布局都应准确对应该 Tab 自身的视图，不会出现切换后面包屑或布局残留上一个 Tab 内容的情况。

**Why this priority**: 多 Tabs 是本项目既有的核心能力（用户明确要求保留），新增的面包屑/栅格布局若与多 Tabs 状态管理冲突会直接损害现有体验，优先级与 US1 同级。

**Independent Test**: 依次打开 5 个不同类型的视图 Tab，随机顺序来回切换，每次切换后核对面包屑文案、卡片区块、表格内容均与当前激活 Tab 一致；关闭/新开/固定 Tab 行为与改造前一致。

**Acceptance Scenarios**:

1. **Given** 用户已打开 3 个不同视图的 Tab，**When** 用户点击切换到任一 Tab，**Then** 面包屑、卡片区块、表格数据在同一渲染帧内更新为该 Tab 对应内容，不出现旧内容闪烁或残留。
2. **Given** 某个视图 Tab 处于 keep-alive（后台保活）状态，**When** 用户切回该 Tab，**Then** 其面包屑与布局状态与离开前一致（如筛选条件、滚动位置不因布局改造而丢失）。
3. **Given** 用户执行新开 Tab / 关闭 Tab / 固定 Tab 操作，**When** 操作完成，**Then** Tab 栏行为（顺序、图钉状态、激活切换）与改造前完全一致。

---

### User Story 3 - 卡片与数据表格按统一栅格重新排布 (Priority: P2)

现有视图中的统计卡片（如 metrics 概览卡片）与数据表格类视图，按 dashboard-01 的栅格间距、卡片圆角/阴影、表格头部工具栏版式重新排布，让不同视图在视觉上呈现一致的"仪表盘"观感，但卡片承载的原始数据、文案、表格列内容保持不变（不新增图表、不新增业务数据）。

**Why this priority**: 视觉一致性提升产品完成度，但不影响功能可用性，可在 US1/US2 落地后独立实施和验证。

**Independent Test**: 对比改造前后同一视图的卡片区块与表格区块，确认栅格列数、卡片间距、圆角/阴影、表格工具栏位置符合 dashboard-01 版式规范，且卡片/表格展示的数据内容与改造前一致。

**Acceptance Scenarios**:

1. **Given** 某视图包含多张统计卡片，**When** 页面按新栅格渲染，**Then** 卡片在桌面宽度下以统一列数网格排布，卡片间距、圆角、阴影符合 `frontend/DESIGN.md` 既有 token。
2. **Given** 某视图包含数据表格，**When** 页面渲染完成，**Then** 表格保留现有筛选工具栏、分页、列可见性等能力，且视觉版式（表头样式、行密度、边框）与 dashboard-01 表格区块对齐。
3. **Given** 某视图当前没有可视化图表，**When** 该视图完成本次改造，**Then** 页面中不出现新增的图表（如面积图/折线图），仅卡片与表格布局发生变化。

---

### Edge Cases

- 视图的层级路径依赖的分组/视图元数据缺失或未注册时，面包屑如何降级显示（不能抛错阻塞页面渲染）？
- 面包屑文案在窄视口/侧边栏展开挤压内容区宽度时如何截断，避免换行破坏顶部布局高度？
- 用户在多个 Tab 打开同一视图但携带不同参数（如不同任务 ID）时，面包屑需体现参数差异（如任务名称），避免多个同类 Tab 面包屑完全相同导致无法区分。
- 某视图当前没有统计卡片或数据表格（纯表单/画布类视图），本次栅格化改造不应强行插入空卡片区块。
- 已有的 033（表格内边距间距）分支若先于本功能合入主干，需在实现阶段以合入后的最新表格样式为基线核对，避免栅格改造覆盖掉已修复的间距问题。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在内容区顶部为当前激活的 Tab 显示面包屑，呈现「项目 > 功能分组 > 当前视图」层级路径。
- **FR-002**: 面包屑内容必须随 Tab 切换实时更新，且各 Tab 独立维护自己的面包屑，不因切换或后台保活而互相覆盖或残留。
- **FR-003**: 视图缺少层级路径元数据时，面包屑必须提供兜底显示（至少展示视图自身名称），不得报错或留空白。
- **FR-004**: 现有左侧导航（icon rail 折叠/展开、7 功能分组、项目切换器）的结构与交互必须保持不变。
- **FR-005**: 现有多 Tabs 工作区能力（新开、关闭、固定/取消固定、切换、keep-alive 保活、Tab 去重）必须保持改造前的行为不变。
- **FR-006**: 统计类卡片区块必须按统一栅格（响应式列数、间距、圆角、阴影）重新排布，卡片原始数据与文案内容不得改变。
- **FR-007**: 数据表格类视图必须保留现有 DataTable 能力（筛选工具栏、分页、列可见性、批量操作），并在视觉版式上与 dashboard-01 表格区块对齐（表头样式、行密度）。
- **FR-008**: 本次改造不得引入新的图表可视化（如面积图、折线图）及配套图表库依赖，仅涉及布局、间距、卡片与表格版式调整。
- **FR-009**: 所有新增/调整的视觉样式必须使用 `frontend/DESIGN.md` 中已登记的设计 token（颜色、圆角、间距、阴影），不得引入未登记的新样式体系。
- **FR-010**: 无统计卡片或数据表格的视图（如纯表单、画布类）不因本次栅格化改造被强行插入空卡片区块或破坏原有布局。

### Key Entities

- **面包屑路径（Breadcrumb Path）**：单个 Tab 在某一时刻对应的层级节点序列（如项目名、功能分组名、视图名，及可选的动态参数如任务名/实例 ID），随 Tab 独立维护，随视图/参数变化重新计算。
- **视图布局区块（View Layout Section）**：视图内容区中的卡片网格区块与数据表格区块，是本次栅格/版式统一改造的作用对象，不改变其承载的原始业务数据。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户打开任意视图 Tab 后，无需查看侧边导航即可在 3 秒内通过顶部面包屑说出自己当前所在的功能分组与视图名称。
- **SC-002**: 抽查项目内全部视图页面，至少 90% 的视图卡片/表格区块间距、圆角、阴影与 `frontend/DESIGN.md` 规范一致，人工走查零明显不一致项。
- **SC-003**: 同时打开 5 个及以上不同视图 Tab 并随机切换 50 次，面包屑与布局内容 100% 准确对应当前激活 Tab，零串台、零残留。
- **SC-004**: 本次改造上线后，前端产物中不出现新增的图表可视化库依赖，相关 bundle 体积增量为 0。
- **SC-005**: 改造前已支持的多 Tabs 操作（新开/关闭/固定/切换/keep-alive）在改造后的自动化或人工回归中 100% 通过，零行为回退。

## Assumptions

- 面包屑所需的"项目 > 功能分组 > 视图"层级数据来自现有 `lib/workspace/` 视图注册与导航分组元数据，无需新增后端接口或数据模型。
- 本次改造复用现有卡片、表格所展示的业务数据（如 metrics 指标、任务实例列表等），不新增业务数据域、不新增查询接口。
- dashboard-01 demo 中的面积图（area chart）部分明确排除在本次范围外，不引入图表库依赖。
- 现有 `032-project-nav` 落地的左侧导航（icon rail + 7 分组 + 项目切换）结构保持不动，本次仅新增顶部面包屑行与内容区卡片/表格版式调整。
- 若 `033-ui-table-frame-spacing`（表格内边距间距修复）先于本功能合入主干，实现阶段以合入后的最新表格样式为基线，避免相互覆盖。

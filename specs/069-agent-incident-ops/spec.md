# Feature Specification: 任务失败智能运维——Agent 自动诊断处置闭环 + 监督席聊天室

**Feature Branch**: `069-agent-incident-ops`

**Created**: 2026-07-13

**Status**: Draft

**Input**: User description: "项目中，任务失败，智能化的运维，复用血缘Agent配置，比如，实例运行失败（周期，手动，实时），Agent自动去读取失败日志，自动评估如何修复，自动的给予日志查找最优雅的解决方案，比如是代码问题，通过dw cli 去改代码，重新发布，如果是数据量问题，并且只有提升内存和CPU可以解决那么修改任务的CPU和内存，如果是源头就有问题，比如，源头数据源存在脏数据的情况，最终的目的是平台要介入的点是Agent没办法解决的比如，数据源用户名密码错误了，这类型的是AI没办法解决的，要有Agent聊天室风格的，面板，可以查看Agent处理列表，Agent，可以自动读取日志列表，处理那些，还有哪些，无法处理，给出建议等等这样 你来思考"

## 背景与动机 *(为什么做这件事)*

066 已把人工告警/事件/质量/工单体系整体清场，故障可见性与处置能力出现刻意留白——留给本特性填补。北极星方向文档（`docs/superpowers/specs/2026-07-03-weft-ai-ops-direction-design.md` §4/§5/§10）已确认产品方向：Weft 的 Trust 层 = 常驻的智能运维处置能力 + 人的监督席，本特性即该方向的第一次落地立项（方向文档 §10 的第 2、3 步合并）。

核心命题：**任务实例失败后，不再等人来看，而是由运维 Agent 自动完成"读日志 → 分型诊断 → 选择最优处置 → 执行 → 验证收口"的闭环**；只有 Agent 判定自己无法解决的故障（如数据源凭据错误、源头脏数据需业务方处理）才升级到人，并附上根因证据与操作建议。人打开平台的第一屏是指挥中心式监督席：Agent 的战况播报横幅 + 全部事故处理步骤混排的实时直播流，每次故障可下钻为一条会话线程（Agent 每一步以消息流全量直播，人可自然语言对话追问与下指令），待审批的修复提案与需人工介入的升级项置顶可达。

Agent 的模型服务不重复建设：**复用 053 已落地的 AI Agent 配置**（协议/端点/模型/凭据的可配置通道），一份配置同时服务血缘抽取与智能运维。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 失败自动感知与分型诊断（Priority: P1）

某周期调度的任务实例在夜间运行失败。无需任何人在场，平台自动为这次失败开立一个「事故（Incident）」：运维 Agent 自动拉取该实例的失败日志、任务定义与脚本内容、近期运行历史，产出一份分型诊断——故障属于哪一类（瞬态抖动 / 资源不足 / 代码缺陷 / 上游数据问题 / 配置凭据问题 / 未知），根因证据是什么（日志中的关键行引用），建议的处置方案是什么。手动触发与实时（流式）任务的失败同样触发这条链路。开发者早上打开平台，每个失败都已附带诊断结论，而不是一堆裸日志。

**Why this priority**: 诊断是整个闭环的地基——没有分型诊断就没有后续的自动处置与升级建议。且诊断本身独立成立：即使不做任何自动修复，"每个失败自动附诊断"已经把排障的第一小时省掉，是可独立交付的最小价值。

**Independent Test**: 制造一个必然失败的任务（如 SQL 语法错误）触发周期/手动/实时三种失败，验证每次失败均自动生成事故记录，且短时间内出现包含故障分型、日志证据引用、处置建议的诊断结论。

**Acceptance Scenarios**:

1. **Given** 一个周期调度任务实例运行失败，**When** 失败终态落定，**Then** 平台自动创建一个事故，且无需人工触发，Agent 在数分钟内产出诊断结论（分型 + 证据引用 + 建议）挂在该事故上。
2. **Given** 手动触发运行与实时（长驻流式）任务分别失败，**When** 失败发生，**Then** 两者同样自动开立事故并获得诊断，覆盖三种触发来源。
3. **Given** 同一任务在短时间内连续失败多次（如重试后再失败），**When** 后续失败发生，**Then** 归并进同一个未收口的事故，不产生一次失败一张单的风暴。
4. **Given** AI Agent 模型配置未启用或端点不可用，**When** 任务失败，**Then** 事故仍被创建并完整保留失败上下文（日志摘要、实例信息），状态标记为"诊断不可用"，配置恢复后可补诊断；主链路（调度、日志）不受任何影响。
5. **Given** 一次失败的日志体量极大，**When** Agent 采集证据，**Then** 采用摘要/截断策略仍能产出诊断，不因日志过大而失败或无限消耗。

### User Story 2 - 可自愈故障的自动处置与验证收口（Priority: P2）

诊断完成后，Agent 按"行动梯度"自动处置：瞬态类（节点抖动、偶发超时）→ 直接自动重跑；资源不足类（数据量增长导致内存溢出，且判断提升资源可解）→ 自动调高该任务的 CPU/内存配置后重跑；实时任务 → 优先从检查点续跑。每次处置后 Agent 跟踪重跑结果：成功则事故自动收口标记"已解决"，再失败则升级到下一梯度或转人工。代码缺陷类走更谨慎的路径：Agent 生成修复变更与证据包，作为"修复提案"提交审批，人一键批准后变更发布生效并重跑验证；修复后的代码保证能从 repo 侧取回，不产生服务端漂移。所有写动作一律经过既有策略闸门并留审计痕迹。

**Why this priority**: 这是用户诉求的主干——"自动评估如何修复、给出最优雅的解决方案并执行"。依赖 US1 的诊断产物，是闭环的"行动"半边；单独交付即可让瞬态与资源类故障（生产中占比最高的两类）实现零人工恢复。

**Independent Test**: 分别制造三类故障——瞬态失败（可通过重跑成功）、内存不足失败（提资源可解）、代码缺陷失败——验证前两类无人工介入自动恢复且事故自动收口，第三类产出带证据的修复提案待批，批准后发布并重跑成功。

**Acceptance Scenarios**:

1. **Given** 诊断分型为瞬态故障，**When** Agent 处置，**Then** 自动重跑该实例，重跑成功后事故自动收口为"已解决"，全过程零人工介入，处置动作与结果在事故线程中可见。
2. **Given** 诊断分型为资源不足且判断提升 CPU/内存可解，**When** Agent 处置，**Then** 任务的资源配置被自动调高（有上限护栏）并重跑，成功后事故收口，资源变更记录在案。
3. **Given** 诊断分型为代码缺陷，**When** Agent 处置，**Then** 生成具体的修复变更与证据包（诊断依据 + 变更内容 + 预期效果）形成修复提案，进入待审批状态而非直接生效；人批准后变更发布并自动重跑验证。
4. **Given** 一个实时（长驻流式）任务失败且存在可用检查点，**When** Agent 处置，**Then** 优先采用从检查点续跑的恢复方式，而非从头重跑。
5. **Given** 某事故的自动处置已连续失败达到上限次数，**When** 再次失败，**Then** Agent 停止自动尝试，事故升级为"需人工介入"并附全部尝试记录——不允许无限重试循环。
6. **Given** Agent 发起的任何写动作（重跑、调资源、发布修复），**When** 动作执行，**Then** 均经过既有策略闸门裁决并留下完整审计记录，不因来源是 Agent 而豁免。
7. **Given** 修复提案被批准且发布生效，**When** 开发者在本地 repo 执行拉取，**Then** 能取回与服务端一致的修复后定义——服务端不存在 repo 追不回的变更。
8. **Given** 人已经在 Agent 处置期间手工重跑/终止了该实例，**When** Agent 准备执行下一步动作，**Then** Agent 感知到人工操作介入，让位并在线程中记录，不与人工操作打架。

### User Story 3 - 无法自愈故障的升级与建议（Priority: P2）

某任务因数据源用户名密码错误而失败——这类故障 Agent 无法自行解决（凭据只有人知道）；又如上游源头存在脏数据，需要业务方在源头修数。Agent 诊断后不做徒劳的重试，直接把事故转为"需人工介入"，附上明确的根因定位（哪一个数据源、什么错误、日志证据）与可执行的操作建议（如"请更新数据源 X 的凭据后点击复验"）。人完成处理后，可以让 Agent 复验（重跑验证），确认恢复后事故收口。

**Why this priority**: 用户明确的产品边界——"平台要人介入的点只剩 Agent 没办法解决的"。升级路径的质量（100% 附建议、不空转重试）决定了人对整个系统的信任度；与 US2 并列为闭环的另一半出口。

**Independent Test**: 配置一个凭据错误的数据源使任务失败，验证 Agent 不进行自动重跑，事故直接进入"需人工介入"队列并附根因与建议；修正凭据后触发复验，任务成功且事故收口。

**Acceptance Scenarios**:

1. **Given** 任务因数据源凭据错误失败，**When** Agent 诊断，**Then** 分型为"配置凭据问题"，不发起自动重跑，事故进入"需人工介入"队列，附根因定位与操作建议。
2. **Given** 诊断分型为上游源头脏数据，**When** Agent 处置，**Then** 不尝试修改上游数据（危险动作），事故升级人工并附受影响范围说明与建议；Agent 至多建议性提示下游影响。
3. **Given** 人按建议完成了源头处理，**When** 在事故线程中触发复验，**Then** Agent 重跑验证，成功后事故收口为"已解决（人工协同）"。
4. **Given** 诊断分型为"未知"（无法归入任何已知类别），**When** 处置决策，**Then** 保守处理：至多一次自动重跑试探，仍失败即升级人工，绝不盲目连续重试。

### User Story 4 - 监督席·指挥中心直播流（产品主页）（Priority: P3）

开发者打开 Weft，第一屏就是监督席指挥中心。顶部是 Agent 实时生成的**战况播报横幅**：一句话综述（"过去 12 小时 9 次失败，我自愈 7 个，2 个等你：1 个待批修复、1 个凭据问题"），内嵌的关键数字可点击过滤，展开即完整「接班报告」。主体是全屏的 **Agent 活动直播流**：所有事故的处理步骤混排成一条实时 live feed，Agent 的每步工作全量直播——「思考中」指示态、工具动作逐个点亮（读取日志 ✓ → 分析代码 ✓ → 生成修复中…）、诊断结论流式打字输出，进行中的事故带呼吸感动效，一眼即知"谁在被处理、处理到哪了"。点击任意条目下钻进该事故的**会话线程**：完整处置过程之外，人可以用自然语言与 Agent 自由对话——追问依据（"为什么判定是 OOM？"）、下指令（"再查一下上游那张表"），Agent 流式回应并可据此发起新动作（写动作照旧过闸门）；批准/驳回/标记已处理/触发复验等关键裁决保留结构化按钮。ops/日志/血缘等既有视图降级为从事故下钻直达的证据页（入口保留）。

**Why this priority**: 这是方向文档 §5 确立的产品主形态与本特性的人机界面总成，承载人对 Agent 的全部观察、对话、审批与协同。依赖 US1-US3 产生的事件与数据，故排 P3；但它决定整个特性的用户感知价值。

**Independent Test**: 触发覆盖不同结局的失败（自动恢复 / 待审批 / 需人工），验证：打开平台第一屏即指挥中心；战况横幅综述与数字正确；直播流实时流式呈现 Agent 步骤；下钻线程可自由对话且 Agent 有效回应；审批/复验在线程内可完成。

**Acceptance Scenarios**:

1. **Given** 打开平台进入工作区，**When** 首屏加载完成，**Then** 默认视图即监督席指挥中心（战况横幅 + 直播流），无需任何点击。
2. **Given** 一个正在处理中的事故，**When** 观察直播流，**Then** Agent 步骤实时陆续出现——「思考中」指示态、工具动作逐项点亮、诊断结论流式打字，进行中条目有可感知的动效区分，全程无需手动刷新。
3. **Given** 战况播报横幅，**When** 查看与点击，**Then** 综述与内嵌数字（处理中/待审批/需人工/已解决）与实际状态一致，点击数字过滤直播流，展开可见完整接班报告。
4. **Given** 下钻进一条事故线程，**When** 用自然语言追问诊断依据或下达排查指令，**Then** Agent 流式回应；由指令引发的新写动作照旧过闸门执行，并作为消息追加进同一线程。
5. **Given** 一条含待审批修复提案的事故线程，**When** 查看提案，**Then** 变更内容与证据包完整可读，可一键批准或驳回；批准后后续执行步骤继续追加进同一线程。
6. **Given** 一条需人工介入的事故，**When** 人在线程中标记"已处理，请复验"，**Then** Agent 执行复验并把结果追加进线程。
7. **Given** 失败风暴导致直播流洪峰，**When** 查看指挥中心，**Then** 可按状态/任务过滤聚焦，"等你处理"（待审批/需人工）的事故始终置顶可达，不被新消息刷走。

### Edge Cases

- **失败风暴**：大面积基础设施故障导致数十个任务同时失败——按任务归并事故、总量限流，Agent 并发处理有上限，不产生模型调用与重跑风暴；超出处理能力的事故排队并如实标记。
- **修复使情况变糟**：修复提案发布后重跑仍失败甚至引入新错误——验证失败即回退到发布前版本，事故升级人工并记录完整过程。
- **诊断与实际不符**：Agent 分型为瞬态但重跑三次仍失败——处置上限触发升级，误诊记录保留在线程中供人核查。
- **人机并发操作**：人工 kill/rerun 与 Agent 动作竞争同一实例——以实例当前状态为准，Agent 动作执行前校验前置状态，不满足即放弃并记录。
- **模型端点中途失效**：诊断进行到一半端点超时——事故停留在"诊断不可用"，保留已采集证据，端点恢复后可续。
- **审批长期挂起**：修复提案数日无人审批——事故保持待审批状态并在队列中持续可见，不自动生效、不静默过期。
- **项目隔离**：事故、日志证据、处置动作严格限定在任务所属项目内；跨项目不可见、不可操作。
- **实时任务无可用检查点**：流式任务失败且无检查点——退化为常规重跑路径，线程中说明原因。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在任务实例失败终态（FAILED，覆盖周期调度、手动触发、实时/长驻流式三种来源）落定后，以及实例进入 SUSPENDED（基础设施重派超限的非终态挂起，正在等待介入）时，自动创建事故记录，无需人工触发；TEST 与补数（BACKFILL）运行不开单；同一任务的连续失败 MUST 归并进同一未收口事故。
- **FR-002**: 运维 Agent MUST 自动采集诊断证据：失败实例日志（含超大日志的摘要/截断策略）、任务定义与脚本内容、近期运行历史；实时任务额外包含外部作业状态。
- **FR-003**: 运维 Agent MUST 产出结构化诊断结论：故障分型（至少区分 瞬态 / 资源不足 / 代码缺陷 / 上游数据问题 / 配置凭据问题 / 未知）+ 日志证据引用 + 处置建议；诊断结论挂在事故上可追溯。
- **FR-004**: 系统 MUST 按行动梯度执行处置：低风险动作（重跑、调整任务 CPU/内存后重跑、检查点续跑）自动执行；修改任务代码类动作 MUST 生成"修复提案"（变更内容 + 证据包）默认经人审批后方可发布生效。
- **FR-005**: 资源调整 MUST 有上限护栏（单任务资源上限、单次调幅上限），超出护栏的诉求转人工。
- **FR-006**: 每次处置动作后系统 MUST 跟踪验证结果：重跑成功 → 事故自动收口"已解决"；失败 → 升级下一梯度或转人工。同一事故的自动处置次数 MUST 有上限，超限强制升级人工，禁止无限循环。
- **FR-007**: 修复提案批准发布后，修复后的任务定义 MUST 可由开发者从 repo 侧完整取回（与服务端一致，无追不回的漂移）；修复验证失败 MUST 可回退到发布前版本。
- **FR-008**: 诊断为不可自愈类（配置凭据错误、上游源头数据问题等）时，Agent MUST 不做徒劳自动重试，事故直接进入"需人工介入"队列，且 100% 附根因定位与可执行操作建议；人工处理后 MUST 支持触发 Agent 复验收口。
- **FR-009**: Agent MUST NOT 直接修改上游/源头数据；上游数据问题只做定位、影响说明与建议。
- **FR-010**: Agent 发起的一切写动作（重跑、调资源、发布修复、复验）MUST 经过既有策略闸门裁决并留下完整审计记录，不因来源是 Agent 而豁免或绕过。
- **FR-011**: 系统 MUST 以「指挥中心直播流」形态提供监督席：全屏实时 live feed 混排全部事故的处理步骤，Agent 工作全量直播——「思考中」指示态、工具动作逐个点亮、结论流式输出、进行中事故的动效区分；点击条目下钻进该事故的会话线程查看完整处置过程；审批（批准/驳回提案）、标记已处理、触发复验等关键裁决 MUST 在线程内以结构化按钮完成；直播流 MUST 支持按状态/任务过滤，待审批与需人工介入的事故 MUST 置顶可达、不被新消息刷走。
- **FR-012**: 运维 Agent 的模型服务配置 MUST 复用既有 AI Agent 配置（053 落地的协议/端点/模型/凭据可配置通道），并提供智能运维功能的独立启停开关；未配置或禁用时，事故仍创建并保留失败上下文（标记"诊断不可用"），调度与日志主链路零影响。
- **FR-013**: 事故、证据、动作、线程消息 MUST 遵守项目级隔离：仅任务所属项目的成员可见可操作。
- **FR-014**: Agent 处置 MUST 与人工操作安全共存：动作执行前校验实例前置状态，人工已介入（kill/rerun）时 Agent 让位并在线程中记录；处置动作 MUST NOT 影响任何非目标实例或正常调度。
- **FR-015**: 失败风暴场景下系统 MUST 限流自保：Agent 并发处理数与模型调用频次有上限，超出的事故排队等待并如实标记状态。
- **FR-016**: 事故线程 MUST 支持自然语言自由对话：人可向 Agent 追问、质疑、下指令（追加排查方向、要求换方案），Agent 流式回应；由对话引发的写动作与自动处置同等对待——过策略闸门、留审计、以消息形式追加进线程。
- **FR-017**: 监督席顶部 MUST 提供 Agent 生成的智能战况播报：一句话战况综述横幅 + 可展开的完整「接班报告」；横幅内嵌的关键数字（处理中/待审批/需人工介入/已解决）MUST 与实际状态一致，点击 MUST 过滤直播流。
- **FR-018**: 监督席 MUST 是工作区默认首屏视图（打开平台第一眼即指挥中心）；既有运维视图（实例/日志/血缘等）保留独立入口，并 MUST 可从事故线程下钻直达对应证据页。

### Key Entities

- **事故（Incident）**: 一次故障响应的一等载体与人机共同的工作单位。属性：所属项目、关联任务与失败实例（可多个，归并）、故障分型、状态（处理中/待审批/需人工介入/已解决/诊断不可用）、开立与收口时间、收口方式（自动/人工协同）。
- **诊断结论**: Agent 对一次失败的结构化判断。属性：故障分型、置信度、日志证据引用、处置建议；隶属于事故。
- **处置动作**: Agent 或人在事故内执行的一次操作。属性：动作类型（重跑/调资源/检查点续跑/发布修复/复验）、发起方（Agent/人）、闸门裁决结果、执行结果、时间；每条动作对应审计记录。
- **修复提案**: 代码缺陷类的谨慎处置载体。属性：变更内容、证据包（诊断依据/预期效果）、审批状态（待批/已批/已驳回）、审批人与时间、发布与验证结果。
- **线程消息**: 事故会话线程中的一条消息。属性：来源（Agent 步骤/人的自由对话与指令/Agent 对话回应/系统事件）、内容、关联的证据或动作、时间序。
- **直播流事件**: Agent 工作过程的细粒度实时事件（思考态开始/结束、工具动作及其完成状态、流式文本分片），是指挥中心 live feed 与线程实时呈现的共同数据源；粗粒度收敛为线程消息持久化。
- **战况播报（接班报告）**: Agent 生成的态势综述。属性：一句话综述、关键数字快照（处理中/待审批/需人工/已解决）、完整报告正文、生成时间；随事故状态变化更新。
- **AI Agent 配置（复用既有）**: 053 已有的模型服务配置（协议/端点/模型/凭据），本特性只增加智能运维用途的启用开关，不新建配置体系。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 模型配置可用时，任务失败后 5 分钟内产出诊断结论（分型 + 证据 + 建议）的事故占比 ≥ 95%。
- **SC-002**: 在覆盖瞬态、资源不足两类可自愈故障的演示集上，无人工介入自动恢复并收口的比例 ≥ 80%。
- **SC-003**: 进入"需人工介入"队列的事故，100% 附带故障分型、根因定位与可执行建议——不存在只有裸日志的升级项。
- **SC-004**: Agent 发起的写动作 100% 有闸门裁决与审计记录可查，抽查零遗漏。
- **SC-005**: 任一事故在面板线程中可完整还原 Agent 的处理过程（证据 → 诊断 → 动作 → 验证 → 收口），第三者无需查看其他页面即可判断处理是否得当。
- **SC-006**: 修复提案从打开到完成批准/驳回的界面操作在 1 分钟内可完成；批准后的修复变更 100% 可从 repo 侧取回。
- **SC-007**: 同一事故自动处置次数不超过配置上限，全量事故中不存在超限仍在自动重试的记录（防循环红线）。
- **SC-008**: 智能运维功能整体禁用时，任务调度、执行、日志链路行为与禁用前完全一致（零侵入验证）。
- **SC-009**: Agent 的处理步骤从发生到出现在指挥中心直播流与事故线程中不超过 2 秒（保持打开状态下，无需手动刷新）。
- **SC-010**: 打开平台后的默认第一屏即监督席指挥中心（战况横幅 + 直播流），无需任何点击；战况横幅数字与事故实际状态抽查零不一致。
- **SC-011**: 线程内自然语言追问在正常模型服务下 5 秒内开始流式回应（P90）；由对话引发的写动作 100% 有闸门裁决与审计记录。

## Assumptions

- **自主度按行动梯度渐进（对齐方向文档 §4.3）**: 用户诉求"自动改代码重新发布"在第一版落地为"Agent 自动生成修复提案 + 人一键批准后自动发布重跑"；重跑/调资源/续跑类低风险动作完全自动。基于成绩单的"连续批准 N 次后升格自动执行"（渐进自主）与经验库属后续特性（方向文档 §10 第 4 步），本特性不含。
- **编队第一版最小化（对齐方向文档 §4.2 判断 3）**: 不做通用多 agent 编排框架；诊断/处置作为单一运维 Agent 的能力交付，事故是唯一决策与审计单位。
- **前置依赖 066 已完成**: 人工告警/事件/质量体系已清场（066 任务 35/35 完成），事故面板是故障可见性的唯一新入口，不与旧告警中心并存。
- **模型服务复用 053 配置**: 不新建模型接入体系；053 的 AI Agent 配置（Anthropic/OpenAI 协议、端点、模型、凭据）是唯一模型服务来源，未配置即降级为"只建单不诊断"。
- **实时任务恢复语义复用 062**: 流式任务的检查点续跑、优雅停止、SUSPENDED 语义沿用 062 既有能力，本特性只做"失败时优先选择续跑"的决策接线。
- **authoring 底线不变（对齐方向文档 §7.1 与 constitution IV）**: 任务创作仍走本地 Skill + CLI；Agent 的代码修复是"带证据的运维变更提案"而非 authoring 界面，修复变更必须回流 repo 可取回。constitution 原则 IV 与服务端运维 Agent 的边界关系在 plan 阶段的 Constitution Check 中正式裁决（方向文档 §7.1 已给出重划口径）。
- **上游脏数据的处置边界**: Agent 只定位与建议，不代人修改上游数据；"hold 下游 / 降级供数"等影响面处置属后续演进，不在本特性范围。
- **诊断时效以分钟计**: 云端模型调用秒级到数十秒，加上证据采集，5 分钟内出诊断是合理预期；事故创建本身不依赖模型（毫秒级）。
- **开单范围裁决**: SUSPENDED 挂起实例纳入开单（其设计意图即等待介入）；TEST 运行与补数（BACKFILL）失败不进入事故通道——补数失败具批量性、由发起人现场跟进，避免开单风暴；周期与手动来源以实例是否携带 cron 表达式区分。
- **直播流与对话的成本边界**: 直播流事件是 Agent 既有工作过程的透出，本身不新增模型调用；自由对话仅在人主动发起时消耗模型，且上下文限定在单事故线程内。战况播报随状态变化生成，非高频轮询。
- **指挥中心视觉遵循既有设计契约**: 作战室美学（动效、呼吸感、流式打字）在实现阶段过 DESIGN.md 设计契约门定稿，规格只约束信息架构与行为，不约束具体视觉语言。

## Dependencies

- **066-remove-alert-incident**（已完成）: 清场是本特性的空间前提；事故面板接棒成为唯一故障入口。
- **053-lineage-llm-agent-schema**: AI Agent 模型配置体系（协议/端点/模型/凭据）被本特性复用。
- **062-streaming-task-ops**: 实时任务的检查点续跑与停止语义被本特性的处置动作复用。
- **既有策略闸门与审计体系**: 所有 Agent 写动作的裁决与留痕通道。
- **既有实例日志与运行历史能力**: 诊断证据的来源。

## Out of Scope

- 经验库（incident 记忆/故障签名/成绩单）与渐进自主升格机制（方向文档 §10 第 4 步，另行立项）。
- 影响评估（血缘爆炸半径 × SLA 时间预算的分诊）与降级供数、hold 下游等影响面处置。
- 确定性 playbook 引擎（签名匹配零成本处置）。
- 通用多 agent 编排框架、动态编队。
- 契约履约看板与数据地图转型（方向文档 §7.3）。
- 修改上游/源头数据的任何自动化。

# Feature Specification: 激活页统计数据自动无感刷新 + 统一手动刷新控件

**Feature Branch**: `031-active-view-refresh`

**Created**: 2026-07-01

**Status**: Draft

**Input**: User description: "系统指标类似于，这种页面系统指标 里面很多指标，如果用户激活的是当前页面，需要支持，里面的数据自动刷新，展示统计的也见面，页面无感，并且统一设计下，Tag手动刷新的功能"

## User Scenarios & Testing *(mandatory)*

工作区是多 tab 形态：用户可以同时打开「系统指标」「业务报表」「数据时效」「运维大盘」等多个统计类页面，并在它们之间切换。本特性聚焦三件事：① 当某个统计页是**当前激活页**时，它的数字应当**自动、定期、无感**地更新，让用户始终看到接近实时的统计；② 非激活（后台隐藏）的页面**不应**持续消耗资源去刷新；③ 所有统计页提供**风格统一**的手动刷新控件，让用户在需要时立刻拉取最新数据。

### User Story 1 - 激活页统计数据自动无感刷新 (Priority: P1)

用户打开「系统指标」页并停留查看。无需任何操作，页面上的各项指标数字按固定节奏自动更新为最新值。更新发生时页面不闪烁、不跳动、不弹出全屏加载态——用户正在看的滚动位置、展开/折叠状态、筛选条件、选中项全部保持不变，只有数值与时间戳安静地原地变化。

**Why this priority**: 这是需求的核心价值——让统计页从「打开那一刻的快照」变成「持续可信的实时面板」，是 MVP 的最小可交付切片。仅此一项即可独立交付价值。

**Independent Test**: 打开系统指标页，后端数据发生变化后，在不触碰页面的情况下，观察指标数字在约定刷新周期内自动更新；同时确认页面滚动位置与交互状态未被重置。

**Acceptance Scenarios**:

1. **Given** 用户停留在激活的系统指标页且后端指标已变化，**When** 经过一个刷新周期，**Then** 页面上的指标数值与「最后更新时间」自动更新为最新值，无全屏加载遮罩、无内容跳动。
2. **Given** 用户在统计页内已滚动到某位置、展开了某分组或设置了某筛选，**When** 自动刷新触发，**Then** 滚动位置、展开态、筛选条件、选中项均保持不变。
3. **Given** 一次自动刷新的数据请求失败，**When** 刷新周期到达，**Then** 页面继续展示上一次成功的数据（不清空、不报错全屏），并以不打断阅读的方式提示「更新失败/数据可能过时」，下一周期自动重试。

---

### User Story 2 - 非激活页暂停刷新，切回即恢复 (Priority: P1)

用户同时开着多个统计页，但同一时刻只看一个。被切到后台（隐藏）的页面应停止自动刷新以节省资源；当用户切回该页时，应立即拉取一次最新数据并恢复自动刷新节奏，避免用户看到陈旧快照。

**Why this priority**: 工作区采用 keep-alive（隐藏而非卸载）策略，若所有打开过的页都持续轮询会造成大量无意义请求与资源浪费；同时「切回即新鲜」是无感体验的一部分。与 US1 同为 P1，二者共同构成「按激活态调度刷新」的完整闭环。

**Independent Test**: 打开两个统计页 A、B，停在 A；观察 B 在后台期间不再发刷新请求；切到 B 时 B 立即刷新一次并随后按周期更新，A 转入后台后停止刷新。

**Acceptance Scenarios**:

1. **Given** 统计页 B 被切换到后台（非激活），**When** 处于后台期间，**Then** B 不再发起自动刷新请求。
2. **Given** 用户从 A 切换到此前在后台的 B，**When** B 成为激活页，**Then** B 立即执行一次刷新，并恢复按周期自动刷新。
3. **Given** 整个浏览器标签页/窗口被切到后台或最小化，**When** 用户重新回到该窗口，**Then** 当前激活的统计页立即刷新一次，后台期间不空转。

---

### User Story 3 - 统一设计的手动刷新控件 (Priority: P2)

用户希望「现在立刻看到最新数据」，不想等下一个自动周期。每个统计页在一致的位置提供一个外观、图标、行为统一的手动刷新控件；点击后立即拉取最新数据，控件给出明确的「正在刷新」反馈，完成后回到常态并更新「最后更新时间」。

**Why this priority**: 自动刷新已覆盖大多数场景，手动刷新是补充的掌控感与即时性。它依赖 US1 建立的刷新能力，故置于 P2，但用户明确点名要「统一设计」，必须纳入本特性。

**Independent Test**: 在任一统计页点击手动刷新控件，确认数据立即更新、控件展示刷新中反馈、「最后更新时间」随之更新；比对多个统计页，确认控件的图标、位置、交互一致。

**Acceptance Scenarios**:

1. **Given** 用户在任一统计页，**When** 点击手动刷新控件，**Then** 立即发起一次数据拉取，控件进入「刷新中」反馈态，完成后恢复并更新「最后更新时间」。
2. **Given** 一次自动刷新正在进行，**When** 用户点击手动刷新，**Then** 不产生重复并发拉取或状态错乱（合并/排队为一次有效刷新）。
3. **Given** 用户浏览多个统计页，**When** 比较各页的刷新控件，**Then** 其图标、相对位置与交互行为在所有纳入范围的页面间保持一致。
4. **Given** 用户在某统计页关闭了「自动刷新开关」，**When** 用户停留在该页，**Then** 该页不再自动刷新，但手动刷新仍可点击拉取最新数据；**When** 用户重开该 tab 或刷新整个页面，**Then** 开关恢复为「开」。

---

### Edge Cases

- **手动刷新冲突自动刷新**：用户点击手动刷新的同一时刻恰逢自动周期触发——必须合并为一次有效拉取，不得并发重复请求或令「刷新中」反馈错乱。
- **慢响应跨越下一周期**：一次刷新请求耗时超过刷新周期——下一周期不得在上一请求未结束时再次发起（避免请求堆叠）。
- **页面在刷新进行中被切到后台**：在途请求应被允许完成或安全丢弃，且不得在用户已离开后改写其它页状态。
- **长时间后台后切回**：页面在后台停留很久（数据严重过时）后被切回——应立即刷新并清晰呈现新「最后更新时间」。
- **持续刷新失败**：连续多次自动刷新失败——保留最后一次成功数据，以不打断的方式提示陈旧/失败状态，不进入全屏错误。
- **空数据/无数据页**：统计页当前无数据时，自动与手动刷新仍可正常工作，不报错。
- **分页/筛选态页面**（如数据时效、运维实例列表）：刷新应在用户当前页码/筛选上下文内更新数据，而非把用户弹回第一页或重置筛选。

## Clarifications

### Session 2026-07-01

- Q: 激活页是否提供「自动刷新开关」让用户暂停/恢复自动刷新？ → A: 提供每页开关，仅会话内（默认开，暂停状态不持久化，重开 tab/刷新页面即回到「开」）。
- Q: 各统计页的自动刷新周期是统一还是按页不同？ → A: 全部统一默认 ~30s（不按页分档；后续若证实某页确需不同节奏再单独评估）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 当某统计类页面是**当前激活页**时，系统 MUST 按一个**统一默认刷新周期（~30 秒）**自动重新拉取并展示该页的统计数据；所有纳入范围的统计页采用同一默认周期，不按页分档。
- **FR-002**: 自动刷新 MUST 为**无感更新**——更新数据时不得展示全屏/遮罩式加载态、不得使页面内容跳动或重排导致阅读位置丢失，且 MUST 保留用户当前的滚动位置、展开/折叠状态、筛选条件、分页位置与选中项。
- **FR-003**: 当页面处于**非激活（后台隐藏）**状态时，系统 MUST 暂停其自动刷新，不再发起刷新请求。
- **FR-004**: 当一个此前在后台的统计页**重新成为激活页**时，系统 MUST 立即执行一次刷新，并恢复按周期的自动刷新。
- **FR-005**: 当整个浏览器标签页/窗口从隐藏/最小化**重新可见**时，系统 MUST 对当前激活的统计页立即刷新一次，且后台期间不空转刷新。
- **FR-006**: 系统 MUST 在每个纳入范围的统计页提供一个**手动刷新控件**，点击即立即拉取最新数据。
- **FR-007**: 手动刷新控件在所有纳入范围的页面间 MUST 保持**统一设计**——一致的图标、相对位置与交互行为（遵循 `frontend/DESIGN.md` 与现有设计系统约束）。
- **FR-008**: 手动刷新与正在进行的自动刷新之间 MUST 互不冲突——同一时刻 MUST NOT 对同一数据源产生重复并发拉取，重叠请求 MUST 合并或排队为一次有效刷新。
- **FR-009**: 一次刷新请求耗时超过刷新周期时，系统 MUST NOT 在上一请求未完成前发起下一次自动刷新（不堆叠请求）。
- **FR-010**: 当一次刷新（自动或手动）失败时，系统 MUST 保留上一次成功的数据、MUST NOT 进入全屏错误态，并 MUST 以不打断阅读的方式提示更新失败/数据可能过时；自动刷新 MUST 在下一周期继续重试。
- **FR-011**: 每个统计页 MUST 展示一个可见的**「最后更新时间」**指示，并在每次成功刷新后更新它，让用户判断数据新鲜度。
- **FR-012**: 手动刷新进行中，控件 MUST 给出明确的「刷新中」反馈，完成后恢复常态。
- **FR-013**: 自动刷新行为 MUST 仅作用于纳入范围的**统计/指标类页面**（见 Assumptions 范围清单），MUST NOT 改变非统计页（如编辑画布、设置、详情面板、已有 SSE 实时流的日志页）的既有行为。
- **FR-014**: 每个统计页 MUST 提供一个**每页的自动刷新开关**，允许用户暂停/恢复该页的自动刷新。约束：① 默认为「开」；② 暂停期间 MUST 停止该页的自动刷新，但**手动刷新仍可用**；③ 该开关状态为**会话内、不持久化**——重开该 tab 或刷新整个页面后 MUST 恢复为「开」；④ 暂停状态下，「最后更新时间」继续如实反映上一次成功刷新的时刻，让用户感知数据可能过时。

### Key Entities *(include if feature involves data)*

- **统计视图（Statistics View）**：一个展示聚合数字/指标/列表的工作区页面（系统指标、业务报表、数据时效、运维大盘、数据质量、告警等）。关键属性：是否激活、刷新周期、最后更新时间、当前数据快照、上一次刷新结果（成功/失败）。
- **刷新调度状态（Refresh Schedule State）**：描述某统计视图当前的刷新生命周期——激活/暂停、用户的自动刷新开关（会话内、默认开）、是否有在途请求、距下次自动刷新的节奏（统一默认 ~30s）。用于实现按激活态启停、按开关启停、防止请求堆叠与并发去重。
- **激活态（Active Tab State）**：工作区中「当前可见的唯一页面」的标识，是刷新调度的输入信号；另叠加浏览器窗口/标签页整体的可见性信号。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户停留在激活统计页时，无需任何操作即可在一个刷新周期内（默认 ≤ 30 秒）看到后端已变化的数据被自动更新。
- **SC-002**: 自动刷新发生时，用户当前的滚动位置、展开/折叠、筛选、分页与选中项 100% 保持不变，且无可感知的内容跳动或全屏加载闪烁。
- **SC-003**: 非激活页在后台期间发起的自动刷新请求数为 0；切回激活时在 1 秒内开始一次刷新。
- **SC-004**: 所有纳入范围的统计页都具备手动刷新控件，且其图标、相对位置、交互行为在各页面间完全一致（设计走查 100% 通过 `DESIGN.md` 约束）。
- **SC-005**: 在自动刷新周期内，对同一数据源的并发重复请求数为 0（手动+自动重叠时合并为一次）。
- **SC-006**: 单次刷新失败不导致页面进入全屏错误态或丢失已展示数据的比例为 100%。

## Assumptions

- **纳入范围（统计/指标类页面）**：系统指标（metrics）、业务报表（reports）、数据时效（freshness）、运维大盘（ops，含汇总卡片与实例列表）、数据质量（quality）、告警（alerts）。这些当前均为「打开时一次性拉取、无自动刷新」。
- **排除范围**：工作流画布/编辑器、设置、占位类页面、纯详情面板；以及已有 SSE 实时流的日志页（instance-log）——其实时性已由现有流式机制满足，不纳入本轮轮询式刷新改造。
- **统一刷新周期**：所有纳入范围的统计页采用同一默认周期 ~30 秒，不按页分档（已澄清）。本特性不提供「用户自定义周期」的设置项（可作为后续增强）；若后续证实个别页确需更快/更慢节奏，再单独评估。
- **自动刷新开关**：提供每页开关，默认「开」，状态仅在会话内有效、不持久化（重开 tab 或刷新整页即回到「开」）（已澄清）。不提供全局开关或跨会话记忆（可作为后续增强）。
- **激活态来源**：复用工作区已有的「当前激活 tab」状态作为刷新调度的主信号；并叠加浏览器页面可见性信号控制整窗后台时的暂停。
- **keep-alive 前提**：工作区切换 tab 采用隐藏（display）而非卸载，因此「按激活态启停刷新」是必要的资源治理手段。
- **设计一致性**：手动刷新控件复用现有设计系统与图标（统一刷新图标），位置/样式以 `frontend/DESIGN.md` 为准；改动前先读 DESIGN.md 并声明采用的约束。
- **数据接口**：复用各统计页现有的数据来源（无需新增后端接口），本特性聚焦前端的刷新调度与统一交互；若个别页需要轻量后端配合（如更廉价的快照端点）于规划期评估。
- **i18n**：新增静态文案（「最后更新时间」「刷新」「更新失败」等）走前端 next-intl，zh-CN/en-US 双语键集一致。

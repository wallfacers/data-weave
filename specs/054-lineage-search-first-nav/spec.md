# Feature Specification: 血缘探索器入口重构——搜索优先 · 数据源降级为分面 · 跨库可辨

**Feature Branch**: `054-lineage-search-first-nav`

**Created**: 2026-07-07

**Status**: Draft

**Input**: User description: "数据血缘，页面操作，必须去选左侧的数据源列表，现实生活中，跨数据库血缘的场景还是比比皆是吧，你深度思考下，这里的操作上能不能优化"

## 背景与问题（Context）

052 把血缘从「Demo」升级成了「可用的探索器」（真实图引擎、双向、原地展开、搜索、影响分析、列级血缘）。但它的**入口/导航范式**仍停留在旧模型：左侧是一棵**数据源 → 表 → 列**的三级树，且这棵树是**唯一的正门**——用户想看某张表的血缘，必须先知道它在哪个数据源、先展开那个数据源、再往里钻。这与「跨数据库血缘」的现实自相矛盾：

1. **数据源被当成孤岛容器**：现实里数据在库之间流动（`mysql.user` → `hive.dwd_user` → `pg.rpt_user`），工程师排障时想的是**「这张表 / 这个指标」本身**，而不是「它在 A 库还是 B 库」。强制先选库，既加了一层无谓心智负担，又把「跨库」这个本该被血缘揭示的东西，在入口处切成了一个个孤岛。
2. **已有的全局搜索被埋没**：052 其实已经实现了跨数据源的资产搜索（`fetchSearch` → `/api/lineage/search`），但它藏在工具栏、不是视觉正门；左侧数据源树仍占据「主入口」的位置。
3. **图上看不出「跨库」**：后端 `GraphNodeView` 目前不携带节点所属**数据源**信息（表节点只有 `parentId` 与 `layer`），前端图里跨库的边和库内的边长得一模一样——「跨库血缘」这个核心价值，恰恰在图上**看不见**。
4. **左侧树的 datasource→table 级还是占位实现**（`lineage-tree.tsx` 内注释：正式实现需 tables-by-datasource 端点，当前用 `fetchColumns` 占位跳过表级）——即以数据源为正门这条路本身都没走通。

**业界对照（对齐目标形态，非实现约束）**：DataHub / OpenMetadata / Atlan / Amundsen / dbt Explorer / Purview 几乎全是**搜索优先 + 资产为中心**——跨所有数据源的全局搜索是正门，数据源（platform/service）降级为**浏览的一个分面**（与分层/主题域/标签/最近并列），绝非必经之门；图上每个节点**自带数据源徽标**，跨源边天然可辨，部分工具（Purview/OpenMetadata）进一步把节点**按数据源分泳道/容器**。

本特性把血缘探索器的入口从「数据源孤岛树优先」重构为**「搜索优先 + 数据源降级为分面 + 图上跨库可辨」**，让跨库血缘从入口到图形都成为一等公民。它不改血缘的查询内核（052 的双向/深度/展开/影响/路径/列级全部复用），只改**如何进入**与**如何呈现数据源边界**。

## Clarifications

### Session 2026-07-07

- Q: 血缘探索器的「正门」入口采用哪种？ → A: **搜索优先，数据源树降级为分面**——把跨全部数据源的全局资产搜索提为主入口（直接打表/列/指标名即可锚定，不必先选库）；左侧数据源树保留但降级为「可切分面之一」，与分层/主题域/标签/最近·我的并列。
- Q: 图上如何呈现「跨库」？ → A: **节点带数据源徽标 + 跨源边强调**——每个数据流节点携带所属数据源（图标/名/配色），跨数据源的边用区别于库内边的样式（虚线/异色）强调；按数据源分泳道/容器作为后续可选深化（P3），非本期主干。
- Q: 实现边界是否允许后端补读侧能力？ → A: **允许**——为让节点显示数据源、跨源边可判定、按分面浏览可用，允许在血缘读侧补齐「节点携带数据源 id/name」「按分面（数据源/分层/…）浏览资产」等**只读**查询能力；写侧与抽取内核不动。
- Q: 保证可用的分面集合是哪些？ → A: **数据源 · 分层 · 最近/我的为保证项**（均有现成数据来源：datasource 节点、table.layer、客户端本地记录）；**主题域 · 标签**为「若资产元数据可得则提供、否则本期从缺」的可选分面，不阻塞交付。
- Q: 「跨源边」如何判定？ → A: 一条边的两端节点解析到的**数据源不同**即为跨源；两端同源为库内边；任一端数据源未知（孤儿/未登记）则按「未知来源」样式呈现，不误判为跨源。
- Q: 分面浏览器（数据源/分层/最近/我的）在本特性里的分量？ → A: **可有可无的辅助，降为可选**——本特性目的聚焦「看血缘 + 跨库展示 + 节点展示样式（数据源 logo/徽标、跨源边、边上字段、下游好查、折叠）」；分面浏览属辅助，降为可选（P3），且**「我的/ownership」不做**（血缘节点无 owner 元数据），近期回访仅保留客户端「最近」，甚至可整体从缺，不阻塞交付。
- Q: 「线字段」/字段级血缘怎么展示？ → A: **表和字段都要连线展示**——表→表画表级连线；表节点展开到列时，列→列的字段级血缘也画成**真实连线**（连到具体字段锚点，如 `user.id → dwd_user.uid`），而非仅高亮列。复用 052 已有列级血缘数据驱动，不新增抽取。这是本特性核心展示能力，非可选辅助。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 搜索优先：不选库直接锚定任意资产（Priority: P1）

数据工程师打开血缘探索器，最显眼的是一个**跨所有数据源的全局搜索框**（而非一棵要求先选库的数据源树）。他只需键入表名/列名/指标名的片段（如 `order_detail`），就能拿到**跨全部数据源**的候选列表，每个候选清楚标注**所属数据源 / 类型 / 分层**；选中任意一项，血缘图即以它为锚点加载并居中——**全程无需先知道、也无需先选择它在哪个数据源**。当尚未锚定任何资产时，画布空态引导「搜一个资产开始」，而不是「去左侧点一个数据源」。

**Why this priority**: 这是用户诉求的主干、也是从「孤岛树优先」到「资产中心」的核心杠杆。它单独交付即构成可用增量——把已有的 `fetchSearch` 从工具栏提到 C 位、把空态与首屏焦点改为搜索，用户就能「想查哪个查哪个」，彻底摆脱「必须先选库」。

**Independent Test**: 首屏未锚定时，搜索框获得视觉主位与输入焦点、画布空态提示「搜一个资产开始」；在搜索框输入一个已存在表名片段，返回带（数据源·类型·分层）标注、跨多个数据源的候选；选中一项后图以该资产为锚点加载并居中——整个过程未与任何数据源列表交互。

**Acceptance Scenarios**:

1. **Given** 用户刚进入血缘探索器且未锚定任何资产，**When** 页面渲染完成，**Then** 全局搜索框位于视觉主入口位置并默认聚焦，画布区展示「搜一个资产开始」空态（非加载态、非报错），不强制用户先与数据源列表交互。
2. **Given** 项目内 `order_detail` 相关表分布在 2 个不同数据源，**When** 用户在搜索框输入 `order_detail`，**Then** 候选列表跨这 2 个数据源返回全部匹配，每项标注所属数据源名、类型（TABLE/COLUMN/METRIC）、分层（若有），限定在当前项目范围内。
3. **Given** 搜索候选列表，**When** 用户点选其中一项，**Then** 血缘图以该资产为锚点加载、该节点居中高亮，且用户**从未**展开或选择任何数据源。
4. **Given** 关键字无任何匹配，**When** 搜索完成，**Then** 展示明确空态提示（非加载态、非报错）。
5. **Given** 用户已锚定某资产，**When** 他清空并重新搜索另一个资产并选中，**Then** 图切换到新锚点，方向/深度/粒度等既有视图状态按 052 既定语义处理。

### User Story 2 - 图上跨库可辨：节点数据源徽标 + 跨源边强调（Priority: P1）

用户在血缘图上，能**一眼看出每个节点属于哪个数据源**（节点带数据源徽标：图标/名/配色），并且**跨数据源的边被明确强调**（区别于库内边的样式，如虚线/异色），配合图例说明。这样「数据从 mysql.user 跨到 hive.dwd_user、再跨到 pg.rpt_user」的跨库流动路径清晰可辨——跨库血缘这个核心价值在图形上终于「看得见」。

**Why this priority**: 这是用户诉求「跨库场景比比皆是」在**呈现层**的兑现、也是本特性的核心目的之一。没有它，即便入口改好了，用户在图上仍分不清哪条边跨了库，跨库血缘等于白做。与 US1 并列为 P1：入口让人**进得来**，本故事让人**看得懂**跨库。可独立于入口改造分别交付与验证。

**Independent Test**: 打开一个横跨 ≥2 个数据源的血缘图；验证每个数据流节点显示其所属数据源徽标；两端数据源不同的边呈「跨源」样式、两端同源的边呈「库内」样式；图例解释两种边与徽标含义；某端数据源未知时按「未知来源」样式呈现而非误判为跨源。

**Acceptance Scenarios**:

1. **Given** 一个锚点的血缘横跨 mysql / hive / pg 三个数据源，**When** 图渲染完成，**Then** 每个表/列节点显示其所属数据源标识（可辨的图标或名或配色），同库节点共享同一数据源标识。
2. **Given** 上述图中存在若干跨源边与若干库内边，**When** 图渲染完成，**Then** 跨源边以区别于库内边的样式呈现（如虚线/异色），并有图例说明二者差异。
3. **Given** 一条边的某一端节点数据源未知（孤儿/未登记），**When** 图渲染，**Then** 该边按「未知来源」样式呈现，不被误判为跨源边。
4. **Given** 用户悬浮或选中一条跨源边，**When** 交互发生，**Then** 详情中可见其两端所属数据源（复用 052 边详情面板承载）。
5. **Given** 数据源数量较多导致配色不足以一一区分，**When** 图渲染，**Then** 至少通过徽标文本/图标保证可辨，不因配色耗尽而出现两库无法区分的歧义。
6. **Given** 两张有列级血缘的表被展开为列清单，**When** 图渲染，**Then** 存在派生关系的列之间画出连到**具体字段**的连线（如 `user.id → dwd_user.uid`），跨库的字段映射以跨源样式强调，且与表级连线视觉可区分。

### User Story 3 - 数据源浏览分面（辅助 · 可选）（Priority: P3）

作为「不知道确切名字、想按维度逛一逛」的**辅助入口**，用户可用左侧分面浏览器在 **数据源 / 分层（ODS·DWD·DWS·ADS）/ 最近** 之间切换：数据源不再是唯一路径，而是与分层/最近**平级的一个分面**；从任一分面选中资产即以它为锚点加载。此故事为**可有可无的辅助**——搜索（US1）已覆盖主入口，本故事仅在有余量时增强探索性浏览，甚至可整体从缺而不影响前两个故事交付；**不含「我的/ownership」**（血缘节点无 owner 元数据）。

**Why this priority**: 用户明确本特性聚焦「看血缘 + 跨库展示 + 节点样式」，分面浏览属辅助功能、可有可无。降为 P3 且可裁剪：它增强而绝不阻断 US1/US2 的核心路径。若实现，则顺带把 052 遗留的 datasource→table 占位实现补成真实 tables-by-datasource。

**Independent Test**: 左侧分面切换器可在「数据源 / 分层 / 最近」间切换；切到「分层」按 ODS/DWD/DWS/ADS 分组、切到「数据源」按库分组并能展开出该库真实的**表**、切到「最近」列出本会话锚定过的资产；从任一分面选一个资产，图以其为锚点加载。

**Acceptance Scenarios**:

1. **Given** 左侧分面浏览器，**When** 用户把分面从「数据源」切到「分层」，**Then** 左侧改为按 ODS/DWD/DWS/ADS 分组列出资产，数据源不再是唯一维度。
2. **Given** 分面切到「数据源」，**When** 展开某个数据源，**Then** 列出该数据源下的**表**（走真实的 tables-by-datasource 能力，而非占位跳过表级）。
3. **Given** 用户此前已锚定过若干资产，**When** 切到「最近」分面，**Then** 列出本会话近期锚定过的资产（客户端本地记录），点击即重新锚定。
4. **Given** 分面浏览整体未纳入本期交付，**When** 用户使用血缘探索器，**Then** 搜索（US1）仍是完整可用的主入口，US1/US2 不受影响。

---

### User Story 4 - 按数据源分组泳道 + 深化空态引导（Priority: P3）

作为跨库可辨的进一步深化，用户可选择把图**按数据源分组为泳道/容器**（同库节点收进同一容器，跨库边成为跨容器连线），在数据源较少、层次清晰时更直观地看清「库与库之间」的数据流；库多时可折叠某些容器聚焦主干。同时首屏与各类空态统一提供「从搜索/最近开始」的引导。

**Why this priority**: 泳道是「跨库可辨」的锦上添花，价值真实但非必需（徽标+跨源边已能表达跨库）；且在数据源多时可能拥挤，需可开关。列为 P3，作为 US3 之上的可选增强，不阻塞前三个故事交付。

**Independent Test**: 在一个横跨多个数据源的图上开启「按数据源分组」，验证同库节点收进同一带标签的容器、跨库边为跨容器连线、可折叠某容器；关闭后回到 US3 的徽标视图。

**Acceptance Scenarios**:

1. **Given** 一个横跨 3 个数据源的血缘图，**When** 用户开启「按数据源分组」，**Then** 节点按所属数据源收进各自带标签的容器，跨库边呈现为跨容器连线。
2. **Given** 已分组视图，**When** 用户折叠某个数据源容器，**Then** 该容器内节点收起、其对外的跨库边以聚合形式或指向容器的方式保留可见。
3. **Given** 分组开启且数据源数量很多，**When** 视图渲染，**Then** 提供关闭分组回到徽标视图的入口，避免拥挤时无法使用。

### Edge Cases

- **孤儿资产**：表/列节点无法解析到数据源（未登记/数据缺失）时，节点显示「未知来源」标识、相关边按未知样式，不阻断图渲染。
- **同名跨库**：不同数据源下存在同名表（如两库都有 `user`），搜索候选与节点必须靠数据源标注区分，不得混淆锚定。
- **搜索空结果 / 分面空结果**：分别给明确空态，区别于加载态与报错态。
- **超大结果**：搜索候选或某分面下资产极多时，遵循 052 既有的分页/截断与截断提示语义，不静默丢弃。
- **数据源极多**：徽标配色不足以区分时，以文本/图标兜底保证可辨。
- **列级粒度下的数据源**：列节点的数据源随其所属表继承，展开列清单时列徽标与表一致。
- **深链恢复**：带 `?anchor=` 深链进入时，直接锚定该资产、不要求先经过搜索或分面（复用 052 深链恢复）。
- **读侧不可用**：血缘存储不可用时（`lineage.store_unavailable`），搜索/分面/图统一走既有不可用态，不各自崩溃。

## Requirements *(mandatory)*

### Functional Requirements

**入口 · 搜索优先（US1）**

- **FR-001**: 血缘探索器首屏未锚定时，System MUST 将跨全部数据源的全局资产搜索作为视觉主入口并默认聚焦；数据源列表 MUST NOT 作为唯一或强制的进入路径。
- **FR-002**: Users MUST be able to 仅凭表/列/指标名片段，在**不选择任何数据源**的前提下检索并锚定任意资产。
- **FR-003**: 搜索候选每项 MUST 标注其**所属数据源**、类型（TABLE/COLUMN/METRIC）、分层（若有），并限定在当前项目范围内。
- **FR-004**: 选中任一搜索候选，System MUST 以该资产为锚点加载血缘图并居中高亮，复用 052 的方向/深度/粒度视图语义。
- **FR-005**: 未锚定时的画布 MUST 呈现「搜一个资产开始」空态（非加载、非报错）；搜索无匹配 MUST 呈现明确空态。

**图上跨库可辨 · 节点与连线展示（US2，核心 P1）**

- **FR-006**: 血缘读侧返回的每个**数据流节点**（表/列）MUST 携带其所属数据源标识（至少 id 与展示名），供前端呈现；孤儿节点标识为「未知来源」。
- **FR-007**: System MUST 在图上为每个节点呈现所属数据源徽标（图标/名/配色其一或组合），同一数据源的节点共享同一标识。
- **FR-008**: System MUST 判定每条边是否**跨源**（两端数据源不同为跨源、相同为库内、任一端未知为「未知来源」），并对跨源边以区别于库内边的样式强调。
- **FR-009**: System MUST 提供图例，说明数据源徽标、跨源边、库内边、未知来源的含义。
- **FR-010**: 跨源边的详情（悬浮/选中）MUST 可见其两端所属数据源，复用 052 边详情面板承载。
- **FR-011**: 当数据源数量导致配色不足以区分时，System MUST 以徽标文本/图标兜底，保证任意两个不同数据源在图上可辨。
- **FR-012**: System MUST 以**真实连线**展示血缘：表级粒度画表→表连线；表节点展开列后，列→列的字段级血缘 MUST 画成连到**具体字段锚点**的连线（如 `user.id → dwd_user.uid`），而非仅高亮列；字段级连线由 052 已有列级血缘数据驱动，不新增抽取。
- **FR-013**: 字段级连线 MUST 与表级连线并存且视觉可区分，且同样参与 FR-008 的跨源判定（跨库的字段映射同样以跨源样式强调）。

**数据源浏览分面（US3，辅助 · 可选）**

- **FR-014**: （可选）System SHOULD 提供分面浏览器，至少支持在 **数据源 / 分层 / 最近** 之间切换；数据源 SHOULD 呈现为「分面之一」而非唯一根维度；**不含「我的/ownership」**。本分面整体可裁剪，裁剪时 US1 搜索仍为完整主入口。
- **FR-015**: （可选）选「分层」分面时按 ODS/DWD/DWS/ADS 分组列出资产；选「最近」时列出本会话近期锚定过的资产（客户端本地记录）。
- **FR-016**: （可选）选「数据源」分面并展开某数据源时，System SHOULD 通过真实的 tables-by-datasource 能力列出该数据源下的**表**（替换 052 遗留的占位跳级实现）；从任一分面选中资产 MUST 与从搜索选中行为一致（锚定并居中加载）。

**分组泳道 · 深化（US4）**

- **FR-017**: System SHOULD 提供可开关的「按数据源分组」视图，将同源节点收进带标签容器、跨库边呈跨容器连线，并允许折叠容器与关闭分组回到徽标视图。

**通用约束**

- **FR-018**: 本特性 MUST NOT 改变 052 已交付的血缘查询内核语义（双向/深度/原地展开/影响/路径/列级血缘）；仅改入口、分面与数据源呈现。
- **FR-019**: 所有界面原语 MUST 遵循 DESIGN.md 强制 reuse-first（搜索输入/下拉/滚动/加载/刷新/卡片/图标复用既有规范组件；图渲染沿用 052 的 `@xyflow/react` 栈），MUST NOT 手写同类原语；新增原语须回填 DESIGN.md 组件目录。
- **FR-020**: 搜索、分面浏览与节点数据源解析 MUST 遵守项目级隔离（仅返回当前项目可见资产），与 052/036 隔离语义一致。
- **FR-021**: 读侧不可用（`lineage.store_unavailable`）时，搜索/分面/图 MUST 统一走既有不可用态处理。

### Key Entities *(include if feature involves data)*

- **Asset（资产）**：可被搜索/浏览/锚定的血缘数据流对象——表 / 列 / 指标。关键属性：id、名称、类型、分层、**所属数据源**、项目归属。
- **Datasource（数据源）**：资产的归属库/连接。关键属性：id、展示名、（用于图上可辨的）稳定标识/配色种子。既是图上节点徽标的来源，也是「数据源」分面的分组键。
- **Facet（分面，可选辅助）**：左侧浏览的维度——数据源 / 分层 / 最近（**不含「我的/ownership」**）。整体可裁剪；每个分面把资产按其维度分组供浏览。
- **SearchCandidate（搜索候选）**：搜索返回项。关键属性：资产 id、名称、类型、分层、**所属数据源名**、（用于同名区分的）数据源标注。
- **GraphNode（图节点，富化）**：052 图节点在本特性中**新增携带所属数据源 id/name**；孤儿节点标记未知来源。
- **Edge（边，跨源判定 + 字段级连线）**：052 边在呈现层新增**跨源/库内/未知来源**分类（由两端节点数据源推导，不改边的存储语义）；**列级派生边以连到具体字段锚点的连线呈现**（表→表连线与列→列连线并存且视觉可区分）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从进入血缘探索器到锚定一个已知资产，**无需与数据源列表发生任何交互**即可完成（0 次「选库」操作）。
- **SC-002**: 对一个已知名字的资产，用户从首屏到血缘图加载完成的操作步数 ≤ 3（键入 → 选候选 → 加载），相较「展开数据源 → 找库 → 展开表 → 选表」的旧路径显著减少。
- **SC-003**: 在横跨 ≥2 个数据源的血缘图上，用户能在**不点开任何节点详情**的情况下，仅凭图面就正确指出哪些边是跨库边、每个节点属于哪个数据源（跨源边与库内边视觉可区分率 100%）。
- **SC-004**: 数据源不再是唯一入口——仅凭搜索即可锚定任意资产；若交付可选分面，用户还可在**数据源 / 分层 / 最近** 中任选维度浏览并锚定（分面裁剪时此项不适用，搜索主入口不受影响）。
- **SC-005**: 同名跨库资产（不同数据源同名表）在搜索候选与图节点上均可被数据源标注唯一区分，无锚定歧义。
- **SC-006**: 052 既有的血缘查询能力（双向/深度/展开/影响/路径/列级）在本特性上线后行为不回退（回归全绿）。
- **SC-007**: 在一张展开了列的血缘图上，有派生关系的字段之间以连线呈现，用户无需打开任何详情面板即可顺着连线看出「哪个字段来自哪个字段」（字段级连线呈现率 100%，跨库字段映射可被跨源样式识别）。

## Assumptions

- **复用 052 查询内核与组件栈**：双向/深度/原地展开/影响/路径/列级血缘、`@xyflow/react` 画布、`FlowCanvasWithPanel`/侧面板/图例/工具栏均复用，本特性不重写它们。
- **搜索已有基座**：跨数据源资产搜索 `fetchSearch` → `/api/lineage/search` 已存在，本特性主要是把它提为主入口并富化候选标注（数据源名），必要时读侧补齐候选字段。
- **节点数据源可解析**：表节点经 `parentId` 关联到 datasource 节点、datasource 节点的展示名可得；读侧据此为数据流节点富化「所属数据源 id/name」。列节点数据源随其表继承。
- **读侧可补只读能力**：为 tables-by-datasource 浏览、按分面浏览、节点数据源富化，允许在血缘读侧新增只读查询能力；neo4j 为唯一血缘存储；写侧与抽取内核不动。
- **分面数据来源**：数据源分面来自 datasource 节点；分层分面来自 table.layer；最近·我的来自客户端本地记录；主题域/标签分面仅在对应资产元数据可得时提供，否则本期从缺（不阻塞交付）。
- **项目级隔离**：搜索/分面/图解析均在当前项目范围内，沿用 052/036 的隔离语义。
- **跨源判定纯呈现层**：边的跨源/库内/未知分类由两端节点数据源在呈现层推导，不改血缘边的存储与既有语义。
- **与 053 无冲突面**：053（血缘 LLM Agent 抽取 + 实时 Schema）改的是**写侧/抽取**通道；本特性改的是**读侧入口/呈现**，二者surface 不重叠，集成时以先落地方为准回归共享读模型。

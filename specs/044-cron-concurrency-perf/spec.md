# Feature Specification: Cron 并发触发链路验证与性能基准

**Feature Branch**: `044-cron-concurrency-perf`

**Created**: 2026-07-05

**Status**: Draft

**Input**: cron 自动触发并发链路此前从未被有效压测 —— 测试脚本误用 5 字段 cron(`*/1 * * * *`),而系统采用 Spring `CronExpression`(6 字段:秒/分/时/日/月/周),导致 cron 自动触发链路长期"零触发",先前所有"压测"实际跑的是手动触发。本 feature 验证大批量同 cron 到期下的触发正确性与并发性能,建立可复现的性能基准,并为后续全链路优化奠基。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 验证 cron 自动触发在并发到期下正确触发 (Priority: P1)

平台开发者要确认:当大量定时任务在同一触发点到期时,系统是否为每一个到期的工作流都正确创建了"由定时驱动"的运行实例 —— 既不漏触发,也不重复触发。

**Why this priority**: 这是后续一切性能优化的前提。若连正确触发都无法验证,吞吐/延迟指标没有意义。此前该链路因表达式约定误解从未跑通,"正确性"是首要未知。

**Independent Test**: 建立一批相同调度频率的在线工作流,等待自然触发点,核对"定时触发"实例数与"工作流数 × 触发点数"是否一致,并核对同一触发点不产生重复实例。

**Acceptance Scenarios**:

1. **Given** 50 个调度频率为每 10 秒的在线工作流, **When** 跨越 N 个触发点, **Then** 系统产生的"定时触发"实例数 ≈ 50 × N(成功率 ≥ 99%),无工作流被漏触发。
2. **Given** 同一工作流的同一触发点, **When** 多个调度节点同时尝试触发, **Then** 该触发点全局仅产生 1 个实例(去重生效,无重复)。

---

### User Story 2 - 量化并发 cron 触发的吞吐与延迟 (Priority: P2)

性能工程师要测得:在不同并发到期规模下,单位时间的触发实例创建吞吐,以及从到期点到实例创建的延迟分布。

**Why this priority**: 正确性确立后,需要量化基线,才能判断"性能压榨"的方向与收益。

**Independent Test**: 在不同并发到期规模(10 / 50 / 100 个工作流)下持续观察若干触发点,统计每触发点的实例创建吞吐与延迟分布,并复核可复现性。

**Acceptance Scenarios**:

1. **Given** 并发到期规模可调(10–100), **When** 持续触发 1 分钟, **Then** 能输出每触发点吞吐、平均/p99 延迟。
2. **Given** 同一规模重复测量, **Then** 结果可复现(吞吐与延迟波动在可接受区间内)。

---

### User Story 3 - 定位并发触发链路的瓶颈点 (Priority: P3)

SRE 要看清:在并发到期下,哪个环节先到顶 —— 触发去重表竞争、预读扫描、认领批量、下发并发、数据库连接。

**Why this priority**: 量化之后需指明瓶颈,才能在后续全链路优化阶段对症下药,避免盲目调参。

**Independent Test**: 在高压并发到期下,同时观测各环节指标(触发去重表写入、调度轮次、下发计数、数据库连接)与资源使用,定位最先饱和的环节并给出证据。

**Acceptance Scenarios**:

1. **Given** 高并发到期持续运行, **Then** 能给出各环节利用率/等待指标,并指出最先饱和环节及定量证据。

---

### Edge Cases

- 触发点恰好落在调度节点扫描间隔之外(misfire):系统应按既定策略补触发或跳过,且行为可观测。
- 多调度节点同时预读同一批触发点:撞键放弃不应导致实例丢失(至少一个节点成功)。
- 秒级触发产生大量实例堆积:测试应在结束后清理,避免污染环境与后续测量。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 按 6 字段 cron 语义(秒/分/时/日/月/周)正确解析定时表达式,并在工作流上线后被调度扫描预读与到点触发。
- **FR-002**: 系统 MUST 在多调度节点同时对同一触发点触发时,通过触发去重记录(工作流 × 到期点 唯一键)保证全局只产生一个实例。
- **FR-003**: 验证手段 MUST 能批量建立可配置并发到期规模的工作流,且触发频率可调(秒级或分钟级)。
- **FR-004**: 验证手段 MUST 能区分并统计"定时触发"与"手动触发"实例,以核对正确性。
- **FR-005**: 验证手段 MUST 能观测触发去重的撞键情况与多调度节点的触发分布。
- **FR-006**: 验证手段 MUST 能量化每触发点的实例创建吞吐,以及到期点到实例创建的延迟分布(平均 / p99)。
- **FR-007**: 验证手段 MUST 能在测试结束后清理所建立的工作流、任务与历史实例。

### Key Entities *(include if feature involves data)*

- **工作流定义(WorkflowDef)**: 含调度类型与 cron 表达式;上线(ONLINE)后纳入调度扫描。
- **工作流实例(WorkflowInstance)**: 一次运行,记录触发类型(定时/手动)、调度到期点、状态。
- **触发去重记录(cron_fire)**: 工作流 × 到期点 的唯一键,用于多调度节点零协调去重。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 50 个同频工作流跨越 ≥ 3 个触发点,定时触发实例创建成功率 ≥ 99%。
- **SC-002**: 多调度节点并发下,触发去重撞键放弃不导致任何触发点实例丢失(每个触发点恰好 1 个实例)。
- **SC-003**: 在 50–100 并发到期规模下,给出稳定的每触发点吞吐与 p99 延迟基线(重复测量波动 ≤ 20%)。
- **SC-004**: 给出并发触发链路最先饱和环节的定量证据(利用率 / 等待指标)。
- **SC-005**: 测试结束后环境可被清理回收,无残留工作流/任务污染。

## Assumptions

- 调度模式为 distributed(多调度节点对等竞争认领),机器资源(20 核 / 35GB)充裕,瓶颈不在硬件。
- 测试采用 ECHO 任务(回显即成功,不依赖数据源/沙箱),以隔离"触发链路"性能、排除执行端差异。
- 调度扫描间隔与预读窗口可在测试环境通过配置覆盖,以提升测试迭代速度;不改动后端代码。
- 入口触发(/run)吞吐优化、执行端优化、调度内核代码改动属后续阶段(阶段二全链路优化),不在本 feature 范围。

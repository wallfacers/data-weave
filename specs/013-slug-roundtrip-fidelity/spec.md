# Feature Specification: 文件契约 slug 唯一性与中文名健壮性修复（round-trip 保真）

**Feature Branch**: `013-slug-roundtrip-fidelity`

**Created**: 2026-06-28

**Status**: Draft

**Input**: E2E 闭环真跑（H2 后端 + 真 dw CLI 驱动 pull→diff→push）抓出的文件契约身份模型对中文名结构性数据丢失缝。详见仓库内存 `weft-slug-collision-roundtrip-break`。

## 背景与问题（实测、根因已锁死）

Weft 的核心价值是 **tasks-as-code**：开发者本地用 AI agent 编辑任务/任务流文件，`dw pull/push/diff` 往返服务器治理与运行。往返保真（本地文件树与服务器状态可无损互转）是这条价值链的地基。

当前地基对中文命名**结构性塌陷**：

- 实体落地为本地文件时，文件名取自实体名的 slug。slug 派生保留连字符 `-`，而中文常见「类别-描述」命名（如 `抽取-拉取订单分区`、`清洗-去重生成宽表`、`指标-GMV 汇总`）经派生后中文部分全被抹除、仅余标点/极少 ASCII 残基，坍缩成 `-` 或 `-gmv` 这类**几乎无区分度**的 slug。
- 「名字全是不可用字符时回落到名字哈希」的保护只在 slug 为**空串**时触发；`-`/`-gmv` 非空，不触发，于是多个不同实体得到**相同 slug**。
- 拉取时文件树以路径为键装配，**同名相互覆盖**；身份匹配（拉取/差异/推送）也以 `(目录路径, slug)` 为键，服务器侧**对称坍缩**。

**实测后果**（种子 demo 项目，某工作流 6 个中文名节点任务）：拉取后服务端 20 个任务在本地只剩 16 个文件，**4 个任务静默丢失**；对这份已丢任务的工作副本运行差异预览，系统却报告**「本地与服务器一致，无差异」**——对丢失完全失明。被丢任务又是在线工作流的引用节点，推回服务器将触发删除守卫使整单被拒、或造成静默不一致。**凡含同 slug 残基中文名的项目都无法 round-trip**，直接打穿核心价值。中文/数据术语命名是平台主力场景，此缺陷影响面接近全量。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 拉取对任意命名无损 (Priority: P1)

开发者拉取一个任务/工作流名为纯中文、中文带连字符、或中英混合的项目，期望本地文件树**完整**呈现服务器上的每一个实体，不因命名而静默丢任何一个。

**Why this priority**: 拉取丢实体是隐蔽的数据丢失，且是后续 diff/push 失真的源头；不修则中文项目根本无法被正确取到本地，tasks-as-code 在主力场景下不成立。

**Independent Test**: 用一个含若干同「类别-中文」命名实体的项目执行一次拉取，比对服务器实体数与本地实体文件数必须相等，且每个服务器实体都能在本地找到唯一对应文件。

**Acceptance Scenarios**:

1. **Given** 服务器某项目含 N 个任务，其中多个名字形如「类别-中文」会落到同一退化 slug，**When** 开发者拉取该项目，**Then** 本地恰好得到 N 个任务定义文件，无一被覆盖丢失。
2. **Given** 两个不同实体的名字会派生出相同的基础 slug，**When** 拉取，**Then** 两者落地为**两个不同文件名**（经确定性消歧），彼此不覆盖。
3. **Given** 一个名字不含任何可用 ASCII 字母数字字符（纯中文/纯标点），**When** 拉取，**Then** 该实体得到一个**确定性、可复现、彼此唯一**的文件名，而非退化为标点。

### User Story 2 - 差异预览忠实、绝不失明 (Priority: P1)

开发者在本地修改（或未修改）后运行差异预览，期望它**忠实**反映本地与服务器的真实差异：零改动必报无差异；任何真实增/改/删必被列出；绝不因命名碰撞把"已经不一致"误报成"一致"。

**Why this priority**: 差异预览是开发者推送前的唯一安全闸；一旦失明，开发者会在错误的"一致"假象下推送，造成不可见的破坏。失真的安全闸比没有更危险。

**Independent Test**: 对刚拉取、零改动的工作副本运行差异预览，必须报告无差异；删除其中一个实体文件后再预览，必须恰好报告该实体为"删除"。

**Acceptance Scenarios**:

1. **Given** 一份刚拉取、未做任何改动的工作副本（含会碰撞的中文命名实体），**When** 运行差异预览，**Then** 报告**无差异**，且本次拉取确实未丢任何实体（与 US1 一致）。
2. **Given** 开发者在本地删除一个实体文件，**When** 运行差异预览，**Then** 恰好该实体被列为"删除"，其余实体不受影响、不被误列。
3. **Given** 开发者改了某实体内容，**When** 运行差异预览，**Then** 恰好该实体被列为"修改"，且不影响同目录其它同源命名实体的判定。

### User Story 3 - 拉取→推送→再拉取身份稳定 (Priority: P2)

开发者拉取、（可不改或改动后）推送、再拉取到一个干净目录，期望两次本地文件树**语义等价**，身份稳定，不产生虚假的"删除/新增"，不误触删除守卫。

**Why this priority**: round-trip 闭合是 tasks-as-code 的完整性证明；身份不稳会让每次往返都漂移，长期不可用。建立在 US1/US2 之上。

**Independent Test**: 拉取→推送（无改动）→再拉取到干净目录，两次文件树（实体集合、每实体内容）语义相等；过程对在线工作流引用的实体不产生误删。

**Acceptance Scenarios**:

1. **Given** 含中文命名实体的项目，**When** 拉取→无改动推送→再拉取到干净目录，**Then** 两次文件树语义等价，无虚假增删。
2. **Given** 工作副本忠实包含某在线工作流的全部引用节点，**When** 无关改动后推送，**Then** 不因身份碰撞把引用节点误判为删除、不误触删除守卫。

### User Story 4 - CLI 按 code 拉取可用 (Priority: P3，已实现并入本特性收口)

开发者用项目 code（而非数字 id）拉取项目，期望命令正常解析并拉取。

**Why this priority**: 同在闭环真跑中发现的往返保真相关缝，1 行级修复、低风险，随本特性收口；不阻塞 P1/P2。

**Independent Test**: 对一个已知 code 的项目执行按 code 拉取，命令成功并拉到正确项目。

**Acceptance Scenarios**:

1. **Given** 服务器存在 code 为 `demo` 的项目，**When** 开发者用 code `demo` 拉取，**Then** 成功解析到该项目并完成拉取（此前永远失败）。

### Edge Cases

- 名字仅由标点/连字符构成（如 `---`）：必须得到确定性、唯一、合法的文件名，不得为空或纯标点。
- 同一目录下 3 个及以上实体派生同一基础 slug：全部互不覆盖、各有稳定唯一文件名。
- 名字含大小写仅大小写不同（如 `ETL` 与 `etl`）：沿用既有大小写碰撞守卫，不得静默坍缩。
- 工作流与任务、标签、目录之间的 slug：本特性的唯一性/消歧规则需在所有以 slug 为身份的实体类型上**一致**适用，避免某一类未覆盖。
- 一个项目里同时存在"会碰撞的中文名"和"正常 ASCII 名"：正常名不应因修复被改写成更难读的形式（避免无谓扰动既有可读文件名）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在拉取时为每一个服务器实体生成**唯一**的本地文件路径——同一目录下任意两个实体不得映射到同一文件路径，杜绝同名覆盖导致的实体丢失。
- **FR-002**: slug 的"退化"判定 MUST 基于"是否含可用区分字符（ASCII 字母或数字）"，而非"是否为空串"；不含任何 ASCII 字母数字的名字 MUST 回落到确定性、可复现的命名方案。
- **FR-003**: 当两个实体在同一目录派生出相同基础 slug 时，系统 MUST 以**确定性**方式消歧（同一输入永远产出同一结果，跨多次拉取稳定），使其落地为不同文件名。
- **FR-004**: 用于身份匹配（拉取装配、差异比对、推送对账）的 slug/路径派生，MUST 与导出（拉取落地文件名）使用**同一套规则**，保证导出侧与匹配侧不产生身份漂移。
- **FR-005**: 差异预览 MUST 忠实：对零改动工作副本报告无差异；对真实的增/改/删恰好报告对应项；任何情形下都不得因命名碰撞把"不一致"误报为"一致"。
- **FR-006**: 拉取 MUST 满足：服务器某项目实体数 = 本地落地的对应实体文件数；每个服务器实体在本地有且仅有一个对应文件。
- **FR-007**: 拉取→推送→再拉取（到干净目录）MUST 身份稳定：两次文件树语义等价，不产生由身份漂移引起的虚假增/删，不因碰撞误触删除守卫。
- **FR-008**: 文件名命名方案 MUST 可移植（限定为各文件系统通用的安全字符集），且对中文/数据术语命名场景为**一等支持**（不得要求用户改名以规避缺陷）。
- **FR-009**: 命名/消歧规则 MUST 在所有以 slug 作为身份的实体类型（任务、工作流、目录、标签等）上一致适用。
- **FR-010**: 本特性 MUST 修复 CLI 按 code 拉取的契约缝——客户端解析项目列表时读取与服务器分页响应**一致**的字段键；并修正会掩盖此缝的测试夹具（夹具须照搬真服务器契约，不得自证错契约）。
- **FR-011**: 非退化、无碰撞的可读 ASCII 名 MUST 保持其现有可读文件名不被本修复改写（最小扰动，避免既有副本无谓 diff）。
- **FR-012**: 命名消歧 MUST 不依赖运行期随机性或时钟，仅依赖实体的稳定属性（名字、稳定标识等），以保证跨进程、跨时间可复现。

### Key Entities *(include if feature involves data)*

- **实体 slug / 文件名**：实体（任务/工作流/目录/标签）在本地文件树中的人类可读标识，由实体名派生，承担"文件路径"与"往返身份键"双重职责。关键属性：可移植字符集、确定性、同目录唯一、跨拉取稳定、对非 ASCII 名健壮。
- **本地工作副本 ↔ 服务器项目 的身份映射**：决定"本地某文件 = 服务器某实体"的对应关系；其正确性直接决定拉取无损、差异忠实、推送不误删。
- **项目同步包（bundle）**：拉取/推送在客户端与服务器之间传输的文件树表示；其文件路径集合必须与服务器实体集合一一对应、无坍缩。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对任意命名构成的项目，拉取后"服务器实体数 = 本地实体文件数"成立率 **100%**（含纯中文、中文带连字符、混合、纯标点名）——零静默丢失。
- **SC-002**: 拉取→（无改动）推送→再拉取到干净目录，两次文件树语义等价率 **100%**。
- **SC-003**: 对刚拉取、零改动的工作副本运行差异预览，报告"无差异"的同时**确实**未丢任何实体——差异失明案例数 **0**。
- **SC-004**: 同一目录下任意多个实体派生相同基础 slug 时，落地文件名两两不同（碰撞率 **0**），且重复拉取得到**完全相同**的文件名集合（确定性 100%）。
- **SC-005**: 含中文命名节点的在线工作流，其引用任务在拉取后**全部**可在本地定位，推送时**无一**被误判删除。
- **SC-006**: 按 code 拉取对存在该 code 的项目成功率 **100%**（修复前为 0%）。

## Assumptions

- 系统处于发布前阶段，无已落库的真实用户项目需迁移；现有服务器数据为种子数据，文件名/身份在本修复后变化是可接受的（不需要历史 slug 兼容层）。
- 本特性聚焦"命名→身份"的正确性与往返保真，不引入新的实体类型、不改变同步 API 的传输形态与端点契约（CLI 字段键修正属对齐既有契约，非新增）。
- 本地 runtime 真跑（`dw run`）、占位符解析、调度执行等子系统不在本特性范围内，已在闭环真跑中验证健康。
- 涉及红线包 `filecontract` / `ProjectSyncService` 的修改已获用户解禁授权（本特性专项）。
- 修复须以"必证伪式"测试覆盖：用真正会碰撞的中文命名夹具断言拉取无损、差异忠实、往返稳定；不得用自证正确契约的 mock 掩盖真缝。

## Clarifications

### Session 2026-06-28

- **Q: slug 退化或需消歧时，文件名采用哪种方案（纯哈希 / ASCII 残基+哈希 / 拼音转写）？** → **A: 纯确定性哈希**。退化（不含 ASCII 字母数字）或同目录撞名的实体，文件名回落为名字派生的确定性哈希（形如 `e<hash>`，与现有空串回落方案一致；该 bundle 中本就已存在此类哈希命名文件，故与现状惯例一致）。不引入 ASCII 残基拼接，不引入拼音转写（避免转写依赖与多音字/一致性复杂度、避免 scope 扩大）。可读性由文件内 `name:` 字段承担。

# Feature Specification: Incident 域模型 + 监督席队列

**Feature Branch**: `043-incident-queue`

**Created**: 2026-07-03

**Status**: Draft

**Input**: User description: "立项——方向文档 §10 第二步：incident 一等领域对象 + 事件流接入 + 监督席队列 UI。按已确认的设计思路：无 agent 参与的确定性分诊工单系统，接入 TASK_FAILED / TASK_TIMEOUT / SLA_BREACH / NODE_OFFLINE 四类信号，工单带爆炸半径与时间预算，队列成为产品主页；为后续多 agent 编队（§10 第三步）预留诊断/提案槽位。"

## 背景与定位

依据方向文档（`docs/superpowers/specs/2026-07-03-weft-ai-ops-direction-design.md`）§4.2/§5/§10：

- **Incident 是一等领域对象**：证据、诊断、决策、提案、审批、结果、复盘全部挂在工单上；它同时是监督席上的一张卡片——未来 agent 的工作空间与人的审查单位是同一个东西。
- **排序即分诊**：本特性与"更好看的告警列表"的本质区别在于每张工单自带确定性分诊信息（影响多少下游 × 距最近下游 SLA 还有多久），队列按紧迫度排序而非时间排序。
- **先造桌子，编队后坐**：本期不含任何 AI/agent 能力（那是 §10 第三步）。agent 将来要写的字段（诊断结论、处置提案）在本期就是可空槽位，编队落地时纯增量写入。
- 现有信号体系（瞬时告警信号 → 健康事件去重流 → 告警规则事件）**没有一层是可受理、可闭环、可聚合多信号的工单**——incident 补的就是这一层。它不替换既有层，与之并行消费同一条信号流。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 故障即工单：失败自动开单、自动愈合、主页可见 (Priority: P1)

开发者的某个任务在夜间调度中失败。系统自动为这次故障开出一张 incident 工单，出现在产品主页的监督席队列中：卡片用人话描述"什么坏了"（任务名 + 故障类别 + 发生次数）。同一任务反复失败不会刷屏——重复故障附着在同一张工单上累加计数。当该任务后来（无论人工重跑还是调度重试）成功了，工单自动转为"已解决"，队列不积压尸体。开发者打开 Weft 首先看到的就是这条队列，而不是各种 dashboard。

**Why this priority**: 这是 incident 作为一等对象的最小成立形态——没有它，后续分诊、处置、编队都无处安放。仅这一条落地，产品主页就从"dashboard 集合"变成"异常队列"，方向文档 §5 的核心承诺即兑现第一步。

**Independent Test**: 制造一次任务失败（或注入失败信号），验证队列中出现对应卡片；再次制造同签名失败，验证不出新卡、计数 +1；让该任务成功，验证卡片自动转已解决。全程不依赖其他 User Story。

**Acceptance Scenarios**:

1. **Given** 一个在线调度的任务，**When** 其实例以失败/超时终态结束，**Then** 监督席队列在短时间内出现一张对应工单卡片，状态为"待处置"，标题含任务名与故障类别。
2. **Given** 一张未关闭的工单，**When** 同一来源发生同类故障，**Then** 不产生新卡片，原工单发生次数 +1、最近发生时间更新。
3. **Given** 一张"待处置/处置中"的工单，**When** 其来源任务的新实例成功结束，**Then** 工单自动转为"已解决"，无需人工操作。
4. **Given** 用户登录 Weft，**When** 打开产品首页，**Then** 第一个看到的常驻视图是监督席队列。
5. **Given** 一张已关闭的工单，**When** 同签名故障再次发生，**Then** 开出一张新工单（不复活旧单）。

---

### User Story 2 - 分诊：爆炸半径 × 时间预算，队列按紧迫度排序 (Priority: P2)

开发者面对多张同时打开的工单，需要立刻知道"先救哪个"。每张卡片展示两项自动推导的分诊信息：**爆炸半径**（这次故障影响多少下游任务）和**时间预算**（距最近的下游 SLA 期限还有多久，倒计时呈现）。队列默认按紧迫度排序——时间预算越紧、爆炸半径越大、严重度越高的排前面——而不是按发生时间排序。

**Why this priority**: 分诊是方向文档定义的决策核心（"把损失降到最低，比修好任务大"）。没有分诊字段的队列只是更好看的告警列表；但它依赖 US1 的工单先存在，故列 P2。

**Independent Test**: 构造一个有多个下游依赖且下游配置了 SLA 的任务，令其失败；验证卡片展示正确的下游数量与最近 SLA 期限；构造两张紧迫度不同的工单，验证队列排序符合紧迫度规则。

**Acceptance Scenarios**:

1. **Given** 一个失败任务存在 N 个下游消费任务，**When** 其工单生成，**Then** 卡片展示"影响 N 个下游"。
2. **Given** 失败任务的下游中存在配置了 SLA 的任务，**When** 查看卡片，**Then** 展示距最近 SLA 期限的剩余时间，且随时间推移递减。
3. **Given** 队列中存在多张打开的工单，**When** 查看队列，**Then** 排序符合紧迫度规则（时间预算紧 > 爆炸半径大 > 严重度高），并列时按最近发生时间。
4. **Given** 一个无下游、无 SLA 关联的失败任务，**When** 其工单生成，**Then** 分诊区显示明确的"无下游影响/无 SLA 约束"缺省态，不显示空白或错误。

---

### User Story 3 - 处置闭环：卡内一键处置、审批内联、全程留痕 (Priority: P3)

开发者在卡片上直接发起处置：一键重跑失败实例（走既有策略闸门——低风险自动执行，高风险生成审批单）。若该工单已有待审批动作，卡片内联展示并可直接批准/驳回。所有发生过的事——原始信号、状态流转、发起的动作、审批结果、人工备注——按时间线记录在工单上，事后可完整追溯"发生了什么、谁做了什么"。对已知问题（如上游供应商已通报的故障），开发者可将工单静默并注明原因，静默的工单退出队列主视图。卡片可下钻到实例日志、血缘、运维视图查看证据。

**Why this priority**: 处置能力让监督席从"看"到"管"，并验证"动作全部走既有闸门、回挂工单"的关键接缝——这正是未来 agent 提案审批的同一条通路。依赖 US1 的工单与时间线，故列 P3。

**Acceptance Scenarios**:

1. **Given** 一张待处置工单，**When** 在卡片上发起重跑，**Then** 动作经既有策略闸门裁决（自动执行或生成待审批），且无论结果如何都作为一条时间线记录回挂工单，工单转"处置中"。
2. **Given** 工单关联了一个待审批动作，**When** 在卡片内批准，**Then** 动作被执行、审批结果记入时间线；驳回同理。
3. **Given** 处置中的工单，**When** 重跑后任务再次失败，**Then** 工单回到"待处置"，计数累加。
4. **Given** 任意一张工单，**When** 展开其时间线，**Then** 能看到从首个信号到当前的全部事件（信号/状态变化/动作/审批/备注），按时间顺序、含操作者。
5. **Given** 一张工单，**When** 将其静默并填写原因，**Then** 工单转"已静默"、退出队列默认视图；静默期间同签名故障仅累加计数不恢复排序。
6. **Given** 一张工单卡片，**When** 点击其来源引用，**Then** 跳转到对应的实例日志/运维视图等证据页。

### Edge Cases

- **信号风暴**：上游大面积故障导致短时间内大量信号涌入——同签名去重 + 同工作流实例内归并（同一工作流实例内的多个任务失败归并为一张工单；该实例引发的 SLA 破约信号归并入已开工单），防止队列爆炸。
- **归并窗口边界**：SLA 破约信号先于任务失败信号到达（或无对应失败工单）时，独立开单，不丢失。
- **分诊信息不可得**：无下游、无 SLA 配置、血缘尚未推导时，卡片显示明确缺省态，排序按剩余可用维度降级。
- **静默过期/误静默**：静默的工单可手动恢复为待处置。
- **已解决后复发**：自动愈合后的工单不复活；同签名新故障开新单，新单可见"历史同签名工单"关联提示（为经验库预热）。
- **服务重启窗口**：信号传递为尽力而为，重启瞬间的信号可能不生成工单——以下一次故障重新触发为准，不做历史回放补偿。
- **手动操作与自动愈合竞争**：人正在处置时任务恰好自愈成功——自动愈合优先，工单直接转已解决，动作记录保留。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**（信号接入）: 系统 MUST 自动消费任务失败、任务超时、SLA 破约、节点离线四类既有信号并生成/更新 incident 工单；指标破约与质量失败信号本期 MUST NOT 接入（待契约转型后另行立项）。
- **FR-002**（故障签名与去重）: 每张工单 MUST 携带可检索的故障签名（由来源对象 + 故障类别构成）；同签名故障在存在未关闭工单时 MUST 附着到该工单（计数累加、更新最近发生时间），不开新单。签名 MUST 设计为未来经验库（故障签名 → 诊断 → 动作 → 效果）的关联键。
- **FR-003**（归并）: 同一工作流实例范围内的多个任务失败信号与由其引发的 SLA 破约信号 MUST 归并为一张工单；跨工作流的血缘级关联本期 MUST NOT 实现（属编队影响评估范围）。
- **FR-004**（生命周期）: 工单 MUST 具备清晰的生命周期：待处置 → 处置中 → 已解决 → 已关闭，另有已静默态；所有状态流转 MUST 留有审计记录（时间、触发者/触发原因）。
- **FR-005**（自动愈合）: 来源对象恢复（如失败任务的新实例成功）时，其未关闭工单 MUST 自动转为已解决，无需人工干预。
- **FR-006**（时间线）: 每张工单 MUST 维护一条只增不改的时间线，记录原始信号、状态变化、处置动作、审批结果、人工备注等全部事件，含操作者与时间。
- **FR-007**（分诊字段）: 工单 MUST 自动携带确定性分诊信息：爆炸半径（受影响下游任务数，由既有任务-表读写关系推导）与时间预算（最近下游 SLA 期限）；不可得时 MUST 呈现明确缺省态。
- **FR-008**（监督席队列）: 系统 MUST 提供监督席队列视图并将其设为产品主页的首个常驻视图；队列默认按紧迫度（时间预算 × 爆炸半径 × 严重度）排序；卡片 MUST 展示标题、严重度、状态、发生次数、分诊信息。
- **FR-009**（处置走闸门）: 从工单发起的一切处置动作（本期为实例重跑）MUST 经既有策略闸门裁决与审计，MUST NOT 新建旁路；动作及其结果 MUST 回挂工单时间线。
- **FR-010**（审批内联）: 工单关联的待审批动作 MUST 可在卡片内查看并批准/驳回，权限沿用既有审批授权规则。
- **FR-011**（静默）: 用户 MUST 能将工单静默（附原因）与恢复；静默工单退出队列默认视图，期间同签名故障仅累加计数。
- **FR-012**（证据下钻）: 卡片 MUST 提供到来源对象既有视图（实例日志、运维、血缘等）的深链，复用既有引用跳转体系。
- **FR-013**（编队槽位预留）: 工单 MUST 预留诊断结论与处置提案两个可空槽位，本期不写入；队列卡片对空槽位呈现占位态，编队落地时无需结构变更即可增量写入。
- **FR-014**（既有面不动）: 本期 MUST NOT 移除或改动既有告警中心与事件中心视图及其后端行为（收敛为 incident 队列另行立项）；既有信号消费链路 MUST 不受新增消费方影响。

### Key Entities

- **Incident（工单）**: 一次故障的受理与闭环单位。属性：故障签名、标题、严重度、状态、来源引用（种类/ID/可读名）、发生次数、首次/最近发生时间、爆炸半径、时间预算、解决方式、解决/关闭时间；预留槽位：诊断结论、处置提案。租户与项目隔离。
- **Incident 时间线条目**: 工单下只增不改的事件记录。属性：所属工单、序号、事件种类（信号/状态变化/动作/审批/备注/agent 结论〔预留〕）、内容、操作者、时间。
- **故障签名**: 标识"同一类故障"的可检索键，由来源对象与故障类别构成；是工单去重的依据，也是未来经验库的关联键。
- **处置动作（既有对象的关联）**: 既有闸门动作记录与工单建立关联，使审批与执行结果可回挂工单，不新建动作模型。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 任务失败后 30 秒内，监督席队列出现对应工单卡片。
- **SC-002**: 同一任务连续失败 5 次，队列中只有 1 张对应卡片且发生次数为 5。
- **SC-003**: 失败任务恢复成功后 30 秒内，其工单自动转为已解决，全程零人工操作。
- **SC-004**: 用户打开产品首页，无需任何点击即看到监督席队列。
- **SC-005**: 从看到卡片到完成一次重跑处置（含闸门自动放行场景）不超过 3 次点击，全程不离开队列视图。
- **SC-006**: 对有下游且下游配置 SLA 的故障任务，卡片 100% 展示下游数量与最近 SLA 剩余时间。
- **SC-007**: 任意工单的完整历史（信号、状态流转、动作、审批）可在卡片内追溯，无一遗漏（与既有审计记录一致）。
- **SC-008**: 一次涉及同工作流实例内 10 个任务失败 + 相应 SLA 破约的故障风暴，队列中新增卡片不超过 1 张。

## Assumptions

- 本期无任何 AI/agent 参与；诊断与提案槽位留空，仅保证结构就绪（方向文档 §10 第三步再启用）。
- 指标破约（METRIC_BREACH）与质量失败（QUALITY_FAILED）信号不接入，待质量体系转型为数据契约（§10 第五步）后再进，避免旧规则噪声灌入新主页。
- 告警中心与事件中心视图保持现状；两者向 incident 队列的收敛是队列站稳后的独立立项。
- 通知触达（钉钉/邮件等）沿用既有事件订阅与告警通道机制，本期不为 incident 新建通知能力。
- 信号传递沿用既有进程内机制，可靠性以现状为准；服务重启窗口内的信号丢失可容忍，不做回放补偿。
- 监督席队列成为主页首个常驻视图已获方向层确认（方向文档 §5，用户已批准）；既有常驻视图（数据新鲜度、系统指标）保留但退居其后。
- 血缘/任务-表读写关系与 SLA 配置为既有能力，分诊信息直接复用其推导结果，不新建元数据。

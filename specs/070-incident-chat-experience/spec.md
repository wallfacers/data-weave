# Feature Specification: 监督席对话体验企业级打磨

**Feature Branch**: `070-incident-chat-experience`

**Created**: 2026-07-14

**Status**: Draft

**Input**: User description: "监督席页面与 demo 无异——边距、审美、多轮对话、聊天体验需达到企业级质感；AI 输入框/对话框/布局参考 workhorse-assistant 的交互模式。设计已逐节确认：docs/superpowers/specs/2026-07-14-incident-console-enterprise-ui-design.md（本特性为两期中的第一期；多人聊天室协作划归 071）"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 可读的 Agent 回复 (Priority: P1)

值班工程师在监督席下钻一个事故线程，Agent 的诊断回复（含根因分析、修复建议、代码/SQL 片段）以结构化富文本呈现：标题、列表、加粗、行内代码清晰分层，代码块带语法高亮和一键复制；回复在流式输出过程中逐步渲染，不闪烁、不跳版式。

**Why this priority**: Agent 诊断是监督席的核心产出物。当前回复以纯文本裸排，长诊断不可读，是「demo 感」的最直接来源；此项独立落地即显著提升日常使用价值。

**Independent Test**: 触发一个会产生含代码块、列表的 Agent 回复的事故对话，观察流式渲染全程与最终版式；点击代码块复制按钮验证剪贴板内容。

**Acceptance Scenarios**:

1. **Given** 事故线程中 Agent 正在流式回复含 Markdown 结构的内容, **When** 内容逐段到达, **Then** 已到达部分即时按富文本渲染，未闭合的代码块不破坏版式，流式结束后版式与一次性渲染结果一致
2. **Given** Agent 回复中含代码块, **When** 用户点击代码块的复制按钮, **Then** 代码原文进入剪贴板且按钮显示约 2 秒的成功确认态
3. **Given** 某条消息的富文本内容异常导致渲染失败, **When** 该消息渲染, **Then** 仅该条消息显示降级提示，线程其余消息不受影响
4. **Given** 用户切换明暗主题, **When** 查看含代码块的回复, **Then** 代码高亮配色随主题正确切换

---

### User Story 2 - 加载与连接状态真实可信 (Priority: P1)

值班工程师打开或刷新监督席时，页面明确区分「正在加载」「连接已断开」「确无事故」三种状态：数据未到达前显示居中加载指示，连接中断时线程顶部有明确提示且已有消息不丢失，只有确认连接正常且确实没有事故时才显示"暂无事故"。

**Why this priority**: 当前首帧数据未到达时直接显示"暂无事故，系统运行正常"，在连接抖动时向值班人员传递虚假的安全信号——这是监控类产品不可接受的可信性缺陷。

**Independent Test**: 冷启动打开监督席观察首帧前状态；人为中断实时连接观察降级提示与消息保留；恢复连接观察自动回到正常态。

**Acceptance Scenarios**:

1. **Given** 监督席刚打开、首帧数据尚未到达, **When** 页面渲染, **Then** 事故列表与战况横幅区域显示居中加载指示（最短展示约 1 秒防闪烁），不显示"暂无事故"
2. **Given** 实时连接中断, **When** 用户停留在事故线程, **Then** 线程顶部出现"连接已断开，重连中"提示条，已加载的消息全部保留
3. **Given** 连接正常且事故列表确为空, **When** 页面渲染, **Then** 显示"暂无事故"空态

---

### User Story 3 - 顺手的发言与打断 (Priority: P2)

值班工程师在事故线程中向 Agent 发言：输入框随内容自动长高（到上限后内部滚动），Enter 发送、Shift+Enter 换行，中文输入法组字期间回车不误发；Agent 正在长篇输出时发送键变为停止键，一键打断本轮输出；发送失败时输入内容保留不丢。

**Why this priority**: 发言是多轮对话的高频动作，输入框的细节（尤其中文输入法保护、打断能力）直接决定"企业级工具"与"demo"的手感差距；依赖 US1/US2 之外可独立交付。

**Independent Test**: 在事故线程中用中文输入法输入并组字回车、多行长文输入、Agent 输出中途点击停止、模拟发送失败，逐项验证行为。

**Acceptance Scenarios**:

1. **Given** 用户使用中文输入法正在组字, **When** 按 Enter 确认候选词, **Then** 消息不被发送，候选词正常上屏
2. **Given** 输入内容超过一行, **When** 继续输入, **Then** 输入框自动长高至上限（约 8 行）后转为内部滚动
3. **Given** Agent 正在流式输出, **When** 用户点击停止键, **Then** 本轮 Agent 输出终止，界面回到可发言状态，且该打断动作留有审计记录、无需人工审批等待
4. **Given** 消息发送请求失败, **When** 错误提示出现, **Then** 提示内容来自服务端本地化消息，输入框中的原文完整保留
5. **Given** 打断请求失败或超时, **When** 停止键点击后无响应, **Then** 停止键回弹为可再次点击并出现错误提示

---

### User Story 4 - 认得出"谁说的、何时说的" (Priority: P2)

多名值班人员先后在同一事故线程发言时，每条人类消息显示真实用户姓名与首字母头像，自己的消息与他人的消息视觉可区分，Agent 消息有专属标识；消息悬停可见精确时间，跨日有日期分隔，同一发言者短时间内的连续消息合并分组展示。

**Why this priority**: 当前发言者硬编码为假名（"ui-user"），消息无时间戳无头像——协作场景下无法归属责任与还原时序。本故事同时是 071 多人聊天室（presence/@提及/未读）的身份地基。

**Independent Test**: 以两个不同登录用户先后在同一事故线程发言，检查双方视角下的姓名、头像、对齐方向；检查悬停时间、跨日分隔与 5 分钟内连续消息的分组。

**Acceptance Scenarios**:

1. **Given** 用户 A 已登录并在事故线程发言, **When** 消息展示, **Then** 消息归属为 A 的真实姓名（来自登录凭证，服务端认定，前端无法伪造），A 自己视角右对齐、他人视角左对齐并带首字母头像
2. **Given** 线程中存在改造前遗留的无归属消息, **When** 历史展示, **Then** 该类消息显示中性兜底称谓（如"操作员"），不显示 "ui-user"
3. **Given** 同一发言者在 5 分钟内连续发送多条消息, **When** 线程展示, **Then** 消息合并为一组，仅首条显示头像与姓名
4. **Given** 线程消息跨越自然日, **When** 浏览历史, **Then** 日期变更处出现日期分隔标识；任意消息悬停可见精确时间
5. **Given** Agent 消息展示完成, **When** 用户悬停该消息, **Then** 浮现"复制原文"操作，点击后有约 2 秒成功确认态

---

### User Story 5 - 稳定的阅读视口与可调布局 (Priority: P3)

值班工程师在 Agent 持续输出时向上滚动查看历史，视口停在原处不被强制拉回；停止跟随后出现"回到底部"按钮；事故列表与线程详情的分栏宽度可拖拽调节且下次打开保持。

**Why this priority**: 滚动抖动与固定分栏是使用一段时间后才暴露的质感短板，价值真实但紧急度低于前四项。

**Independent Test**: Agent 长输出期间上滑阅读、点击回底按钮、拖拽分栏并刷新页面验证宽度保持。

**Acceptance Scenarios**:

1. **Given** Agent 正在流式输出且用户位于线程底部, **When** 新内容到达, **Then** 视口平滑跟随到最新内容
2. **Given** 用户主动向上滚动离开底部, **When** 新内容继续到达, **Then** 视口保持在用户所在位置，同时出现"回到底部"按钮；点击后恢复跟随
3. **Given** 用户拖拽事故列表与线程的分栏, **When** 刷新或重新打开监督席, **Then** 分栏宽度保持用户上次调整值，且列表面板不能被压缩到最小可用宽度以下

---

### Edge Cases

- 实时连接断开重连期间到达的消息：重连后通过历史补齐机制恢复，线程不出现重复或缺失（沿用既有按序去重）。
- 存量历史消息发言者为空或为遗留假名：统一显示中性兜底称谓，不阻塞渲染。
- 打断请求到达时本轮输出恰好自然结束：界面以最终落库消息为准，不出现"已打断"与完整回复并存的矛盾态。
- 单条消息富文本渲染崩溃：仅该条降级，线程与页面其余部分不受影响。
- 分栏被拖到极限：列表面板保有最小可用宽度，不会被压没。
- 明暗主题切换瞬间存在流式中的代码块：高亮主题即时切换，无残留错色。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须区分并呈现三种页面数据状态：加载中（首帧未达）、连接降级（已断开重连中）、正常（含真空态）；"暂无事故"空态只允许在连接正常且列表确为空时出现。
- **FR-002**: 加载指示必须使用平台统一的居中加载组件，具备防闪烁的最短展示时长；连接降级提示不得清空已加载消息。
- **FR-003**: Agent 消息、Agent 步骤、修复提案正文与战况横幅报告必须以富文本（标题/列表/强调/行内代码/代码块）渲染，并支持流式过程中的渐进渲染，未闭合结构不破坏版式。
- **FR-004**: 代码块必须提供语法高亮（跟随明暗主题）、语言标签与一键复制；复制成功须有短暂确认反馈。
- **FR-005**: 单条消息的富文本渲染失败必须被隔离降级，不影响线程其余消息与页面。
- **FR-006**: 发言输入框必须支持随内容自动增高至上限后内部滚动、Enter 发送、Shift+Enter 换行，且输入法组字期间的 Enter 不触发发送。
- **FR-007**: Agent 流式输出期间，发送控件必须切换为停止控件；触发停止即请求打断本轮 Agent 输出；打断失败或超时时控件回弹可重试并给出提示。
- **FR-008**: 打断 Agent 输出属于写操作，必须经平台统一写闸门并留审计记录，同时其策略等级必须配置为免人工审批直接执行（否则停止能力失效）。
- **FR-009**: 消息发送失败时，错误提示必须直接使用服务端返回的本地化消息，且用户已输入内容必须保留。
- **FR-010**: 人类消息的发言者身份必须由服务端依据登录凭证认定并随消息持久化（用户标识+显示名），不得信任客户端自报身份；改造后不得再产生假名发言者。
- **FR-011**: 消息展示必须区分三类主体：本人（右对齐）、他人（左对齐+首字母头像+姓名）、Agent（专属标识）；存量无归属消息显示中性兜底称谓。
- **FR-012**: 线程必须提供时间可读性：消息悬停显示精确时间、跨日插入日期分隔、同一发言者 5 分钟内连续消息合并分组（仅首条带头像与姓名）。
- **FR-013**: Agent 消息完成后必须提供悬停操作：复制消息原文，带约 2 秒成功确认态，重复点击幂等。
- **FR-014**: 线程视口必须实现跟随式滚动：位于底部时平滑跟随新内容；用户上滑后停止跟随并出现"回到底部"按钮；全程无渲染抖动。
- **FR-015**: 事故列表与线程详情的分栏必须可拖拽调节，列表面板具备最小宽度护栏，宽度偏好本地持久化。
- **FR-016**: 界面构件必须复用平台设计系统的既有组件与设计令牌（卡片、输入、加载态等），消除手写同类原语；整体维持平台中性主题，不引入新配色体系。
- **FR-017**: 所有新增界面文案必须同时提供中英双语，两语言键集保持一致。
- **FR-018**: 发送采用服务端回显模型：消息以服务端落库并回流的版本为唯一展示来源，客户端不做乐观插入。

### Key Entities

- **消息（Message）**: 事故线程内的单条记录；关键属性：序号（排序与去重依据）、类型（人类发言/Agent 回复/Agent 步骤/动作/提案/系统）、内容（富文本原文）、发言者标识与显示名、时间。
- **发言者（Actor）**: 消息归属主体；来源于服务端认定的登录身份（用户标识+显示名）或 Agent；存量数据允许为空并以兜底称谓展示。
- **连接状态（Connection Phase）**: 页面级实时通道状态：连接中/正常/降级；驱动加载态、空态与降级提示的展示逻辑。
- **打断请求（Cancel Action）**: 针对某事故当前 Agent 输出轮次的终止指令；具备审计留痕，策略等级为直接执行。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 首帧数据未到达时，用户看到的是加载指示——"暂无事故"在连接未确认前出现的概率为 0。
- **SC-002**: 含代码块的 Agent 回复 100% 以高亮代码块呈现，复制按钮点击后剪贴板内容与原文一致率 100%。
- **SC-003**: 中文输入法组字期间按回车导致的误发送次数为 0。
- **SC-004**: 改造后新产生的人类消息 100% 归属到真实用户显示名，界面上不再出现 "ui-user" 字样。
- **SC-005**: Agent 流式输出期间用户上滑阅读历史时，视口被强制拉回底部的次数为 0。
- **SC-006**: 用户可在 Agent 输出中途打断且打断动作 100% 留有审计记录、0 次进入人工审批等待。
- **SC-007**: 双语文案键集一致性检查、类型检查与既有全量回归保持通过（当前基线 369 例全绿）。
- **SC-008**: 设计合规复查通过：无手写加载文案、无绕过设计系统的手写卡片/输入原语。

## Assumptions

- 事故线程的多轮对话持久化、按序去重、流式直播机制（069 已落地）保持为既有骨架，本特性不重构数据流。
- 用户身份来自平台既有登录体系；服务端可从请求凭证解析出用户标识与显示名。
- 多人协作能力（在线状态、正在输入、@提及、未读、认领/移交、引用回复、附件、置顶）明确划归下一期特性 071，本期仅落地身份地基。
- 发送采用服务端回显（不做乐观插入）是已确认的产品决策，回流延迟在可接受范围（约 200 毫秒内）。
- 视觉基调维持平台既有中性主题；参考对象（workhorse-assistant）仅移植交互模式，不移植其配色。
- 详细技术选型与逐节确认记录见 `docs/superpowers/specs/2026-07-14-incident-console-enterprise-ui-design.md`；技术方案在 plan 阶段展开。

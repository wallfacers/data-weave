# Feature Specification: 虚拟管家监督席(Virtual Butler Companion)

**Feature Branch**: `071-virtual-butler-companion`

**Created**: 2026-07-15

**Status**: Draft

**Input**: User description: "用虚拟人/虚拟管家形象替代传统 Agent 嵌入式对话:一个科幻风格的虚拟智能体负责整个系统的巡检(任务失败、机器状态、数据质量、代码质量),主动完成工作并汇报。布局:中央虚拟人 + 右侧可关闭的汇报卡片(卡片内可继续对话) + 底部对话输入;后续支持语音输入与播报。底层对话大脑评估复用 workhorse-agent;巡检循环参考 Claude loops 的 proactive loop 模式,由平台自身调度承载。成熟后直接替代 070 监督席。原型已验证(tmp/companion-prototype):原创代码生成机器人形象 + 五状态动画 + 卡片栈布局。"

## Clarifications

### Session 2026-07-15

- Q: 管家视图的场景背景如何处理平台主题?(原型为固定深色科幻背景) → A: 背景必须兼容项目当前的亮/暗两种主题——场景随平台主题切换,两种主题下形象、状态色与文案均须可读可辨。
- Q: 汇报卡片的「关闭」作用域是个人级还是项目级? → A: 项目级共享——关闭即团队已阅处置,一人关闭全员消失,与监督席 incident 的项目级处置模型一致。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 管家视图与状态呈现 (Priority: P1)

值班工程师打开「虚拟管家」视图:画面中央是原创机器人管家形象(悬浮、有表情、有状态动画),顶部是巡检概况(今日巡检轮次、待处理异常数、下轮巡检时间)。管家的形态实时反映系统真实健康状态——系统正常时待机/微笑,巡检进行中呈扫视姿态,存在未处理异常时进入警觉形态(变色、锐利眼型)。工程师瞟一眼管家,就知道系统现在"有没有事"。

**Why this priority**: 这是整个特性的存在理由——把"系统状态"从一堆图表压缩成一个有情绪的形象,一眼可读。没有它,后续的汇报和对话无处安放。它本身即是可交付的最小价值:一个映射真实系统状态的"情绪化仪表盘"。

**Independent Test**: 打开管家视图;人为制造一个失败任务实例,观察管家在下一次状态刷新时进入警觉形态、顶部异常计数 +1;清除异常后管家回落待机形态。全程不需要卡片或对话功能。

**Acceptance Scenarios**:

1. **Given** 用户具备监督席访问权限, **When** 从导航打开管家视图, **Then** 3 秒内呈现管家形象与顶部巡检概况,形象带待机动画(呼吸、眨眼、注视跟随)。
2. **Given** 系统存在未处理的失败任务, **When** 管家视图打开或状态推送到达, **Then** 管家呈警觉形态且顶部显示对应异常计数。
3. **Given** 管家处于警觉形态, **When** 所有异常被处理完毕, **Then** 管家在下一次状态更新时回落待机形态,异常计数归零。
4. **Given** 用户无监督席权限, **When** 尝试打开管家视图, **Then** 显示权限不足提示,不渲染任何运行数据。

---

### User Story 2 - 主动巡检与汇报卡片 (Priority: P2)

管家按计划在后台巡检四个领域:任务失败、机器状态、数据质量、代码质量。每轮巡检产出结构化「汇报」,以卡片形式推入视图右侧的汇报栈:卡片标注严重度(危险/警告/正常/信息)、时间与摘要,可单独关闭;有新汇报到达时,管家进入播报形态并用一句话字幕提示要点,右上角「汇报」入口显示未读计数。工程师不在场时汇报也会持续累积,回来后可补看。

**Why this priority**: "主动去完成一些事情"是虚拟管家区别于传统被动问答的核心。巡检-汇报闭环让管家真正承担值班工作,是本特性的主体业务价值。

**Independent Test**: 配置一条巡检例程并触发一轮巡检;验证汇报卡片按严重度正确出现在右侧、可关闭、计数正确;管家同步播报。不依赖对话功能。

**Acceptance Scenarios**:

1. **Given** 巡检例程已启用, **When** 到达计划时间或被手动触发, **Then** 该轮巡检执行并产出至少一张汇报卡片(即使结论是"一切正常")。
2. **Given** 一轮巡检发现异常, **When** 汇报推入视图, **Then** 卡片按严重度标色置顶、管家播报一句话摘要、未读计数增加。
3. **Given** 汇报卡片已显示, **When** 用户点击卡片关闭按钮, **Then** 该卡片移除、计数减一,已关闭的汇报不再自动重现。
4. **Given** 用户离开视图期间产生了多轮汇报, **When** 用户重新打开视图, **Then** 未处理的汇报仍按时间序完整可见。
5. **Given** 某轮巡检执行失败(超时/大脑不可用), **When** 到达汇报时机, **Then** 产出一张"巡检未完成"的信息卡片说明原因,不静默丢失。

---

### User Story 3 - 与管家对话并派活 (Priority: P3)

工程师通过两个入口与管家对话:① 底部全局输入框,直接对管家下达指令("帮我重跑失败的任务""给我一份数据质量周报");② 任一汇报卡片内展开的迷你对话,上下文自动锚定该汇报的主题。管家流式回复(可打断),对涉及系统变更的动作(重跑、改调度、修复)自动走平台审批闸门,并把"已提交审批/已执行/被驳回"的结果回报给用户。管家形态随对话联动:接到指令进入思考形态,回复时进入播报形态。

**Why this priority**: 对话把管家从"汇报机器"升级为"能派活的同事"。依赖 US1 的形象载体与 US2 的汇报上下文,故列 P3;但它是替代 070 监督席对话能力的必要条件。

**Independent Test**: 在底部输入框下达一条查询类指令,验证流式回复与形态联动;在一张汇报卡片内追问,验证回复内容锚定该汇报主题;下达一条写操作指令,验证进入审批流并正确回显 pending 状态。

**Acceptance Scenarios**:

1. **Given** 管家视图已打开, **When** 用户在底部输入框发送指令, **Then** 管家进入思考形态,随后以流式文本回复并进入播报形态,回复渲染为富文本(与监督席同级的可读性)。
2. **Given** 一张汇报卡片处于展开状态, **When** 用户在卡片内提问, **Then** 管家的回答基于该汇报的巡检上下文,而非泛泛作答。
3. **Given** 管家正在流式输出, **When** 用户点击停止, **Then** 输出立即中止且会话可继续。
4. **Given** 用户指令涉及系统变更(如重跑任务), **When** 管家执行该动作, **Then** 动作进入平台审批闸门,管家明确告知当前审批状态,闸门通过前不产生任何实际变更。
5. **Given** 对话大脑服务不可用, **When** 用户发送消息, **Then** 用户收到明确的本地化错误提示(非静默失败、非空白回复)。

---

### User Story 4 - 巡检例程的平台化治理 (Priority: P4)

巡检例程本身是平台上可治理的计划任务:管理员可以按领域启用/停用例程、调整巡检频率与范围;每轮巡检的执行历史、耗时、产出可追溯;巡检动作与普通任务一样受项目隔离与审计约束。

**Why this priority**: 自举(管家的本职工作即平台任务)保证巡检不是黑盒旁路,而是被平台自身的调度、隔离、审计体系治理。价值在长期运维,可最后交付。

**Independent Test**: 停用某领域例程后验证该领域不再产出新汇报;调整频率后验证下轮巡检时间随之变化;查询巡检执行历史可见每轮记录。

**Acceptance Scenarios**:

1. **Given** 管理员停用「代码质量」巡检例程, **When** 下一个计划周期到达, **Then** 该领域不执行巡检、不产出汇报,其余领域不受影响。
2. **Given** 巡检例程执行完成, **When** 查看执行历史, **Then** 可见本轮的触发时间、耗时、结论摘要与产出汇报的关联。
3. **Given** 多项目环境, **When** 项目 A 的巡检执行, **Then** 其读取范围与产出汇报严格限定在项目 A 内。

---

### Edge Cases

- 状态/汇报的实时通道断线:界面明确提示连接状态,已有汇报与消息不清空;重连后补齐离线期间的增量。
- 管家大脑进程不可用:视图仍可打开(形象与既有汇报可见),对话入口给出明确的降级提示;巡检产出"未完成"卡片。
- 一轮巡检产出超量汇报(如批量任务同时失败):同领域汇报聚合为一张卡片(计数 + 明细),不刷屏。
- 用户在管家播报中途发送新指令:播报中止,新指令优先处理。
- 汇报涉及的对象已消失(如任务已被删除):卡片内跳转/操作给出"对象已不存在"的友好提示。
- 审批闸门驳回管家提交的动作:管家明确回报驳回结果与原因,不重复自动重试。
- 多人/多标签页同时打开管家视图:汇报关闭为项目级,任一处关闭后其余打开的视图实时同步移除,未读计数一致(以服务端状态为准)。
- 中文输入法组字中按 Enter:不误发消息(延续监督席已建立的 IME 保护标准)。
- 用户在视图打开状态下切换平台主题:3D 场景与全部 UI 即时跟随,无残留旧主题元素、无需刷新页面。

## Requirements *(mandatory)*

### Functional Requirements

**视图与形象**

- **FR-001**: 系统 MUST 提供「虚拟管家」视图,注册进多标签 Workspace 并纳入导航与权限体系;访问权限与监督席一致。
- **FR-002**: 视图 MUST 以本项目原创的机器人管家形象为中心呈现,形象具备至少五种可区分的形态:待机、巡检中、发现异常、思考中、播报中,形态间切换有动画过渡。
- **FR-003**: 管家形态 MUST 由真实系统状态驱动:存在未处理异常 → 警觉;巡检执行中 → 巡检;对话处理中 → 思考/播报;其余 → 待机。
- **FR-004**: 视图顶部 MUST 展示巡检概况:今日巡检轮次、待处理异常数、下轮巡检时间。
- **FR-005**: 管家形象与全部界面文案 MUST 支持平台双语(zh-CN/en-US),文案 key 双 bundle 齐全;形象本体不含任何内嵌文字。

**巡检与汇报**

- **FR-006**: 系统 MUST 支持四个巡检领域:任务失败、机器状态、数据质量、代码质量;每个领域为独立巡检例程,可单独启停。
- **FR-007**: 巡检例程 MUST 由平台自身的调度体系触发(计划触发 + 手动触发),而非旁路定时器;每轮执行留有可追溯的历史记录。
- **FR-008**: 每轮巡检 MUST 产出结构化汇报:领域、严重度(危险/警告/正常/信息)、标题、摘要、产生时间、关联对象;巡检失败时产出"未完成"汇报而非静默。
- **FR-009**: 汇报 MUST 持久化,并实时推送到已打开的管家视图;用户离线期间的汇报在重新打开视图后完整可见。
- **FR-010**: 汇报卡片 MUST 支持:按时间倒序堆叠、按严重度标识、单独关闭、未读计数;整个汇报栈可收起/展开。关闭为**项目级共享**:任一成员关闭后对项目内全员消失、持久且不自动重现。
- **FR-011**: 新汇报到达时管家 MUST 播报一句话摘要(字幕形式),并切换到对应形态;同领域短时间内的多条异常聚合为一张卡片。

**对话与执行**

- **FR-012**: 用户 MUST 能通过底部全局输入框与管家对话;管家回复为流式富文本,渲染质量不低于监督席既有标准(富文本、代码块、渲染崩溃隔离)。
- **FR-013**: 每张汇报卡片 MUST 可展开上下文对话,对话内容自动锚定该汇报的巡检上下文。
- **FR-014**: 管家流式输出 MUST 可被用户即时打断,打断后会话保持可用。
- **FR-015**: 管家执行的一切系统变更动作 MUST 经过平台统一审批闸门并留审计痕迹,闸门放行前不产生实际变更;管家 MUST 向用户如实回报动作的审批/执行/驳回状态。
- **FR-016**: 对话大脑不可用时,系统 MUST 给出明确的本地化降级提示;视图的形象、概况与既有汇报不受影响。
- **FR-017**: 对话与汇报数据 MUST 遵守项目隔离:用户仅能看到当前项目范围内的汇报与会话。
- **FR-018**: 输入框 MUST 具备与监督席同级的输入体验:自动增高、中文 IME 组字保护、发送-停止状态机。

**演进与替代**

- **FR-019**: 管家视图与 070 监督席在过渡期 MUST 并存互不干扰;管家视图具备替代监督席所需的事故处置入口(查看异常明细、批准/驳回管家提案)或到既有处置界面的直达跳转。
- **FR-020**: 语音输入与语音播报 MUST 在界面上预留入口(以"规划中"呈现),本期不实现其功能。

**主题兼容**

- **FR-021**: 管家视图(含 3D 场景背景、氛围效果、卡片与文案)MUST 兼容平台现有亮/暗两种主题:随平台主题即时切换、无需刷新;两种主题下管家形象轮廓、五种状态色、汇报严重度色与全部文案均保持可读可辨。

### Key Entities

- **巡检例程 (Patrol Routine)**: 某一领域的计划巡检定义——领域、启停状态、频率、范围;它是平台可调度、可治理的任务,拥有执行历史。
- **巡检汇报 (Patrol Report)**: 一轮巡检的结构化产出——领域、严重度、标题、摘要、产生时间、关联对象(任务/机器/表/脚本)、处置状态(未读/已读/已关闭,**项目级共享**,含关闭人与关闭时间)。
- **管家会话 (Companion Conversation)**: 用户与管家的对话流——全局会话或锚定某汇报的上下文会话;消息含发言者真实身份、时间与内容。
- **管家状态 (Companion State)**: 管家当下的形态(待机/巡检/警觉/思考/播报)及其驱动源(异常计数、巡检进行中、对话进行中),是可推送的实时状态。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 打开管家视图后 3 秒内完成首屏呈现(形象 + 概况 + 既有汇报)。
- **SC-002**: 从异常发生到管家形态变化与播报,延迟不超过一个巡检周期 + 30 秒;实时通道在线时状态变化 5 秒内反映到形象。
- **SC-003**: 值班工程师不查看任何图表,仅凭管家形态与顶部概况即可在 5 秒内判断"当前系统是否有事、有几件事"。
- **SC-004**: 用户从看到汇报卡片到就该汇报发出第一条追问,不超过 2 次点击。
- **SC-005**: 对话首个可见回复片段在发送后 5 秒内出现(流式);打断操作 1 秒内生效。
- **SC-006**: 管家发起的系统变更动作 100% 留有审计记录,0 条绕过审批闸门。
- **SC-007**: 巡检汇报零静默丢失:每轮已触发的巡检要么产出正常汇报,要么产出"未完成"说明。
- **SC-008**: 两周试运行内,管家视图承接监督席 ≥80% 的日常值班查看场景(以视图打开次数对比衡量),验证"替代 070"的可行性。

## Assumptions

- **形象资产**: 管家形象采用本项目原创的代码生成机器人(已在原型验证,版权归项目,见 `tmp/companion-prototype/NOTICE.md`),不引入任何第三方 3D 模型/贴图资产;具体成像技术由 plan 阶段决定,原型采用的方案可直接沿用。
- **对话大脑**: 大脑能力由独立的 agent 进程(候选:workhorse-agent,经评估其 HTTP+SSE 多会话、工具调用、MCP 接入能力满足需要)以 sidecar 形态提供;平台服务端不内嵌推理逻辑,仅做编排、治理与通道——与 constitution 原则 IV 的"服务端无 AI 大脑"内核一致,详细合规论证在 plan 阶段 Constitution Check 完成。
- **巡检循环**: agent 进程本身无定时能力,巡检的"proactive loop"由平台既有调度体系触发(参考 Claude loops 的 proactive 模式:计划/事件触发 + 明确停止条件),这是有意的自举设计。
- **权限**: 复用监督席既有的权限模型,不新增独立权限域。
- **交互标准继承**: 070 监督席已打磨的对话交互标准(流式富文本、打断、IME 保护、身份服务端认定、连接状态三态)作为管家对话的既定基线,不在本 spec 重复展开。
- **范围外**: 语音输入/播报的实际功能、多人协作(presence/@提及)、移动端适配、070 监督席的正式下线均不在本期范围;070 下线待 SC-008 验证后另行决策。
- **数据保留**: 巡检汇报与会话记录遵循平台既有的运行数据保留策略,不为本特性单设策略。

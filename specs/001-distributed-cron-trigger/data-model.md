# Phase 1 Data Model: 分布式 Cron 精确触发

实体来自 spec 的 Key Entities + research 的 schema 事实。改动:`workflow_def` 加列、新增 `master_nodes`、`cron_fire` 不变。DDL 须 H2 + PG 双兼容。

## E1. workflow_def（既有表，加列）

承载 cron/周期工作流定义,是扫描的输入源。现有字段见 `schema.sql:301-325`(`schedule_type` / `cron` / `schedule_start` / `schedule_end` / `status` / `last_fire_time` / `priority` …)。

**新增列**:

| 列 | 类型 | 约束 | 含义 |
|----|------|------|------|
| `next_trigger_time` | TIMESTAMP | NULL | 下一次计划触发时间(单值持久化,PowerJob 模型)。扫描以 `next_trigger_time ≤ now + lookahead` 为捞取条件。NULL = 尚未初始化,首轮扫描据 `last_fire_time`/创建时刻回填 |
| `schedule_interval_ms` | BIGINT | NULL | FIXED_RATE / FIXED_DELAY 的间隔(毫秒)。CRON 类型为 NULL |

**`schedule_type` 取值扩展**: 现 `MANUAL` / `CRON` / `DEPENDENCY` → 增 `FIXED_RATE` / `FIXED_DELAY`。

**字段使用规则**:
- `CRON`: 用 `cron`(6 字段含秒)算 `next_trigger_time`;`schedule_interval_ms` 必为 NULL。
- `FIXED_RATE`/`FIXED_DELAY`: 用 `schedule_interval_ms`;`cron` 可为 NULL。
- `status=ONLINE` 才进入扫描;`schedule_start/end` 之外的触发点被跳过;超出生效期 → `next_trigger_time=NULL` 并停止排程。
- **向后兼容**: 既有分钟级 CRON 工作流 `next_trigger_time` 初始为 NULL,首轮扫描据 `last_fire_time` 计算并回填,无需数据迁移。

**索引**: 加 `idx_workflow_def_scan (deleted, schedule_type, status, next_trigger_time)`,支撑「按类型/状态 + next_trigger_time 范围」高频扫描;分片查询追加 `id` 取模过滤(`WHERE deleted=0 AND status='ONLINE' AND schedule_type IN (...) AND next_trigger_time <= ? AND MOD(id, ?) = ?`)。

**状态/生命周期**(就 cron 触发而言):
```
[创建/上线] status=ONLINE, next_trigger_time=NULL
   → 首轮扫描回填 next_trigger_time
   → 到点触发(cron_fire 去重 + 下游 trigger) → 重算 next_trigger_time
   → ... 循环 ...
   → 超出 schedule_end / 下线 → next_trigger_time=NULL,退出扫描
```

## E2. cron_fire（既有表，不变）

去重真相。DDL `schema.sql:567-577`,唯一键 `uq_cron_fire (workflow_id, scheduled_fire_time)`。本特性**不改结构**,继续:到点触发前 `save()`,撞唯一键(`DataIntegrityViolationException`)即放弃。

**新增运维约定**(FR-011):归档/清理保持有界 —— 定期删除 `fired_at < now - retention`(默认保留窗口,如 30 天)的历史记录,避免无界增长拖慢去重 INSERT。清理本身幂等、可由 `@Scheduled` 低频任务执行(与触发路径解耦)。

## E3. master_nodes（新增表，分片用，Batch B）

记录活 master 成员,供哈希分片计算「本机序号 / 活 master 数」。照搬 `worker_nodes` 心跳惯例。

```sql
CREATE TABLE master_nodes (
    id              BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    master_code     VARCHAR(96) NOT NULL,        -- hostname + "-" + pid，进程唯一
    master_uri      VARCHAR(256),                -- http://host:port（可选，便于诊断）
    incarnation     BIGINT,                      -- 进程启动纪元（重启即变）
    status          VARCHAR(32) NOT NULL,        -- ONLINE / OFFLINE
    last_heartbeat  TIMESTAMP,                   -- 心跳时刻；超时(默认30s)判离线
    created_at      TIMESTAMP,
    updated_at      TIMESTAMP,
    CONSTRAINT uq_master_nodes_code UNIQUE (master_code)
);
```

| 列 | 含义 |
|----|------|
| `master_code` | 本机唯一标识,自注册键(upsert) |
| `incarnation` | 启动纪元,用于识别重启 |
| `status` / `last_heartbeat` | 成员存活;`MasterRegistry` 周期续约,超时由任一 master 标记 OFFLINE |

**活集合语义**: 「活 master 列表」= `status=ONLINE AND last_heartbeat > now - threshold`,按 `master_code` 稳定排序得到每个 master 的 `index`;`activeMasterCount = 列表长度`。分片 `shard(workflow_id) = hash(workflow_id) % activeMasterCount`,本 master 预读 `shard == myIndex` 的工作流。集合变化期的不一致由 `cron_fire` 唯一键兜底(见 research D3)。

## E4. 计时策略模型（运行态，无表）

`TimingStrategy` 策略接口 + 3 实现,纯计算,不落新表:

| 策略 | `next(prevScheduledFire, lastCompletion, now)` |
|------|-----------------------------------------------|
| `CRON` | `CronExpression.parse(cron).next(prevScheduledFire ?? now)` |
| `FIXED_RATE` | `prevScheduledFire + interval`(首轮 `创建时刻 + interval`) |
| `FIXED_DELAY` | `lastCompletion + interval`(首轮 `创建时刻 + interval`) |

**misfire 归一**(所有策略):若算得 `next ≤ now`(过期)→ 触发引擎以 delay=0 立即触发该 `next`,随后再 `next(now, …)` 推进到未来最近点;`scheduler.cron-misfire=skip` 时不补、仅推进基准。

## E5. 预读触发窗口（运行态，进程内，无表）

每个 master 本地的 `ScheduledExecutorService` + 已装载点集合。仅含「本分片、未来 lookahead 窗口内」的到期点;不跨进程共享;到点去重仍落 `cron_fire`。崩溃丢失的内存点由下一轮 15s 扫描 + 过期补触发恢复(延迟有界,满足 SC-001/SC-007)。

## E6. 触发观测指标（运行态，Micrometer，无表）

经 `SchedulerMetrics`(现有 instrumentation owner)暴露(FR-014):

| 指标 | 类型 | 含义 |
|------|------|------|
| `dw.cron.trigger.latency` | Timer | cron 时刻 → 实例创建延迟(验证 SC-001/SC-005) |
| `dw.cron.misfire.count` | Counter | 过期补触发次数(验证 SC-003) |
| `dw.cron.window.size` | Gauge | 当前预读窗口装载点数(本 master 本分片) |
| `dw.cron.shard.workflows` | Gauge | 本 master 负责的工作流数(验证 SC-007 单机负载) |

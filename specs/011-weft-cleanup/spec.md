# Feature Specification: Weft 掉头后代码库净化

**Feature Branch**: `011-weft-cleanup`

**Created**: 2026-06-28

**Status**: Draft

**Input**: User description: "项目中由于改造，必然会存在大量的老代码不需要的，你全盘扫描，确认是不需要的可以删除掉，避免上下文污染"

## 背景与动机

Weft 平台经历了一次产品方向掉头：从"服务端 AI 编排平台"转向"任务即代码（Tasks-as-Code）平台"。掉头过程中删除了服务端 AI 编排、内联任务写入工具等大批功能，但清理未彻底，代码库中残留大量"已无调用方、却仍占据空间与注意力"的死代码。这些残留会误导后续维护者（含 AI 编程 agent）误判功能边界、增加理解成本、甚至掩盖潜在缺陷。本特性对此做一次系统化净化。

**受众说明**：与面向业务用户的特性不同，本特性是工程内部的技术债务清理，利益相关者是"在 codebase 上工作的开发者与 AI agent"。故 spec 采用工程视角，但仍遵循 WHAT/WHY 导向；具体文件级清单与删除步骤留待 plan/tasks。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 清除服务端 AI 编排残留链 (Priority: P1)

掉头删除了服务端 AI 编排（AG-UI 聊天、CopilotKit 集成），但"聊天文件附件上传/下载"的整条链路（接口 → 服务 → 存储 → 数据）以及编排器的返回类型仍完整保留在代码库，全仓库无任何调用方。

**Why this priority**: 这是已删除功能最完整的残留——一整条端到端链路悬空。它最易让维护者误以为聊天/AI 能力还在，是上下文污染最严重的来源。

**Independent Test**: 删除该链路后，后端构建通过、配套测试一并移除、全仓库搜索无对该链路任何符号的引用。

**Acceptance Scenarios**:

1. **Given** 聊天附件链路与编排器返回类型已删除，**When** 执行全量后端构建，**Then** 零编译错误。
2. **Given** 上述残留已删除，**When** 全仓库搜索相关符号，**Then** 零命中。
3. **Given** 已删除的 AG-UI 端点，**When** 检查鉴权白名单，**Then** 白名单中不再包含该已删端点的放行条目。

---

### User Story 2 - 清除前端孤儿视图与死文案 (Priority: P1)

掉头后前端改为 Workspace 多标签架构，旧的独立视图组件（实例表、日志侧栏、任务定义列表、任务搜索栏、设置面板、血缘图）、配套的基础 UI 组件、以及 AI 时代遗留的聊天/驾驶舱/诊断/审批/发现等 i18n 文案、已不再使用的图标与 markdown 渲染依赖、聊天专用样式——均已无任何引用。

**Why this priority**: 体量最大的上下文污染源；孤儿 i18n 文案尤其会让维护者以为这些界面仍在产品中。

**Independent Test**: 删除后前端类型检查通过、两种语言文案包键集合一致、全仓库无孤儿导入。

**Acceptance Scenarios**:

1. **Given** 孤儿视图组件与配套 UI 组件已删除，**When** 执行前端类型检查，**Then** 零错误。
2. **Given** AI 遗留 i18n 命名空间已删除，**When** 校验文案包，**Then** 两种语言键集合一致且每个键都有代码引用。
3. **Given** 未使用依赖已从清单移除，**When** 安装依赖并构建，**Then** 无缺失、无残留引用。

---

### User Story 3 - 消除阶段性实现桩与 create_task 残留 (Priority: P2)

早期阶段的 worker 执行桩与正式实现并存，在单进程模式下两者注册同一路由（重复注册）；已永久移除的"任务定义内联创建"工具（create_task）在领域服务里的兼容方法 `createAndOnline` 仍残留，E 重塑后已无生产调用方，仅被测试用作建表夹具。

> **plan 核实 v1→v2 纠正（2026-06-28）**：v1 曾因在过时树（6177ffa，E 之前）核实，误判 `createAndOnline` 为活代码并移出清单；v2 在真 main（e568c38）复验确认其为死代码（E 已把 PROJECT_PUSH 换成 `projectSyncService.push`，`createAndOnline` 无生产调用），恢复入清单。

**Why this priority**: 路由重复注册是潜在运行期缺陷；已删工具的残留方法让维护者误以为这些写入路径仍可用。

**Independent Test**: 删除桩后单进程模式不再有重复路由注册；删除 `createAndOnline` 后生产代码无引用（测试改用现行建表途径）。

**Acceptance Scenarios**:

1. **Given** 执行桩已删除，**When** 单进程模式启动，**Then** 不存在同一路由的重复注册。
2. **Given** `createAndOnline` 已移除，**When** 全仓库搜索生产代码引用，**Then** 零命中。
3. **Given** 测试原依赖被删方法建表，**When** 改用现行建表途径，**Then** 测试仍通过。

---

### User Story 4 - 退役已被取代的工具与空骨架 (Priority: P3)

任务级冻结工具已被节点级冻结取代（仍在写库但调度门不再读取，行为已失效）；告警能力仅为空骨架，全仓库零引用。

**Why this priority**: 退役工具的行为已无效却仍占用工具列表与维护成本；空骨架制造"能力已实现"的假象。优先级低于 P1/P2，因其不主动制造错误，只是冗余。

**Independent Test**: 退役后工具列表不再含被取代项；空骨架移除后构建通过、无残留装配引用。

**Acceptance Scenarios**:

1. **Given** 任务级冻结工具已退役，**When** 查看工具清单，**Then** 不再含该被取代项。
2. **Given** 告警骨架已移除，**When** 全量构建，**Then** 零错误且无对该骨架的装配引用。

---

### User Story 5 - 迁移并移除已废弃的任务查询端点 (Priority: P3)

旧的任务查询端点已标记废弃（运维不再以任务为主体），但仍有前端面板在使用；需先将该面板迁移到现行的工作流查询端点，再移除废弃端点。

**Why this priority**: 消除废弃端点的长期维护负担、统一查询入口。优先级 P3，因涉及前端迁移，工作量大于纯删除。

**Independent Test**: 前端面板改用现行端点后功能不变；废弃端点删除后无引用。

**Acceptance Scenarios**:

1. **Given** 前端面板已迁移到现行工作流查询端点，**When** 使用该面板，**Then** 数据展示与原行为一致。
2. **Given** 废弃端点已删除，**When** 全仓库搜索其引用，**Then** 零命中。

---

### User Story 6 - 归档已落地的规格目录 (Priority: P3)

specs/ 下 001–010 对应的特性均已落地，应移入归档区保留可索引，并同步更新知识库导航中的链接。

**Why this priority**: 让 active 规格区只保留进行中的工作，避免维护者误判还有未完成特性。

**Independent Test**: 归档后 active 区不含已落地特性；导航链接指向新位置。

**Acceptance Scenarios**:

1. **Given** 001–010 已移入归档区，**When** 查看 active 规格区，**Then** 仅含进行中的特性。
2. **Given** 知识库导航链接已更新，**When** 点击链接，**Then** 指向归档区的新位置。

### Edge Cases

- **反射/动态引用风险**: 某项看似零引用，实则被字符串名、反射、配置或动态导入引用——编译能过但运行期才断。**对策**: 静态文本扫描之外，对反射扫描（Spring 组件/JDBC repo）、next/dynamic 动态导入、事件字符串等盲区加针对性验证（启动期装配检查、动态导入追踪）后方可删除（见 FR-014）。
- **测试夹具依赖被删代码**: 测试用被删方法建表/构造数据。**对策**: 同步把测试迁移到现行构造途径，不可只删生产代码。
- **删除引起装配/路由变化**: 移除模块或控制器后 bean 装配与路由表变化。**对策**: 单进程与分布式两种模式都需启动验证。
- **归档规格导致导航链接失效**: 知识库导航多处引用规格路径。**对策**: 归档时同步更新所有链接。
- **跨特性残留**: 本 worktree 基于 main，不含并行收尾中的 D/E 实现；若 D/E 引入了新的可删项，需在实现阶段同步主线后复查。
- **连带配置悬空**: 删代码后 seed/存储路径/policy_rule 可能悬空——删除时须同步检查并清理其直接牵连的配置项（见 FR-011）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: MUST 移除服务端 AI 编排的全部残留——聊天文件附件整条链路（接口/服务/存储/数据）、编排器返回类型、以及已删 AG-UI 端点在鉴权白名单中的放行条目。
- **FR-002**: MUST 移除前端所有零引用的视图组件、配套基础 UI 组件、AI 遗留 i18n 命名空间、未使用的图标与 markdown 渲染依赖、以及聊天专用 CSS 块。
- **FR-003**: MUST 移除早期阶段的 worker 执行桩，消除单进程模式下的路由重复注册。
- **FR-004**（**改判 defer，2026-06-28 统筹裁定**）: `createAndOnline`/`recordLineage`/`buildEdges`（含 A×B 交叉校验）虽生产零调用，但 **暂留不删**——产品定调设计期血缘后期将在任务发布/创建时基于图库（neo4j）重做，此 JDBC 实现为过渡品，随 neo4j 版一并退役；现在删除须为即将重写的子系统迁移/丢弃血缘测试夹具、并丢失 neo4j 版需参考的 A×B 语义，得不偿失。〔历史：v1 在过时树误判撤销 → v2 在 e568c38 确认为死代码恢复“删” → v3 据产品 neo4j 方向改判“defer”〕。**followup**：(a)「建任务即建血缘」在 `push` 真写路径未落血缘（`ProjectSyncService.push` 不调 `recordDesignTimeIo`，受红线阻断，非本特性范围）；(b) 血缘终态 = 发布期 + neo4j 图库。
- **FR-005**: MUST 退役已被节点级冻结取代的任务级冻结工具及其已失效的写库链。
- **FR-006**: MUST 移除零引用的告警能力骨架，包括模块声明与所有装配引用。
- **FR-007**: MUST 将仍在使用废弃任务查询端点的前端面板迁移到现行工作流查询端点，随后移除废弃端点及其领域方法。
- **FR-008**: MUST 将已落地的 specs/001–010 移入归档区，并同步更新知识库导航中所有指向这些路径的链接。
- **FR-009**: 每一批删除后 MUST 通过后端编译、前端类型检查、相关测试，且全仓库无对已删项的残留引用。
- **FR-010**: 删除 MUST NOT 改变任何仍在使用的功能行为（纯净化，零行为回归）。
- **FR-011**: MUST 同步清理代码删除直接导致的悬空配置（seed、存储路径、policy_rule），避免"代码删了配置还在"的半拉子状态；独立的配置注释与杂项不在本次范围。
- **FR-012**: 删除 observability 类孤儿组件（run logs / DAG / lineage 相关）后 MUST 通过浏览器回归验证——run logs 流与 DAG 实例视图功能不回归、lineage 视图 placeholder 状态不被破坏，作为宪法 IV（observability 不可损害）的守护。
- **FR-013**: 死代码判定聚焦生产调用方——生产路径零调用即判死；仅被测试引用的代码亦判死（测试视为可迁移夹具，删时同步迁移到现行构造途径）。
- **FR-014**: 涉及反射扫描（Spring 组件/JDBC repo 扫描）、动态导入（next/dynamic）、事件/字符串引用的删除项，MUST 在静态引用扫描之外加针对性验证（启动期装配检查、动态导入追踪）后方可删除。
- **FR-015**: 退役 freeze_task 前 MUST 验证无外部 MCP 客户端调用（搜索客户端代码、检查调用记录）；若发现仍有调用方，该项暂缓并升级告知，不擅自删除。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 清理完成后，全量后端构建与前端类型检查均零错误（功能零回归）。
- **SC-002**: 清理完成后，代码库中对任何已删项的引用数为零（孤儿引用归零）。
- **SC-003**: 单进程运行模式下不再存在任何路由的重复注册。
- **SC-004**: active 规格区仅保留进行中的特性；001–010 全部进入归档区且导航链接可达。
- **SC-005**: 两种语言的 i18n 文案包键集合一致，且每个键都被代码引用（孤儿文案归零）。
- **SC-006**: 代码层净减少约 2900 行死代码（含主代码、测试与骨架，含恢复的 `createAndOnline` 删除；具体数值在实现期据 `git diff --stat` 核定）。
- **SC-007**: observability 视图（run logs 流、DAG 实例视图、lineage）在删除孤儿组件后，运行期行为零回归（浏览器验证）。

## Assumptions

- 清理范围包含"代码层死代码 + 四项灰色裁决 + 连带死配置"（连带死配置 = 代码删除直接导致的悬空 seed、存储路径、policy_rule，作为删除的必要收尾）；**不**含独立配置注释清理、demo-data.sql 整文件死表、过时文档章节、openspec 历史结构。
- 已删除功能（服务端 AI 编排、任务定义内联创建工具）的删除决策是最终决定，不会回滚。
- 退役项（任务级冻结、告警骨架）的退役决策已由用户确认；告警能力未来如需实现将作为全新特性重起，不复用本骨架。
- "死代码"的判定基线聚焦**生产调用方**：生产路径零调用即判死；仅被测试引用的代码（如已废功能的残留方法）亦判死，测试引用视为可迁移夹具，删除时同步迁移到现行构造途径。对存在反射/字符串/配置动态引用风险的项，单独验证后方可删除。
- 本次为纯本地操作，不涉及推送到远程；weft 改造尚未推送远程的事实不影响本地清理。
- 本特性在独立 worktree（`dw-011-weft-cleanup`）上进行，基准树 `e568c38`（真 main，已 `git reset --hard main`，含 D/E/揭红）。main 是 moving target（收尾 AI 持续推进），若 main 再前进需复查基准与行号——非微移，是可能的结构性变化（如 E 重塑整段重写了 McpToolRegistry / DefaultPlatformActionExecutor）。

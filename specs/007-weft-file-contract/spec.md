# Feature Specification: 文件化定义契约(Weft File Contract)

**Feature Branch**: `007-weft-file-contract`

**Created**: 2026-06-27

**Status**: Draft

**Input**: 子特性 B「文件化定义契约」—— Weft「任务即代码」转型(总纲 `specs/005-weft-pivot/spec.md`,宪法 `.specify/memory/constitution.md`)的地基子特性。定义 project / catalog / task / workflow 四类实体在本地磁盘上的确定性纯文本表示,以及与服务端治理模型之间的序列化/反序列化「往返(round-trip)」契约。

## 背景与定位

Weft 把任务与任务流从「Web UI 配置 + 数据库行」转变为「本地 git 仓库里的纯文本文件」。本子特性(B)只定义**文件格式本身与往返契约**,是后续子特性的地基:C(pull/push API)负责把这套格式在服务器与本地之间传输并生成版本快照;D(CLI + 本地 runtime)负责读写并执行;E(MCP 工具)负责暴露给本地 AI agent。

宪法原则 I(文件优先)、II(服务器为治理真相源)直接约束本特性:文件是**可读、可 diff、可 code review、对 AI 编程 agent 友好**的源定义;服务器掌握版本快照与租户/项目隔离。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - AI agent 把项目当代码读懂并手改 (Priority: P1)

开发者把一个项目从服务器拉到本地后,得到一棵纯文本目录树。本地 AI 编程 agent(Claude Code / Codex)或开发者本人,**无需运行任何服务**,仅凭文件就能理解每个任务的类型、脚本体、参数、调度,以及任务流的节点与依赖拓扑,并能像改代码一样直接编辑这些文件(新增任务 = 新建文件,移动任务 = 移动文件,改依赖 = 改一行)。

**Why this priority**: 这是「任务即代码」的核心价值与全部后续能力的前提。没有一套人类与 AI 都能无歧义读写的格式,pull/push、本地 run、MCP 工具都无从谈起。

**Independent Test**: 给定一份符合契约的项目目录,人(或 agent)在不连服务器的情况下能正确回答「这个任务跑什么脚本、什么类型、什么参数」「这个任务流有哪些节点、依赖关系如何」,并能新建/编辑文件使其仍然合法。

**Acceptance Scenarios**:

1. **Given** 一个含若干任务与一个任务流的项目目录, **When** 人或 agent 仅阅读文件(不连服务器), **Then** 能完整还原每个任务的类型/脚本体/参数/超时/重试,以及任务流的全部节点与依赖边及其依赖强度。
2. **Given** 一个合法项目目录, **When** 新建一个任务文件(元数据 + 脚本体)并放入某个类目目录, **Then** 该任务被识别为属于该类目,且文件本身自洽合法(无需服务器校验即可被解析)。
3. **Given** 一个任务的脚本体文件(`.sql`/`.sh`/`.py`), **When** 用普通编辑器或 AI agent 编辑脚本, **Then** 脚本以其原生语言的纯文本形式存在(语法高亮、lint、diff 全部适用),不被转义、包裹或编码。

---

### User Story 2 - 往返语义无损,支撑 push/pull (Priority: P1)

服务器把一个项目的治理模型导出为文件(序列化),再从文件解析回模型(反序列化),**语义无损**:任务类型、脚本体、参数、调度、类目归属、任务流拓扑与依赖强度全部一致还原。反向同理:从文件解析为模型、再序列化为文件,产物**逐字节稳定**(idempotent)。

**Why this priority**: 往返契约是 C(pull/push)能安全工作的数学保证。pull 是「模型→文件」,push 是「文件→模型→版本快照」;若往返有损,push 会悄悄改变任务语义,违背宪法原则 II。与 US1 并列 P1,因为二者共同定义「契约」本身。

**Independent Test**: 用一组覆盖各实体与各字段的样例,跑两条往返链:① 模型→文件→模型,断言模型等价;② 文件→模型→文件,断言文件逐字节相同。两条链均通过即满足。

**Acceptance Scenarios**:

1. **Given** 一个含全字段任务的治理模型, **When** 序列化为文件再解析回模型, **Then** 所有语义字段一致(脚本体逐字符相同,参数语义相同,空值与零值可区分)。
2. **Given** 一份合法项目文件, **When** 解析为模型再序列化回文件, **Then** 输出与输入逐字节一致(键有序、无随机、无时间戳漂移)。
3. **Given** 两个仅某一字段不同的等价定义, **When** 各自序列化, **Then** 产物差异**恰好**集中在对应位置(最小 diff),不产生噪声差异。

---

### User Story 3 - 确定性、最小 diff、可 code review (Priority: P2)

同一逻辑定义每次序列化输出字节稳定(键有序、集合顺序确定、不含机器生成的易变值如自增 id/绝对时间戳);改动一个字段只在一个文件的一处产生 diff;AI agent 的批量编辑产生干净、可审阅的 diff。

**Why this priority**: 决定「可 code review」与 git 协作体验。确定性是 US2 往返字节稳定的延伸,单列因为它额外约束「跨等价输入的稳定性」与「diff 局部性」,是 AI 友好的关键。

**Independent Test**: 对同一定义重复序列化多次,产物完全相同;打乱模型内集合顺序后序列化,产物仍相同;修改单一字段,git diff 仅显示该处。

**Acceptance Scenarios**:

1. **Given** 同一模型, **When** 反复序列化, **Then** 每次输出完全一致。
2. **Given** 任务的参数集合或任务流的节点/边集合以不同内部顺序提供, **When** 序列化, **Then** 输出顺序由稳定规则(如名称/key 字典序)决定,与输入顺序无关。

---

### User Story 4 - 非法/残缺/未知输入的可预期处理 (Priority: P3)

解析遇到非法或残缺文件时给出明确、可定位、对 AI 友好的错误(哪个文件、哪一处、缺什么/不合法在哪);遇到**未知字段**时按前向兼容规则处理(保留或忽略,见 FR),使旧解析器能读新文件、新解析器能读旧文件。

**Why this priority**: 健壮性与前向兼容,保证格式可演进且 agent 的不完美产物能被温和纠正。优先级低于核心契约,但为长期可维护性所必需。

**Independent Test**: 喂入一批构造的坏样例(缺必填字段、类型错误、依赖指向不存在的任务、目录与元数据矛盾、未知字段),断言每个都得到预期分类的明确错误或预期的兼容处理。

**Acceptance Scenarios**:

1. **Given** 一个缺必填字段的任务文件, **When** 解析, **Then** 报出指明文件与缺失字段的错误,不静默吞掉。
2. **Given** 一个任务流的依赖边指向不存在的节点/任务, **When** 解析, **Then** 报出指明该边与无效引用的错误。
3. **Given** 一个含未来版本新增的未知字段的文件, **When** 用当前解析器解析, **Then** 按前向兼容规则处理且不崩溃。

---

### Edge Cases

- **目录与元数据冲突**:任务文件所在目录路径(=类目归属)与元数据中的任何冗余类目信息矛盾时,以谁为准?(契约须有唯一真相源,见 FR-008。)
- **脚本体超长**:服务端 `content` 当前为有限长度(VARCHAR 4000)。文件侧脚本体是独立文件、天然无此限;契约须明确文件→模型时超出服务端承载上限如何对待(拒绝并报错,而非静默截断)。
- **空类目目录**:一个不含任何任务的目录(空类目)如何在文件中表达且 round-trip 不丢失。
- **文件名/类目名含文件系统非法字符或大小写敏感性差异**:跨平台(Linux/macOS/Windows)命名与大小写折叠如何约束。
- **同一类目下任务重名**:两个任务在同一目录下需要稳定区分(文件名即标识)——重名如何被禁止或消歧。
- **任务流引用尚未发布/不存在的任务**:文件层是源定义(草稿语义),允许引用同项目内任意任务定义;指向项目外或根本不存在的任务须报错。
- **虚拟节点(VIRTUAL,无关联任务)**:任务流中无 task 的锚点节点在文件中如何表达。
- **依赖强度缺省**:边未标注强度时的默认语义(STRONG)须在契约中固定。
- **未分类资产**:不属于任何类目的任务(服务端 `catalogNodeId=NULL`)在目录树中落在何处(如项目根)。

## Requirements *(mandatory)*

### Functional Requirements

**格式与布局**

- **FR-001**: 契约 MUST 为 project / catalog / task / workflow 四类实体定义确定性的纯文本磁盘表示,且元数据与脚本执行体**物理分离**:元数据用人类可读的结构化文本(键值,可 diff),脚本执行体为独立的原生语言文件(`.sql`/`.sh`/`.py` 等),以其原始字节存放,不转义、不内联包裹。
- **FR-002**: 一个 project MUST 有确定的根布局,包含项目级元数据、类目目录树、任务与任务流文件的落位规则,使整棵树自包含(self-contained)、可独立于服务器被解析。
- **FR-003**: 本地**目录树结构** MUST 直接映射 catalog(类目)树:目录即类目节点,目录的嵌套即类目父子关系;任务/任务流文件所在目录即其类目归属。空类目以空目录表达。
- **FR-004**: task 的文件表示 MUST 完整承载其语义字段:名称、任务类型(SQL/SHELL/PYTHON/DATA_SYNC/ECHO 等)、脚本体、参数、超时、重试、优先级、描述、数据源引用、冻结标志等;脚本体在独立文件中,元数据通过约定关联到脚本体。
- **FR-005**: workflow 的文件表示 MUST 用纯文本表达节点集合与依赖边集合(DAG 拓扑),可读可 diff:每个节点声明其类型(TASK/VIRTUAL)及(TASK 节点)所引用的任务;每条边声明源、目标与依赖强度(STRONG/WEAK,缺省 STRONG);并承载工作流级调度配置(调度类型 MANUAL/CRON/DEPENDENCY、cron 表达式、生效时间范围、超时、可抢占等)。
- **FR-006**: 标签(tag)MUST 在文件中可表达且可 round-trip(标签名、色值,以及任务/任务流与标签的关联);因一个资产可属多标签而只居一个目录,标签 MUST NOT 用目录表达,而作为资产元数据或项目级清单的一部分。

**身份与引用(跨 clean-slate 边界)**

- **FR-007**: 因 Weft 为 clean-slate(宪法硬规范 No Legacy Migration,服务端自增 id 在 push 时重新分配),文件中实体之间及文件对实体的引用 MUST NOT 依赖服务端数字主键;MUST 改用**文件路径/文件名作为实体身份**:类目身份 = 目录相对路径,任务身份 = 其所属目录内的文件名 slug,任务流节点对任务的引用 = 任务的相对路径/slug。元数据中 MUST NOT 另存独立的逻辑 key 字段(身份唯一由文件位置承载,无两套真相源)。**移动/改名文件即视为更换身份**:push 后服务器按新文件位置重建归属,文件层不承担「跨改名追踪同一实体」的语义(该跨版本追踪若需要,归服务器治理或后续子特性)。
- **FR-008**: 类目归属 MUST 有唯一真相源:任务/任务流的类目由其**文件所在目录**决定;元数据中 MUST NOT 重复存可与目录矛盾的类目字段。任何冗余的派生信息(如服务端物化 path)在序列化时省略、反序列化时由目录结构重算。
- **FR-009**: 数据源引用 MUST 以稳定的**逻辑 code/name**(而非数字 id)出现在任务文件中;具体连接串/凭据的解析是**环境与服务器的职责**,不在 B 展开。任务定义因此保持可移植、不含环境耦合(同一任务文件在不同环境按 code 解析到不同物理数据源)。数据源**自身定义**不纳入本契约(留作环境配置/后续子特性);B 仅保证该逻辑引用在 round-trip 中保真。

**往返与确定性**

- **FR-010**: 序列化 MUST 确定性:同一逻辑定义任意次序列化产出逐字节相同;键有序、集合按稳定规则排序、不含易变值(自增 id、绝对时间戳、随机数)。
- **FR-011**: 往返 MUST 双向成立:① 模型→文件→模型语义无损(往返幂等);② 文件→模型→文件逐字节稳定。脚本体在往返中逐字符保真。
- **FR-012**: 契约 MUST 区分「空值/未设置」与「零值/空串」,使二者各自 round-trip 不被混淆(如未设超时 vs 超时=0)。
- **FR-013**: 单一语义字段的改动 MUST 只在单个文件的单一位置产生 diff(diff 局部性);跨等价输入(仅集合内部顺序不同)序列化结果一致。

**版本与范围边界**

- **FR-014**: 文件契约的范围 MUST 限定为实体的**源定义(可编辑态)**:文件只表达源/草稿定义。已发布版本快照(TaskDefVersion / WorkflowDefVersion)是**服务器在 push 时生成的治理产物,绝不进文件**(对齐宪法原则 II);本地若需查看历史版本,经服务器查询而非读文件。任务流节点对任务版本的「钉死」(发布快照里的 taskVersionNo)同属服务器发布行为,文件层只表达「引用哪个任务」,不表达「钉死哪个版本」。
- **FR-015**: 解析 MUST 对非法/残缺输入给出明确、可定位的错误(指明文件与具体问题),不静默吞错、不静默截断、不静默丢字段。
- **FR-016**: 契约 MUST 定义未知字段的前向兼容规则,使格式可演进:旧解析器读到新版本新增字段不崩溃,新解析器能读旧文件。契约自身 MUST 携带可识别的版本/格式标记以支持演进。
- **FR-017**: 本特性 MUST 复用既有领域语义而非另起一套:任务/任务流/类目的字段含义、版本快照语义、类目树的父子/移动/防环规则均与现有治理模型一致(B 定义其文件投影,不重新定义其语义)。

### Key Entities *(include if feature involves data)*

- **Project 文件表示**:一个项目在磁盘上的根布局与项目级元数据(项目代码/名称等);是类目树、任务、任务流、标签的容器。租户/项目隔离的归属由服务器治理,文件侧承载可移植的项目身份。
- **Catalog 目录树**:由真实目录构成的类目树;目录=类目节点,嵌套=父子;任务/任务流文件落在对应目录中表达归属。无独立的「类目清单文件」与目录争夺真相源。
- **Task 文件**:一个任务的元数据(类型、参数、超时、重试、优先级、描述、数据源引用、冻结等)+ 关联的脚本体文件(原生语言纯文本)。任务身份由其文件标识决定(见 FR-007)。
- **Workflow 文件**:一个任务流的元数据(调度配置)+ 节点集合(TASK/VIRTUAL)+ 依赖边集合(源/目标/强度),以纯文本表达 DAG;节点按稳定标识引用任务。
- **Tag 与关联**:标签定义(名/色)及任务/任务流到标签的多对多关联;以元数据/项目级清单表达,不占目录。
- **格式版本标记**:嵌入文件契约的版本/格式标识,支撑前向兼容演进(FR-016)。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在不运行任何服务器的前提下,一个熟悉契约的开发者或 AI agent 能仅凭文件正确还原一个含 ≥10 个任务、≥2 个任务流的项目的全部语义(任务类型/脚本/参数/调度、任务流节点与依赖),还原正确率 100%。
- **SC-002**: 对一组覆盖四类实体全部字段的样例,「模型→文件→模型」语义等价率 100%,「文件→模型→文件」逐字节一致率 100%。
- **SC-003**: 同一定义重复序列化 100 次,产物 100% 逐字节相同;打乱集合内部顺序后序列化结果与原序列一致。
- **SC-004**: 修改任意单一语义字段后,产生改动的文件数 = 1 且改动处 = 1(diff 局部性)在全字段样例上成立率 100%。
- **SC-005**: 一组覆盖各类错误的非法/残缺样例,100% 得到指明文件与具体问题的明确错误,0 例静默吞错或静默截断。
- **SC-006**: 含未来新增未知字段的样例被当前解析器解析时 0 崩溃,且已知字段全部正确解析。

## Assumptions

- **范围**:本特性只定义文件格式与往返契约本身,**不**实现传输(pull/push REST API —— C)、**不**实现 CLI 与本地执行(D)、**不**实现 MCP 工具(E)、**不**决定 Web 编辑器存废(总纲倾向只读保留)。
- **No Legacy Migration(宪法硬规范)**:不考虑既有数据库存量数据的迁移;存量直接删除走 clean-slate。因此契约无需兼容任何历史磁盘格式,且不得依赖既有自增 id 的稳定性。
- **版本快照归属服务器**:已发布版本快照(TaskDefVersion / WorkflowDefVersion)是服务器在 push 时生成的治理产物;文件侧只表达源/草稿定义(FR-014 已裁定)。
- **身份 = 文件位置**:实体身份由文件路径/文件名承载,不存独立逻辑 key;移动/改名即换身份(FR-007 已裁定)。
- **数据源 = 逻辑 code 引用**:任务以数据源逻辑 code 引用,连接解析归环境/服务器,数据源自身定义不在本契约(FR-009 已裁定)。
- **复用既有内核语义**:类目树(父子/移动/防环)、任务/任务流字段语义、版本快照语义、DAG 节点稳定标识(nodeKey)等既有概念直接复用,B 定义其文件投影。
- **目标读者**:文件主要由本地 AI 编程 agent 与开发者读写,以及由服务器序列化/反序列化程序处理;格式优先保证人类可读与 git diff 友好。
- **依赖**:本特性依赖 A(服务端 AI 拆除,已完成并合入 main `449fe12`)。完成后解锁 C(pull/push API)。

## Constitution Alignment

- **原则 I(文件优先)**:本特性即该原则的直接落地——元数据/脚本分离、目录树=类目树、纯文本可 diff/review、对 AI agent 友好(FR-001..006)。
- **原则 II(服务器为治理真相源)**:版本快照与租户/项目隔离归服务器;文件是工作副本的源定义(FR-014、Assumptions)。
- **原则 V(内核复用而非重写)**:复用既有类目/任务/任务流/版本快照语义,只定义文件投影(FR-017)。
- 与原则 III(本地两条腿调试)、IV(AI 归位本地)不直接冲突;本契约为其提供可被本地 runtime 与本地 agent 消费的文件基底。

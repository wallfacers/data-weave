# Feature Specification: 统一数据健康事件中心

**Feature Branch**: `027-event-center`

**Created**: 2026-06-30

**Status**: Draft

**Input**: 现有功能完善：平台已有多处产出健康信号（`SlaService` → `SLA_BREACH`、`QualitySignalEmitter` → 质量断言失败、`InstanceStateMachine`/`LeaseReaper` → 任务失败/租约过期，以及 024 列血缘 `CONFLICT`），它们当前各自流入告警引擎做「即时通知」，但**没有一个统一、可持久、可回溯、可订阅的事件归宿**——运维无法在一个地方看「这个资产/任务今天发生了哪些数据健康事件」。另注：现存两个 `AlertSignal` 类（`domain/signal` 与 `quality/domain`）语义重复，需统一信号契约。

> **范围边界**：本特性把散落的数据健康信号**汇聚成一个统一事件中心**——持久化事件流 + 按资产/类型/严重度/时间检索的视图 + 订阅（命中订阅经既有通道分发）。它**消费** 021/026 的告警分发能力与现有信号 emit 点，**不重做**告警规则引擎、**不改**各信号的业务判定逻辑。**不做**：复杂事件关联分析、告警风暴抑制/聚合归因（标记为后续）、新信号源的业务规则（只接现有 emit 点 + 预留 024 CONFLICT 接缝）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 一个地方看全所有数据健康事件 (Priority: P1)

运维打开事件中心，看到一条按时间倒序的统一事件流：SLA 违约、质量断言失败、任务失败、（预留）列血缘冲突，每条含发生时间、来源类型、严重度、关联对象（任务/指标/表）、摘要。可按类型 / 严重度 / 关联资产 / 时间范围筛选。

**Why this priority**: 这是事件中心的存在理由——把今天散落在告警日志、质量页、SLA 各处的信号收敛到单一可回溯视图；没有它，运维要在多个页面拼凑「今天出了什么事」。

**Independent Test**: 制造一次 SLA 违约 + 一次质量断言失败，断言两者都作为事件持久化并出现在事件流；按「严重度=高」筛选，断言只返回匹配事件。

**Acceptance Scenarios**:

1. **Given** 某任务实例触发 SLA_BREACH, **When** 打开事件中心, **Then** 出现一条 SLA 类事件，含时间/严重度/关联任务/摘要。
2. **Given** 某质量断言失败, **When** 打开事件中心, **Then** 出现一条质量类事件，关联到对应资产。
3. **Given** 事件流含多类事件, **When** 按类型「质量」+ 严重度「高」筛选, **Then** 仅返回匹配事件，计数正确。
4. **Given** 一条事件, **When** 点击其关联对象, **Then** 深链跳转到对应任务/指标/血缘视图。

---

### User Story 2 - 统一信号契约（消除 AlertSignal 重复）(Priority: P1)

各信号源经**单一信号契约**写入事件中心：来源类型、租户、严重度、关联对象引用、上下文、发生时间、去重指纹。现存重复的两个 `AlertSignal` 类收敛为一套契约，新事件源接入只需产出该契约。

**Why this priority**: 没有统一契约，事件中心要为每种信号写适配代码、字段不齐、去重不一致；统一契约是「可持续接入新信号源」的地基，且消除现有技术债（两个 AlertSignal）。

**Independent Test**: 用统一契约从两个不同来源（SLA、质量）各写一条事件，断言两条在事件中心结构一致、可被同一套筛选/订阅逻辑处理。

**Acceptance Scenarios**:

1. **Given** SLA 与质量两个来源, **When** 各 emit 一个信号, **Then** 两者以同一契约结构落库，字段齐备（类型/严重度/关联/指纹）。
2. **Given** 同一来源对同一对象短时间重复 emit, **When** 落库, **Then** 按指纹去重/合并，不产生重复事件刷屏。
3. **Given** 既有告警引擎对这些信号的即时分发, **When** 引入事件中心, **Then** 即时分发行为零回归（事件中心是旁路持久化 + 视图，不夺走告警链路）。

---

### User Story 3 - 订阅事件并经既有通道触达 (Priority: P2)

用户/租户可订阅某类事件（如「我的资产的任何高severity质量事件」）。命中订阅时，事件经既有通道（邮件/Webhook，复用 026）分发给订阅者。

**Why this priority**: 让事件中心从「被动可查」升级为「主动触达」，但价值依赖 US1/US2 已立住，故 P2。

**Independent Test**: 配一条订阅（类型=质量、severity≥高、资产=X），触发一个匹配事件，断言订阅者经配置通道收到通知；再触发不匹配事件，断言不通知。

**Acceptance Scenarios**:

1. **Given** 一条订阅（类型+severity+资产维度）, **When** 命中事件产生, **Then** 经订阅配置通道分发给订阅者。
2. **Given** 不匹配订阅条件的事件, **When** 产生, **Then** 不向该订阅者分发。
3. **Given** 通道分发失败, **When** 订阅触达, **Then** 不阻断事件持久化与其它订阅，失败有记录（复用 026 分发记录）。

---

### Edge Cases

- **024 列血缘 CONFLICT 信号尚未 emit**：预留事件源接缝；024 落地后接入即出现在事件流，本期不强依赖 024。
- **同一根因短时间产生大量事件**：按指纹去重/合并，避免刷屏（完整风暴抑制归后续）。
- **关联对象已删除**：事件仍保留，关联深链优雅降级为「对象已不存在」。
- **订阅者无任何可用通道**：跳过分发 + 记录，不报错。
- **历史信号（事件中心上线前已发生）**：不回填，仅记录上线后的事件。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 把以下来源的健康信号持久化为统一事件：SLA 违约、质量断言失败、任务失败/租约过期，并预留列血缘 CONFLICT 接缝。
- **FR-002**: 每条事件 MUST 含发生时间、来源类型、严重度、租户、关联对象引用（任务/指标/表）、摘要、去重指纹。
- **FR-003**: 系统 MUST 提供按来源类型、严重度、关联资产、时间范围的事件检索，结果按时间倒序。
- **FR-004**: 事件的关联对象 MUST 可深链跳转到对应业务视图；对象不存在时优雅降级。
- **FR-005**: 各信号源 MUST 经**单一统一信号契约**写入；现存重复的两个 `AlertSignal` 类 MUST 收敛为该契约。
- **FR-006**: 同一来源对同一对象的重复信号 MUST 按指纹去重/合并，不产生刷屏事件。
- **FR-007**: 引入事件中心 MUST 对既有告警引擎的即时分发零回归（事件中心为旁路持久化 + 视图 + 订阅）。
- **FR-008**: 用户/租户 MUST 能按类型/严重度/资产维度订阅事件，命中时经既有通道（复用 026）分发。
- **FR-009**: 订阅分发失败 MUST 不阻断事件持久化与其它订阅，失败 MUST 有记录。
- **FR-010**: 事件写入与检索 MUST 按租户隔离。

### Key Entities

- **HealthEvent**：一条数据健康事件——时间、来源类型（SLA/质量/任务/血缘）、严重度、租户、关联对象引用、摘要、指纹。
- **UnifiedSignal（契约）**：信号源产出的统一结构，事件中心据此落库；替代重复的两个 AlertSignal。
- **EventSubscription**：订阅条件（类型 + severity 阈值 + 资产维度）+ 目标通道 + 订阅者/租户。
- **关联对象引用**：指向任务实例 / 指标 / 表（血缘节点）的轻量引用，用于深链。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 任一受支持来源产生的信号，在事件中心可于产生后即时检索到（≤ 数秒）。
- **SC-002**: 运维在单一视图即可按资产/类型/严重度回溯近期所有数据健康事件，无需切换多个页面。
- **SC-003**: 重复信号去重后，同一根因在事件流中不超过合理条数（不刷屏）。
- **SC-004**: 命中订阅的事件触达率：通道可用时 ≥ 99%；通道不可用时事件仍 100% 持久化。
- **SC-005**: 引入后既有告警即时分发与各信号业务判定零回归（回归测试全绿）。

## Assumptions

- 信号源沿用现有 emit 点（`SlaService`/`QualitySignalEmitter`/`InstanceStateMachine`/`LeaseReaper`），本期只统一契约与增加旁路持久化，不改业务判定。
- 通道分发复用 026 告警引擎收口的真实通道与分发记录；建议排在 026 之后。
- 024 列血缘 CONFLICT 作为预留接缝，不强依赖 024 落地。
- 前端事件中心为新视图（Workspace 新增一个 tab），复用现有视图框架与筛选组件范式。
- 复杂事件关联、告警风暴抑制/聚合归因为后续特性，不在本期。

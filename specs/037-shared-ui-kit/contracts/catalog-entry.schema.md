# Contract: 公共组件目录条目 Schema

**约束对象**：`frontend/DESIGN.md` 的 `## 公共组件目录` 章节中每一条组件条目。
**满足时机**：编目种子条目时、以及未来任何"未命中→新建组件→回填目录"时（FR-003）。

## 必备字段

每个条目 MUST 提供：

```
- 原语类别:      <九类之一 | 其它>            # 必填
- 规范组件:      <组件名> @ <真实文件路径>     # 必填，唯一
- 用途/何时用:   <一句话适用场景>             # 必填
- 何时不用/禁写: <边界；"存在时禁止手写同类">  # 必填
- 关键 props/变体: <如 closable / centered|overlay / server|client>  # 必填
- token/间距引用: <如 --card-spacing；禁止字面魔法值>   # 涉及间距时必填
- 深章节链接:    <指向 DESIGN.md 既有规范段>   # 有则填，避免重复
```

## 有效性规则（评审校验）

1. **唯一性**：同一"原语类别"在目录中 MUST 只有一个规范组件条目（FR-005）。多实现须先收敛（如 Tabs 卡片/下划线属两个类别变体，不算冲突）。
2. **可定位**：`规范组件` 的路径 MUST 真实存在于 `frontend/`（评审时可点开）。
3. **无魔法值**：`token/间距引用` MUST 引用 token（`--card-spacing` 等），禁止写死 `20px`/`p-5` 类页面级字面值（FR-010）。
4. **禁写边界明确**：`何时不用/禁写` MUST 能让下一个 agent 判断"该不该手写"——即复用优先的判定依据（FR-002）。
5. **主题一致**：条目若含视觉规范，MUST 在明/暗主题下一致（FR-013），不得只描述单主题。
6. **回填闭环**：新建组件的 PR MUST 同一提交内更新本目录条目（防目录-实现漂移，Edge Case）。

## 9 类原语最小覆盖（FR-004 验收）

目录 MUST 至少包含以下类别的条目（种子值见 `../data-model.md` 实体 2）：
滚动条 · Tabs(卡片) · Tabs(下划线) · 表格 · 下拉 · 弹框 · 日期 · 加载 · 刷新 · 卡片容器。

> Tabs(下划线) 条目依赖 030 的 `tabs.tsx` 落地（见 research R3）；若 030 未合入，该条目 MUST 标注依赖状态而非指向不存在的组件。

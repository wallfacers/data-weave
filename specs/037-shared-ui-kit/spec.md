# Feature Specification: 复用优先的公共前端组件契约与目录

**Feature Branch**: `037-shared-ui-kit`

**Created**: 2026-07-02

**Status**: Draft

**Input**: User description: "多 Agent、多厂商模型开发，前端组件多、每次实现页面都像抽卡，要经过多轮对话补充风格细节（加载动画、垂直水平居中、业务日期主键、tabs 写死分割、滚动条、下拉、弹框、日期、刷新按钮位置与居中动画、主卡片内边距 20/20/20 等）。目标：① 建立公共组件；② 所有实现先看公共组件，只有没有才实现功能组件——把这些原语统一，避免每页微调。"

## Clarifications

### Session 2026-07-02

- Q: 本特性对存量已有页面的迁移范围到哪？ → A: 新页面强制使用公共组件 + 存量页面登记「差距清单」后增量迁移（已有对应公共组件的原语优先），不要求一次性重写全部历史页面。
- Q: 「复用优先/统一」用什么手段强制？ → A: 以 DESIGN.md 公共组件目录 + 复用优先约定 + 实现后自查/评审为主要强制手段；自动化 lint/CI 视为可选增强，不作本特性硬前提。
- Q: 公共组件目录（agent「先去看」的入口）以什么形态交付？ → A: 在既有 `frontend/DESIGN.md` 增设「公共组件目录」章节作为单一真相源（纯文本可 diff、按需加载），不另起独立文档或组件画廊。

## User Scenarios & Testing *(mandatory)*

> 本特性的"用户"是**在本仓库构建/修改前端页面的开发者与 AI 编程 agent**（直接受益人），以及**看到最终界面的终端用户**（间接受益人：观感一致）。核心痛点：多个厂商模型各自"抽卡式"造轮子，导致同类原语（Tabs、表格、下拉、弹框、日期、加载/刷新、卡片留白）每页样式不一，需反复多轮对话才能对齐。

### User Story 1 - 复用优先：先查公共目录，命中即复用 (Priority: P1)

一个 agent（或开发者）接到"新建/改造某页面"的任务。在动手实现任何界面原语（Tabs、表格、下拉、弹框、日期选择、滚动区、加载/刷新、卡片容器等）之前，它先查阅一个**单一、权威的公共组件目录**：目录清晰列出"已有哪些公共组件、各自的用途、如何使用、禁止再手写的场景"。若目录中已存在对应原语，agent **必须直接复用**，不得重新手写一套；只有当目录中确无对应能力时，才允许实现一个新的功能组件，并**将其回填进目录**供后续复用。

**Why this priority**: 这是用户两个诉求的核心（"所有实现先去看公共组件，除非没有才实现"）。没有这条"复用优先 + 单一真相源"的约定，其余原语统一都会被下一次抽卡打回原形。它本身即可独立交付价值——即便只落地目录与约定、暂不新增组件，也能立刻减少重复造轮子。

**Independent Test**: 给一个未参与过本项目的 agent 一个"新建含表格与筛选下拉的页面"任务，仅向其提供公共组件目录入口；验证它能在不询问额外样式细节的前提下，从目录选中表格/下拉/卡片容器等公共组件完成页面，且未手写重复原语。

**Acceptance Scenarios**:

1. **Given** 存在一个权威公共组件目录且已收录表格组件，**When** agent 需要在新页面展示数据表格，**Then** 它能在目录中定位到表格公共组件、直接复用，无需就边框/留白/分页/加载占位再发起澄清对话。
2. **Given** 目录中不存在某个所需原语（如某种特殊图表卡片），**When** agent 判定确无可复用组件，**Then** 它实现该新组件后，目录随之新增一条对应条目（用途 + 用法 + 归属），使下一次同类需求可直接复用。
3. **Given** 一次页面实现中 agent 手写了一个目录里已存在的原语（重复造轮子），**When** 进行实现后自查/评审，**Then** 该重复实现可被明确识别为"违反复用优先约定"，并被替换为公共组件。

---

### User Story 2 - 统一原语目录覆盖用户点名的高频组件 (Priority: P1)

用户点名的一批高频界面原语——**滚动条、Tabs、表格、下拉框、弹框、日期选择、加载/刷新动画、主卡片内容留白**——在公共目录中各有**唯一一种规范外观与用法**。任意页面使用这些原语时呈现完全一致：Tabs 不再各自"写死分割"、表格统一边框包裹、下拉/弹框风格一致、卡片内容留白统一（如四周统一内边距）、滚动条外观统一。

**Why this priority**: 这是用户逐一列举的痛点清单，也是"复用优先"约定要保护的具体资产。与 US1 并列 P1：目录若空有约定而无这批组件，agent 仍无可复用之物。

**Independent Test**: 在 3~5 个不同页面分别使用 Tabs、表格、下拉、弹框、卡片容器，逐一比对同类原语在各页面的外观（边框、圆角、留白、间距、滚动条样式）是否像素级/视觉一致，且无任一页面出现"写死的一次性样式"。

**Acceptance Scenarios**:

1. **Given** 两个不同页面都使用 Tabs，**When** 并排查看，**Then** 两处 Tabs 的选中态、分隔、Tab 条到内容的间距完全一致，不存在某页"写死 tab 分割线/等分宽度"的一次性实现。
2. **Given** 任意含表格的页面，**When** 查看表格外观，**Then** 表格统一被一个边框容器包裹，边框/圆角/容器内留白一致（与既有全站表格统一诉求对齐）。
3. **Given** 任意页面的下拉框与弹框，**When** 打开它们，**Then** 触发器、浮层、圆角、内边距、滚动条外观均取自同一套公共规范，跨页一致。
4. **Given** 任意主内容卡片，**When** 查看其内容四周留白，**Then** 内边距取自统一的间距规范（如统一的卡片内容内边距），不再出现每页各异的手填数值。

### User Story 3 - 业务日期为默认主键，带时间为统一变体 (Priority: P2)

涉及日期的字段与展示，默认统一使用**业务日期（bizDate，只到"日"粒度）**作为主键/主展示口径；当某场景确需精确到时间时，使用**同一套日期组件的"带时间"变体**，其格式、时区口径、控件外观与业务日期保持统一，而非各页各自决定格式与粒度。

**Why this priority**: 用户明确点名"日期主键统一用业务日期，如果带时间的，需要等组件统一风格"。日期口径不一致会直接造成数据对齐歧义，但相较 US1/US2 覆盖面略窄，列为 P2。

**Independent Test**: 在两个不同页面分别展示/输入一个日期字段与一个日期时间字段；验证日期字段一律呈现业务日期粒度与统一格式，日期时间字段一律使用统一的"带时间"变体格式，两页无差异。

**Acceptance Scenarios**:

1. **Given** 一个以日期为主键/主口径的字段，**When** 在任意页面展示或选择，**Then** 其粒度为业务日期（到日）、格式统一，跨页无差异。
2. **Given** 一个确需精确到时间的字段，**When** 在任意页面展示或选择，**Then** 其使用统一的"带时间"日期组件变体，格式与时区口径一致。
3. **Given** 一个新页面需要日期输入，**When** agent 从目录选取日期组件，**Then** 默认得到业务日期粒度，需带时间时通过同一组件的显式变体开启，无需自造格式。

---

### User Story 4 - 加载与刷新交互全站统一 (Priority: P2)

页面/区域的等待状态统一为**居中的转圈动画**（垂直+水平居中于其加载容器内），不再是孤立的"加载中"文字；页面的**刷新入口固定在统一位置**，刷新进行中的动画同样居中呈现。用户在任何页面看到的"加载/刷新"反馈一致、可预期。

**Why this priority**: 用户点名"加载中动画、垂直水平居中""刷新按钮位置、刷新动画垂直水平居中"。属于高频体验细节，但可在 US1/US2 骨架之上增量落地，列为 P2。此故事与既有"加载状态统一转圈动画"诉求同源，本特性将其纳入公共目录统一收口。

**Independent Test**: 打开若干需异步加载的视图，验证加载占位为居中转圈动画；点击各页刷新入口，验证刷新按钮位置一致、刷新中动画居中；比对不同页面加载/刷新反馈是否一致。

**Acceptance Scenarios**:

1. **Given** 任意需要加载数据的视图，**When** 数据请求中，**Then** 加载容器内显示垂直水平居中的转圈动画，而非纯文字或非居中占位。
2. **Given** 任意含刷新能力的页面，**When** 查看刷新入口，**Then** 刷新按钮位于统一约定的位置，跨页一致。
3. **Given** 用户触发刷新，**When** 刷新进行中，**Then** 刷新反馈动画居中呈现，动画规范与首次加载一致。

### Edge Cases

- **目录与实现漂移**：公共组件被改动但目录条目未更新（或反之），如何避免目录失真、误导下一个 agent？（需有"改公共组件即更新目录"的约束与可核对手段。）
- **既有页面遗留原语**：本特性落地前已存在的手写一次性原语如何处理——是全部立即迁移，还是保留但登记在"待迁移清单"逐步收敛？
- **公共组件不满足个别页面**：当某页面确有目录组件覆盖不到的特殊需求时，如何区分"合理的新组件"与"又一次抽卡式造轮子"，避免约定被滥用绕过？
- **减少动画偏好**：用户开启系统"减少动画"时，转圈/刷新动画应有静态可访问的降级表现。
- **暗色主题一致性**：所有公共原语在明/暗主题下都须保持统一外观，不得出现某原语仅在一种主题下被打磨。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供一个**单一、权威的公共组件目录**（真相源，落于 `frontend/DESIGN.md` 的「公共组件目录」章节），登记每个公共组件的：名称、用途、适用场景、使用方式、以及"存在该组件时禁止手写同类原语"的约束。
- **FR-002**: 面向前端页面实现的工作流程 MUST 确立"**复用优先**"约定：实现任何界面原语前，先查公共组件目录；命中则必须复用，未命中方可实现新组件。该约定 MUST 对 agent 可发现（沉淀于设计契约/任务创作指引中）。
- **FR-003**: 当确无可复用组件而新增一个功能组件时，流程 MUST 要求将该新组件**回填进公共组件目录**，使其对后续实现可复用。
- **FR-004**: 公共组件目录 MUST 覆盖用户点名的高频原语集合，至少包括：滚动条/滚动区、Tabs、数据表格、下拉框（select）、弹框（dialog/modal）、日期选择、加载动画、刷新入口与刷新动画、主内容卡片容器。
- **FR-005**: 上述每个原语 MUST 有**唯一一种规范外观与用法**；跨页面使用时呈现一致，禁止页面级"写死"的一次性样式（如写死的 Tab 分割/等分宽度、页面自造的表格边框或卡片留白）。
- **FR-006**: Tabs 公共组件 MUST 统一选中态、分隔与"Tab 条到内容"的间距，消除各页各自写死分割的实现。
- **FR-007**: 数据表格 MUST 统一采用"边框容器包裹"的外观（边框/圆角/容器内留白一致），并与既有全站表格统一诉求（033）对齐、共用同一规范。
- **FR-008**: 下拉框与弹框 MUST 共用同一套浮层规范（触发器、浮层圆角、内边距、滚动条外观一致）。
- **FR-009**: 滚动条/滚动区 MUST 有统一的外观规范，跨页与跨原语（表格、浮层、内容区）一致。
- **FR-010**: 主内容卡片容器 MUST 使用**统一的内容内边距规范**（单一间距 token，取代各页手填的 20/20/20 类魔法数值）。
- **FR-011**: 日期能力 MUST 默认以**业务日期（bizDate，到日粒度）**为主键/主展示口径；需要精确到时间时，MUST 通过同一日期组件的显式"带时间"变体提供，格式与时区口径统一。
- **FR-012**: 加载与刷新状态 MUST 统一为**居中（垂直+水平）的转圈动画**；刷新入口 MUST 位于统一约定的位置；两者动画规范一致，并与既有"加载统一转圈"诉求（035）对齐收口。
- **FR-013**: 所有公共原语 MUST 在明/暗主题下外观统一，且 MUST 遵循既有设计系统的语义 token 与间距规范（不得引入游离于设计契约之外的一次性数值/颜色）。
- **FR-014**: 加载/刷新等动画 MUST 在用户开启"减少动画"偏好时提供静态降级，并对辅助技术可读其状态。
- **FR-015**: 系统 MUST 提供一份**覆盖清单/差距清单**，登记现有页面对公共原语的采用状态与待迁移项，使"统一到什么程度"可核对、可验收（避免"看起来统一了"实则残留一次性实现）。

### Key Entities *(include if feature involves data)*

- **公共组件目录（Component Catalog）**: 前端公共组件的单一真相源清单。每条目包含组件名、用途、适用/不适用场景、用法、归属，以及"禁止手写同类原语"的约束。与"设计契约"及"任务创作指引"关联，是复用优先约定的落点。
- **公共组件条目（Catalog Entry / Primitive）**: 目录中的单个原语（如 Tabs、表格、下拉、弹框、日期、加载/刷新、卡片容器、滚动区）。承载该原语的唯一规范外观与用法。
- **间距/外观规范 token（Design Token）**: 统一的间距（如卡片内容内边距）、边框、圆角、滚动条与动画规范，供各公共组件引用，取代页面级魔法数值。归属既有设计系统。
- **覆盖/差距清单（Adoption Inventory）**: 现有页面对各公共原语的采用与待迁移状态记录，用于验收统一程度。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 一个未预先了解本项目样式细节的 agent，仅凭公共组件目录即可完成一个"含表格 + 筛选下拉 + 卡片容器"的新页面，**无需就外观细节发起额外澄清对话**（对比现状的多轮"抽卡"补充，澄清轮次降至 0）。
- **SC-002**: 抽查任意 5 个使用同类原语（Tabs/表格/下拉/弹框/卡片留白）的页面，同类原语外观**100% 一致**，无任一页面存在写死的一次性原语实现。
- **SC-003**: 目录覆盖的原语集合中，用户点名的 9 类原语（滚动条、Tabs、表格、下拉、弹框、日期、加载、刷新、卡片留白）**全部有唯一规范条目**且可复用。
- **SC-004**: 全站日期字段中，作为主键/主口径的日期**100% 以业务日期粒度统一呈现**；带时间字段 100% 使用统一"带时间"变体格式。
- **SC-005**: 全站需加载/刷新的视图，其加载与刷新反馈**100% 为居中转圈动画**，刷新入口位置一致。
- **SC-006**: 新增或改造前端页面时，因样式/原语不统一而产生的返工与澄清对话轮次相比现状显著下降（目标：同类页面从"多轮"降到"一轮内"）。
- **SC-007**: 覆盖/差距清单可查，明确列出已迁移与待迁移页面，验收方可据此判定统一程度而非凭主观印象。

## Assumptions

- **收编而非重造既有点状特性（已核实各自着陆状态，2026-07-02）**：`033-ui-table-frame-spacing`（表格边框包裹）与 `035-loading-spinner`（加载居中转圈）**已合入 main**，直接作种子条目编目；`034-dashboard-layout-shell` 仅卡片栅格保留、面包屑已撤回，本特性**不恢复面包屑**，卡片栅格并入"卡片容器"条目；`030-unify-tab-styles` 的下划线 `Tabs` 组件（`components/ui/tabs.tsx`）**未合 main**——且 main 已另行用"页内内联下划线"做过视觉统一（ops/alerts 仍手写），二者路线分叉，故 037 **仅文件级取用 030 的 `tabs.tsx` 组件**（不整提交 cherry-pick，避免与 main 内联实现冲突），视图迁移在当前 main 上重做。本特性把这些收敛进单一目录 + 复用优先约定，并补齐缺口原语（滚动条、下拉、弹框、日期口径、刷新按钮位置），而非重复其已完成的实现。
- **目录/约定落点（已确认，Session 2026-07-02）**：公共组件目录以既有**设计契约 `frontend/DESIGN.md` 增设「公共组件目录」章节**的形态交付，作为单一真相源与 agent 可发现的单一入口（纯文本可 diff、按需加载）；不另起独立目录文档，也不引入 Storybook 式组件画廊。任务创作指引引用该章节。
- **迁移范围（已确认，Session 2026-07-02）**：新建页面 MUST 遵循复用优先并使用公共组件；既有页面通过"覆盖/差距清单"登记后**增量迁移**（已有对应公共组件的原语——如表格/加载/Tabs——优先迁移），不要求一次性重写全部历史页面。
- **强制手段（已确认，Session 2026-07-02）**：以"单一真相源目录 + 设计契约约束 + 实现后自查/评审"为主要强制手段；自动化 lint/CI 校验（如检测页面手写魔法间距/一次性原语）视为**可选增强**，不作为本特性交付的硬前提。
- **技术栈沿用**：公共组件基于既有前端栈与设计系统（shadcn base 风格、hugeicons、next-intl、语义 token），不引入新的 UI 框架；"业务日期/带时间"口径沿用后端既有 bizDate 语义。
- **范围边界**：本特性聚焦前端界面原语的统一与复用治理，不改动后端 API、数据模型或调度/权限内核。

# Feature Specification: 调度器节点容错闭环（节点剔除·任务转移·不丢不失败）

**Feature Branch**: `060-node-eviction-failover`

**Created**: 2026-07-10

**Status**: Draft

**Input**: User description: "调度器节点容错闭环：假节点/死节点自动剔除、任务向健康节点转移、无健康节点时保持等待不判死、系统恢复主动抽干等待、RUNNING 任务靠真续约区分长跑与状态丢失、终极人工停止重跑兜底，全程不丢不双跑不误判终态"

## 背景与问题（Why）

分布式调度下，"节点在线（`status=ONLINE`）"被当成"节点可用"来分派任务，但二者并不等价：一个进程纪元（incarnation）反复抖动的**假节点**、一个心跳正常但执行端不可达的**坏节点**、一个刚重启还没稳定的节点，都会通过这道弱门槛并被派上任务。结果是任务被派到用不了的节点上，进而被回收判失败；更糟的是，"节点故障"这类**基础设施问题**会消耗任务自身的**业务重试次数**，最终把一个从未真正跑过的任务判成终态失败。

与此对称的另一端是**运行中任务的状态可信度**：系统无法区分"任务确实在长时间运行"与"任务状态已经丢失（worker 死了但 master 仍以为在跑）"，因而既可能误杀长跑任务，也可能让丢失的任务永久滞留。此外，**外部托管的长驻作业**（如 Flink 流式，真实执行运行在外部引擎集群、生命周期与 worker 进程解耦）天然一直处于运行态，若套用"节点故障→重派"的朴素模型会**重复提交作业**并**误杀集群里健康运行的作业**——需要一套区别于进程内/有界任务的容错语义。

本特性要建立一个**闭环**：坏/假节点自动剔除 → 任务自动转移到健康节点 → 基础设施故障不消耗业务重试 → 无健康节点时安全等待而非判死 → 系统恢复后主动唤醒抽干等待 → 运行中任务靠真实租约续约区分长跑与丢失 → 外部托管长驻作业按句柄重连而非重提交 → 最无奈时人工停止重跑兜底。**全程保证：不丢任务、同一实例不双跑、不因基础设施原因误判终态。**

## Clarifications

### Session 2026-07-10

- Q: 崩溃循环（poison task）的保护边界 → A: 对单实例的连续基础设施重派设上限；超限则挂起该实例并告警交人工，不自动判终态失败（防毒任务无限重派危害机群）。
- Q: 任务级最大运行时长（max-runtime timeout）是否纳入本特性 → A: 纳入本特性。
- Q: 网络分区下进程内任务的物理双跑如何避免 → A: worker 在续约窗口内失联 master 即**自我中止**本地执行；外部托管长驻作业豁免此中止。
- Q: 外部托管长驻作业（Flink 流式）的 failover 语义 → A: 立为独立任务类别；提交成功即持久化外部作业句柄；failover 为按句柄 **reattach 重连**、**绝不重提交**；豁免 max-runtime 与 worker 自我中止；任务实例在作业存活期间保持运行态，句柄作为未来实时任务卡片等扩展的统一挂载点。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 坏节点不再拖垮任务（Priority: P1）

作为平台运维，当集群里存在一个不可用的节点（假节点、执行端坏掉、或刚崩溃）时，我希望调度器**根本不把任务派给它**；万一任务已经在它上面且它出了故障，任务要**自动转移到健康节点**继续，而不是因为节点的问题被判失败。

**Why this priority**: 这是"不失败"的核心。当前每次基础设施抖动都在消耗业务重试并制造无谓的失败，是用户直接观测到的痛点。单独实现它即构成可用的 MVP。

**Independent Test**: 造一个坏节点（心跳在线但执行端不可达 / incarnation 抖动），提交任务，验证：① 坏节点从不承接任务；② 已在坏节点上的任务被自动转移到健康节点并最终成功；③ 任务的业务重试次数（`retry_max`）未被这次转移消耗。

**Acceptance Scenarios**:

1. **Given** 集群有 1 个健康节点和 1 个 incarnation 反复抖动的假节点，**When** 提交若干任务，**Then** 全部任务只派给健康节点，假节点承接数为 0。
2. **Given** 一个任务已 DISPATCHED 到某节点，**When** 该节点进程重启（incarnation 变化），**Then** 该任务被判为基础设施回收并转移到另一健康节点执行，**且** 其业务重试计数不变。
3. **Given** 一个节点连续多次下发失败/回收，**When** 超过熔断阈值，**Then** 该节点进入隔离（quarantine）不再被派任务；稳定一段时间或成功执行一次后自动恢复候选资格。
4. **Given** 某任务的执行反复搞垮承接它的 worker（毒任务），**When** 该实例的连续基础设施重派超过单实例上限，**Then** 该实例被挂起并告警交人工，不再继续重派，机群不再被它持续拖垮。

---

### User Story 2 - 无健康节点时安全等待，恢复即自动跑（Priority: P2）

作为平台运维，当此刻没有任何健康节点可用时，我希望就绪的任务**安全地等待**（不被判失败、不消耗重试），并且**一旦有健康节点恢复，等待的任务立即开始执行**，系统健康时不残留任何"该跑却没跑"的等待。

**Why this priority**: "等待"的语义是等资源/等系统状态，而非等判决。保证恢复后无残留等待，是"不丢"的另一半。依赖 US1 的健康判定，故 P2。

**Independent Test**: 让所有节点不可用，提交任务，验证任务停在等待态且不判失败、不消耗重试；随后恢复一个健康节点，验证等待任务在既定时限内全部开始执行，无残留。

**Acceptance Scenarios**:

1. **Given** 集群无任何健康节点，**When** 任务就绪，**Then** 任务保持等待态，既不判终态失败也不消耗业务重试。
2. **Given** 若干任务因无健康节点而等待，**When** 一个健康节点恢复（上线/隔离解除/纪元稳定），**Then** 系统主动唤醒，等待任务在设定时限内被重新认领并开始执行。
3. **Given** 任务长期无健康节点可用，**When** 等待超过设定阈值，**Then** 系统发出告警（可见性），但**不**自动判死。

---

### User Story 3 - 长跑/流式任务不被误杀，状态丢失可兜底（Priority: P3）

作为平台运维，我希望**确实在长时间运行**的任务（含 Flink 流式这类外部托管长驻作业）不被调度器误当成"失联"而杀掉或重复提交；而当任务状态**真的丢失**时，系统能自动兜底（进程内任务重派、外部作业重连）；对于既非正常长跑、又无法自动判定的疑难情形，我能通过**人工停止 + 重跑**作为最终兜底。

**Why this priority**: 处理 RUNNING 态的可信度边界（含外部托管作业），是闭环的收尾。它需要 US1/US2 的机制铺垫，且部分依赖运行时真实续约与外部句柄，故 P3。

**Independent Test**: ① 跑一个运行时长明显超过一个租约周期的进程内任务，worker 正常存活，验证它不被回收；② 杀掉 worker 进程模拟状态丢失，验证进程内任务在租约到期后被自动重派并最终完成；③ 提交一个 Flink 流式作业后杀掉承载 worker，验证新 worker 按外部句柄重连恢复监控、集群里**不**出现第二个作业；④ 对一个卡死的运行中实例执行人工停止，再重跑，验证其重试计数被重置且能重新执行。

**Acceptance Scenarios**:

1. **Given** 一个进程内任务的实际运行时长 > 一个租约周期且 worker 持续存活，**When** 租约窗口反复到来，**Then** 该任务不被回收、持续运行至自然结束。
2. **Given** 一个 RUNNING 的进程内任务的 worker 进程崩溃，**When** 租约在续约窗口内未被续上，**Then** 该任务被判为基础设施回收并自动重派到健康节点，无需人工介入。
3. **Given** 一个 Flink 流式作业（外部托管长驻）正常运行、其外部作业句柄已持久化，**When** 承载它的 worker 故障，**Then** failover 为另一 worker 按句柄**重连**恢复监控，集群中该作业的重复实例数为 0；且作业本身在集群中不受影响持续运行。
4. **Given** 一个进程内任务配置了最大运行时长，**When** 运行超过该时长，**Then** 该任务被判超时终态（区别于基础设施回收）；外部托管长驻作业**不**受此约束。
5. **Given** 一个疑似卡死、无法自动判定的 RUNNING 实例，**When** 运维执行人工停止再重跑，**Then** 实例进入终态后被重置为可重新认领，业务重试计数归零，并重新开始执行。

---

### Edge Cases

- **worker 真重启（任务正 RUNNING）**：旧实例必须被恰好回收一次并转移；刚重启的节点在稳定窗内不被立即再派任务。
- **对等 master 并发**：两个 master 同时回收同一实例、或同时熔断同一节点时，结果必须与串行等价——不重复回收、不把隔离状态覆盖成更宽松。
- **迟到成功回报**：任务在被回收/重派之后，旧 worker 才回报成功，该回报必须被丢弃，不能把已重派的实例错误标成成功。
- **下发命令滞留**：下发命令在队列中滞留超租约被回收重派后，旧命令即使迟发也必须被识别为过期而跳过，不得双投。
- **基础设施反复失败 / 长期无节点**：任务可多次基础设施重派（带退避），但**永不**因此判终态失败；长期卡住必须以告警形式暴露，杜绝静默黑洞。
- **毒任务 / 崩溃循环**：某实例的执行反复搞垮 worker，达到**单实例重派上限**后必须挂起 + 告警 + 人工，不得无限重派危害机群。
- **网络分区物理双跑**：进程内任务在旧 worker "存活但失联"时被 master 重派，旧 worker MUST 在续约窗口内失联即自我中止，避免与重派实例并发物理执行。
- **外部长驻作业 worker 故障**：Flink 流式作业的监控 worker 挂掉，failover MUST 为按句柄重连，绝不重新提交；作业本身在集群中持续运行不受影响。
- **长跑 vs 状态丢失**：仅凭"运行时间长"不得作为回收依据；进程内任务的回收依据是"续约在窗口内缺失"，外部托管作业的存活依据是"按句柄探测作业仍在"。

## Requirements *(mandatory)*

### Functional Requirements

**节点可用性与剔除（对应 US1）**

- **FR-001**: 系统 MUST 在派发候选中排除**心跳不新鲜**的节点（超过离线阈值未上报）。
- **FR-002**: 系统 MUST 在派发候选中排除**进程纪元不稳定**的节点，即当前 incarnation 的持续时间未达到稳定窗（stabilization window）的节点（专治抖动型假节点与刚重启未稳定的节点）。
- **FR-003**: 系统 MUST 对节点维护基于**近期基础设施故障计数**的熔断：连续故障跨过阈值即将节点隔离（quarantine）一段退避时间，隔离期内不参与派发。计入熔断计数的故障 MUST 限于"下发不可达 / 运行中执行异常回收"这类归因于节点自身的事件；节点**正常重启**（incarnation 变化）由稳定窗（FR-002）处置，不重复计入熔断计数。
- **FR-004**: 系统 MUST 在满足以下任一条件时自动解除节点隔离：隔离退避到期后节点心跳新鲜且纪元稳定，或该节点成功执行了一次任务；解除后故障计数复位。
- **FR-005**: 节点可用性过滤 MUST 收口在派发候选的**唯一来源处**（一处过滤，避免多点判定不一致）。
- **FR-006**: 节点健康状态的所有更新 MUST 以原子/单调的方式进行（故障计数原子自增、隔离到期时刻只增不减），保证对等 master 并发更新同一节点时结果与串行等价。

**任务转移与重试语义（对应 US1）**

- **FR-007**: 系统 MUST 区分**基础设施回收**（节点丢失/节点重启/下发不可达）与**业务失败**（任务真正运行后失败）两类失效。
- **FR-008**: 系统 MUST 保证基础设施回收**不消耗任务的业务重试次数**（`retry_max`），且**永不**因基础设施回收把任务判成终态失败。
- **FR-009**: 系统 MUST 仅在任务**真正进入过运行态之后**发生的失败才计入业务重试；业务重试耗尽方可判终态失败。
- **FR-010**: 任务被回收重派时，系统 MUST 清除其上一次的节点归属，使下一轮认领可以选择**不同的健康节点**（实现转移）。
- **FR-011**: 系统 MUST 保留一个**严格单调递增、永不回退的下发纪元令牌**（每次下发递增，无论基础设施还是业务原因），作为防止同一实例双投/双跑的栅栏；该令牌与业务重试计数 MUST 相互独立。
- **FR-012**: 系统 MUST 对**单个实例**的连续基础设施重派设上限；超过上限时 MUST 挂起该实例（停止自动重派）并发出告警交人工处置，MUST NOT 自动判终态失败。此为毒任务/崩溃循环的机群保护边界。

**等待与恢复抽干（对应 US2）**

- **FR-013**: 当无任何健康节点可承接时，就绪任务 MUST 保持等待态，既不判终态失败也不消耗任何计数。
- **FR-014**: 当节点健康状态发生**恢复性变化**（上线 / 隔离解除 / 纪元稳定达标）时，系统 MUST 主动触发唤醒，使等待任务在设定时限内被重新认领，而非仅依赖兜底轮询。系统健康时 MUST NOT 残留"就绪却未被认领"的等待。
- **FR-015**: 当任务因无健康节点而等待超过设定阈值时，系统 MUST 发出可见告警，但 MUST NOT 因此自动判终态失败。

**运行中任务可信度（对应 US3）**

- **FR-016**: worker MUST 在心跳中真实上报其当前正在运行的实例标识集合（而非空集），供 master 续约。
- **FR-017**: 系统 MUST 仅在（进程内）运行中实例的**租约在续约窗口内未被续上**时才回收它；MUST NOT 仅以"运行时间长"作为回收依据。
- **FR-018**: 运行中任务的 worker 持续存活并续约时，系统 MUST NOT 回收该任务（保证长跑不被误杀）。
- **FR-019**: 当（进程内）运行中任务因 worker 失联而租约到期时，系统 MUST 将其判为基础设施回收并自动重派。
- **FR-020**: 系统 MUST 支持任务级**可选**最大运行时长（max-runtime timeout，本特性范围内实现）：仅在任务配置了该值时生效，超时判为超时终态（区别于基础设施回收）；未配置时 MUST NOT 按墙钟强杀正在续约的长跑任务。外部托管长驻作业（FR-023~026）豁免此约束。
- **FR-021**: 对**进程内执行**的任务（在 worker 进程内运行的 SQL/Shell/Python 等），worker MUST 在其租约续约窗口内失联 master 超过阈值时**自我中止**本地执行，使网络分区下不发生并发物理双跑。外部托管长驻作业豁免此自我中止。

**外部托管长驻作业（对应 US3 / Flink 流式，Clarification Q4）**

- **FR-022**: 系统 MUST 区分**外部托管长驻作业**（真实执行运行在外部引擎集群、生命周期与 worker 进程解耦，如 Flink 流式）与**进程内/有界任务**两类，并按类别施加不同的容错语义。
- **FR-023**: 提交外部托管长驻作业成功后，系统 MUST 持久化其**外部作业句柄**（引擎作业标识 + 集群 / tracking 端点）到任务实例，作为重连、状态展示与后续扩展（实时任务卡片、savepoint/取消、指标）的统一挂载点。
- **FR-024**: 当承载外部托管长驻作业的 worker 失联/故障时，failover MUST 为按句柄**重连（reattach）恢复监控**，MUST NOT 重新提交作业（避免在集群里产生重复作业）；仅当按句柄探测确认外部作业确实已不存在/已失败时，方可按业务重试重新提交。
- **FR-025**: 外部托管长驻作业的任务实例 MUST 在作业存活期间保持运行态；其"存活/续约"信号来自"监控 worker 存活 **且** 按句柄探测外部作业仍在运行"，而非固定墙钟或子进程阻塞。
- **FR-026**: 外部托管长驻作业 MUST 豁免任务级最大运行时长（FR-020）与 worker 自我中止（FR-021）。

**人工兜底（对应 US3）**

- **FR-027**: 系统 MUST 提供人工**停止**运行中/疑难实例并将其置为终态的能力（复用既有停止能力）；对外部托管长驻作业，停止 MUST 同时按句柄取消集群侧作业。
- **FR-028**: 系统 MUST 提供人工**重跑**终态实例的能力，重跑时 MUST 同时重置下发纪元令牌与业务重试计数，并触发重新认领。

**一致性总不变量（横切全部）**

- **FR-029**: 所有失效回收与状态推进 MUST 通过乐观 CAS（以当前状态为条件）裁决，保证并发下"单赢"——同一实例不被重复回收、不被同时推进到冲突状态。
- **FR-030**: 系统 MUST 保证任一时刻同一任务实例最多有一个下发命令被判为"当前"，过期下发命令 MUST 被识别并跳过（不双投）；worker 侧幂等 MUST 以下发纪元令牌为键，保证同一实例不双跑。
- **FR-031**: 本特性对 `task_instance` / 节点健康 / 任务定义（max-runtime、任务类别）的任何列变更 MUST 按项目约定同步提升 schema 版本。

### Key Entities *(include if feature involves data)*

- **节点健康（Worker Node Health）**：描述一个 worker 节点当前是否可承接任务。关键属性：当前进程纪元的起始时刻（用于稳定窗判定）、近期连续基础设施故障计数、隔离到期时刻。与"心跳新鲜度"共同决定该节点是否进入派发候选。
- **任务实例（Task Instance）**：关键属性区分两个独立计数——**下发纪元令牌**（每次下发递增、只增不减，作双投栅栏）与**业务重试计数**（仅真正运行后失败才增，与 `retry_max` 比较）；**单实例基础设施重派计数**（用于毒任务上限与挂起判定）；**租约到期时刻**（进程内运行中任务续约的判据）；**失效原因/状态**（基础设施回收 / 业务失败 / 超时 / 停止 / 挂起 的区分与可见性）；以及外部托管作业的**外部作业句柄**引用。
- **外部作业句柄（External Job Handle）**：外部托管长驻作业提交成功后的稳定标识——引擎作业标识 + 集群 / tracking 端点（+ 作业类别）。持久化在任务实例上，是 failover 重连、状态探测、以及未来实时任务卡片等 UI 扩展的统一挂载点。
- **任务类别（Task Class）**：将任务区分为"进程内/有界"与"外部托管长驻"，据以选择容错语义（重派 vs 重连、是否受 max-runtime/自我中止约束）。
- **失效分类（Failure Classification）**：把一次实例失效归类为"基础设施回收"/"业务失败"/"超时"，据此决定是否消耗业务重试、是否可判终态、是否触发转移或重连。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 集群中存在坏/假节点时，该节点承接的任务 **100%** 自动转移到健康节点并最终完成，因节点原因导致的终态失败数为 **0**。
- **SC-002**: 幽灵/纪元抖动节点承接的任务数为 **0**（从不被派发）。
- **SC-003**: 无健康节点期间，就绪任务判终态失败数为 **0**、业务重试消耗为 **0**；健康节点恢复后，等待任务在 **不超过一个兜底轮询周期** 内全部开始执行，且系统健康稳态下"就绪未认领"的残留等待数为 **0**。
- **SC-004**: 实际运行时长超过一个租约周期、且 worker 持续存活的长跑（进程内）任务，被误回收的比例为 **0**。
- **SC-005**: worker 崩溃丢失的运行中进程内任务，**100%** 在租约到期后自动重派并最终完成，无需人工介入（业务真失败或配置超时除外）。
- **SC-006**: 在坏节点注入 + 对等 master 并发 + 下发滞留 + 迟到回报的组合压力下，同一任务实例双跑次数为 **0**、就绪任务丢失数为 **0**、因基础设施原因误判终态数为 **0**。
- **SC-007**: 外部托管长驻作业（Flink 流式）在承载 worker 故障后，集群中该作业的重复实例数为 **0**（failover 走 reattach 而非重提交）；监控 worker 接管后作业状态可见性恢复。
- **SC-008**: 毒任务（执行反复搞垮 worker）在达到单实例基础设施重派上限后 **100%** 被挂起并告警、停止继续重派；机群不因单个毒任务持续受损。
- **SC-009**: 对疑难卡死实例，运维可在一次"停止 + 重跑"操作内使其重新开始执行，重跑后业务重试计数归零。

## Assumptions

- **任务幂等由任务自身负责**：平台对基础设施故障提供的是 at-least-once 的自动重派语义；被转移/重派的（进程内）任务可能从头重跑，任务的幂等性由任务作者保证。
- **续约信号来自 worker 心跳**：区分"长跑"与"状态丢失"依赖 worker 如实上报正在运行的实例集合；worker 存活即视为其上报的进程内实例仍在运行。
- **外部引擎可重连**：外部托管长驻作业的引擎（如 Flink 集群）支持按作业标识查询/重连（reattach），且提交时能取回稳定的作业标识。当前 Flink 执行器为"阻塞子进程 + 1h 超时、未记 job id、未重连"范式，本特性需将其演进为"提交记句柄 + 探测/重连"模型；此演进属本特性范围。
- **复用既有能力**：人工停止/重跑复用现有运维接口；节点心跳与租约、SKIP LOCKED 认领、乐观 CAS、固定锁序（task→workflow）、事务内落状态/事务外下发等既有调度不变量保持不变，本特性在其上叠加，不得破坏（对齐 constitution 原则 V "内核复用"）。
- **保守默认 + 可配置**：稳定窗、熔断阈值与退避曲线、无节点等待告警阈值、续约窗口、单实例基础设施重派上限等数值先取保守默认并可配置，具体调参在实测阶段完成。
- **幽灵 worker-local 的部署侧修复已单独落地**（master 容器不再注册内嵌 worker，已于基线 commit `2b137e5`）；本特性解决的是更一般的"任何坏/假/丢失节点"的调度容错，不依赖该部署修复，但与之互补。
- **未来 UI 扩展点**：外部作业句柄的持久化即为后续"工作流实例/任务实例上的实时任务卡片"预留的数据挂载点；本特性只负责产出并维护该句柄与状态，不含 UI 实现。

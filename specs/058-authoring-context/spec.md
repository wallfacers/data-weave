# Feature Specification: 数据开发 LSP —— 血缘/依赖接地的创作上下文服务

**Feature Branch**: `058-authoring-context`

**Created**: 2026-07-08

**Status**: Draft

**Input**: User description: "在 cli 写代码的过程，可以基于血缘、解析、上游任务等操作做推荐（AI 推荐），并有更高的代码检查和复用正确率。类比现在 AI 编程工具基于 LSP 语法树解析做到更好的意图识别——企业级数据中台任务链路很长，vibecoding 数据开发任务时若知道血缘、任务依赖关系，大模型很快就能识别意图。"

## 背景与动机 *(为什么做这件事)*

企业级数据中台的任务链路很长：一张目标表往往由多层上游任务/表逐级加工而来。开发者在 CLI 里用 AI 编码 agent「vibecoding」一个数据任务时，agent 只能看到当前这一个脚本文件，看不到「这张表的上游是谁、下游谁依赖、已有哪些任务产出过类似结果」。缺了这层语义，大模型对**意图**的识别只能靠脚本字面猜测，导致：引用错上游、重复造已有的加工逻辑、改动破坏下游列契约却无从察觉。

通用 AI 编程工具靠 **LSP + 语法树**获得「跳定义 / 找引用 / 诊断」这组确定性语义能力，从而更准地识别代码意图。本特性给**数据开发**提供等价物：一个 **血缘/依赖接地的创作上下文服务**，把「上游/下游任务依赖、表/列血缘、复用候选、一致性诊断」作为一组**确定性、按需查询**的能力，喂给 CLI 回路里的 AI 编码 agent，让它**秒懂当前任务在整条数据链路中的位置与意图**，从而提高代码检查与复用的正确率。

**关键立场（边界，先钉死）**：

- 「LSP」是**类比**，不是线协议、不接编辑器。本特性提供的是一组请求/响应式的**查询 + 诊断**能力。
- **唯一的「AI」是开发者手上的编码 agent**。本服务只产出**结构化事实 + 诊断**（全程确定性、可复现、防幻觉）；agent 消费这些事实后自行推理与推荐。本服务内部不调用大模型。
- 本特性是**纯读 + 诊断**能力，**不含任何写/创作能力**——创作仍只走既有 CLI/Skill 黄金路径与 push 策略闸门。

## Clarifications

### Session 2026-07-08

- Q: 创作上下文包默认遍历血缘邻域多远？ → A: 多跳；遍历深度作为**调用方（消费的 AI agent）自决参数**，默认给足广度，仅对极端广度按邻域截断并显式标注。
- Q: 什么触发一条复用候选？ → A: 以**写表目标重叠**为主（同目标表或高列重叠），名称相似度仅作打分加权，不单独触发候选。
- Q: 草稿分析的输入形态是？ → A: **整个本地工作副本**（可含多个任务草稿一起分析，草稿间未 push 的跨任务依赖也纳入）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创作上下文接地：上游/下游依赖 + 表/列血缘（Priority: P1）

数据开发者在 CLI 里编辑一个数据任务（脚本或 SQL，可能尚未 push）。开发者（或其 AI 编码 agent）请求「这个任务的创作上下文」，立即拿到一个结构化的**意图接地包**：该任务读的表及其**上游生产者**（哪些任务/表产出它们）、写的表及其**下游消费者**（哪些任务/表依赖它们）、这些读写表的**列级血缘**、以及任务绑定数据源的真实 schema。上下文对**本地草稿**同样有效——不需要先 push。AI agent 拿到这个接地包后，无需翻阅其他任务文件即可判断当前任务在整条链路中的位置与意图。

**Why this priority**: 这是整个特性的地基——「模型知道血缘与依赖才能秒懂意图」正是用户诉求的核心。复用推荐与一致性检查都从这个接地包取数。单独交付即构成可用增量：agent 有了链路视野就能显著减少「引用错上游 / 误解目标」。

**Independent Test**: 准备一条多层链路（表 A ← 任务 T1 ← 表 B ← 任务 T2）。对 T1 请求创作上下文，返回必须包含：T1 读表 B、B 的上游生产者 T2、T1 写表 A、A 的下游消费者、以及 A/B 的列血缘；对**未 push 的 T1 草稿文件**重复请求得到等价接地包——即为通过。

**Acceptance Scenarios**:

1. **Given** 一个已 push、绑定数据源的数据任务，**When** 请求其创作上下文，**Then** 返回读表→上游生产者、写表→下游消费者、读写表列血缘、绑定数据源 schema，均带来源标记与接地状态（存在/推断/未接地）。
2. **Given** 一份含该任务草稿的**本地工作副本**（尚未 push，可能还含其他相关任务草稿），**When** 请求创作上下文，**Then** 返回与已 push 版本语义等价的接地包，草稿间未 push 的跨任务依赖也被纳入，且**不产生任何服务端写入或持久化**。
3. **Given** 任务读了一张平台从未见过、也无法从数据源接地的表，**When** 请求上下文，**Then** 该表以「未接地/无已知生产者」状态返回，不虚构上游（防幻觉）。
4. **Given** 上下文服务的图数据源暂时不可用，**When** 请求上下文，**Then** 返回可获取的部分事实并明确标注缺失部分，绝不整体报错拖垮 CLI 回路。

### User Story 2 - 复用推荐：已有定义中的重叠输出候选（Priority: P2）

开发者正要写一个新任务去产出某张表/某些列。系统主动指出「已有任务/表已产出重叠的输出」，给出按**重叠度 + 名称相似度确定性排序**的复用候选清单（含各候选产出什么、与当前意图重叠在哪）。开发者据此决定复用既有定义、还是确有必要新建，从而减少企业内重复建设的加工逻辑。

**Why this priority**: 直接兑现用户诉求里的「更高的复用正确率」。它建在 P1 的接地包之上（复用候选来自同一套血缘/定义视图），可独立交付：即使不做一致性检查，复用推荐本身就能压低重复任务的产生率。

**Independent Test**: 库中已存在任务 X 产出表 `dw.user_daily(id,name,amt)`。对一个意图产出 `dw.user_daily` 或高度重叠列集的新草稿请求复用推荐，返回候选须包含 X 并说明重叠列——而对一个产出完全无关表的草稿返回空候选——即为通过。

**Acceptance Scenarios**:

1. **Given** 已有任务产出与当前草稿目标表/列高度重叠的输出，**When** 请求复用推荐，**Then** 返回该候选并给出重叠度与重叠的具体表/列，按确定性分数排序。
2. **Given** 当前草稿目标与任何已有定义均无重叠，**When** 请求复用推荐，**Then** 返回空候选，不臆造复用项。

---

### User Story 3 - 一致性诊断：声明依赖 vs 实际数据血缘（Priority: P3）

开发者改完任务，请求「一致性检查」。系统把**创作代码隐含的血缘**（从脚本/SQL 抽取）与平台已有的两类事实比对，产出诊断清单：① 读了无已知生产者/未接地的表（悬空上游）；② 写的表其下游消费者期望的列被破坏（列契约破坏）；③ 重复了已有定义（复用提示）；④ **声明的工作流 DAG 依赖与实际数据血缘背离**——声明了上游依赖却无数据流经过（僵依赖），或存在数据流却未声明依赖（缺依赖）。诊断为**建议性**，帮助开发者与 agent 在 push 前修正，绝不自动阻断创作。

**Why this priority**: 兑现「更高的代码检查正确率」。第 ④ 类诊断（声明 DAG vs 实际数据血缘背离）是本平台「双依赖源」（开发者声明的控制流 + 数据推导的血缘）独有的高价值检查。它建在 P1 抽取 + 接地之上，最后交付。

**Independent Test**: 造一个任务，其脚本实际读取表 B（有数据流），但其所在工作流未声明对产出 B 的上游任务的依赖；请求一致性检查须报出「缺依赖」诊断；再造一个声明了上游依赖但脚本根本不读其产物的任务，须报出「僵依赖」——即为通过。

**Acceptance Scenarios**:

1. **Given** 任务脚本读取某表但工作流未声明对其生产者的依赖，**When** 请求一致性检查，**Then** 报出「缺依赖」诊断，指明涉及的表与建议的上游任务。
2. **Given** 工作流声明了某上游依赖但任务并不读取其产物，**When** 请求一致性检查，**Then** 报出「僵依赖」诊断。
3. **Given** 任务写的表被下游消费者按固定列集消费，而本次改动移除/改名了这些列，**When** 请求一致性检查，**Then** 报出「下游列契约破坏」诊断，指明受影响的下游。
4. **Given** 一个各方面都一致的任务，**When** 请求一致性检查，**Then** 返回空诊断集（无误报）。
5. **Given** 任一诊断被报出，**When** 开发者继续 push，**Then** 诊断为建议性、不阻断 push（push 仍由既有策略闸门独立裁决）。

---

### Edge Cases

- 草稿脚本语法无法解析/方言不支持时，上下文与诊断如何降级？（应返回可得的表级事实并标注「列级不可用」，不整体失败。）
- 目标表/上游表跨多个数据源时，依赖与血缘如何归属与去重？
- 同一语义键被多通道产出（确定性抽取 vs 推断）时，上下文里以哪条为准、如何标注来源可信度？
- 链路极深/极宽（上游数百）时，上下文包如何限定广度以保证可用（如按邻域跳数截断并标注「已截断」，绝不静默丢失）？
- 请求的任务/草稿引用了软删除或历史版本的定义时，依赖如何呈现？
- 复用候选与当前草稿属于不同项目/租户时，是否跨界（默认：租户+项目隔离，不跨界）。
- 工作副本内多个草稿互相引用（未 push 的跨任务新依赖）时，依赖以草稿为准解析，再回退服务端图谱（FR-019）。
- 调用方要求的遍历深度过深导致上下文包过大时，如何在"给足广度"与可用延迟间平衡（按广度截断并标注，深度不设硬顶但延迟由调用方承担）。

## Requirements *(mandatory)*

### Functional Requirements

**上下文接地（US1，P1）**

- **FR-001**: 系统 MUST 能对一个数据任务（按标识）或一份**本地工作副本**（可含多个任务草稿，各带内容/类型/绑定数据源元数据）返回其**创作上下文包**；血缘邻域**遍历深度由调用方（消费的 AI agent）作为参数自决**，默认给足广度（多跳）。
- **FR-002**: 创作上下文包 MUST 包含：读表及其上游生产者（任务/表）、写表及其下游消费者（任务/表）、读写表的列级血缘、绑定数据源的真实列 schema。
- **FR-003**: 每条事实 MUST 带**来源标记**与**接地状态**（存在/推断/未接地），未接地的表 MUST NOT 被虚构出上游。
- **FR-004**: 对本地草稿的分析 MUST 为**无状态**：不产生服务端写入、不改动血缘图谱、不落任何持久化记录。
- **FR-005**: 任一底层事实源（血缘图/数据源/定义库）不可用时，系统 MUST 返回可得的部分事实并标注缺失，MUST NOT 整体失败。
- **FR-006**: 上游/下游依赖 MUST 合并两个来源——开发者**声明的工作流 DAG 依赖**与**数据血缘推导的依赖**——并分别标注来源。

**复用推荐（US2，P2）**

- **FR-007**: 系统 MUST 能对一个任务/草稿返回**复用候选**清单——已有定义中，其**写表目标**（产出的表/列）与当前草稿写表目标重叠者；名称相似度仅作打分加权，MUST NOT 单独触发候选。
- **FR-008**: 复用候选 MUST 按**确定性分数**（写表目标的表/列重叠度为主 + 名称相似度加权）排序，每条 MUST 说明其产出与重叠的具体表/列。
- **FR-009**: 无重叠时系统 MUST 返回空候选，MUST NOT 臆造复用项。

**一致性诊断（US3，P3）**

- **FR-010**: 系统 MUST 能对一个任务/草稿返回**一致性诊断**清单，至少覆盖：悬空上游、下游列契约破坏、重复已有定义、声明依赖 vs 实际血缘背离（缺依赖/僵依赖）。
- **FR-011**: 每条诊断 MUST 指明涉及的实体（表/列/任务）与可执行的修正建议。
- **FR-012**: 一致性诊断 MUST 为**建议性**，MUST NOT 自动阻断 push 或创作（push 仍由既有策略闸门独立裁决）。
- **FR-013**: 各能力（上下文/依赖/复用/诊断）MUST **确定性**、不依赖大模型，相同输入产出相同结果。

**暴露面与隔离（贯穿）**

- **FR-014**: 上述能力 MUST 由**同一后端能力**产出，并通过**两个面**薄封装暴露：① CLI 命令（供 agent 在 vibecoding 回路里按需调用、读结构化输出）；② MCP 只读工具（供自动化查询）。两面语义等价。
- **FR-015**: 借本特性 MUST **修正既有 MCP 血缘查询的漂移**——现有 MCP 血缘查询仍停留在「指标→SQL→物理表」的旧模型，MUST 升级/补齐为表/列级血缘与依赖只读查询。
- **FR-016**: 所有能力 MUST 遵守既有**租户 + 项目隔离**；上下文、依赖、复用候选、诊断 MUST NOT 跨越请求身份的租户/项目边界。
- **FR-017**: 既有创作 Skill MUST 被扩展，教会 AI 编码 agent 何时调用这组能力（编辑前取上下文、改完做检查）。
- **FR-018**: 遍历深度由调用方自决（FR-001）；当某一层**广度**超过可用阈值（如单表上游数百）时，系统 MUST 按邻域截断并**显式标注已截断**，MUST NOT 静默丢失事实。
- **FR-019**: 分析本地工作副本时，草稿间的跨任务依赖 MUST 先在工作副本内的草稿之间解析（未 push 的新依赖可见），再回退到服务端已 push 图谱；同一任务同时存在草稿与已 push 版本时，MUST 以草稿为准。

### Key Entities *(include if feature involves data)*

- **创作上下文包（Authoring Context）**：某任务/草稿的意图接地视图。属性：读表集、写表集、每表的上游生产者与下游消费者、列级血缘、绑定数据源 schema、各事实的来源与接地状态。
- **任务依赖（Task Dependency）**：任务间的依赖关系，含**声明来源**（工作流 DAG 边）与**推导来源**（数据血缘），带方向（上游/下游）。
- **复用候选（Reuse Candidate）**：一条已有定义，与当前目标存在输出重叠。属性：候选定义标识、产出的表/列、与当前意图的重叠集、确定性相似分。
- **一致性诊断（Consistency Diagnostic）**：一条建议性发现。属性：类别（悬空上游/列契约破坏/重复定义/依赖背离）、涉及实体、修正建议、严重度。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对一条典型多层链路（≥3 层）中的任务，AI 编码 agent 在**仅**获得创作上下文包、不读取其他任务文件的前提下，能正确说出该任务的直接上游与下游，正确率 ≥ 90%。
- **SC-002**: 对本地工作副本请求创作上下文，在企业级规模（数千任务定义）、**默认遍历深度**下 **5 秒内**返回；调用方显式加深遍历时延迟随之增长（可接受），结果始终可复现（同输入同输出）。
- **SC-003**: 在存在可复用定义的场景中，复用推荐将「本可复用却重复新建」的任务占比相对无此能力时**下降 ≥ 50%**。
- **SC-004**: 一致性诊断对「声明依赖 vs 实际血缘背离」的检出：在预置夹具上召回 ≥ 90% 的真实背离，且对一致任务的误报率 ≤ 5%。
- **SC-005**: 创作上下文/诊断中呈现的每一条表级事实均可追溯到确定性来源，**虚构（无法在脚本/图/schema 中定位）的事实占比 = 0%**。
- **SC-006**: CLI 与 MCP 两面对同一任务/草稿返回**语义一致**的结果（同一后端能力，无双实现漂移）。

## Assumptions

- **定位为类比而非线协议**：本特性不实现编辑器 LSP 线协议、不接 IDE；「数据开发 LSP」指一组请求/响应式的血缘/依赖查询 + 诊断能力。
- **消费方是既有编码 agent**：推荐/意图识别由开发者手上的 AI 编码 agent 完成，本服务只供确定性事实与诊断，内部不调用大模型；「AI 推荐」= agent 消费这些事实后的产物。
- **纯读 + 诊断，无写能力**：不新增任何创作/写入路径；创作仍走既有 CLI/Skill 黄金路径与 push 策略闸门。诊断为建议性、永不自动阻断 push。
- **复用既有血缘与依赖地基**：直接复用平台已有的表/列血缘图、目录接地（存在真伪裁决）、脚本/SQL 血缘抽取、以及工作流 DAG 依赖，不新建独立血缘存储。
- **工作副本分析无状态**：对未 push 的本地工作副本（可含多个任务草稿）的分析不落任何持久化、不改血缘图谱；遍历深度由消费的 AI agent 自决、默认给足广度。
- **确定性优先**：复用排序与依赖合并用确定性规则（重叠度/名称相似/图查），不引入需训练的推荐模型；云 AI 抽取（异步富化）不在本特性的同步查询路径内。
- **隔离沿用既有机制**：租户 + 项目隔离沿用平台既有约束，本特性不放宽边界。
- **分期交付**：全景写入本 spec，实现按 P1（上下文+依赖+双面暴露+修 MCP 漂移）→ P2（复用推荐）→ P3（一致性诊断）增量落地，每期独立可测、独立可用。
- **隔离于并行特性**：本特性在 worktree `dw-058-authoring-context` 隔离开发；与在做的 054（测试集 B）、057（system-settings）无共享改面，集成前按跨特性规则复核。

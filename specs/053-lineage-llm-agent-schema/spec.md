# Feature Specification: 血缘解析扩展——可配置云 AI Agent 抽取通道 + 数据源实时 Schema 解析

**Feature Branch**: `053-lineage-llm-agent-schema`

**Created**: 2026-07-07

**Status**: Draft

**Input**: User description: "血缘解析章节，有支持小模型解析血缘的功能，这里在这个基础之上，可以可配置的方式，兼容 anthropic 和 openai 两种协议的 AI Agent，直接调用云厂商；比如有些场景 `select * from user`，这里 user 表的字段其实我是不知道，但是任务上绑定了数据源，这里需要通过数据源去获取到 user 表的字段——通过当前模块的数据源信息链接数据库应该可以拿到；你再想想还有哪些场景，把这个补全。"

## 背景与动机 *(为什么做这件事)*

现有血缘解析已具备三条通道：Calcite 语法树解析 SQL 表/字段血缘、脚本内嵌 SQL 抽取（EmbeddedSql）、规则接口抽取（ApiPattern），以及一条本地小模型推断通道（ModelExtractor，走独立推理 sidecar）。041-R/047 的实证负结果表明：合成训练的本地小模型在真实 OOD 脚本上因**记忆泄漏**急剧退化（真实 precision 0.27、方向混淆），单靠小模型无法把长尾脚本血缘做准。

本特性在既有可插拔抽取器地基（`ScriptLineageExtractor` 契约）之上补两块能力，二者互相强化：

1. **可配置的云 AI Agent 抽取通道**：以配置驱动的方式接入兼容 **Anthropic 协议** 与 **OpenAI 协议** 的大模型服务，直接调用云厂商（如任意 OpenAI-兼容 / Anthropic-兼容端点），作为小模型之外的、准确率更高的抽取通道，补长尾。协议、端点、模型名、凭据、启用开关均可配置，无需改代码即可换供应商。
2. **数据源实时 Schema 解析**：当解析需要知道某张表有哪些列、而列元数据未预先登记时（典型：`SELECT *`），通过**任务绑定的数据源连接信息直连数据库**取回真实列清单，喂给确定性列级血缘引擎，同时作为大模型的 schema 上下文以抑制幻觉。

两块能力落地后，血缘解析既有"更聪明的推断通道"，又有"能看到真实 schema 的地基"，共同把 `SELECT *`、脚本长尾等历史盲区补齐。

## Clarifications

### Session 2026-07-07

- Q: 云 AI Agent 抽取通道相对 push 的执行时机？ → A: **异步后台富化**——push 立即返回不阻塞；AI 边由后台任务在 push 之后补齐并入图谱（带来源标记）。云端大模型单次调用需数秒~数十秒，不得阻塞交互式 push。
- Q: 同一血缘语义键被多条通道同时产出、冲突时的消解优先级？ → A: **确定性来源优先**——按可信度排序：内嵌 SQL/Calcite > 规则 > AI 推断 > 本地小模型；高确定性通道覆盖低确定性通道的同键边。
- Q: 云 AI Agent 通道适用的任务类型范围？ → A: **脚本 + Calcite 解析失败的 SQL**——Python/Shell/Spark 脚本长尾，外加因方言/不支持语法而 Calcite 解析失败的 SQL 任务兜底；Calcite 能正常解析的 SQL 不叠加 AI。
- Q: 数据源实时解析到的列 Schema 缓存的新鲜度策略？ → A: **重 push 失效 + TTL 兜底**——任务重新 push 时使其引用表的列缓存失效重解析，另设 TTL 兜底防长期漂移。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 可配置的云 AI Agent 血缘抽取通道（Priority: P1）

数据平台管理员在数据源/连接配置界面新增一个"血缘 AI Agent"配置：选择协议类型（Anthropic 或 OpenAI），填入 base URL、模型名、API Key，保存并启用。此后，脚本任务（Python/Shell/Spark）以及因方言/不支持语法而 Calcite 解析失败的 SQL 任务，其规则与内嵌 SQL 通道覆盖不到的读写表场景，交由这条云 AI Agent 通道推断。因云端大模型调用需数秒~数十秒，该通道**异步**执行：push 立即返回不被阻塞，AI 边在 push 之后由后台补齐并入图谱，带"AI 推断"来源标记与置信度，与既有规则/小模型边走同一套防幻觉校验、冲突消解、宁缺毋滥与人工修正机制。

**Why this priority**: 这是用户诉求的主干，也是补 041/047 长尾失败的最直接手段。它以既有 `ScriptLineageExtractor` 可插拔契约为地基，新增一条通道即可，存储与展示层零改动，单独交付即构成可用增量——管理员配好一个云端点就能立刻用上更强的推断能力。

**Independent Test**: 配置一个 OpenAI-兼容云端点（协议=OpenAI），push 一个规则/内嵌 SQL 均不覆盖、但大模型能识别写表意图的 Python 任务，push 立即返回后短时内血缘图谱出现来源标记为"AI 推断"的边（带置信度）；把配置协议切成 Anthropic 指向 Anthropic-兼容端点，重复同一任务仍得到等价的抽取结果——即为通过。

**Acceptance Scenarios**:

1. **Given** 管理员保存并启用了一条 OpenAI 协议的 AI Agent 配置，**When** push 一个规则不覆盖的写表脚本，**Then** push 立即返回，且短时内血缘图谱异步出现来源为"AI 推断"的读/写表边（含可解析的字段边），边上标注来源与置信度。
2. **Given** 同一份脚本，配置切换为 Anthropic 协议并指向 Anthropic-兼容端点，**When** 重新 push，**Then** 抽取结果的表/方向与 OpenAI 协议下语义一致（协议差异对用户不可见）。
3. **Given** AI Agent 未配置或被禁用，**When** push 脚本，**Then** 该通道整体旁路，其余通道（内嵌 SQL / 规则 / 本地小模型）结果不受影响，push/上线流程不阻断。
6. **Given** 一个 Calcite 因方言/不支持语法解析失败的 SQL 任务，**When** push，**Then** AI Agent 通道异步兜底抽取其读写表血缘，产出带"AI 推断"来源标记的边；而 Calcite 能正常解析的 SQL 任务不触发 AI 通道（不做冗余外呼）。
4. **Given** 云端点超时、鉴权失败或返回非法结构，**When** push 脚本，**Then** 该通道降级为空产物并留痕（可查看的提示），绝不抛异常拖垮主链路，也不产出错误边。
5. **Given** 大模型返回了脚本文本中根本不存在的表名，**When** 组装血缘边，**Then** 该边被防幻觉校验拒收并留痕（表名必须能在脚本/SQL 文本中字面定位）。

### User Story 2 - 数据源实时 Schema 解析补全未知列（Priority: P2）

数据开发者的任务里出现 `INSERT INTO dw.user_snapshot SELECT * FROM user`，而 `user` 表的列此前未被登记进列元数据目录。任务绑定了数据源，平台复用该数据源的连接信息直连数据库，取回 `user` 表的真实列清单（含列名、序号、数据类型），把 `SELECT *` 展开为具体列，从而产出完整的字段级血缘——无需开发者手工登记 schema。

**Why this priority**: 这是用户诉求的第二支柱，直接解决 `SELECT *` 等"解析器不知道表有哪些列"的确定性盲区。它依赖任务已绑定数据源这一既有事实，价值独立于 AI 通道；即使不做 US1，US2 也能显著提升 SQL 类任务的字段级血缘完整度。

**Independent Test**: 准备一个绑定了数据源、且该数据源库中真实存在 `user` 表（列元数据未预先 seed 进目录）的 SQL 任务，脚本为 `INSERT INTO dw.snap SELECT * FROM user`；push 后血缘图谱中 `dw.snap` 的字段级血缘按 `user` 的真实列逐列建立，而非整体降级——即为通过。

**Acceptance Scenarios**:

1. **Given** 一个绑定数据源的任务含 `SELECT *`，源表列未预先登记，**When** push 解析，**Then** 系统经数据源连接取回真实列清单并展开 `SELECT *`，产出逐列的字段级血缘。
2. **Given** `INSERT INTO t SELECT ...` 目标表未写列清单，**When** 解析，**Then** 系统取回目标表列清单，按位置映射源列→目标列。
3. **Given** 多表 JOIN 中出现不带表限定的裸列（如 `SELECT id FROM a JOIN b ...`），**When** 解析，**Then** 系统借真实列清单判定该裸列归属哪张表，消歧后建立正确字段边；仍无法唯一确定时不猜、记提示。
4. **Given** 引用的是数据库视图，**When** 解析，**Then** 系统取回视图的输出列清单参与列溯源。
5. **Given** 数据源不可达 / 无该表 / 无读元数据权限，**When** 解析，**Then** 该表的列解析降级（沿用既有列级降级路径）、留可查看提示，不阻断其余语句与表级血缘。
6. **Given** 任务未绑定任何数据源，**When** 出现 `SELECT *`，**Then** 沿用既有降级行为（无列元数据即列级降级），不因缺连接而报错。

### User Story 3 - Schema 上下文喂给 AI Agent 并回填目录缓存（Priority: P3）

平台把 US2 取回的真实 schema 作为上下文注入 US1 的 AI Agent 提示，使大模型"看着真实列名"抽取，显著降低列/表幻觉（正面回应 047 记忆泄漏问题）。同时，凡是经数据源实时解析到的列元数据，回填进列元数据目录缓存，后续对同表的解析直接命中缓存、不再重复连库——把"实时解析"沉淀为"越用越准的地基"。

**Why this priority**: 这是把 US1 与 US2 拧成一股绳的增益：schema 接地是抑制大模型幻觉最有效的手段之一，缓存回填则解决实时连库的性能与稳定性顾虑。它依赖 US1、US2 均已打通，故排 P3。

**Independent Test**: 对一个源列真实存在但目录未登记的脚本任务开启 AI Agent + 实时 schema，push 后 AI 通道产出的字段边全部落在真实列集合内（无越界幻觉列）；再次 push 同表任务时，日志/提示显示列元数据命中缓存而非重新连库——即为通过。

**Acceptance Scenarios**:

1. **Given** AI Agent 已配置且源表 schema 可经数据源取回，**When** push 脚本，**Then** AI 提示中包含真实列清单，模型产出的字段边不含真实列集合之外的幻觉列（越界列被拒收留痕）。
2. **Given** 某表列已被实时解析并回填缓存，**When** 后续任务再次引用该表，**Then** 列元数据命中缓存、不重复连库，且血缘结果与首次一致。
3. **Given** 源库该表结构发生变更（新增/删除列）且缓存已过期，**When** 再次解析，**Then** 系统按既定新鲜度策略刷新列元数据，血缘随之更新。

### User Story 4 - AI 外呼的治理与安全护栏（Priority: P3）

数据治理/安全负责人需要确保"直连云厂商"这件事可控：AI Agent 配置默认关闭、按项目/租户显式开启；凭据加密存储、界面脱敏、不落日志；每次外呼计入审计轨迹；具备超时、并发/频次上限与失败降级，避免 push 被外部服务拖慢或拖垮；可选地对送往云端的脚本/SQL 片段做敏感信息约束提示。

**Why this priority**: 直连外部云厂商引入数据出境、成本与稳定性风险，必须有治理面兜底，否则 US1 不可安全上线。但它的价值依附于 US1 已存在的通道，故与 US3 同级 P3。

**Independent Test**: 在未开启 AI Agent 的项目里 push 脚本，确认无任何外呼且流程正常；开启后 push，确认审计轨迹记录了一次 AI 外呼、配置界面不回显明文 Key、且构造超时/失败时 push 仍在预算内完成并降级——即为通过。

**Acceptance Scenarios**:

1. **Given** 某项目未开启 AI Agent，**When** push 脚本，**Then** 系统不发起任何云外呼。
2. **Given** 管理员保存了 API Key，**When** 再次查看配置，**Then** Key 以脱敏形式呈现、不回显明文，且不出现在任何日志中。
3. **Given** AI Agent 通道被调用，**When** 抽取完成或失败，**Then** 生成一条包含配置标识、协议、耗时、结果状态（成功/降级/拒收）的审计记录。
4. **Given** 云端点响应缓慢，**When** 超过配置的超时/预算，**Then** 通道降级、push 流程在预算内返回，不无限等待。

### Edge Cases

- `SELECT *` 涉及跨库/跨数据源 JOIN，而任务只绑定单一数据源——仅能解析绑定源侧的表，其余表沿用降级并留痕。
- 同名表在不同 schema/catalog 下并存——依连接的默认 schema/当前库做限定名规范化；仍歧义则不猜、留提示。
- 源表列数与目标表列数不一致的 `INSERT ... SELECT *`——按可映射部分建边，多/缺列记提示，不臆造。
- AI Agent 与本地小模型同时启用——两条模型来源的边按 FR-004a 优先级消解（AI 推断 > 本地小模型），同键冲突时 AI 覆盖小模型，被覆盖来源留痕。
- 云端点返回超长/畸形 JSON、限流 429、鉴权 401——分别归为降级并留痕，不同错因给出可区分提示。
- 大模型输出的表名字面存在于脚本但方向（读/写）判断存疑——沿用宁缺毋滥，方向不确定不建有向边。
- 实时连库拉取大宽表（数千列）——需有列数/耗时上限保护，超限截断或降级并留痕。
- 数据源密码轮换后旧连接失效——schema 解析降级，不影响既有已缓存元数据的复用。
- 敏感库表 schema 被送往云端的合规顾虑——受 US4 开关与脱敏约束，默认关闭。

## Requirements *(mandatory)*

### Functional Requirements

**AI Agent 抽取通道（US1）**

- **FR-001**: 系统 MUST 提供一条以配置驱动的 AI Agent 血缘抽取通道，作为既有 `ScriptLineageExtractor` 可插拔契约的一个实现，新增该通道不改动血缘存储与展示层。该通道 MUST 覆盖 Python/Shell/Spark 脚本任务，并额外兜底"因方言/不支持语法而 Calcite 解析失败"的 SQL 任务；Calcite 能正常解析的 SQL 任务 MUST NOT 触发该通道（不做冗余外呼）。
- **FR-002**: 该通道 MUST 同时兼容 Anthropic 协议与 OpenAI 协议两种调用形态，协议类型由配置选择；对同一脚本，两种协议下的抽取语义结果 MUST 等价（协议差异对用户不可见）。
- **FR-003**: 配置项 MUST 至少包含：协议类型、服务 base URL、模型名、鉴权凭据、启用开关；MUST 支持指向任意兼容端点的云厂商，换供应商无需改代码或重新部署。
- **FR-004**: 该通道 MUST 产出带独立"AI 推断"来源标记与置信度的血缘边，且与规则/内嵌 SQL/本地小模型边共用同一血缘语义键去重、冲突消解、宁缺毋滥与人工确认/剔除机制。
- **FR-004a**: 同一血缘语义键被多条通道产出而冲突时，系统 MUST 按确定性来源优先级消解：内嵌 SQL/Calcite > 规则 > AI 推断 > 本地小模型；高优先级通道的边覆盖同键低优先级边，被覆盖来源可留痕。
- **FR-004b**: AI Agent 通道 MUST 异步执行——push/上线流程立即返回、不被云外呼阻塞；AI 边由 push 之后的后台流程补齐并入图谱。后台失败/超时 MUST 不影响已入图谱的确定性血缘。
- **FR-005**: 该通道 MUST 执行防幻觉校验——输出结构合法且表名能在脚本/SQL 文本中字面定位方可建边，否则拒收并留可查看提示。
- **FR-006**: 该通道 MUST 在未配置、被禁用、超时、鉴权失败或返回非法结构时降级为空产物并留痕，绝不抛异常拖垮 push/上线主链路，也不影响其它通道。
- **FR-007**: 系统 MUST 允许 AI Agent 通道与本地小模型通道并存，两者产出的边分别标注来源。

**数据源实时 Schema 解析（US2）**

- **FR-008**: 当列级血缘解析需要某表的列而列元数据目录查不到时，系统 MUST 复用任务绑定数据源的连接信息直连数据库，取回该表真实列清单（列名、序号、数据类型）。
- **FR-009**: 系统 MUST 用取回的列清单展开 `SELECT *` 与 `t.*`，产出逐列字段级血缘。
- **FR-010**: 当 `INSERT INTO t SELECT ...` 目标表未显式写列清单时，系统 MUST 取回目标表列清单并按位置映射源列→目标列。
- **FR-011**: 多表 JOIN 中出现不带表限定的裸列时，系统 MUST 借真实列清单判定其归属表以消歧；无法唯一确定时不猜、记提示。
- **FR-012**: 系统 MUST 能解析数据库视图的输出列清单参与列溯源。
- **FR-013**: 数据源不可达、无该表、无读元数据权限或未绑定数据源时，系统 MUST 沿用既有列级降级路径并留痕，不阻断其余语句与表级血缘。
- **FR-014**: 实时 schema 解析 MUST 有列数与耗时上限保护，超限时截断或降级并留痕，避免大宽表拖慢 push。
- **FR-015**: 系统 MUST 依连接的默认 schema/当前库对裸表名做限定名规范化，与既有血缘坐标解析同源。

**Schema 接地与缓存回填（US3）**

- **FR-016**: 当 AI Agent 通道与实时 schema 解析均可用时，系统 MUST 把真实列清单作为上下文注入 AI 提示；AI 产出的字段边 MUST 落在真实列集合内，越界列被拒收留痕。
- **FR-017**: 经实时解析取回的列元数据 MUST 回填进列元数据目录缓存，后续对同表解析优先命中缓存、不重复连库。
- **FR-018**: 缓存的列元数据 MUST 采用"重 push 失效 + TTL 兜底"新鲜度策略——任务重新 push 时使其引用表的列缓存失效并重解析，另设 TTL 兜底防长期漂移；缓存失效或过期后再次解析 MUST 反映最新 schema。

**治理与安全（US4）**

- **FR-019**: AI Agent 配置 MUST 默认关闭，按项目/租户显式开启；未开启项目 MUST 不发起任何云外呼。
- **FR-020**: 鉴权凭据 MUST 加密存储、界面脱敏（不回显明文）、且不出现在任何日志中——与既有数据源密码的保护姿态一致。
- **FR-021**: 每次 AI 外呼 MUST 计入审计轨迹，记录配置标识、协议、耗时与结果状态（成功/降级/拒收）。
- **FR-022**: 系统 MUST 为 AI 外呼提供超时与失败降级；因异步执行，外呼超时/失败 MUST 只影响该 AI 边的补齐、不阻塞或回退 push，后台任务 MUST 在配置预算内结束而非无限等待。
- **FR-023**: 系统 SHOULD 提供并发/频次上限，避免批量 push 时对云端点造成过载或成本失控。

### Key Entities *(include if feature involves data)*

- **AI Agent 抽取配置**：一条可配置的大模型接入定义。关键属性：协议类型（Anthropic/OpenAI）、base URL、模型名、加密凭据、启用开关、超时/频次预算、归属租户/项目。
- **解析出的列 Schema**：某表在某数据源下的列清单快照。关键属性：表限定名、数据源坐标、列（名/序号/类型）、来源（实时解析/预登记）、新鲜度时间戳。
- **AI 推断血缘边**：AI Agent 通道产出的读/写表或字段边。关键属性：血缘语义键（任务+方向+表坐标[+字段]）、来源标记（AI 推断）、置信度、人工确认/剔除状态。
- **AI 外呼审计记录**：一次外呼的留痕。关键属性：配置标识、协议、耗时、结果状态、关联任务、时间戳。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理员在配置界面完成一条云 AI Agent 接入（选协议、填端点/模型/Key、启用）并对一个脚本任务生效，全流程可在 5 分钟内完成，无需改代码或重启服务。
- **SC-002**: 对既有规则/内嵌 SQL 通道均不覆盖的脚本长尾样本集，启用云 AI Agent 后正确抽取的读写表边覆盖率相较仅本地小模型显著提升（在评估样本集上至少 +30 个百分点），且新增幻觉边比例低于 5%。
- **SC-003**: 对含 `SELECT *`、源表列未预登记但已绑定数据源的 SQL 任务，字段级血缘完整解析率达 95% 以上（即绝大多数 `SELECT *` 不再整体降级）。
- **SC-004**: 单次实时 schema 解析对 push 的额外耗时在设定预算内（典型 ≤ 2 秒）；AI Agent 通道异步执行、对 push 耗时零增加；且任一外部依赖（数据源库、云端点）不可用时 push 成功率不下降、100% 走降级而非失败，AI 边在依赖恢复后可补齐。
- **SC-005**: 未开启 AI Agent 的项目发生的云外呼次数为 0；开启项目的每次外呼均有对应审计记录，配置界面 100% 不回显明文凭据。
- **SC-006**: 同一表二次及以后的解析命中列元数据缓存、不重复连库的比例达 90% 以上。

## Assumptions

- **接入形态复用既有可插拔契约**：AI Agent 作为 `ScriptLineageExtractor` 的第 4 条通道接入（与 EmbeddedSql/ApiPattern/本地小模型并列），沿用其"绝不抛异常、宁缺毋滥、来源标注、人工修正"约束；不替换本地小模型，二者可并存，冲突按确定性优先级消解（AI 推断 > 本地小模型，见 FR-004a）。
- **实时 schema 解析同时服务确定性与 AI 两条路径**：以数据源连接为后端的列元数据来源，作为既有列元数据目录（`ColumnLineageCatalog`）的一个新实现/兜底层——既补全 Calcite 确定性列级血缘的 `SELECT *`，又为 AI Agent 提供 schema 接地；用户 `select * from user` 示例即落在确定性路径。
- **连接信息与凭据保护复用既有能力**：直连数据库复用现有数据源连接与解密能力（含上传驱动隔离）；AI 凭据复用与数据源密码同源的加密存储与脱敏姿态。
- **合规姿态沿用宪法原则 IV**：推断不嵌入平台服务端进程——本地小模型走独立 sidecar，云 AI Agent 走远端 HTTP 调用，均在平台进程之外。
- **数据出境为用户显式选择**：用户明确要求"直连云厂商"，故外呼被视为已授权；本特性以"默认关闭 + 显式开启 + 审计 + 脱敏"作为安全护栏，具体到某库表是否可外呼由项目治理决定，不在本期做逐字段脱敏改写。
- **解析时机分两档**：确定性通道（Calcite/内嵌 SQL/规则）与数据源实时 schema 解析沿用 push/上线的既有同步血缘解析入口、带预算与降级；云 AI Agent 通道则由 push 触发、异步后台补齐（澄清 Q1），不阻塞 push、无独立用户触发面。
- **列元数据缓存复用既有目录/同步机制**：回填与新鲜度沿用列元数据目录与已同步行的既有存储与刷新通道（specs 024/025 建立的能力），不新建平行存储。
- **协议兼容以"OpenAI-兼容 / Anthropic-兼容端点"为目标**，覆盖主流云厂商网关；不承诺适配任一厂商的私有非标扩展。

# Data Model: 血缘目录接地（Catalog Grounding）

## 1. 新增审计表 `lineage_grounding_disposition`

单一权威 DDL 落 `backend/dataweave-api/src/main/resources/schema.sql`；schema 版本 **0.11.0 → 0.12.0**。

```sql
CREATE TABLE lineage_grounding_disposition (
    id             BIGINT GENERATED BY DEFAULT AS IDENTITY,
    tenant_id      BIGINT NOT NULL,
    project_id     BIGINT NOT NULL,
    task_def_id    BIGINT,                               -- 关联任务
    candidate      VARCHAR(512) NOT NULL,                -- 原始候选表名（规范化前）
    direction      VARCHAR(8)  NOT NULL,                 -- READS | WRITES
    source_channel VARCHAR(16) NOT NULL,                 -- 来源通道（Source 枚举名）
    datasource_id  BIGINT,                               -- 核验所用数据源（null=未绑定）
    verdict        VARCHAR(16) NOT NULL,                 -- PRESENT | ABSENT | UNKNOWN | SYSTEM_EXCLUDED
    disposition    VARCHAR(16) NOT NULL,                 -- ADOPTED | DROPPED | EXCLUDED | RETAINED
    reason         VARCHAR(256),                         -- 处置原因摘要（脱敏）
    created_at     TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT pk_lineage_grounding_disposition PRIMARY KEY (id),
    CONSTRAINT ck_grounding_verdict CHECK (verdict IN ('PRESENT','ABSENT','UNKNOWN','SYSTEM_EXCLUDED')),
    CONSTRAINT ck_grounding_disposition CHECK (disposition IN ('ADOPTED','DROPPED','EXCLUDED','RETAINED')),
    CONSTRAINT ck_grounding_direction CHECK (direction IN ('READS','WRITES'))
);
CREATE INDEX idx_grounding_disp_task ON lineage_grounding_disposition (tenant_id, project_id, task_def_id);
```

同步在 schema.sql 头部加 `DROP TABLE IF EXISTS lineage_grounding_disposition;`（DROP 顺序与既有一致）、版本注释追加一行、`INSERT INTO schema_version` 加 `0.12.0`。

**verdict × disposition 对应关系**：

| verdict | 场景 | 来源类 | disposition | 边处理 |
|---|---|---|---|---|
| `PRESENT` | 目录命中 | 任意 | `ADOPTED` | 保留，confidence 升 CONFIRMED（catalog-verified） |
| `ABSENT` | 可达目录确认缺席 | 推断类 | `DROPPED` | 从边集剔除 + 连带列级边 |
| `ABSENT` | 可达目录确认缺席 | 确定性/声明 | `RETAINED` | 保留（FR-011 不剔除），仅留痕 |
| `SYSTEM_EXCLUDED` | 落系统命名空间 | 推断类 | `EXCLUDED` | 从边集剔除 |
| `SYSTEM_EXCLUDED` | 落系统命名空间 | 确定性/声明 | `RETAINED` | 保留，仅留痕（确定性不剔） |
| `UNKNOWN` | 未绑定/连不上/超时/规范化失败 | 任意 | `RETAINED` | 原样保留，零惩罚 |

> 留痕策略：`DROPPED`/`EXCLUDED` 必落表（SC-006 追溯剔除原因）；`ADOPTED` 落表用于正向审计；`RETAINED`(UNKNOWN) 为控噪可仅 debug 级、不必逐条落表（实现阶段决定采样，SC-006 只要求"被剔除/排除/采纳"可追溯）。

## 2. 领域实体（内存值对象，不落库或落 neo4j 边元数据）

### TableExistence（enum，新增）
`application/lineage/grounding/TableExistence.java`
- `PRESENT` — 目录（cache/neo4j/live）确认表存在。
- `ABSENT` — 数据源可达且成功查询、目录中确认无此表。
- `UNKNOWN` — 无法查证（未绑定数据源 / 连接失败 / 超时 / 解密失败 / 非 JDBC / 名称规范化失败）。

### GroundingDisposition（值对象，新增）
`application/lineage/grounding/GroundingDisposition.java` — 一次候选处置的内存结果，1:1 映射审计表行：
`candidate, direction, sourceChannel(Source), datasourceId, verdict(TableExistence|SYSTEM_EXCLUDED), disposition(ADOPTED|DROPPED|EXCLUDED|RETAINED), reason`。

### 复用既有实体（不改结构）
- **`IoEdge`**（`domain.lineage`）：grounding 输入/输出单元。`PRESENT` 采纳时把 `confidence` 由 `UNVERIFIED` 升为 `CONFIRMED`（唯一字段变更语义，构造新 record 而非改结构）。
- **`ColumnEdge`**：某表被 `DROPPED` 时，`srcTable`/`dstTable` 命中该规范化表名的列级边连带剔除。
- **`Source`**（枚举，不改）：来源分类判据——`{SCRIPT_INFERRED, SCRIPT_MODEL, SCRIPT_AGENT}`=推断类（可剔），其余 + `null`=确定性/声明（不可剔）。
- **`Confidence`**（枚举，不改）：复用 `CONFIRMED` 承载 catalog-verified。
- **`TableSchema` / `ColumnMeta`**：`probeTable` 命中时仍可返回（存在性核验不依赖列，但 live probe 顺带回填列目录复用 053 路径）。

## 3. 状态流转（单个候选表在 grounding 阶段）

```
候选边(direction, source, 规范化名)
      │
      ├─ 系统命名空间? ──是──► SYSTEM_EXCLUDED
      │                          ├ 推断类 → EXCLUDED(剔除)
      │                          └ 确定性 → RETAINED(留痕)
      └─否─► probeExistence(三态)
               ├ PRESENT ──► ADOPTED（保留 + confidence=CONFIRMED）
               ├ UNKNOWN ──► RETAINED（原样保留，零惩罚）
               └ ABSENT
                    ├ 推断类 → DROPPED（剔除 + 连带列级边）
                    └ 确定性 → RETAINED（留痕不剔，FR-011）
```

## 4. 数据完整性 / 规则

- 规范化：比对前按数据源默认 catalog/schema 规范化（复用 `DatasourceSchemaResolver.parseQualifiedName`）；规范化失败→UNKNOWN→RETAINED。
- 隔离：审计表按 `tenant_id/project_id` 落库，沿用既有隔离；查询走同租户上下文。
- 幂等：grounding 随 enricher 每次 push 重跑；`recordTaskIo` keyed replace 保证边集幂等；审计表按 push 追加（历史处置可回溯，不去重）。
- 新鲜度：ABSENT/UNKNOWN 不缓存三态；PRESENT 依赖底层 catalog TTL 缓存（默认 6h）+ 重 push evict。

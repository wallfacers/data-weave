# Feature Specification: 管家会话列表模式 + Markdown 回复

**Feature Branch**: `073-companion-conversation-list`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "Vega 的回复支持 markdown；汇报卡片改为会话列表模式，支持看对话历史、支持看问题列表等，而不是现在这种卡片方式——我还得一个一个地去对话。"

## 背景与约束

管家视图（`companion`，071 交付）当前形态：中央 3D 机器人 Vega + 顶部巡检概况 + 中央瞬时字幕气泡 + **右侧汇报卡片栈** + 底部全局输入框。两个现实痛点：

1. **回复不留存、无版式**：Vega 的主回复仅以中央「字幕气泡」纯文本呈现，数秒后消失；表格、列表、代码块等结构化内容塌陷成一行纯文本，且无法回看。
2. **一张张卡片、逐个对话**：每张汇报卡片各自展开一个独立迷你对话，用户要与多个问题交流必须逐张打开卡片、上下文彼此割裂，无统一的对话历史可回看。

**已确认的设计边界（用户拍板）**：

- 中央 3D 机器人 Vega **保持视图中心**，五状态动画 / 氛围层 / 字幕气泡不变——071「机器人为中心」身份与 DESIGN.md 071 豁免条目不改。
- 右侧面板由「卡片栈」改为**单一会话列表面板**：上半区=可折叠的「待处理问题」列表（原卡片压缩为行），下半区=一条留存的**统一会话线程**（全历史 + Markdown）。
- Vega 回复以 Markdown 渲染并留存于线程；中央字幕气泡保留，仅作一句话即时播报。

**技术地基（已具备，本特性零后端改动）**：Markdown 渲染组件 `ChatMarkdown`（react-markdown + remark-gfm + 复用项目 Shiki 双主题代码块、流式安全闭合围栏、单条崩溃隔离）已存在并在卡片迷你对话中在用；会话历史拉取端点 `GET /api/companion/messages`（`reportId?` / `before?` / `limit?`，对应 `chatService.history`）已存在；SSE 单通道（snapshot/state/report/briefing/message/delta/end）与 `sendChat({content, reportId?})` / `cancelChat()` 均已就绪。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 统一会话线程 + Markdown 回复 (Priority: P1)

值班工程师在管家视图底部输入框向 Vega 提问或下达指令，Vega 的回复以 **Markdown** 渲染（标题、列表、表格、代码块、粗体等按版式呈现）并**留存在右侧面板下半区的会话线程**中，按时间正序排列。工程师连续提问，多轮问答积累成一条可上下滚动回看的对话历史；重新打开视图或刷新页面后，历史对话仍可加载回看。流式输出过程中版式实时、稳定（不因半截 Markdown 塌陷），可随时打断，打断后线程仍可继续。

**Why this priority**: 这是用户最痛的两点（「输入你好没反应/看不到回复」的延续 + 「回复无版式、看不了历史」）的正面解法，独立即构成可用 MVP——即便不做问题列表改造，一条能回看、有版式的对话线程本身就交付核心价值。

**Independent Test**: 打开管家视图，在底部输入框依次发送两条查询指令（其中一条让 Vega 返回含表格/代码块/列表的结构化内容）；验证回复以 Markdown 版式渲染、两轮问答均留存在右侧会话线程且可滚动回看；刷新页面后线程能加载出历史对话；流式中点停止可中断且线程可继续。全程不依赖问题列表功能。

**Acceptance Scenarios**:

1. **Given** 管家视图已连接, **When** 用户在底部输入框发送一条指令, **Then** Vega 回复以 Markdown 渲染并作为一条消息追加到右侧会话线程尾部，用户消息也在线程中，二者按时间正序。
2. **Given** Vega 返回含表格、有序/无序列表、代码块、粗体的内容, **When** 回复完成, **Then** 各元素按版式正确呈现（代码块含语言标签与复制按钮、复用项目双主题高亮），不塌陷成纯文本。
3. **Given** 已有多轮问答, **When** 用户滚动会话线程, **Then** 可回看全部历史消息，顺序稳定。
4. **Given** 用户曾与 Vega 对话后关闭/刷新了视图, **When** 重新打开管家视图, **Then** 会话线程加载并展示既往对话历史（不为空白）。
5. **Given** Vega 正在流式输出, **When** 用户点击停止, **Then** 输出立即中止、线程中该条消息保留已产出部分并标记中断，会话可继续。
6. **Given** Vega 大脑不可用返回兜底报错, **When** 该次对话结束, **Then** 错误以可见消息呈现在会话线程中（非静默、非仅瞬时字幕）。

### User Story 2 - 待处理问题列表（卡片栈退役） (Priority: P2)

右侧面板上半区呈现「待处理问题」列表：每个未关闭汇报压缩为一行（严重度色点 + 标题 + 领域 + 聚合计数 + 本地化时间），按时间倒序，显示未读计数，整块可折叠。工程师一眼扫过就能看清当前有哪些待处理问题，无需逐张展开卡片。每行支持项目级关闭（一人关闭全员消失，与 071 语义一致）与「查看详情」跳转。

**Why this priority**: 把「一张张卡片」收敛为可扫读的问题清单，是用户「不想一个一个去对话」诉求的另一半；依赖 US1 的线程作为对话落点，故次于 P1。

**Independent Test**: 触发一轮产出多条不同严重度汇报的巡检；验证问题列表按严重度色点 + 倒序正确成行、未读计数正确、整块可折叠；点某行关闭后该行消失且计数减一、刷新后不复现；点「查看详情」正确跳转。

**Acceptance Scenarios**:

1. **Given** 存在多条未关闭汇报, **When** 打开管家视图, **Then** 上半区以列表行（非卡片）呈现全部未关闭问题，按时间倒序、标严重度色点与聚合计数，并显示未读计数徽标。
2. **Given** 问题列表非空, **When** 用户点某行的关闭, **Then** 该行移除、未读计数相应减少、项目内全员消失且刷新后不自动重现。
3. **Given** 问题列表已展开, **When** 用户点折叠, **Then** 列表收起仅留标题与计数，再次点击可展开。
4. **Given** 无未关闭汇报, **When** 打开视图, **Then** 上半区显示空态文案，会话线程仍可正常使用。

### User Story 3 - 点问题锚定进会话追问 (Priority: P3)

工程师在问题列表点击某个问题，该问题被「锚定」到下半区会话线程：后续发送的消息自动关联该问题的巡检上下文（Vega 基于该问题作答而非泛泛回应），会话头部显示当前锚定的问题标题与「取消锚定」入口。切换到另一问题或取消锚定后，输入恢复为全局对话。锚定问题下的历史往来消息在切换锚定时可被拉取展示。

**Why this priority**: 打通「问题 → 对话」的上下文衔接，是从「浏览问题」到「处置问题」的闭环，但 US1（能对话）与 US2（能看问题）已各自可用，故为 P3 增强。

**Independent Test**: 点问题列表某行；验证会话头部出现该问题标题、随后发送的提问获得基于该问题上下文的回答；点「取消锚定」后输入恢复全局；切换到另一问题时其既往对话可被加载。

**Acceptance Scenarios**:

1. **Given** 问题列表有若干问题, **When** 用户点击某问题, **Then** 会话线程进入「已锚定该问题」态（头部显示问题标题 + 取消锚定），此后发送的消息关联该问题上下文。
2. **Given** 已锚定某问题, **When** 用户提问, **Then** Vega 的回答基于该问题的巡检上下文。
3. **Given** 已锚定某问题, **When** 用户点「取消锚定」, **Then** 会话回到全局对话态，后续消息不再关联任何问题。
4. **Given** 某问题此前已有往来对话, **When** 用户锚定该问题, **Then** 该问题的既往对话被加载并展示于线程。

### Edge Cases

- **历史为空 / 首次使用**：会话线程无历史时显示友好空态提示（引导用户开口），而非空白或加载假死。
- **历史加载失败**：`GET /messages` 失败时线程给出可重试提示，不影响新对话发送。
- **流式半截 Markdown**：delta 逐字到达时未闭合的代码围栏 / 表格不得使后续正文塌陷（复用 `completePartialMarkdown` 安全闭合）。
- **单条消息渲染崩溃**：某条消息 Markdown 触发渲染异常时降级为纯文本原文，不拖垮整个线程（复用 `MarkdownBoundary`）。
- **锚定的问题被他人关闭**：当前锚定问题被项目内他人关闭/消失时，会话给出「该问题已处置」提示并回落全局态，不阻断已在进行的对话。
- **超长会话历史**：线程消息量很大时按需分页/上滚加载（`before` 游标），不一次性拉全导致卡顿。
- **主题切换**：亮/暗主题即时切换时，会话线程、Markdown 版式（含代码块高亮）、问题列表色点与文案均保持可读，无需刷新。
- **消息去重**：SSE 实时消息与 `GET /messages` 历史消息按 id 合并去重，重连/刷新不产生重复条目。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 右侧面板 MUST 由「汇报卡片栈」重构为**单一会话列表面板**：上半区「待处理问题」列表 + 下半区「统一会话线程」，机器人中央舞台、氛围层、顶部概况条、底部全局输入框保持不变。
- **FR-002**: Vega 的对话回复 MUST 以 Markdown 渲染（标题/列表/表格/代码块/粗体/链接/引用等），复用既有 `ChatMarkdown`（含项目 Shiki 双主题代码块与复制），不得塌陷为纯文本。
- **FR-003**: 会话线程 MUST 留存本次会话的全部往来消息（用户与 Vega），按时间正序、可上下滚动回看；不再随字幕气泡数秒消失。
- **FR-004**: 打开管家视图或刷新后，会话线程 MUST 通过既有 `GET /api/companion/messages` 加载既往对话历史；实时 SSE 消息与历史消息按 id 合并去重。
- **FR-005**: 会话线程 MUST 支持流式渲染与即时打断，打断后保留已产出内容并标记中断，会话可继续（复用现有 SSE delta/end + `cancelChat`）。
- **FR-006**: Vega 大脑兜底报错（SYSTEM 消息）MUST 在会话线程中作为可见消息呈现，不静默丢失。
- **FR-007**: 中央字幕气泡 MUST 保留，仅承担「一句话即时播报」（新汇报到达 / 最新回复要点），完整内容以线程 Markdown 为准。
- **FR-008**: 「待处理问题」列表 MUST 将每条未关闭汇报压缩为一行，含严重度色点、标题（缺省回落领域名）、领域、聚合计数、本地化时间，按时间倒序，并显示未读计数徽标；整块可折叠/展开。
- **FR-009**: 问题列表每行 MUST 保留 071 的项目级共享关闭语义（任一成员关闭后项目内全员消失、持久、不自动重现）与「查看详情」跳转。
- **FR-010**: 用户 MUST 能点击问题列表某项将其**锚定**到会话线程：此后发送的消息关联该问题上下文（`sendChat({content, reportId})`），会话头部展示锚定问题标题与「取消锚定」入口；取消后回落全局对话。
- **FR-011**: 锚定切换时，被锚定问题的既往对话历史 MUST 可按 `reportId` 加载并展示于线程。
- **FR-012**: 所有新增/改造文案 MUST 走 next-intl 双语（zh-CN / en-US 键集一致），时间 MUST 走 `useFormatDateTime`（无裸 ISO 的 `T` 分隔与微秒），不得使用省略号 `…` 表示进行中状态。
- **FR-013**: 会话线程、Markdown 版式、问题列表 MUST 兼容平台亮/暗两种主题，随平台主题即时切换、无需刷新，两主题下均保持可读可辨。
- **FR-014**: 本特性 MUST 为纯前端改造，复用既有后端端点与 SSE 通道，不新增/变更后端接口、数据库 schema 或 SSE 事件契约。

### Key Entities *(include if feature involves data)*

- **会话消息（Message）**：一次往来的一条消息，含 id、角色（USER / AGENT / SYSTEM）、内容（Markdown 文本）、可选锚定的问题 id（reportId）、时间。来源：SSE 实时流 + `GET /messages` 历史。
- **待处理问题（Report）**：一条未关闭汇报，含 id、领域、严重度、标题、摘要、聚合计数、未读/已读状态、时间。列表行的数据源，与会话通过 reportId 锚定关联。
- **锚定上下文（Anchor）**：会话线程当前锚定的问题（reportId | 全局），决定 `sendChat` 是否携带 reportId 与线程头部展示。纯前端状态。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户发送一条指令后，Vega 的回复以 Markdown 版式呈现并留存在会话线程；含表格/代码块/列表的回复各元素 100% 按版式渲染、零塌陷为纯文本。
- **SC-002**: 用户无需逐张展开卡片，即可在一屏内同时看到「待处理问题清单」与「对话历史」，与多个问题交流全部落在同一条可回看的会话线程中。
- **SC-003**: 刷新页面或重开视图后，既往对话历史可加载回看（非空白），实时与历史消息无重复条目。
- **SC-004**: 亮/暗两种主题下，会话线程、Markdown（含代码高亮）、问题列表色点与全部文案均保持可读可辨，主题切换即时生效无需刷新。
- **SC-005**: 从问题列表点击某问题到在会话线程对其追问、并获得基于该问题上下文的回答，全流程在 3 步操作内完成。

## Assumptions

- 会话历史端点 `GET /api/companion/messages`（`reportId?` / `before?` / `limit?`）已存在且按时间返回消息，reportId 省略时返回全局 + 各问题消息；无需新增后端接口。
- Markdown 渲染沿用既有 `ChatMarkdown`（react-markdown + remark-gfm + 项目 Shiki 双主题 + 流式安全闭合 + 崩溃隔离），不引入新的重型流式 Markdown 库。
- 管家视图为 DESIGN.md 071 书面豁免的沉浸式表面；本特性延续该豁免，会话面板作为面板容器复用既有半透明玻璃卡样式，不套标准 `Card`；机器人中央地位与五状态色 token 不变。
- 「对话历史」指当前项目范围内、当前用户可见的消息，遵守 071 已有的项目隔离（`X-Project-Id` / TenantContext）。
- 语音输入/播报仍为后续占位（071 既定），本特性不涉及。
- 汇报的产生、巡检例程治理、项目级关闭的后端语义均沿用 071，不在本特性范围内变更。

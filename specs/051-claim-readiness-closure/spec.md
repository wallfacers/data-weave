# Feature Specification: 认领就绪态物化 + 性能链收口

**Feature Branch**: `051-claim-readiness-closure`

**Created**: 2026-07-06

**Status**: Draft

**Input**: User description: "最近一个大的技术债：崩溃注入 + 四不变量审计（046/048/049 三连欠）、慢任务压测补 slot_util 验证、以及 claim 的彻底解——unmet 计数就绪态物化（把就绪判定从"每轮扫描"变 O(1)，与 WAITING 规模完全脱钩），适合作为下一个 spec。"

## 背景（044–049 性能链现状）

044→049 逐层解除调度瓶颈：触发层并行化（045）→ dispatch 并行化（046）→ claim 批量化（048）→ selectRunnable 去 NOT EXISTS（049）。到 049 为止，就绪判定仍是**每轮扫描**：`collectReady` 取一窗候选 → Java `batchUpstreamReady` + `batchCrossCycleReady` 过滤 → 就绪量不足则按 `(updated_at,id)` 游标翻窗兜底（`claimMaxWindows` 有界）。这一路径的成本随 WAITING 堆积量上升——宽 DAG + 慢任务时，候选窗被大量"未就绪老实例"占满，靠游标翻窗缓解饿死，但 round 时长仍与堆积规模挂钩（源码已自留伏笔：`SchedulerKernel` 第 226 行"彻底解需就绪态物化(unmet 计数)，留结构演进"）。

同时，046/048/049 三个 spec 的 P3 收尾任务（崩溃注入真跑 + 死锁四不变量审计）**均未勾选**（046 T016/T017/T018、048 T013/T014、049 T013/T014），是三连欠；slot_utilization 此前仅以**快任务**复测，未用**慢任务**证明槽位真实占用与不饿死。本 feature 是这条性能链的**收口**：结构终解（就绪态物化）+ 正确性收口（崩溃+不变量）+ 容量证据（慢任务 slot_util）。

## Clarifications

### Session 2026-07-06

- Q: 就绪态相对上游终态的一致性保证？ → A: 最终一致——上游终态转移仅提交自身状态（+ 廉价 outbox/事件），下游 unmet 递减异步、可批量地离峰进行；周期性对账清扫在有界窗口内收敛漂移。理由：把扇出写挪出完成/聚合热路径（046 已点名的下一瓶颈）、锁更干净（上游事务不触碰下游行，天然不违反不变量③）、吞吐更稳；代价是就绪态有界滞后（不重新引入 O(WAITING) 扫描，SC-001 仍成立）。
- Q: 跨周期依赖门是否一并纳入就绪态物化？ → A: 是——DAG 内上游门与跨周期门（048 `batchCrossCycleReady`）一并纳入**统一 unmet 计数**，使所有就绪门 O(1)、无残留扫描，达成"与 WAITING 规模完全脱钩"的彻底解。跨周期依赖的满足（上周期实例到达 SUCCESS）与 DAG 内上游满足同构，纳入同一异步递减 + 对账模型。
- Q: 就绪态滞后的可接受上界？ → A: 就绪实例从"依赖满足"到"变为可认领"的滞后 **p99 < 3s**（比 046–049 的 5s 认领延迟信封更严，异步递减不得突破）；任何残留漂移由周期对账在有界清扫间隔内（≤ 数十秒量级）收敛，保证无永久卡死。
- Q: slot_util 验证的达成口径（"接近饱和"量化）？ → A: 双判据——① **slot 成为绑定约束**（持续加压到 claim 吞吐不再随负载提升、瓶颈从认领转移到 worker 槽位）② **稳态 slot_utilization ≥ 80%**（慢任务持续占槽，留调度/爬坡/碎片余量）。两条须同时满足。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 就绪判定 O(1)，与 WAITING 规模完全脱钩 (Priority: P1)

作为调度系统，当大量任务实例在 WAITING 堆积（50 万+）时，认领单轮的**就绪判定**耗时应恒定在低位、不随堆积量增长；就绪实例应被立即认领而非被"未就绪老实例"占满候选窗后靠翻窗兜底。当前（049 后）就绪判定是每轮候选扫描 + Java 上游门 + 窗口游标兜底，round 时长与堆积规模挂钩（049 R 段实测堆积后 round_duration 反弹、claim 速率崩到十几 inst/s）。

**Why this priority**: 这是本 feature 的核心工程与最大价值——把就绪判定从"每轮扫描 O(WAITING)"变为查物化就绪态 O(就绪数)，让认领速率与堆积规模彻底脱钩，同时消除 049 引入的窗口游标兜底 hack。不解决则 044–049 整链在真实大堆积下仍会退化，性能收敛不闭合。

**Independent Test**: 1000wf `*/2s` 跑稳后人为让 WAITING 堆积到 50 万+，核对单轮就绪判定耗时与堆积量无关（早期与堆积后一致、稳定低位），且就绪实例延迟被认领的比例不因堆积上升。

**Acceptance Scenarios**:

1. **Given** WAITING 堆积 50 万+，**When** 调度器执行单轮认领的就绪判定，**Then** 就绪判定耗时稳定在低位、不随堆积量线性增长（与 049 的候选扫描 + 翻窗形态相比无退化点）。
2. **Given** 就绪实例（上游已到终态）与大量未就绪实例混在同一堆积中，**When** 认领，**Then** 就绪实例被及时认领，不因未就绪老实例占满候选窗而饿死。
3. **Given** 上游实例到达终态（SUCCESS，或 WEAK 依赖下任意终态），**When** 其下游的就绪态被异步更新，**Then** 下游在与堆积规模无关的时延内变为可认领（就绪滞后 p99 < 3s）。
4. **Given** 就绪态物化上线，**When** 与 049 基线（同压测档）对比，**Then** claim 吞吐跟上触发层物化、WAITING 稳态不堆积，且不退化 046/048/049 已达成的指标。

---

### User Story 2 - 崩溃注入 + 四不变量审计（046/048/049 三连欠收口）(Priority: P2)

作为可靠性工程师，需对**物化→认领→下发**全链（含 US1 新增的就绪态物化）做一次真崩溃注入 + 死锁四不变量代码审计，一次性收口 046/048/049 三个 spec 遗留的未勾 P3 任务。认领必须保持"恰好一次 / 不丢 / 无死锁"，且新增的就绪态计数在崩溃与并发下不漂移（不多不少、可自愈）。

**Why this priority**: 正确性是不可协商底线，且 US1 改变了就绪判定形态（引入需维护的物化状态），必须重新核对不变量并把三连欠的真跑验证补齐。仅次于 US1，因为它验证的正确性对象包含 US1 的产出。

**Independent Test**: 高负载认领运行中注入崩溃（认领事务提交后、下发前 `docker kill` 一个 master 节点再重启），重启后核对：无实例被重复认领、无实例丢失（已认领未下发实例被重派）、日志/actuator 无 deadlock；并对就绪态物化做一致性核对（崩溃/并发后计数与真实上游终态一致，无永久漂移）。

**Acceptance Scenarios**:

1. **Given** 高负载认领中同一实例可能被多 master 并发可见，**When** 认领，**Then** 该实例恰好被认领一次（无重复下发）。
2. **Given** 认领事务已提交但下发尚未完成，**When** master 崩溃后重启，**Then** 已认领未下发实例被正确重派或恢复（不丢）。
3. **Given** 持续高负载长跑 + 崩溃注入，**When** 认领事务并发推进，**Then** 无死锁/活锁（任一事务有限时间内推进），死锁防御四不变量全部保持：① 认领仅经 SKIP LOCKED；② 状态推进均走乐观 CAS（`WHERE state=?`）；③ 固定锁顺序 task→workflow；④ 状态仅在事务内持久化、下发在事务外。
4. **Given** US1 的就绪态物化在崩溃/并发/重复上游事件下，**When** 崩溃后重启并核对，**Then** 就绪态计数与真实上游终态一致（无永久漂移；漂移可被检测并自愈），不会导致就绪实例永久卡 WAITING 或未就绪实例被误认领。

---

### User Story 3 - 慢任务压测补 slot_util 验证 (Priority: P3)

作为性能工程师，需用**真跑慢任务**（持续占用 worker 槽位数十秒的 sleep/长任务）的压测，证明：① slot_utilization 在认领链路解除瓶颈后真实升高、槽位被有效占满；② 就绪/认领路径不饿死慢任务（慢任务不会因候选窗被后来者反复挤出而长期不被认领）。此前 slot_util 复测以快任务为主，快任务瞬时释放槽位，无法证明"槽位真实容量被占用"与"慢任务不饿死"这两点。

**Why this priority**: 容量证据与真实占用语义，是判断"下一层瓶颈是否转移到 worker 执行端"的依据；相较 US1/US2 是验证性工作，故 P3。但它是本收口 spec 声明"性能链收敛"必须补齐的最后一块证据。

**Independent Test**: 用真跑慢任务（每任务 sleep N 秒占槽）的压测档持续加压，记录 slot_utilization 曲线与慢任务从进入待认领到被认领的延迟分布；核对 slot_util 随认领吞吐上升到接近饱和，且慢任务认领延迟分布不出现长尾饿死。

**Acceptance Scenarios**:

1. **Given** 真跑慢任务（占槽数十秒）持续压测，**When** 认领链路稳态运行，**Then** slot 成为绑定约束（加压到 claim 吞吐不再提升、瓶颈转移到 worker 槽位）且稳态 slot_utilization ≥ 80%（证明 slot 真实容量被占用，非快任务瞬时释放的假象）。
2. **Given** 慢任务与快任务混合堆积，**When** 认领持续进行，**Then** 慢任务认领延迟分布无长尾饿死（不因候选/就绪路径被后来者反复占据而长期得不到认领）。
3. **Given** slot_utilization 接近饱和，**When** 继续加压，**Then** 记录 slot 饱和点与是否成为新瓶颈（量化认领解除后瓶颈是否转移到 worker 执行端）。

### Edge Cases

- **上游被重跑 / 补数（BACKFILL）**：上游实例被重置或重跑时，已置为就绪的下游就绪态如何回退，避免下游在上游重跑期间被误认领。
- **多上游 / 宽 DAG**：一个下游有 N 个上游，部分 SUCCESS 部分 RUNNING 时就绪态如何正确反映"仍有未满足上游"。
- **STRONG vs WEAK 依赖**：WEAK 依赖上游到任意终态即放行、STRONG 需 SUCCESS——就绪态更新须区分两种放行语义（见 `workflow_dependency.strength`）。
- **跨周期依赖（跨 cycle）**：跨周期依赖满足由"上周期实例到达 SUCCESS"触发递减，须正确关联到本周期后继实例（node+bizdate 映射）；上周期实例迟到/缺失时后继就绪态如何反映"仍有未满足跨周期依赖"。
- **首周期豁免**：`earliest_biz_date` 之前的首周期跳过跨周期检查，就绪态初值须正确豁免。
- **重复 / 乱序上游终态事件**：同一上游多次或乱序上报终态时，就绪态更新须幂等（不重复递减、不越界）。
- **实例物化时**：新 task_instance 物化时其初始未满足上游数须被正确计算（含无上游节点直通就绪的单节点/单跑实例）。
- **崩溃在就绪态更新与上游落终态之间**：两者若非同一原子边界，崩溃后就绪态与真实上游状态如何对账自愈。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 为每个 WAITING task_instance 维护一个可 O(1) 判定的**统一就绪态**（"未满足依赖数"计数或等价物化标记），**涵盖 DAG 内上游门与跨周期门两类依赖**，使单轮认领的就绪判定耗时与 WAITING 堆积规模无关、无残留扫描门。
- **FR-002**: 系统 MUST 在 task_instance 物化时按**权威状态**正确初始化其统一就绪态——只计**尚未满足**的依赖数（DAG 内上游 + 适用跨周期），而非依赖总数。特别地，物化时上游/上周期实例**可能已到终态**（跨周期上周期常已 SUCCESS、乱序/延迟物化下 DAG 内上游也可能已完成），此类已满足依赖 MUST 不计入初值——否则将无后续信号触发、实例卡就绪滞后直到对账。无未满足依赖的节点直通就绪（含首周期跨周期豁免）。
- **FR-003**: 系统 MUST 在依赖被满足时增量更新其下游/后继的就绪态——DAG 内上游实例到达放行终态（区分 STRONG=SUCCESS / WEAK=任意终态），以及跨周期依赖的上周期实例到达 SUCCESS，二者同构地递减同一 unmet 计数。该更新采用**最终一致**模型：满足方的终态转移仅提交自身状态（+ 廉价 outbox/事件记录），下游 unmet 递减异步、可批量地离峰进行，不进入完成事务的关键路径。
- **FR-004**: 就绪态的增量更新 MUST 幂等（重复或乱序的上游终态事件不导致就绪态漂移；不重复递减、不越界为负），使异步/重投递下的递减安全可重放。
- **FR-005**: 认领候选查询 MUST 直接以统一就绪态作为过滤条件（就绪的实例才进入认领），取代 049 的"候选扫描 + Java 上游门（`batchUpstreamReady`）+ 跨周期门（`batchCrossCycleReady`）+ 窗口游标兜底"整套路径，并移除窗口游标饿死兜底 hack（就绪态物化后候选即就绪，无需翻窗、无需 Java 侧就绪门）。
- **FR-006**: 就绪态物化 MUST 保留死锁防御四不变量：① 认领仅经 SKIP LOCKED；② 状态推进走乐观 CAS；③ 固定锁顺序 task→workflow；④ 状态仅事务内持久化、下发事务外——就绪态的异步递减不得引入新的跨表锁路径或反向锁顺序（上游完成事务不得为更新下游就绪态而锁下游行）。
- **FR-007**: 系统 MUST 在上游被重跑/重置/补数导致其不再处于放行终态时，正确回退相应下游的就绪态（避免下游被误认领）。
- **FR-008**: 系统 MUST 提供就绪态与真实上游终态的**周期性一致性对账**能力，在有界窗口内收敛漂移：崩溃/并发/重复事件/异步递减丢失后可检测并自愈就绪态漂移，避免就绪实例永久卡 WAITING 或未就绪实例被误认领。对账是最终一致模型的正确性护栏（非仅崩溃兜底）。
- **FR-009**: 系统 MUST 通过真崩溃注入（认领事务提交后、下发前杀 master 再重启）验证认领"恰好一次 / 不丢 / 无死锁"，一次性收口 046/048/049 三连欠的未勾 P3 验证任务。
- **FR-010**: 系统 MUST 通过真跑慢任务（持续占槽）的压测验证 slot_utilization 真实升高至接近饱和，且慢任务认领延迟无长尾饿死。
- **FR-011**: 就绪态物化的实现 MUST H2/PG 双兼容（与既有调度测试基线一致）。
- **FR-012**: 本 feature MUST 不退化 046/048/049 已达成的吞吐/延迟指标（dispatch 吞吐、claim 吞吐、dispatch_latency、round_duration）。

### Key Entities *(include if feature involves data)*

- **task_instance 就绪态**：附着于每个 WAITING 任务实例的可 O(1) 判定字段——未满足上游依赖数（unmet count）或等价就绪标记。为 0（或已就绪）表示该实例上游门已满足、可进入认领。随上游终态增量维护，不在认领时重扫上游。
- **上游放行事件**：某上游实例到达放行终态（STRONG=SUCCESS / WEAK=任意终态）这一事实，触发其下游就绪态的增量更新；须幂等承接重复/乱序事件。
- **就绪态对账**：就绪态与真实上游终态之间的一致性校验与自愈机制，作为崩溃/并发下的正确性护栏。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: WAITING 堆积 50 万+ 时，单轮认领的就绪判定耗时稳定在低位、不随堆积量线性增长（与堆积量无相关性；相较 049 堆积后 round 反弹形态消除退化点）。
- **SC-002**: 1000wf `*/2s` 全负载连续 3 分钟，claim 吞吐 ≥ 触发层物化速率、WAITING 稳态不堆积（就绪判定不再是瓶颈）。
- **SC-003**: 认领链路解除后，用真跑慢任务压测满足双判据——① slot 成为绑定约束（加压到 claim 吞吐不再随负载提升、瓶颈转移到 worker 槽位）② 稳态 slot_utilization ≥ 80%；且慢任务认领延迟分布无长尾饿死。
- **SC-004**: 全部压测与崩溃注入场景中，认领"恰好一次"（无重复下发）、"不丢"（已认领未下发实例重派）、"无死锁"，死锁防御四不变量全部保持。
- **SC-005**: 就绪实例从依赖满足到变为可认领的滞后 p99 < 3s；崩溃/并发/重复上游事件后，就绪态与真实上游终态在有界对账窗口内（≤ 数十秒）收敛一致（无永久漂移；可检测并自愈），无就绪实例永久卡 WAITING、无未就绪实例被误认领；就绪态滞后不导致 SC-002 稳态堆积。
- **SC-006**: 046/048/049 三连欠的崩溃注入 + 四不变量审计任务全部收口通过；本 feature 不退化前序性能指标。
- **SC-007**: 记录慢任务压测下 slot 饱和点与瓶颈是否转移到 worker 执行端（为后续优化提供依据）。

## Assumptions

- **单 spec 三线并作收口**：用户明确"适合作为下一个 spec"（单数）——就绪态物化（结构终解）+ 崩溃/不变量审计（三连欠）+ 慢任务 slot_util（容量证据）作为 044–049 性能链的统一收口。三个 user story 各自独立可交付、独立可测：仅落 US1 即得结构终解的可用切片；US2/US3 是围绕它的正确性与容量护栏。
- **范围边界——就绪态物化覆盖两类就绪门（彻底解）**：US1 物化的统一 unmet 计数**同时涵盖** DAG 内上游就绪门（原 NOT EXISTS / `batchUpstreamReady`，堆积时 327–493ms）与跨周期就绪门（`batchCrossCycleReady`，按 node+bizdate 查上周期）。目标是所有就绪门 O(1)、无残留扫描，认领候选查询与 WAITING 规模完全脱钩；认领路径不再保留任何 Java 侧就绪门。
- **压测复用既有档位**：复用 045/046 的 cron-stress 压测框架（1000wf `*/2s` 极限档、可人为堆积 WAITING）与 fault-injection 手段（`docker kill` master 节点）；慢任务档需 worker 真跑 sleep 任务（rebuild worker image，ShellTaskExecutor 真占槽）。
- **崩溃注入为真跑**：按三连欠原任务口径（046 T017 / 048 T013 / 049 T013），崩溃注入在真实多 master docker 环境执行（`docker kill dataweave-master-2 && docker start`），非仅单测级幂等核对；四不变量审计含代码层审查 + 长跑无死锁核对。
- **不改指标语义**：性能对比复用 046/048/049 已定义的 metrics（dispatch_latency、claim/dispatch 吞吐、round_duration、slot_utilization、queue 深度、markClaimExtraWindow 等）；就绪态物化若引入新可观测量（如就绪态漂移计数），按指标不可变约定新增而非改写。
- **基线**：基于 main 含 044–049 全量收尾（`ddd306a`/`302a683` 之后）；不回退任何已合并的调度收尾修复。
- **实现细节留 plan**：就绪态的物化载体（task_instance 列 / 独立表 / 计数触发点）、初始化与增量更新的原子边界、对账机制的触发频率与自愈方式，均属 HOW，留 plan.md/research.md 决策；本 spec 聚焦 WHAT/WHY。

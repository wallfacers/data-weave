# Feature Specification: SQL 脚本重梳理与严格 Schema 版本设计

**Feature Branch**: `017-sql-schema-versioning`

**Created**: 2026-06-29

**Status**: Draft

**Input**: User description: "项目中的 SQL 脚本，由于大改造，需要重梳理，并按从 0.0.1 起的严格版本设计（而非 v1 这种大版本），SQL 版本与项目版本对应；过时不要的直接删除；后续 alert 这类改表脚本直接压缩到同一个表（单一权威 schema）；老数据不考虑兼容性，可以直接删除并覆盖。"

## Clarifications

### Session 2026-06-29

- Q: schema 版本与项目发布版本应是什么关系？ → A: **两者相同**——schema 版本即项目发布版本，单一版本号、**不设独立映射**。任何 schema 结构变更必须伴随项目版本升级；省去映射可避免"哪天漏更新、不方便查找"的隐患。
- 据此决定（存储方式，informed default）：版本字面量写在权威 schema 文件头部，同一文件内顺带写入单行 `schema_version` 表（值=项目版本），同源一份文件、天然防漂移、库内可查。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 单一权威 schema，无散落增量脚本 (Priority: P1)

平台维护者打开数据库脚本目录时，应只看到**一份权威的 schema 定义**与配套的种子数据，而不是一份基线加上一堆历史增量改表脚本（`db/migration/*.sql` 那种）。任何一张表的当前结构，都能在这一份文件里一眼看全，不需要在脑中把基线 + N 个 ALTER 叠加起来推算。

**Why this priority**: 这是本次"重梳理"的核心价值。当前散落的增量脚本既不被启动流程加载（无 Flyway/Liquibase），又与权威 schema 重复，制造"哪份才算数"的认知负担和漂移风险。收口成单一真相源后，后续所有改表（包括 alert 这类）都直接改这一份，不再新增孤立脚本。

**Independent Test**: 全新环境从零启动后端，数据库被这一份权威 schema 完整建好、可正常运行；脚本目录中不再存在未被加载的孤立增量脚本；对照实际建出的表结构与权威 schema 文件，二者完全一致（无漂移）。

**Acceptance Scenarios**:

1. **Given** 一个空数据库，**When** 后端以默认配置启动，**Then** 全部业务所需的表/索引/约束被一份权威 schema 一次性建好，应用正常运行、健康检查通过。
2. **Given** 已梳理完成的脚本目录，**When** 维护者查找任意一张表的完整定义，**Then** 该定义只存在于唯一一份权威 schema 文件中，目录里不存在描述同一张表的其它增量脚本。
3. **Given** 一份历史增量脚本（如曾经的某个 ALTER 改表），**When** 梳理完成，**Then** 它的效果已被合并进权威 schema 的对应表定义中，原始增量脚本文件已被删除。

---

### User Story 2 - 严格 SemVer 版本，与项目版本对应 (Priority: P1)

平台维护者需要知道"当前运行的库结构属于哪个版本"。schema 采用从 **0.0.1** 起的严格语义化版本号（不是 v1/v2 这类粗粒度大版本），且**该版本即项目发布版本**——同一个版本号，不另设映射。任何人查看 schema 文件头部或库内的版本记录，都能读到当前 schema 版本，它就是当前项目版本。

**Why this priority**: "版本严格设计、与项目版本对应"是用户的明确诉求。没有版本标识，就无法判断库结构是否与代码匹配、发布是否需要重建库。版本号是后续每次改表发布的锚点。采用"二者相同"而非独立映射，是为了避免映射哪天漏更新、不方便查找。

**Independent Test**: 在权威 schema 文件头部与数据库中都能读到一个符合 `MAJOR.MINOR.PATCH` 格式、初始为 `0.0.1` 的版本号，且该值与当前项目发布版本一致（同一个号）。

**Acceptance Scenarios**:

1. **Given** 重梳理后的权威 schema，**When** 维护者查看其版本标识，**Then** 读到一个严格 SemVer 版本号，本次基线为 `0.0.1`，与当前项目版本相同。
2. **Given** 后端已用该 schema 初始化数据库，**When** 维护者查询库内的 schema 版本记录，**Then** 返回的版本与权威 schema 文件头部声明的版本、以及当前项目版本三者一致。
3. **Given** 一次后续的改表发布，**When** 改表合并进权威 schema，**Then** 项目版本（即 schema 版本）按变更性质递增（破坏性→MAJOR、加表/加列→MINOR、修正→PATCH）——**任何 schema 结构变更都不得在不升项目版本的情况下发布**。

---

### User Story 3 - 过时脚本与死内容清除 (Priority: P2)

平台维护者完成本次大改造后，希望脚本目录里**不残留**任何过时、废弃、不再使用的 SQL 内容——包括不再使用的表/列定义、指向已废弃设计文档的注释、以及任何不被任何环境加载的孤立脚本。

**Why this priority**: 残留的死脚本会被误认为"还在用"，误导后续维护，也是漂移源头。清除属于"重梳理"的必要组成，但相较 P1 的收口与版本化，优先级稍低。

**Independent Test**: 梳理后逐一核对脚本目录，每一份文件都能说明其用途（被某环境加载或作为权威真相源）；不存在指向已删除/废弃文档的真相源引用；删除的内容在代码与配置中无任何残留引用。

**Acceptance Scenarios**:

1. **Given** 脚本目录，**When** 梳理完成，**Then** 不存在任何"零环境加载、零代码引用"的孤立 SQL 文件。
2. **Given** 权威 schema 文件头部注释，**When** 梳理完成，**Then** 不再引用已废弃的设计真相源文档；真相源指向当前有效的文档（或自身即真相源）。
3. **Given** 任意被判定为过时而删除的表/列，**When** 全仓检索，**Then** 后端代码、配置、种子数据中均无对它的引用（即删除不会破坏构建或启动）。

---

### User Story 4 - 不兼容老数据的覆盖式发布 (Priority: P3)

平台维护者在本阶段发布时，**不需要**为老数据做迁移兼容。发布模型为"删旧建新、直接覆盖"：启动即按权威 schema 重建结构，老数据可被直接丢弃。维护者据此能放心地把改表直接写进权威 schema，而不必编写数据迁移逻辑。

**Why this priority**: 用户明确表示"老数据不考虑兼容性，可以直接删除并覆盖"。这是一项被采纳的工作约束/发布策略，使前三个故事得以用最简方式落地；它本身不是新增能力，故列为最低优先级，主要作为边界声明。

**Independent Test**: 在已有任意旧数据的库上重新启动，结构被按权威 schema 重建，启动成功；不存在为保留旧数据而存在的迁移/兼容代码路径。

**Acceptance Scenarios**:

1. **Given** 一个已含旧结构与旧数据的数据库，**When** 后端以默认配置启动，**Then** 旧结构被丢弃、按权威 schema 重建，应用正常运行（不保证旧数据保留）。
2. **Given** 一次破坏性改表（删列/改类型），**When** 它被写进权威 schema，**Then** 无需附带任何老数据迁移脚本即可发布。

---

### Edge Cases

- **权威 schema 与实际建表漂移**：若文件声明与库内实际结构不一致，应被视为缺陷（验收要求二者一致）；本特性结束态必须无漂移。
- **删除的表/列仍被引用**：若某张被判过时的表实际仍被代码/种子数据使用，则它不算"过时"，不应删除——必须先在全仓确认零引用再删，避免破坏启动。
- **种子数据引用被删表**：删表/删列时，配套种子数据（`data.sql` 等）中的相关 INSERT 必须同步清理，否则初始化报错。
- **H2 与 PostgreSQL 兼容**：权威 schema 需同时在默认 PostgreSQL 与 H2（测试/本地）下可建；任何收口/删除不得破坏两库兼容。
- **版本递增方向歧义**：当一次变更同时含加表与删列时，按"最高破坏级别"决定版本段递增（取 MAJOR）。
- **库内版本记录缺失**：旧库无版本记录时，重建后应被写入当前版本（`0.0.1`），不依赖旧库已有版本。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 将当前散落的增量改表脚本（`db/migration/` 下未被启动流程加载的历史 ALTER/CREATE 文件）的效果**合并进单一权威 schema**，使每张表的完整结构在权威 schema 中一处可见。
- **FR-002**: 系统 MUST 在合并完成后**删除**这些被收口的孤立增量脚本文件，目录中不再保留与权威 schema 重复的散落脚本。
- **FR-003**: 系统 MUST 为后续所有改表确立"**直接修改权威 schema**"的工作约定——不再新增独立增量/迁移脚本（用户以 alert 类改表为例）。
- **FR-004**: 系统 MUST 为权威 schema 赋予一个**严格 SemVer**（`MAJOR.MINOR.PATCH`）版本标识，本次基线为 **`0.0.1`**，不使用 v1/v2 这类粗粒度版本。
- **FR-005**: schema 版本 MUST **等于项目发布版本**（同一个版本号，**不设独立映射**）；**任何 schema 结构变更 MUST 伴随项目版本升级**，不得在不升版本的情况下改表发布。
- **FR-006**: 系统 MUST 使**库内可查询到当前 schema 版本**，且该值与权威 schema 文件头部声明的版本、当前项目版本三者一致。
- **FR-007**: 系统 MUST 定义**版本递增规则**（递增的是项目版本 = schema 版本）：破坏性变更（删表/删列/改类型）递增 MAJOR、新增表/列等向后可加变更递增 MINOR、修正性变更递增 PATCH；一次变更含多类时取最高级别。
- **FR-008**: 系统 MUST **删除过时、不再使用的 SQL 内容**（废弃表/列、孤立脚本），且删除前 MUST 确认其在后端代码、配置、种子数据中**零引用**，删除后不破坏构建与启动。
- **FR-009**: 系统 MUST **清理权威 schema 中指向已废弃设计文档的真相源注释**，改为指向当前有效文档或声明自身为真相源。
- **FR-010**: 系统 MUST 采用**覆盖式（drop-and-recreate）发布模型**——启动按权威 schema 重建结构，**不为老数据提供迁移/兼容**逻辑。
- **FR-011**: 系统 MUST 在删除任意表/列时**同步清理配套种子数据**中的相关条目，保证初始化不报错。
- **FR-012**: 重梳理后的权威 schema MUST 同时在默认 PostgreSQL 与 H2 下成功建库，**两库兼容性不被破坏**。
- **FR-013**: 结束态 MUST **无 schema 漂移**——权威 schema 文件声明的结构与从零启动后实际建出的结构一致。
- **FR-014**: 本特性 MUST **不引入新增业务能力、不改动后端业务逻辑/API 行为**——仅为 SQL 脚本的梳理、版本化与清理（结构等价收口）。

### Key Entities *(include if feature involves data)*

- **权威 Schema（Authoritative Schema）**：项目数据库结构的唯一真相源。包含全部表、索引、约束的完整定义，携带一个严格 SemVer 版本标识；是启动初始化加载的对象，也是后续一切改表的唯一落点。
- **Schema 版本（Schema Version）**：`MAJOR.MINOR.PATCH` 形式的版本号，基线 `0.0.1`，**与项目发布版本是同一个号**；既声明于权威 schema 文件头部，也以单行 `schema_version` 表形式在库内可查；三处取值恒等。
- **过时脚本（Obsolete Script）**：零环境加载、零代码引用的孤立 SQL 文件，或描述已废弃表/列的内容；本特性的清除对象。
- **种子数据（Seed Data）**：随 schema 一同加载的初始化数据；删表/删列时必须同步清理其相关条目。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 重梳理后，描述同一张表结构的 SQL 文件数量为 **1**（单一权威 schema）；散落的历史增量改表脚本数量为 **0**。
- **SC-002**: 全新环境从零启动，数据库结构 **100%** 由权威 schema 建成，健康检查通过，应用功能与梳理前**等价**（无功能回归）。
- **SC-003**: 任意维护者在 **1 处**（权威 schema 文件头部或库内版本记录）即可读到当前 schema 版本，且为合法 SemVer，基线 `0.0.1`；该值与文件头部声明、当前项目版本**三者相等**（无第二处需对照的映射）。
- **SC-004**: 权威 schema 文件声明的结构与实际建出的结构对照，差异项为 **0**（无漂移）。
- **SC-005**: 脚本目录中"零环境加载且零代码引用"的孤立 SQL 文件数量为 **0**。
- **SC-006**: 在已含旧数据的库上重启，结构重建成功率 **100%**，无需任何老数据迁移步骤。
- **SC-007**: 权威 schema 在 PostgreSQL 与 H2 两套环境下建库成功率均为 **100%**。
- **SC-008**: 被删除的过时表/列在全仓检索中的残留引用数为 **0**（构建与启动不受影响）。

## Assumptions

- **发布模型为覆盖式而非迁移式**：依据用户"老数据不考虑兼容、直接删除覆盖"，本特性采用单一权威 schema + 启动 drop-and-recreate，**不引入** Flyway/Liquibase 式的版本化迁移链；"版本"是权威 schema 的标签 + 项目版本映射，而非一串增量迁移。
- **"alert" 为举例而非具体表**：当前不存在 alert 相关表，dataweave-alert 模块无独立 SQL。用户以"alert 这种改表"泛指"后续增量改表脚本"，本特性将其约定为"直接改权威 schema"，无需为 alert 建表。
- **当前权威 schema 已基本收口**：现有 `schema.sql` 已包含散落增量脚本的大部分效果且被启动加载；本特性重点是删除死的增量目录、对齐残余漂移、补版本标识与清理过时内容，而非从头重写全部 DDL。
- **schema 版本 = 项目版本（同一个号，无映射）**：依据澄清，schema 不另设独立编号，其版本恒等于项目发布版本（基线 `0.0.1`）；任何改表必伴随项目版本升级。库内以单行 `schema_version` 表持有该值，且 INSERT 与版本字面量同写在权威 schema 文件内，与文件头部声明同源、不会漂移。当前项目版本：后端 `0.0.1-SNAPSHOT`、前端 `0.0.1`，schema 基线取 `0.0.1`。
- **真相源指向需更新**：现有 schema 头部引用的设计真相源文档（openspec 旧路径）已废弃，需改指当前有效文档或声明 schema 自身为真相源。
- **种子数据与演示数据在收口范围内**：`data.sql` 等种子/演示数据随结构调整同步清理，但其"中文豁免 i18n"等既有约定不变。
- **不改业务行为**：本特性为结构等价的脚本治理，不新增能力、不改 API/内核语义；验收以"功能等价 + 无漂移"为准。

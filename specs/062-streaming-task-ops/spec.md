# Feature Specification: 实时任务运维（Streaming Task Ops）

**Feature Branch**: `062-streaming-task-ops`

**Created**: 2026-07-11

**Status**: Draft

**Input**: User description: "任务实例后面新增一个实时任务卡片，运维的实时任务长期运行，可停止，可 checkpoint 继续重跑，实时任务的一些相关特性需要支持，最新的日志等"

## Clarifications

### Session 2026-07-11

- Q: 首版"检查点续跑"是否要求真实引擎 checkpoint/savepoint 能力（依赖 061），还是首版先交付完整 UX+状态模型、续跑随引擎就绪（全量重跑兜底过渡）？ → A: **首版必须真续跑**——续跑能力（US4 / FR-007 / SC-003）硬依赖 061 Flink 真实 checkpoint 落地；不采用全量重跑兜底过渡（Option A）。其余 US（视图 / 日志 / 停止 / 监控）不因此阻塞，可先于 061 交付。
- Q: 一个实时任务的检查点如何保留与过期（影响数据模型与续跑 UX）？ → A: **保留最近 N 个成功检查点（默认 N=3，可配置），滚动淘汰更早的；续跑时可从中选择任一个作为回滚恢复点**（Option B）。
- Q: SUSPENDED（底层故障挂起）的实时任务恢复时走哪条路径？ → A: **续跑优先**（reattach 引擎侧仍活任务 / 从检查点恢复），仅当无有效检查点时降级为全量重跑；两条路径都在，不把运维卡死在单一路径（Option C）。
- Q: 停止 / 续跑 / 终止实时任务的操作门控等级（是否需审批）？ → A: **全部 L1 直接执行、不设审批、仅留审计痕迹**——运维自主、效率优先（Option A）。

### Session 2026-07-11（实现前代码核实——060 地基缺口）

- 发现（代码实证）：本特性假设「060 已提供可用的 long_running 服务端地基」**不成立**。① `task_def.long_running` 列是 060 加的（DEFAULT FALSE），但**无任何产品路径能置 TRUE**——`TaskDef` 实体不映射该字段、push 映射（`ProjectSyncService`）不复制它、`TaskDoc`/`TaskMapper` 文件契约无该字段；② 服务端下发链路（`SchedulerKernel`→`DispatchCommand`→`WorkerExecController`）**零处**读/传 long_running，`WorkerExecController` 用 `EngineSubmitRef` 7 参向后兼容构造硬编码 `longRunning=false`；仅 CLI `dw run --long-running`（`LocalRunMain`）走真 long_running。→ **服务端 cron/手动下发永不产生 long_running 实例**，060 detached/reattach 与 061 T038 真跑均经 localrun/夹具触达，非服务端产品链路。
- Q: 062 面板按 `long_running=TRUE` 过滤会永远为空——如何定范围？ → A: **Option C（折中，用户 2026-07-11 批准）**：在 062 Foundational 内并入**最小「创作→下发」接通链路**——`TaskDoc`+`TaskMapper` 加 longRunning（文件契约往返）、`TaskDef` 实体 + push 复制（授权持久化）、`DispatchCommand`+`WorkerExecController`+两 gateway 传播 long_running（下发生效）。不做完整创作 UX（无前端 toggle / 无 MCP 专用工具）。使实时任务经产品链路真实可创建、可下发为 detached 长驻，面板方有真实数据源。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 实时任务独立运维视图（Priority: P1）

运维人员在运维中心，除了现有的"任务实例"（一次性批处理任务）之外，能看到一个**紧随其后、与之并列的独立"实时任务"区域**，集中列出所有正在长期运行的流式/实时任务（如实时 ETL、实时同步作业）。每个实时任务展示其核心运行信号：运行状态、已连续运行时长。把长期运行的实时任务从"淹没在一次性批实例记录里"提升为**一等运维对象**——运维一眼就能看清当前有哪些实时任务在跑、各自什么状态。

**Why this priority**: 这是整个特性的入口和价值前提。没有独立视图，停止 / 续跑 / 日志 / 监控都无处挂载。仅交付这一条就独立产生价值——运维第一次能集中看到所有在跑的实时任务及其状态，而不用从一堆已结束的批任务记录里翻找。

**Independent Test**: 运维打开运维中心，在"任务实例"旁看到独立的"实时任务"区域，其中只列出长期运行的实时任务，且与一次性批任务在视觉上可明确区分，每条带状态与已运行时长。

**Acceptance Scenarios**:

1. **Given** 平台中存在若干长期运行的实时任务与若干一次性批任务，**When** 运维进入运维中心，**Then** 在"任务实例"之后看到一个独立的"实时任务"区域，且该区域只包含实时任务、不含已结束的批任务
2. **Given** 一个实时任务正在运行，**When** 运维查看实时任务区域，**Then** 能看到它的运行状态（运行中）与已连续运行时长
3. **Given** 运维不熟悉平台，**When** 首次查看，**Then** 能仅凭界面区分"这是一次性批任务"还是"长期运行的实时任务"，无需培训

---

### User Story 2 - 实时任务最新日志（Priority: P1）

运维点开任一实时任务，能看到它**最新、正在产生**的运行日志，并持续近实时刷新，无需手动重新查询。实时任务与批任务不同——批任务看的是"已结束的历史日志"，实时任务看的是"正在产生的最新输出"。因此日志必须始终反映任务当前时刻的运行情况，且在任务连续运行数天甚至数周后，日志查看体验不退化、不卡顿。

**Why this priority**: 实时任务的排障几乎完全依赖"看最新日志"（数据不进来了？延迟堆积了？反序列化报错了？）。没有最新日志，实时任务出问题时运维等于盲人摸象。它和 US1 一样是基础能力，独立交付即可显著降低排障时间。

**Independent Test**: 打开一个正在运行的实时任务，其最新日志在数秒内自动刷新出最新输出；连续运行多日的任务，日志查看依旧流畅。

**Acceptance Scenarios**:

1. **Given** 一个实时任务正在持续产出日志，**When** 运维打开该任务，**Then** 数秒内看到最新产生的日志行，并随任务运行持续自动刷新
2. **Given** 一个已连续运行多日的实时任务，**When** 运维打开其日志，**Then** 日志加载与滚动查看依然流畅，不被历史日志量拖垮
3. **Given** 实时任务与日志通道短暂断连，**When** 连接恢复，**Then** 日志自动续接，不丢失断连期间的关键输出、不重复回放

---

### User Story 3 - 优雅停止实时任务并保留进度（Priority: P1）

运维能对一个运行中的实时任务发起"停止"，停止时系统**尽可能保留任务已处理的进度（检查点）**，使任务安全、干净地停下来而不丢失已处理的状态与数据。这与"强制终止"（force kill，立刻掐断、不保证状态一致）是两种不同操作——日常的停止（发版、资源腾挪、参数调整）应走优雅停止，强制终止仅作为优雅停止不可用时的兜底。

**Why this priority**: 实时任务天然需要频繁停止（每次发版、扩缩容、调试）。若每次停止都丢状态、下次只能从头跑，则直接违背流式处理"不丢数据、不重复处理"的核心承诺，导致下游数据缺口或重复。优雅停止是实时任务区别于批任务的关键运维能力。现有平台只有强制终止，这是必须补齐的缺口。

**Independent Test**: 对一个运行中的实时任务发起"停止" → 任务进入"已停止且保留了可恢复进度"的状态；同时"强制终止"作为独立兜底操作依旧可用。

**Acceptance Scenarios**:

1. **Given** 一个运行中的实时任务，**When** 运维发起"停止"，**Then** 系统尝试保留可恢复进度后将任务置为已停止，过程中已处理的数据不丢失
2. **Given** 运维需要对一个实时任务下线，**When** 选择停止方式，**Then** 能明确区分"停止（保留进度）"与"强制终止"两种操作，并知晓各自后果
3. **Given** 优雅停止因故无法完成，**When** 系统判定无法保留进度，**Then** 明确提示运维，并可改用强制终止兜底

---

### User Story 4 - 从检查点续跑（恢复）实时任务（Priority: P2）

实时任务被停止或因异常中断后，运维能从一个**保留的检查点恢复续跑**，使任务从上次停止/中断的进度之后继续处理，已处理的部分既不重做、已落地的数据也不丢失。这与"全量重跑"（清空进度、从零开始）是两条不同路径——续跑用于"停了想接着跑"，全量重跑用于"彻底重来"。

**Why this priority**: 这是 US3"优雅停止保留进度"所兑现的价值面——"停下来能接着跑"才是完整闭环。定为 P2 是因为它依赖 US3 先产出可恢复的检查点，且实际使用频率低于日常的"看 / 停"；但它是"可 checkpoint 继续重跑"这一核心承诺的兑现，价值很高。

**Independent Test**: 停止一个实时任务（已保留检查点）→ 发起"恢复续跑" → 任务从检查点之后继续处理，不重做已处理部分。

**Acceptance Scenarios**:

1. **Given** 一个已停止且保留了多个检查点的实时任务，**When** 运维发起"恢复续跑"并从中选择一个检查点，**Then** 任务从所选检查点对应的进度之后继续，不重做该点之前已处理的数据
2. **Given** 一个因异常中断、但留有有效检查点的实时任务，**When** 运维发起恢复续跑，**Then** 任务恢复运行而非从零开始
3. **Given** 一个任务的检查点已失效或不存在，**When** 运维尝试恢复续跑，**Then** 系统明确告知"无可用检查点"，并仅提供全量重跑作为可选路径

---

### User Story 5 - 实时任务健康监控与挂起（SUSPENDED）一等化（Priority: P2）

运维在实时任务视图能看到每个任务的健康监控信号：处理进度 / 堆积延迟、最近一次检查点是否成功及其时效、作业运行期间是否发生过重启。尤其地，当实时任务因底层基础设施故障被系统**挂起**（SUSPENDED——既非正常运行也非失败终止，而是"卡住等待人工处置"）时，这一状态在实时任务视图中被作为**一等状态明确标识**，运维能直接从这里把它转出（恢复续跑或终止），不留下无人察觉、长期悬挂的任务。

**Why this priority**: "长期运行"要能持续可靠，监控信号是前提；SUSPENDED 一等化是修复现状缺口（当前该状态在运维界面不是一等公民，易被忽视）。定为 P2 是因为首版可先覆盖核心信号、健康监控的深度可迭代，但它对"减少实时任务静默堆积 / 静默挂起的线上事故"价值明确。

**Independent Test**: 实时任务视图展示健康信号（进度 / 延迟 / 检查点状态 / 重启次数）；一个因底层故障被挂起的实时任务在视图里被明确标识，并可直接恢复或终止。

**Acceptance Scenarios**:

1. **Given** 多个实时任务正在运行，**When** 运维查看实时任务视图，**Then** 能看到每个任务的进度 / 堆积延迟、最近检查点状态与时效、是否重启过
2. **Given** 一个实时任务因底层节点故障被系统挂起（SUSPENDED），**When** 运维查看视图，**Then** 该任务以"已挂起"状态被明显标识，而非被误显示为运行中或失败
3. **Given** 一个处于 SUSPENDED 的实时任务，**When** 运维决定处置，**Then** 能直接从实时任务视图发起恢复（优先续跑，无有效检查点时降级全量重跑）或终止，无需借助其他入口

---

### Edge Cases

- 实时任务因底层节点故障被系统挂起（SUSPENDED）时，如何在实时任务视图中呈现，并确保运维能从这里恢复或转出？
- 检查点失败、过期或不存在时，"恢复续跑"如何降级——是否明确提示"无可用检查点，仅可全量重跑"？
- 实时任务连续运行数天 / 数周后，历史日志量爆炸，如何不拖垮日志查看与存储？
- 运维同时对多个实时任务发起停止 / 恢复时的并发与限流如何处理？
- 实时任务运行所在节点短暂失联（detached）期间，视图如何呈现——"断连但任务可能仍在引擎侧运行"，避免误判为已停止？
- 引擎侧任务已自行结束或失败，但平台侧状态尚未同步（状态漂移），视图如何体现并纠正？
- "停止（保留进度下线）"与"暂停 / 恢复（临时挂起、不释放长期资源）"的语义边界——本特性聚焦前者，后者如何界定为非目标？
- 优雅停止耗时较长（等待检查点完成）期间，运维发起的停止请求如何反馈进度，避免被误以为无响应？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在运维中心提供与"任务实例"并列、紧随其后的独立"实时任务"运维入口
- **FR-002**: 实时任务视图必须只纳入长期运行的流式 / 实时任务，并在视觉上与一次性批任务明确区分
- **FR-003**: 系统必须为每个实时任务展示其运行状态（运行中 / 已停止 / 恢复中 / 异常 / 已挂起）与已连续运行时长
- **FR-004**: 用户必须能从实时任务视图进入该任务的最新日志，且日志近实时刷新、长期运行下不退化
- **FR-005**: 用户必须能对运行中的实时任务发起"停止"，且停止必须尽可能保留可恢复进度（检查点）
- **FR-006**: 系统必须提供与"停止（保留进度）"相区分的"强制终止"兜底操作，用于优雅停止不可用时
- **FR-007**: 用户必须能从保留的多个检查点中**选择一个**来"恢复续跑"一个已停止 / 异常的实时任务，且不重做所选检查点之前已处理的进度
- **FR-008**: 当无可用检查点时，系统必须明确告知用户，并仅提供全量重跑作为可选路径
- **FR-009**: 系统必须将 SUSPENDED（底层故障挂起）作为实时任务的一等状态呈现，并支持从此状态恢复或终止；恢复时**优先续跑（reattach 引擎侧仍活任务 / 从检查点恢复），仅当无有效检查点时降级为全量重跑**（澄清 2026-07-11）
- **FR-010**: 系统必须为实时任务展示健康监控信号：处理进度 / 堆积延迟、最近检查点状态与时效、作业重启情况
- **FR-011**: 所有停止 / 恢复 / 终止 / 重跑操作经统一操作门控为 **L1 直接执行（不设审批）**，并必须留有审计痕迹（澄清 2026-07-11）
- **FR-012**: 系统必须为每个实时任务滚动保留最近 N 个（N≥1，默认 3，可配置）成功检查点，超出 N 的更早检查点自动淘汰

### Key Entities *(include if feature involves data)*

- **实时任务实例（Streaming Task Instance）**: 长期运行的流式作业实例，具备外部引擎句柄、可检查点、可恢复；与一次性批实例同源，但运维生命周期不同——长期运行、可停止保留进度、可恢复续跑、不重复处理已落地数据
- **检查点（Checkpoint）**: 实时任务在某时刻的可恢复进度快照，是"停止后续跑、不丢不重"的依据。一个实时任务**保留最近 N 个成功检查点（默认 N=3，可配置），滚动淘汰更早的**（澄清 2026-07-11）；续跑时运维可从中选择任一个作为恢复点。检查点具有时效性，过期或缺失则不可用于续跑
- **实时任务运行状态**: 运行中 / 已停止（保留检查点）/ 恢复中 / 异常 / 已挂起（SUSPENDED）；其中 SUSPENDED 为底层故障导致的非终态保护性状态，需人工恢复或终止

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 运维人员能在 30 秒内从运维中心进入实时任务视图并定位到一个目标实时任务
- **SC-002**: 运维打开实时任务后，最新日志在 10 秒内反映任务最新输出，且连续运行 7 天的实时任务其日志查看体验不退化
- **SC-003**: 运维对一个实时任务发起"停止"后，能随后从保留的检查点恢复续跑，且已处理的数据既不丢失也不重复处理
- **SC-004**: 新运维人员无需培训即可在实时任务视图中区分"运行中 / 已停止 / 已挂起"等状态并知晓下一步可执行动作
- **SC-005**: 底层故障导致实时任务被挂起时，100% 在实时任务视图中被明确标识并可转出（恢复或终止），不遗留无人察觉的悬挂任务
- **SC-006**: 实时任务的健康信号（进度 / 延迟 / 检查点状态）使运维能在问题影响下游之前主动发现——以"实时任务静默堆积 / 静默挂起导致的线上事故数"下降为定性验证目标

## Assumptions

- **底层流式执行运行时已具备**长期运行作业的执行机制（detached 提交 + 失联后 reattach + SUSPENDED 非终态保护），由 060 提供并入主干。**但服务端「创作→下发→long_running 执行」的接线在 060 未接通**（见 Clarifications 2026-07-11 代码核实）——故本特性按 Option C 在 Foundational 内并入最小接通链路（`TaskDoc`/`TaskMapper`/`TaskDef` 实体 + push 复制 + `DispatchCommand`/`WorkerExecController`/gateway 传播），再在其上构建可观测与可控面。
- **"检查点续跑"硬依赖 061（首版不兜底过渡）**：真实流式引擎（如 Flink）的检查点 / savepoint 与运行状态查询依赖"大数据任务类型真实引擎验证"（061，进行中、尚未合入主干）。**续跑（US4 / FR-007 / SC-003）必须在 061 提供真实 checkpoint 能力后才交付——首版不采用"全量重跑兜底 + 提示"的过渡方案**（澄清 2026-07-11）。其余 US（视图 / 日志 / 停止 / 监控）不因此阻塞，可先于 061 交付。
- **复用既有 SSE 日志流基础设施**（已支持近实时刷新、断点续传、自动滚动、多视图 keep-alive），实时任务的"最新日志"在其上挂载而非重建。
- **复用既有的操作门控与审计机制**——所有停止 / 恢复 / 终止 / 重跑操作走统一的审批与留痕流程，不另立旁路。
- **检查点 / 续跑消费而非重新实现引擎机制**：本特性依赖流式引擎自身的检查点能力（周期性 checkpoint / 手动 savepoint），平台负责触发、追踪与展示，不重新实现引擎级状态快照。
- **范围边界——"暂停 / 恢复（临时挂起资源）"为非目标**：本特性聚焦"停止（保留进度下线）+ 恢复续跑"。与"暂停 / 恢复"（临时挂起但不释放长期资源、快速唤醒）的语义边界以运维习惯为准，后者作为后续可选，不在本特性范围。
- **首版监控深度可迭代**：健康监控首版覆盖核心信号（进度 / 延迟 / 检查点状态 / 重启），更深入的流式作业内部指标（算子级反压、水位线细节等）留待后续。

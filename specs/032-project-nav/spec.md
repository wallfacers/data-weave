# Feature Specification: 企业项目左侧导航（按功能模块划分目录）

**Feature Branch**: `032-project-nav`

**Created**: 2026-07-01

**Status**: Draft

**Input**: User description: "项目最左侧新增一个企业项目导航 按功能模块划分目录"

## 概述

当前应用为「Tab 栏 + 右侧辅助面板」的双栏布局，全部功能视图（运营中心、数据开发、资产目录、告警中心、系统指标等十余个）只能通过 Tab 栏右侧「+」启动菜单平铺打开。视图数量增长后，平铺菜单缺乏分类，用户难以快速建立「这个平台有哪些功能、各自归属哪个领域」的整体心智地图，找功能靠记忆与试错。

本特性在应用**最左侧新增一条常驻的企业项目导航栏**，把全部功能视图**按功能模块（领域）分组成目录**呈现。导航栏让用户一眼看清平台能力全景，并按业务领域（数据开发、运维监控、告警事件、数据治理、数据资产、系统管理等）快速定位并打开任一功能，作为打开视图的主入口。导航顶部提供**企业项目切换器**，用户可在当前租户下的多个项目间切换，切换后全平台数据按所选项目重新取数。

## Clarifications

### Session 2026-07-01

- Q: 切换企业项目时，工作区已打开的标签页如何处理？ → A: 保留已开功能标签并按新项目重新取数；带旧 projectId 参数的上下文详情标签自动关闭。
- Q: 用户首次进入（尚无上次选择记录）时，默认选中哪个项目？ → A: 按稳定排序取用户可访问项目列表的第一个。
- Q: 左侧导航「收起」后的形态是什么？ → A: 仅图标的窄导航条（icon rail），悬停/展开显示文字。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 按功能模块浏览并打开功能（Priority: P1）

作为平台使用者，我希望在应用最左侧看到一条常驻导航栏，其中所有功能按业务领域分组成若干目录（如「数据开发」「运维监控」「数据治理」等），每个目录下列出该领域的功能项；我点击任一功能项即可在主工作区打开对应视图。

**Why this priority**: 这是本特性的核心价值——把分散的功能按领域结构化呈现，并成为打开功能的主入口。没有它，左侧导航就不存在，特性无从谈起；单独实现它即构成可用的 MVP。

**Independent Test**: 仅实现本故事即可独立验证——打开应用后左侧出现分组导航，点击任一分组下的功能项能在工作区打开正确的视图，且分组覆盖全部可作为入口的功能视图。

**Acceptance Scenarios**:

1. **Given** 用户已登录进入应用，**When** 应用首屏加载完成，**Then** 应用最左侧展示一条常驻导航栏，功能项按功能模块分组成多个目录，每个目录有清晰的中文分组标题。
2. **Given** 左侧导航已展示，**When** 用户点击某目录下的某个功能项（如「运维监控」下的「运营中心」），**Then** 主工作区打开（或激活）对应视图，与现有「+」菜单打开同一视图的结果一致。
3. **Given** 用户点击的功能视图此前已在工作区打开，**When** 用户再次从导航点击该功能项，**Then** 工作区直接激活已存在的标签页，不重复创建新标签页。
4. **Given** 左侧导航已展示，**When** 用户查看导航，**Then** 每个功能项带有可辨识的图标与中文名称，分组顺序稳定（每次进入顺序一致）。

### User Story 2 - 切换企业项目（Priority: P2）

作为隶属多个项目的使用者，我希望在左侧导航顶部看到「当前项目」并能切换到本租户下的其他项目；切换后，平台各功能视图展示的是所选项目的数据。

**Why this priority**: 「企业项目导航」名副其实需要项目维度——让同一租户下的多项目用户在统一入口切换工作上下文。它独立于功能目录本身，可在 P1 之上增量交付；但因牵动全局取数上下文，范围与风险较大，故置于核心目录能力之后。

**Independent Test**: 在导航顶部打开项目切换器，选择另一个项目后，验证工作区内依赖项目维度的视图（如资产目录、运营中心）重新加载为所选项目的数据；切换偏好在刷新后保留。

**Acceptance Scenarios**:

1. **Given** 用户已登录且其租户下存在多个项目，**When** 用户查看左侧导航顶部，**Then** 显示当前所选项目名称，并提供可展开的项目切换入口列出可访问的项目。
2. **Given** 项目切换器已展开，**When** 用户选择另一个项目，**Then** 当前项目上下文切换为所选项目，**已打开的功能标签页保留并按新项目重新取数**，而**带旧项目参数的上下文详情标签页（如某条实例日志）自动关闭**。
3. **Given** 用户切换了项目，**When** 用户刷新或重新进入应用，**Then** 应用恢复到用户上次所选的项目上下文。
4. **Given** 用户**首次进入**应用且无上次选择记录，**When** 首屏加载，**Then** 默认选中其可访问项目列表中按稳定排序的**第一个**项目作为当前项目上下文。
5. **Given** 用户租户下仅有一个项目，**When** 用户查看导航顶部，**Then** 展示该项目名称，切换器可表现为只读/禁用态，不报错。

---

### User Story 3 - 标识当前所在功能（Priority: P2）

作为平台使用者，当我在工作区切换不同标签页时，我希望左侧导航能高亮标识「我当前正在看的功能」，从而始终知道自己在平台中的位置。

**Why this priority**: 提升方向感与导航闭环，但不阻塞核心的「浏览并打开」能力；在 P1 之上增量增强。

**Independent Test**: 在工作区激活不同标签页，观察左侧导航中对应功能项是否同步高亮为选中态；激活非导航入口的上下文详情视图时，高亮行为符合既定规则。

**Acceptance Scenarios**:

1. **Given** 工作区当前激活的是某功能视图，**When** 用户查看左侧导航，**Then** 该功能对应的导航项呈现选中/高亮态，其所属目录可辨识为当前所在分组。
2. **Given** 用户在工作区切换到另一个标签页，**When** 切换完成，**Then** 左侧导航的高亮项随之更新到新的当前功能。
3. **Given** 工作区当前激活的是带参数打开的上下文详情视图（如某条实例日志），**When** 用户查看左侧导航，**Then** 高亮落在其所归属的功能模块/父功能上（或无高亮），不出现误导性的错误高亮。

---

### User Story 4 - 收起/展开导航以让出空间（Priority: P3）

作为在小屏或需要更大工作区的使用者，我希望能把左侧导航收起为**仅图标的窄条（icon rail）**，需要时再展开，且我的偏好被记住。

**Why this priority**: 体验优化项，对核心导航能力非必需；在主流程稳定后增量交付。

**Independent Test**: 点击收起/展开控件，验证导航在「展开（图标+文字）」与「收起（仅图标窄条）」之间切换，工作区宽度相应变化；刷新页面后保留上次的展开状态。

**Acceptance Scenarios**:

1. **Given** 左侧导航处于展开态，**When** 用户点击收起控件，**Then** 导航切换为**仅图标的窄条**，主工作区获得更多横向空间，功能项图标仍可一键直达。
2. **Given** 左侧导航处于收起（仅图标）态，**When** 用户点击展开控件（或悬停），**Then** 分组目录与功能项文字重新可见。
3. **Given** 用户设置了某个展开/收起状态，**When** 用户刷新或重新进入应用，**Then** 导航恢复到用户上次选择的状态。

---

### Edge Cases

- **占位/未实现功能**：导航中出现尚未实现的占位功能项（如「数据集成」「数据服务」）时，照常列出，点击打开占位视图，与现状一致；不隐藏，以便预告能力版图。
- **上下文详情视图**：某些视图只能携带参数从其他视图跳转打开（如实例日志、工作流实例详情），不适合作为独立导航入口——这些视图不在左侧导航中列为独立功能项。
- **权限差异**：本期不做基于权限的过滤，全部功能照常展示；权限收口为后续特性。
- **空分组**：若某功能模块下当前没有任何可见功能项，该分组目录不应渲染为空标题。
- **长列表**：当分组功能项较多导致导航整体超出视口高度时，导航区域可纵向滚动，分组标题的可读性不被破坏。
- **与「+」启动菜单并存**：左侧导航成为主入口后，原 Tab 栏「+」菜单与深链 `?open=` 打开方式仍可用，三者打开同一功能时行为一致（同一视图只存在一个标签页）。
- **切换项目时的已开标签页**：切换项目后，已打开的功能视图标签**保留**并以新项目上下文重新取数；带旧项目参数的上下文详情标签页（如某条实例日志）**自动关闭**，避免展示跨项目的过期数据。
- **单项目/无项目**：租户下仅一个项目时切换器为只读；理论上无可访问项目时，导航顶部给出明确空态提示而非崩溃。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在应用主界面**最左侧**展示一条**常驻**的导航栏，位于现有主工作区之左，登录后所有页面均可见。
- **FR-002**: 导航栏必须将全部「可作为入口的功能视图」**按功能模块（业务领域）分组**为若干目录，每个目录有明确的中文分组标题。
- **FR-003**: 每个功能项必须以「图标 + 中文名称」呈现，名称与现有视图标题保持一致（复用既有视图标题）。
- **FR-004**: 用户点击任一功能项时，系统必须在主工作区打开对应视图；若该视图已打开，则激活其现有标签页而非重复创建（与现有打开机制一致）。
- **FR-005**: 分组与组内功能项的排列顺序必须**稳定且可预期**（每次进入应用顺序一致）。
- **FR-006**: 导航必须**标识当前激活的功能**：工作区当前激活标签页对应的功能项在导航中呈现选中/高亮态，并随工作区标签页切换实时更新。
- **FR-007**: 对于只能携带参数从其他视图上下文跳转打开的详情类视图（如实例日志、工作流实例详情），系统**不得**将其列为左侧导航的独立功能入口；当此类视图为当前激活页时，导航高亮按既定归属规则处理而不产生误导性高亮。
- **FR-008**: 左侧导航必须与现有的「+」启动菜单及深链 `?open=` 打开方式**并存且行为一致**——同一功能无论从哪个入口打开，工作区中至多存在一个对应标签页。
- **FR-009**: 用户必须能**收起/展开**左侧导航；收起态为**仅图标的窄导航条（icon rail）**（悬停/展开恢复文字），功能图标在收起态仍可一键直达；该偏好在刷新/重进后被**记住**。
- **FR-010**: 导航的视觉风格必须遵循项目既有设计系统（语义化配色 token、间距、图标体系），与现有工作区视觉一致，区域间以留白区分、不使用分割线。
- **FR-011**: 当导航内容超出视口高度时，导航区域必须可纵向滚动，且分组结构始终清晰可辨。
- **FR-012**: 全部分组标题与功能项文案必须走前端国际化（zh-CN / en-US 两套 key 集一致），不得硬编码中文。
- **FR-013**: 导航顶部必须展示**当前所选企业项目**，并提供切换入口列出当前租户下用户可访问的项目供选择。
- **FR-014**: 用户切换项目后，系统必须将**当前项目上下文**更新为所选项目，平台内按项目维度取数的功能视图必须以新项目重新取数（替代当前硬编码的单一项目上下文）。
- **FR-015**: 所选项目上下文必须在刷新/重进后被**记住并恢复**。
- **FR-016**: 项目列表必须限定在**当前租户/当前用户可访问**范围内，不得跨租户暴露其他项目。
- **FR-017**: 当用户可访问项目为 1 个时，切换器以只读/单项形态呈现；为 0 个时给出明确空态提示，均不得报错或阻塞导航其余部分。
- **FR-018**: 切换项目时，系统必须**保留已打开的功能视图标签页并按新项目重新取数**，同时**自动关闭带旧项目参数的上下文详情标签页**（如实例日志、工作流实例详情），不得展示跨项目的过期数据。
- **FR-019**: 用户**首次进入**且无历史选择记录时，系统必须默认选中其可访问项目列表中**按稳定排序的第一个**项目作为当前项目上下文。

### 功能模块分组（建议目录结构）

以下为基于现有视图归纳的建议分组（最终命名与归属可在 plan 阶段微调）：

| 功能模块（目录） | 包含功能视图 |
|------------------|--------------|
| 数据开发 | 数据开发（工作流画布） |
| 运维监控 | 运营中心、系统指标、集群机器、数据新鲜度 |
| 告警事件 | 告警中心、事件中心 |
| 数据治理 | 资产目录、数据质量、数据血缘 |
| 数据资产 | 指标市场、数据源管理、数据集成、数据服务 |
| 数据分析 | 业务报表 |
| 系统管理 | 系统设置 |

> 上下文详情类视图（实例日志、工作流实例详情）不作为独立目录入口，由相关视图内跳转打开。

### Key Entities *(include if feature involves data)*

- **功能模块（Module / 目录）**：一个业务领域分组，含中文标题、稳定排序、若干功能项。本特性的分组结构属于前端呈现层组织，不要求新增后端数据。
- **功能项（Nav Item）**：对应一个可打开的功能视图，含图标、名称（复用既有视图标题）、目标视图标识、所属功能模块。
- **企业项目（Project）**：租户下的逻辑工作空间，含名称等标识；用户可访问其租户下的若干项目（后端已有 `/api/projects` 与 Project 概念）。导航顶部据此呈现切换器。
- **导航/项目状态（Nav & Project State）**：当前高亮的功能、当前所选项目上下文、导航展开/收起偏好——属于需持久化的前端 UI/上下文状态。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户进入应用后，**无需点击任何菜单**即可在左侧看到平台全部功能模块分类，建立功能全景认知。
- **SC-002**: 用户从「想打开某功能」到「该功能在工作区打开」，**点击次数 ≤ 2 次**（展开态下点击功能项即 1 次）。
- **SC-003**: 左侧导航覆盖 **100%** 的「可作为入口的功能视图」，无遗漏、无重复入口。
- **SC-004**: 在工作区切换任意标签页后，左侧导航高亮项在 **1 秒内**同步到当前功能，准确率 100%（上下文详情视图按既定规则）。
- **SC-005**: 收起/展开偏好在刷新后 **100%** 被正确恢复。
- **SC-006**: 任意功能无论从左侧导航、Tab「+」菜单或深链打开，工作区最终**至多保留一个**对应标签页（无重复标签）。
- **SC-007**: 用户切换企业项目后，按项目维度取数的视图在 **2 秒内**刷新为所选项目数据，且仅展示该项目（及其租户）范围内的数据，无跨项目/跨租户串数。
- **SC-008**: 所选项目上下文在刷新后 **100%** 被正确恢复。

## Assumptions

- **「企业项目导航」理解**：本期理解为「按功能模块组织的全平台功能目录（左侧导航栏）+ 顶部企业项目切换器」。项目切换器**在本期范围内**：用户可在当前租户下其可访问的多个项目间切换，切换后平台按所选项目重新取数——替代前端当前硬编码的单一项目上下文（projectId=1）。后端已有 `/api/projects` 及项目成员概念可复用，无需新建项目管理能力。
- **与「+」菜单的关系**：本期左侧导航作为**主入口新增**，现有「+」启动菜单与深链 `?open=` 打开方式**保留并存**，不在本期移除；三入口行为一致。
- **分组归属**：分组命名与各视图归属采用上表的合理默认，plan 阶段可微调，不改变「按功能模块划分」的核心结构。
- **占位视图**：未实现的占位功能（数据集成、数据服务）照常在导航中列出，点击打开占位视图，与现状一致。
- **权限过滤**：本期不做基于角色/权限的导航项过滤，全部功能对已登录用户可见。
- **复用现有机制**：导航点击打开视图复用现有工作区打开/激活标签页机制，不新增后端接口；功能项名称复用既有视图标题。
- **响应式**：以桌面端为主要目标，窄屏下提供收起能力让出空间；移动端专门适配不在本期范围。

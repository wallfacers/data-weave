**Feature Branch**: `041-script-lineage-extraction`

**Created**: 2026-07-03

**Status**: Draft

**Input**: User description: "任务血缘解析扩展：当前基于 Apache Calcite 语法树解析 SQL 任务的表/字段血缘；希望扩展到 Python、Shell 脚本类任务——当脚本代码中存在写表/读表场景时，也能解析出其涉及的表和字段血缘，并考虑训练一个小模型来处理脚本类血缘抽取"

## Clarifications

### Session 2026-07-03

- Q: 脚本中抽取出的表，其数据源坐标如何确定？ → A: 沿用任务绑定数据源（写侧优先 target 数据源），未绑定时降级到租户级身份——与既有 SQL 任务血缘坐标解析完全同源。
- Q: 被人工剔除的推断边，重复 push 时用什么键匹配抑制？ → A: 血缘语义键（任务 + 读/写方向 + 表坐标，字段级再加字段名）；脚本改写不使剔除失效，若后续出现对该表的真实写入需人工撤销剔除。（用户暂离，采纳推荐答案）
- Q: 谁有权对推断边执行确认/剔除？ → A: 具备项目管理权限的成员（沿用既有项目角色体系）；普通成员只读可见来源标注。（用户暂离，采纳推荐答案）
- Q: "未解析提示"在哪里查看？ → A: 血缘图谱详情面板内呈现（图谱中任务以读写边呈现，入口为该任务对应边的详情），不新建独立视图。（用户暂离，采纳推荐答案；plan 阶段核实图谱无任务节点后措辞由"任务节点"修正为"边详情"）
- Q:（改判）小模型是否纳入本期？ → A: **纳入**。用户明确推翻首问裁决：小模型全流程（选型、数据集构建、训练、评估、接入）由本特性交付；模型与数据集托管用户 Hugging Face 账号；训练环境为本机 12G 显存 GPU；模型选型与数据集来源授权实现方决定。推理以独立服务进程提供、不嵌入平台服务端进程（宪法原则 IV 合规姿态，见 plan Complexity Tracking）。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 脚本内嵌 SQL 的血缘自动解析 (Priority: P1)

数据开发者的 Python/Shell 任务里，绝大多数读写表行为是通过内嵌 SQL 语句完成的（如 Python 中 `spark.sql("INSERT INTO ...")`、`cursor.execute("INSERT ...")`，Shell 中 `hive -e "INSERT ..."`、`psql -c "..."`）。开发者 push 任务后，平台自动从脚本源码中定位这些内嵌 SQL 片段，复用既有 SQL 血缘解析能力，把脚本任务涉及的读表、写表（含字段级血缘，若 SQL 可解析出）纳入血缘图谱——与 SQL 类型任务享受同等待遇。

**Why this priority**: 内嵌 SQL 覆盖脚本任务读写表场景的主体，且能复用已验证的 SQL 解析链路，投入产出比最高；单独交付即可消除大部分脚本血缘盲区，构成可用 MVP。

**Independent Test**: push 一个含 `INSERT INTO dw.orders SELECT id, amount FROM ods.orders` 内嵌 SQL 的 Python 任务，在血缘图谱中能看到该任务节点与 `ods.orders`→任务→`dw.orders` 的表级边（及字段边），即为通过。

**Acceptance Scenarios**:

1. **Given** 一个 Python 任务，脚本内含字符串常量形式的 `INSERT INTO t2 SELECT a, b FROM t1`，**When** 开发者 push 并上线该任务，**Then** 血缘图谱出现 t1（读）与 t2（写）的表级血缘边，且字段 a、b 的字段级血缘一并呈现。
2. **Given** 一个 Shell 任务，脚本内通过 `hive -e "..."` 执行写表 SQL，**When** push 并上线，**Then** 血缘图谱同样出现对应读写表边。
3. **Given** 一个脚本内含多条 SQL 语句（建表、插入、查询混合），**When** push 并上线，**Then** 每条可解析语句的读写表均被抽取并合并到该任务的血缘中，不可解析的语句被跳过且不阻断其余语句的抽取。
4. **Given** 一个不含任何读写表行为的纯计算脚本，**When** push 并上线，**Then** 不产生任何血缘边，也不误报。

### User Story 2 - 非 SQL 代码写表场景的血缘推断 (Priority: P2)

部分脚本不写 SQL，而是通过程序接口读写表（如 Python 的 `df.write.saveAsTable("dw.orders")`、`df.to_sql("orders", ...)`，Shell 中 `sqoop export --table orders`）。平台对这类"代码写表"场景进行血缘推断：识别出目标表（及可确定的字段），生成带"推断"标记和置信度的血缘边。推断依据由内置识别规则覆盖常见接口形态；推断通道以可插拔抽取器形式实现（FR-010），规则抽取器先行交付，训练的小模型作为第三个抽取器在本特性内接入（见 User Story 4），覆盖规则触达不到的写法。

**Why this priority**: 非 SQL 写表是剩余盲区，但形态发散、准确率天然低于 SQL 解析，须在 P1 打通链路后再扩展；且其价值依赖 P1 已建立的"脚本任务可有血缘"的展示与存储通道。

**Independent Test**: push 一个用 `saveAsTable("dw.result")` 写表的 Python 任务，血缘图谱出现指向 `dw.result` 的写边且带"推断"来源标记，即为通过。

**Acceptance Scenarios**:

1. **Given** 一个 Python 任务用程序接口写表且表名为字符串常量，**When** push 并上线，**Then** 血缘图谱出现对应写表边，边上标注来源为"推断"及置信度。
2. **Given** 写表调用中字段以显式列清单形式静态可枚举（如选取指定列后写入），**When** push 并上线，**Then** 对应字段级血缘边一并出现，同样带"推断"来源标注。
3. **Given** 一个脚本的表名由运行时变量拼接、静态无法确定，**When** push 并上线，**Then** 系统不生成错误的血缘边，并将"存在疑似写表但无法确定目标"记录为可查看的提示，而非静默丢弃。
4. **Given** 推断通道自身异常或不可用，**When** push 并上线，**Then** push/上线流程不受阻断，内嵌 SQL 通道（P1）的抽取结果不受影响。

---

### User Story 3 - 血缘来源区分与人工修正 (Priority: P3)

数据治理人员在血缘图谱中查看某张表的上下游时，需要区分每条血缘边的来源可信度：来自 SQL 语法解析的边（确定）与来自脚本推断的边（可能有误）。对推断边，治理人员可以人工确认（转为已确认）或标记错误（从图谱剔除），修正结果持久生效，同任务重新 push 后不重复出现已剔除的错误边。

**Why this priority**: 推断存在错报/漏报，缺少人工兜底会污染血缘图谱的可信度；但它依赖 P2 产生推断边后才有意义。

**Independent Test**: 对一条推断来源的血缘边执行"标记错误"，刷新图谱后该边消失，重新 push 同任务后该边仍不出现，即为通过。

**Acceptance Scenarios**:

1. **Given** 血缘图谱中同时存在 SQL 解析边与脚本推断边，**When** 治理人员查看图谱，**Then** 两类边在视觉与详情上可明确区分来源。
2. **Given** 一条推断边被人工标记为错误，**When** 该任务后续重新 push 且推断结果不变，**Then** 该边不再出现在图谱中。
3. **Given** 一条推断边被人工确认，**When** 查看图谱，**Then** 该边呈现为"已确认"状态并保留推断来源记录。

---

### User Story 4 - 小模型推断通道：训练、评估与接入 (Priority: P4)

规则模式表天然只覆盖枚举过的接口形态。平台方（本特性内由实现方全流程执行）构建一个专用小模型：从公开代码/SQL 数据集合成带标注的"脚本→表/字段读写"训练数据（辅以本特性语料库为种子与评估集），在 12G 显存 GPU 上微调开源代码小模型，使其能从 Python/Shell 源码中抽取表/字段引用及读写方向；通过评估门槛后，以独立推理服务接入为推断通道的第三个抽取器——模型产出的边带独立来源标记（模型推断）与置信度，与规则边走同一冲突消解、宁缺毋滥、人工修正机制。模型与数据集发布到用户 Hugging Face 账号，版本与评估报告可追溯。

**Why this priority**: 模型通道的价值在补规则的长尾，其接入面（可插拔抽取器、来源标注、修正机制）全部由 US1-US3 建立；训练与评估可与前三个故事并行推进，但接入验收依赖 US2 的推断通道基建。

**Independent Test**: 在 held-out 评估集上模型通道达到发布门槛（SC-006）；push 一个规则模式表不覆盖、但模型可识别的写表脚本，图谱出现来源为"模型推断"的虚线边；停掉推理服务后重复 push，流程不受阻且自动降级为纯规则结果。

**Acceptance Scenarios**:

1. **Given** 训练数据管线与评估集就绪，**When** 执行训练与评估，**Then** 产出模型在 held-out 集上达到 SC-006 门槛，评估报告与模型工件发布到 Hugging Face（未达门槛则不接入，特性以规则通道交付并记录差距）。
2. **Given** 一个规则未覆盖的写表形态（如自定义封装函数内的表名常量），**When** push 该任务，**Then** 模型通道产出对应写表边，来源标注为"模型推断"、置信度 UNVERIFIED（虚线），可被人工确认/剔除。
3. **Given** 推理服务不可用或单任务推理超时，**When** push 任务，**Then** push 零阻断，血缘结果自动退化为"内嵌 SQL + 规则"两通道产物，失败留痕。
4. **Given** 同一读写点被规则与模型同时命中且结论冲突，**When** 抽取完成，**Then** 按 SQL 解析 > 规则推断 > 模型推断的优先序消解，低优先结果不覆盖高优先结果。

---

### Edge Cases

- 表名由变量、f-string、字符串拼接、环境变量在运行时决定：静态抽取无法确定目标 → 不得猜测生成边，须记录"未能解析的疑似读写点"供人工查看。
- 脚本中的 SQL 出现在注释、文档字符串、打印语句里（并非真正执行）→ 应尽量避免误报；无法区分时按推断边（低置信度）处理而非确定边。
- 一个脚本内同一张表既读又写（自依赖）→ 血缘边应如实呈现，不因去重丢失方向。
- 超大脚本（数千行）→ 抽取须有时长上限，超时降级为"未解析"并留痕，不阻断 push/上线。
- 脚本语法本身有错误（无法构成合法 Python/Shell）→ 抽取失败不阻断 push（push 是否放行由既有门禁决定，血缘抽取不新增阻断点）。
- 同一任务多次 push 内容变化 → 血缘以最新版本为准，旧版本产生的边被替换而非累加。
- 推断通道对同一写表点给出与内嵌 SQL 通道冲突的结果 → SQL 解析结果优先，推断结果不覆盖确定结果。
- 模型输出格式非法（非约定结构）或幻觉出脚本中不存在的表名 → 输出校验拒收（表名必须能在脚本文本中定位到），拒收样本留痕供迭代，不入图。
- 模型对同一脚本重复推理结果不稳定 → 推理用确定性解码（温度 0），同版本模型同输入必须产出同结果。
- 训练数据中的表名/库名泄露进模型输出 → 输出校验的"脚本内可定位"规则同时兜住此项。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在脚本类任务（Python、Shell）push/上线时对脚本源码执行血缘抽取，抽取范围包括读表、写表；抽取结果并入既有血缘图谱，与 SQL 任务的血缘在同一图谱中统一呈现。
- **FR-002**: 系统 MUST 从脚本源码中识别内嵌 SQL 片段（字符串常量形式的 SQL，含常见执行载体：Python DB/计算接口调用、Shell 内嵌 SQL 命令行），并复用既有 SQL 血缘解析能力抽取表级与字段级血缘。
- **FR-003**: 系统 MUST 支持对非 SQL 的程序接口读写表形态进行推断抽取（至少覆盖：表名为字符串常量的常见读写接口调用），推断粒度至少到表级；对脚本中字段可静态枚举的形态（如显式列清单、显式列参数），系统 MUST 同时抽取字段级血缘并纳入验收；字段无法静态枚举时仅出表级边，不猜测字段。
- **FR-004**: 每条血缘边 MUST 记录来源类型（SQL 解析 / 脚本推断）与置信度；推断边在图谱查询与展示中 MUST 可与解析边区分。
- **FR-005**: 血缘抽取 MUST 零阻断：任何抽取失败、超时、推断通道不可用，均不得使 push、上线、任务运行失败；失败须留痕可查。
- **FR-006**: 对静态无法确定读写目标的疑似读写点，系统 MUST 记录为"未解析提示"（含任务、脚本位置线索），在血缘图谱详情面板（该任务对应边的详情）内可查看，不得静默丢弃，也不得生成猜测性血缘边；不新建独立视图。
- **FR-007**: 具备项目管理权限的成员 MUST 能对推断边执行人工确认与标记错误（普通成员只读）；剔除抑制按血缘语义键匹配（任务 + 读/写方向 + 表坐标，字段级边再加字段名），与脚本具体写法无关——被标记错误的边在同任务后续重复推断中不再进入图谱，除非人工撤销剔除。
- **FR-008**: 同一任务重新 push 后，其脚本血缘 MUST 以最新脚本内容为准整体替换（人工确认/剔除状态按 FR-007 保留），不产生历史残留边。
- **FR-009**: 当内嵌 SQL 解析结果与推断结果对同一读写点冲突时，系统 MUST 以 SQL 解析结果为准。
- **FR-010**: 系统 MUST 以可插拔抽取器方式实现脚本推断通道——新增识别规则或模型抽取器不改动血缘存储与展示层；本期内该扩展点由模型抽取器（FR-012）实际兑现验证。
- **FR-011**: 脚本抽取出的表 MUST 按任务绑定的数据源解析坐标（写侧优先 target 数据源），任务未绑定数据源时降级到租户级身份——坐标解析逻辑 MUST 与既有 SQL 任务血缘同源，保证同一物理表在图谱中对齐为同一节点；本期不做脚本内多数据源识别。
- **FR-012**: 系统 MUST 提供模型推断抽取器：调用独立推理服务从脚本源码抽取表/字段读写引用，产出边来源标注为"模型推断"、置信度 UNVERIFIED，走与规则通道完全相同的冲突消解（优先序：SQL 解析 > 规则推断 > 模型推断）、宁缺毋滥、人工修正与展示机制。模型输出 MUST 经结构与"表名在脚本文本中可定位"双重校验，校验不过不入图。
- **FR-013**: 模型推理 MUST 以独立服务进程提供，不嵌入平台服务端进程；推理服务不可用、超时或未部署时 MUST 自动降级为"内嵌 SQL + 规则"通道，push/上线零阻断，降级留痕。推理 MUST 使用确定性解码保证结果可复现。
- **FR-014**: 训练全流程 MUST 可复现：数据管线（公开数据集来源 + 合成生成 + 种子语料）脚本化并入库版本管理；评估使用与训练不重叠的 held-out 集；达到 SC-006 发布门槛方可接入，未达门槛记录差距、特性以规则通道交付。
- **FR-015**: 模型工件、数据集与评估报告 MUST 发布到用户 Hugging Face 账号（私有 repo），带版本标识；平台侧记录接入的模型版本，推理结果可回溯到模型版本。

### Key Entities

- **脚本血缘抽取结果**: 一次 push/上线针对一个脚本任务产生的抽取产物；包含任务标识、脚本版本、抽取时间、各通道（内嵌 SQL / 推断）的成败与耗时。
- **血缘边（扩展）**: 既有血缘边增加来源类型（解析/推断）、置信度、人工状态（无/已确认/已剔除）属性。
- **未解析提示**: 静态无法确定目标的疑似读写点记录；包含任务标识、脚本位置线索、疑似形态描述、产生时间。
- **人工修正记录**: 项目管理权限成员对推断边的确认/剔除动作；包含操作人、时间、目标边的血缘语义键（任务 + 方向 + 表坐标 [+ 字段名]），剔除记录按语义键抑制同边重复进入图谱。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对表名为字符串常量的内嵌 SQL 脚本任务，表级血缘抽取正确率（正确边 / 应抽取边）达到 95% 以上，与既有 SQL 任务解析能力持平。
- **SC-002**: 对非 SQL 常见读写接口（表名字符串常量）场景，表级推断召回率达到 80% 以上，且错报边占推断边总数低于 10%；对字段静态可枚举的写表形态，字段级抽取正确率达到 90% 以上。
- **SC-003**: 脚本血缘抽取引入后，push/上线端到端耗时增加不超过原耗时的 20%，且抽取失败导致 push 失败的发生率为 0。
- **SC-004**: 治理人员在血缘图谱中能在 3 次以内的交互内完成"识别推断边→查看依据→确认或剔除"的完整修正流程。
- **SC-005**: 所有推断边 100% 带有来源与置信度标注；被人工剔除的边在后续重复 push 中重现率为 0。
- **SC-006**（模型发布门槛）: 在 ≥200 条与训练不重叠的 held-out 标注样本上，模型通道表级抽取 precision ≥ 85%；在"规则模式表未覆盖形态"子集上表级召回 ≥ 60%（体现模型对规则的增量价值）；幻觉率（输出的表名无法在脚本文本中定位的比例）≤ 2%（校验层拦截前的原始输出计）。
- **SC-007**: 推理服务停机场景下 push 成功率 100% 且自动降级；单任务模型推理超时上限内完成率 ≥ 95%（超时即弃不阻断）。

## Assumptions

- 「学员解析」按上下文理解为「血缘解析」；本特性是既有 SQL 血缘能力（表级 + 字段级）向脚本类任务的扩展，不改变既有 SQL 任务的血缘行为。
- 抽取时机为设计态（push/上线时对脚本源码静态抽取）；脚本任务的运行态行级指标不在本特性范围（脚本进程内部执行的语句平台无法像 SQL 任务那样逐条上报）。
- 覆盖的脚本任务类型为 Python 与 Shell 两类；其他类型（如 Spark 提交脚本中的内嵌 SQL）如形态一致可顺带受益，但不作为验收范围。
- 血缘图谱沿用既有单一图存储与既有图谱展示入口，本特性只新增边属性与来源区分展示，不新建独立的脚本血缘视图。
- 推断通道的第一优先级是"宁缺毋滥"：无法确定时记录提示而非生成边，图谱可信度优先于召回率。
- 小模型纳入本期（用户改判）：定位为推断通道内的第三个可插拔抽取器，输出与规则抽取器走同一置信度/来源标注与人工修正机制，推理服务不可用时自动退化为"内嵌 SQL + 规则"。人工修正记录持续积累为后续模型迭代的标注数据。
- 训练环境为本机单卡 12G 显存 GPU；模型选型以该预算内可微调、可低成本部署推理为约束。推理服务部署形态（GPU 常驻或 CPU 量化）由 plan 决定，平台只依赖其 HTTP 契约。
- 训练数据以公开数据集（许可合规）+ 程序化合成为主，本特性语料库样例作为种子与评估集来源；不使用租户生产脚本训练（隐私边界）。
- 用户 Hugging Face 账号可用（经 MCP 认证授权），模型/数据集私有托管其下。
- 模型未达 SC-006 门槛不阻塞特性交付：US1-US3 照常上线，US4 记录差距后可独立补交。

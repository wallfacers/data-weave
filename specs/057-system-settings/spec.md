# Feature Specification: 全局系统设置——AI Agent 配置统收（项目级 → 全局单例）

**Feature Branch**: `057-system-settings`

**Created**: 2026-07-08

**Status**: Draft

**Input**: User description: "在系统设置中新增，AiAgent 的配置，把血缘上的 AI Agent [配置移到] 系统设置中，目的是，全局所有项目共用一个即可"

## 背景与动机 *(为什么做这件事)*

血缘智能抽取目前提供一条「云 AI Agent」通道：通过兼容 Anthropic / OpenAI 协议的云端大模型，对脚本长尾与确定性解析失败的 SQL 做兜底推断。该通道的配置（协议、端点、模型、凭据、启用开关、运行参数）今天**按项目隔离**——每个项目各配一份，挂在**血缘探索器工具栏**的一个弹窗入口里，互不复用。

实际使用中，同一个租户内的 AI 供应商与凭据几乎是同一套：按项目重复配置带来不必要的管理负担，也容易各项目配置漂移。本特性把这份配置收拢为**租户级全局单例**——全租户所有项目共用同一份，配一次即处处生效；并把它的管理入口从血缘工具栏弹窗迁到**已有的「系统设置」视图**——在其中**新增一个顶级「配置」tab**，该 tab 内部采用**左侧导航 + 右侧内容**的布局（与「数据开发」视图同风格），把 AI Agent 作为左导航的首个分区；未来其它全局/平台配置可作为新分区持续挂入同一套左导航，无需再改容器。

血缘的抽取内核（协议适配、加密、限流、异步富化、防幻觉校验、审计）不在本特性重做——本特性聚焦**位置迁移 + 作用域从项目提升为全局**，外加**系统设置里一套可生长的配置分区外壳**。

## Clarifications

### Session 2026-07-08

- Q: AI Agent 全局配置如何在已有「系统设置」视图中承载，并兼顾后续更多全局配置？ → A: 在系统设置**新增一个顶级「配置」tab**（既有 用户/角色/项目 三个 tab 不动）；该 tab 内部采用**左导航 + 右内容**布局（参照「数据开发」视图风格），AI Agent 作为左导航的**首个分区**，未来其它全局配置以新分区挂入同一左导航。
- Q: 这套「左导航 + 右内容」是只服务 AI Agent 还是可复用？ → A: 作为**可复用能力**建设——左导航分区列表 + 右侧内容区是一个通用外壳，新增配置类型只需注册一个分区项 + 其右侧内容组件，不必再改容器结构。

> 其余模糊点（权限门槛、旧入口处置、既有数据迁移、并发编辑）以合理默认写入下方 Assumptions，未作为本轮提问；plan 阶段可再固化。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 全局唯一的 AI Agent 配置（Priority: P1）

平台管理员进入「系统设置 → 配置 tab → AI Agent 分区」，配置**一份**云端大模型通道：选择协议（Anthropic 或 OpenAI 兼容），填入服务端点、模型名、API 凭据，按需调整运行参数（超时、速率上限、最大列数），保存并启用。这份配置对当前租户内**所有项目**立即生效——任一项目的血缘智能抽取都用这同一份配置，无需逐项目重复填写，也不存在项目间漂移。

**Why this priority**: 用户诉求的核心主干——把分散在每个项目的 AI Agent 配置收拢为全局一份，直接消除重复配置与不一致。它以既有血缘 AI Agent 通道为地基，只改变配置的「存放位置」与「作用域」，单独交付即构成可用增量。

**Independent Test**: 在 系统设置 → 配置 → AI Agent 保存并启用一份全局配置，然后在一个**从未单独配过** AI Agent 的项目里 push 一个需要 AI 抽取的脚本任务，其血缘图谱短时内出现来源标记为「AI 推断」的边——即通过。

**Acceptance Scenarios**:

1. **Given** 管理员在 AI Agent 分区保存并启用了一份全局配置，**When** 租户内任一项目触发需要 AI 抽取的任务，**Then** 该任务使用这份全局配置（与其它项目所用的完全一致）。
2. **Given** 全局配置已存在，**When** 管理员修改它（如更换模型或调整限流），**Then** 所有项目后续的 AI 抽取立即按新配置执行，无需逐项目改动。
3. **Given** 全局配置未配置或被禁用，**When** 任务触发，**Then** AI 通道整体旁路，其余血缘通道（内嵌 SQL / 规则 / 本地小模型）不受影响，任务上线/push 不阻断。
4. **Given** 多个项目同时触发 AI 抽取，**When** 并发外呼，**Then** 共用同一份全局配置与速率额度，且每条调用审计能追溯到来源项目与来源任务。

### User Story 2 - 在系统设置新增「配置」Tab：左导航 + 右内容（Priority: P1）

「系统设置」视图已存在（用户/角色/项目三个顶级 tab，左侧 admin 导航组，按 `project:manage` 权限可见）。本特性**新增第 4 个顶级「配置」tab**（既有三个不动）；该 tab 内部采用**左侧导航 + 右侧内容**布局——与「数据开发」视图同风格：左侧是一个配置分区列表（圆角卡片、可拖拽调宽、常驻），右侧是选中分区的配置内容区。AI Agent 作为左导航的首个分区；未来其它全局配置可作为新分区挂入，无需改动容器。这是一套**可复用的配置外壳能力**。

**Why this priority**: 它与 US1 并列为核心：US1 定义「全局一份配置」的语义，US2 定义「在系统设置里以一个可持续生长的配置分区外壳承载它」。复用既有 settings 入口与权限、不动既有三个 tab，使范围与风险最小。

**Independent Test**: 以具 `project:manage` 权限的身份进入系统设置，切到「配置」tab，左导航出现「AI Agent」分区并可选中，右侧呈现其配置内容、可完成一次保存——即通过。

**Acceptance Scenarios**:

1. **Given** 具 `project:manage` 权限的用户进入系统设置，**When** 查看顶级 tab 条，**Then** 看到「配置」与 用户/角色/项目 并列（下划线式 tab 风格一致），既有三个 tab 行为不变。
2. **Given** 切到「配置」tab，**When** 浏览，**Then** 看到左侧分区导航 + 右侧内容区的左右布局；左导航首项为「AI Agent」。
3. **Given** 选中「AI Agent」分区，**When** 查看右侧，**Then** 呈现协议、端点、模型、凭据、启用开关及运行参数等字段（表单内联渲染）。
4. **Given** 未来新增另一项全局配置，**When** 注册为新分区，**Then** 它作为左导航新条目出现，右侧切换到其内容，容器结构无需改动。
5. **Given** 无 `project:manage` 权限的用户，**When** 查看导航，**Then** 系统设置入口不可见（沿用既有过滤），无法触及「配置」tab。

### User Story 3 - 连通性测试与凭据脱敏（Priority: P2）

管理员在保存前能对当前填入的配置发一次最小探活外呼，立即得到成功（含延迟）或失败（含脱敏原因）的反馈；凭据保存后以脱敏掩码回显，明文绝不回传界面、日志或审计。

**Why this priority**: 提升配置体验与安全性，是「可用且敢用」的保障层。依赖 US1 已建立的全局配置语义，价值独立，但缺少它配置流程仍可用。故列为 P2。

**Independent Test**: 在 AI Agent 分区填入一套配置，点「测试连接」即时看到成功/失败与延迟；保存含凭据的配置后重新打开，凭据以掩码呈现、明文不可见——即通过。

**Acceptance Scenarios**:

1. **Given** 管理员填好配置字段，**When** 点击「测试连接」，**Then** 立即返回成功（含延迟毫秒数）或失败（含脱敏后的技术原因）。
2. **Given** 已保存含凭据的全局配置，**When** 重新打开 AI Agent 分区，**Then** 凭据以脱敏掩码显示，明文不出现在任何回显、日志或审计。
3. **Given** 编辑已有配置时凭据字段留空，**When** 保存，**Then** 原凭据保留不变（留空 = 不修改语义）。

### Edge Cases

- **既有按项目配置的迁移**：升级到全局单例后，各项目已存在的 AI Agent 配置择一收拢为全局、其余作废并留痕，不得静默丢失（细节 plan 阶段固化）。
- **血缘工具栏旧入口**：原「血缘探索器工具栏 → AI Agent 配置弹窗」移除，单一真相源在 系统设置 → 配置 → AI Agent。
- **全局误改的爆炸半径**：全局配置被一次保存误改或误禁时，所有项目即时受影响。缓解：启用开关 + 完整审计 + 管理员权限门槛，变更可追溯、可快速回退。
- **并发编辑**：两位管理员同时编辑同一份全局配置时，默认后写覆盖 + 审计；是否加乐观锁提示在 plan 阶段细化。
- **免鉴权网关**：协议/端点/模型为必填，但凭据可空（当云端点由网关代签时）。
- **权限边界**：系统设置沿用既有 `project:manage` 门槛；普通项目成员不可见、不可改全局配置，但其项目仍受全局配置驱动。
- **空分区/无配置态**：AI Agent 分区在尚未配置时，右侧呈现明确的空态与「去配置」引导，而非空白。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 复用已有的「系统设置」入口（左侧 admin 导航组，按 `project:manage` 权限过滤可见），**不新增**导航入口。
- **FR-002**: 系统设置必须**新增一个顶级「配置」tab**（与 用户/角色/项目 并列，沿用下划线式 tab 风格）；既有三个 tab 的功能与布局保持不变。
- **FR-003**: 「配置」tab 内部必须采用**左侧导航 + 右侧内容**布局：左侧为配置分区列表，右侧为选中分区的配置内容；整体观感与「数据开发」视图同风格（圆角卡片、可拖拽调宽的左侧常驻面板、留白分区而非分割线）。
- **FR-004**: 「AI Agent」必须作为「配置」tab 左导航的**首个分区**；右侧渲染其配置内容（协议、服务端点、模型名、凭据、启用开关、运行参数：请求超时/速率上限/最大处理列数），表单内联渲染。
- **FR-005**: 这套「左导航分区 + 右侧内容」必须作为**可复用外壳能力**——新增一种全局配置只需注册一个分区项与其右侧内容组件，不必改动容器结构。
- **FR-006**: AI Agent 配置必须以**租户级全局单例**存储——整个租户只有一份，对所有项目共同生效，不再支持按项目单独配置。
- **FR-007**: 任一项目的血缘智能抽取必须读取并使用这份全局配置；项目上下文中不得存在独立的 AI Agent 配置覆盖。
- **FR-008**: 系统必须在保存前提供「测试连接」——对当前填入（或覆盖既有）的配置发一次最小探活外呼，返回成功/失败与延迟，且该探活不落库、不落日志明文。
- **FR-009**: 凭据必须加密存储；读取时仅以脱敏掩码回显，明文不得出现在任何回显、日志或审计。
- **FR-010**: 编辑既有配置时，凭据字段留空提交必须保留原凭据不变（留空 = 不修改）。
- **FR-011**: 系统必须记录每次 AI Agent 外呼的审计——含来源项目、来源任务、协议、耗时、结果、产出边数；**配置虽全局，调用按项目/任务可溯源**。
- **FR-012**: AI Agent 配置的读/写沿用系统设置既有的 `project:manage` 权限门槛；非授权用户不得访问「配置」tab 与其读写端点。
- **FR-013**: 当全局 AI Agent 配置缺失或被禁用时，AI 抽取通道必须整体旁路，不得阻断任务上线/push，其余血缘通道不受影响。
- **FR-014**: 升级到全局配置时，系统必须妥善处置既有的按项目配置，不得静默丢失（默认择一收拢、其余作废留痕；细节 plan 固化）。
- **FR-015**: 血缘探索器工具栏内原有的项目级 AI Agent 配置弹窗入口必须移除，单一真相源在 系统设置 → 配置 → AI Agent。

### Key Entities *(include if feature involves data)*

- **全局 AI Agent 配置**：租户级单例；含协议、服务端点、模型、加密凭据、启用开关、运行参数；为租户内所有项目的血缘智能抽取共同使用。
- **AI Agent 调用审计**：每次外呼一条；记录来源项目、来源任务、协议、耗时、结果、产出边数。配置虽全局，调用按项目/任务溯源。
- **配置分区（Config Section）**：「配置」tab 左导航的一项，对应一种全局/平台配置；含稳定标识、显示名、图标、其右侧内容组件。AI Agent 是首个分区，结构上可持续扩展。
- **系统设置「配置」tab**：既有系统设置视图新增的顶级 tab，承载上述左导航+右内容外壳。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理员只需配置**一次** AI Agent，即可让租户内全部项目的血缘智能抽取生效——配置动作从「每项目一次」降为「全局一次」。
- **SC-002**: 任一项目触发 AI 抽取时所用配置与 系统设置 → 配置 → AI Agent 中的全局配置**完全一致**（项目级配置偏差为 0）。
- **SC-003**: 从工作区进入系统设置 → 配置 → AI Agent 并完成一份配置（含一次成功测试）可在 **2 分钟**内完成。
- **SC-004**: 凭据在任何回显/日志/审计中均以脱敏形式出现——明文泄漏事件数为 **0**。
- **SC-005**: 全局配置缺失或禁用时，任务上线/push 链路 **100%** 不被阻断（AI 通道旁路，其余通道正常）。
- **SC-006**: 新增一种全局配置时，**无需改动**「配置」tab 的容器结构（仅注册一个分区项 + 内容组件）即可上线。

## Assumptions

- 目标管理者为具备 `project:manage` 权限的用户（既有系统设置的可见人群）；普通项目成员对系统设置不可见。
- 「全局共用一份」= **纯全局单例**，不保留任何按项目覆盖能力（用户明确「共用一个即可」）。
- 复用现有血缘 AI Agent 的协议适配、凭据加密、速率限流、异步富化、防幻觉校验与外呼审计机制——本特性聚焦「位置迁移 + 作用域提升 + 配置外壳」，不重做抽取内核。
- 「配置」tab 的左导航+右内容外壳在视觉与交互上对齐「数据开发」视图（圆角卡片 + `border` + `shadow-lg` + `--radius-lg`、可拖拽调宽并 localStorage 持久化的左侧面板、`DwScroll` 滚动、留白分区而非分割线），并复用既有公共组件目录（reuse-first）。
- 新「配置」顶级 tab 沿用既有下划线式 tab 风格（非 closable）；其内部的左导航是配置分区列表（nav rail），不与 Tabs 组件的双风格规则冲突。
- 「配置」tab 与「AI Agent」分区的最终 i18n 文案在 plan 阶段确定，归入既有 `settingsView` / `views.settings` 命名空间，zh-CN / en-US 双 bundle 同步。
- 血缘工具栏原有的项目级 AI Agent 配置弹窗入口移除，单一真相源在 系统设置 → 配置 → AI Agent。
- 既有按项目配置的迁移：默认择一收拢为全局（如取最近更新的一份），其余作废并留痕；细节 plan 阶段细化，必须保证不静默丢失。
- 涉及的存储结构变更遵循本项目「权威 DDL + 版本号升级」约定。
- 本特性不改变 AI 抽取的执行时机（仍为 push 后异步后台富化）、冲突消解优先级（确定性通道优先于 AI 推断）与防幻觉规则。
- 并发编辑默认后写覆盖 + 审计；是否加乐观锁提示留 plan 阶段。

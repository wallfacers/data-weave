# 052「高召回 + 人工复核残差」部署信封——诚实结论（测试集 A / held-out）

**日期**：2026-07-07 · **路径**：把「3B 全自动追平 m2」重构为**人在环分层部署**
**测试集**：gold A（139 py/sh，非空金标 59，真表 141）——对蒸馏模型 held-out（污染审计 train∩test=0）

## 为什么改路径：全自动门在真实分布上不可达（US1 已证）

单模型独跑 recall 0.66 / precision 0.36、方向 0.60，未达严格门（recall≥0.80 等）。
根因不止语料规模——本轮进一步定位到**机制性天花板**：

- **连确定性 AST 抽取器在真实野生脚本上都不干净**：全量启发式 SQL 通道 precision 仅 **0.345**，
  污染源是**非执行 SQL**——Postgres 文档关键字列表、Databricks `# MAGIC` markdown、
  Python docstring 里的示例 `CREATE TABLE`，全被当血缘抽出。
- **执行上下文门控 + 治理级名字消毒**把 SQL 通道 precision 抬到 **0.488（全）/0.677（非空）、幻觉归零**，
  但**再往上就是对本 gold 过拟合**：残余 FP 是 temp/staging/view（约定判断依赖）+ CTE 片段泄漏，
  不是可无损自动化的。→ **「什么算血缘」本身约定/判断依赖，这是必须人在环的机制性原因。**

## 可部署形态：SQL-gated ∪ 蒸馏模型，通道一致性分层

复核者看到**并集候选**（高召回不漏）；双通道**一致层**（SQL-AST ∧ 模型都认）最高置信、可自动采纳/快速批。

| 指标 | 值 | 判定 | 含义 |
| --- | --- | --- | --- |
| **并集召回** | **0.837** | ✅ ≥0.80 | 真血缘 surface 给复核者的比例（高召回目标达成，靠通道并集） |
| **一致层 precision** | **0.952** | ✅ 治理级 | 双通道都认的表的正确率 → 自动采纳安全 |
| 一致层 recall | 0.142 | — | 自动采纳只覆盖 14% 真表（省人力有限） |
| 复核负载 | **2.64 候选/脚本** | — | 复核者每脚本需过目的候选数 |
| 一致/候选 | 21/156 | — | 一致层占全部候选比例 |

- Tier 1（自动采纳）= SQL-AST exec-gated：precision 0.488（全）/0.677（非空）、**幻觉 0.000**、
  自动覆盖非空真表 ~23%；确定性、方向 AST 保证。
- Tier 2（人工复核残差）= Tier 1 打空的脚本（41/59，69.5%）：模型 surface 候选、人拍板；不自动入库 → 无需 precision 达标。

## 诚实判定

- **可作「人在环复核助手」生产部署**：并集召回 0.837（≥0.80 门）、每脚本仅 2.64 候选、
  附 95%-precision 快批层 → 显著省复核者人力且不漏 84% 真血缘。
- **不可全自动入库**：一致层（可自动）只覆盖 14% 真表，其余 86% 需人确认。
- **HF 卡「不可用于生产血缘」结论更新**：不可**自动入库**（错误血缘有后果，成立）；
  但可作**复核助手**部署——这是本特性交付的真实、诚实的「生产可用」形态。

## 已知残差（诚实记录，非模型能力问题）

1. **CTE 片段泄漏**：关键字锚定的片段化丢掉 `WITH cte AS(...)` 前缀 → `FROM cte` 被当读表。
   语句级 CTE 排除无能为力；根治需整串（exec-sink 到闭合）抽取（未做，记为升级项）。
2. **temp/staging/view 约定依赖**：`temp.*`/`*_tmp`/视图是否算血缘随站点约定而变；未硬编码（避免过拟合本 gold）。
3. 测试集为 A（n=59 非空）；测试集 B（非空≥100，agent 初裁+maintainer 抽查）未建——全自动门未达故未投入建 B。

## 加性零破坏

现有已发布 HF 模型与后端 sidecar 原样未动；`exec_gated`/`strict` 为**默认关**的新参数，
050 及既有 `extract_sql_lineage` 调用者行为逐字不变（回归测试锁定）。
